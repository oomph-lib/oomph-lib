3\mainpage Example problem: Adaptive simulation of finite-Reynolds number entry flow into a 3D tube
4
5In this example we shall demonstrate the spatially adaptive solution of the
6steady 3D Navier-Stokes equations using the problem of developing pipe flow.
7

11\section example The example problem

13<CENTER>
14<TABLE>

16<TD>
17<CENTER>
18<B>The 3D developing pipe flow in a quarter-tube domain.</B>
19</CENTER>
20Solve the steady Navier-Stokes equations:
21\f[
22Re\phantom{i}u_j\frac{\partial u_i}{\partial x_j} =
23- \frac{\partial p}{\partial x_i} +
24\frac{\partial }{\partial x_j} \left(
25\frac{\partial u_i}{\partial x_j} +
26\frac{\partial u_j}{\partial x_i} \right),
27 \ \ \ \ \ \ \ \ \ \ (1)
28\f]
29and
30\f[
31\frac{\partial u_i}{\partial x_i} = 0,
32\f]
33in the quarter-tube domain \f$
34D = \left\{x_1 \geq 0, x_2 \geq 0, x_1^2+x_2^2 \leq 1, x_3 \in [0,L]
35\right\}\f$,
36subject to the Dirichlet boundary conditions:
37\f[
38\left. \mathbf{u}\right|_{\partial D_{wall}}=(0,0,0),
39\ \ \ \ \ \ \ \ \ \ (2)
40\f]
41on the curved wall
42\f$ \partial D_{wall} = \{(x_1,x_2,x_3) \ | \ x_1^2+x_2^2=1\}, \f$
43\f[
44\left. \mathbf{u}\cdot\mathbf{n}\right|_{\partial D_{sym[1,2]}}=0,
45\ \ \ \ \ \ \ \ \ \ (3)
46\f]
47(where \f$ \mathbf{n} \f$ is the outer unit normal vector) on the
48symmetry boundaries
49\f$ \partial D_{sym[1]} = \{(x_1,x_2,x_3) \ | \ x_1=0\} \f$
50and
51\f$ \partial D_{sym[2]} = \{(x_1,x_2,x_3) \ | \ x_2=0\}, \f$
52\f[
53\left. \mathbf{u}\right|_{\partial D}=(0,0,1-(x_1^2+x^2)^\alpha),
54\ \ \ \ \ \ \ \ \ \ (4)
55\f]
56on the inflow boundary,
57\f$ \partial D_{inflow} = \{(x_1,x_2,x_3) \ | \ x_3=0\}, \f$ and finally
58\f[
59\left. u_1\right|_{\partial D_{outflow}}=
60\left. u_2\right|_{\partial D_{outflow}}=0,
61\ \ \ \ \ \ \ \ \ \ (5)
62\f]
63(parallel flow) on the outflow boundary \f$ \partial D_{outflow} =
64\{(x_1,x_2,x_3) \ | \ x_3=L\}.\f$
65Note that the axial velocity component,
66\f$u_3\f$, is not constrained at the outflow. Implicitly, we are therefore
67setting the axial component of traction on the fluid to zero,
68\f[
69t_3|_{\partial D_{outflow}} = \left. \left(-p + 2 \,
70\frac{\partial u_3}{\partial x_3}\right)
71\right|_{\partial D_{outflow}} = 0.
72\f]
73Since \f$ \partial u_1 / \partial x_1 = \partial u_2
74/\partial x_2 = 0\f$ in the outflow cross-section [see
75(5], and the flow is incompressible,
76this is equivalent to (weakly) setting the pressure at the outflow to zero,
77\f[
78p|_{\partial D_{outflow}}=0.
79\f]

88\subsection results Results for Taylor-Hood elements
89
90The figure below shows the results computed with \c oomph-lib's 3x3x3-node
913D adaptive Taylor-Hood elements for the
92parameters \f$ \alpha=20\f$, \f$ L=7\f$ and \f$Re=100 \f$.
93The large exponent \f$ \alpha=20\f$ imposes a very blunt inflow
94profile, which creates a thin boundary layer near the wall.
95Diffusion of vorticity into the centre of the tube smooths the
96velocity profile which ultimately approaches a parabolic Poiseuille
97profile. If you are viewing these results online you will be able
98to see how successive mesh adaptations refine the mesh -- predominantly
99near the entry region where the large velocity gradients in the
100boundary layer require a fine spatial discretisation. Note also that
101on the coarsest mesh, even the (imposed) inflow profile is
102represented very poorly.
103

123\section main The driver code
124Since the 3D computations can take a long time, and since all demo codes
125are executed during \c oomph-lib's self-test procedures,
126we allow the code to operate in two modes:
127- By default, we specify error targets for which the code
128 refines the mesh near the inflow region, and allow up to
129 five successive mesh adaptations. The code is executed
130 in this mode if the executable is run without any command
131 line arguments.
132- If the code is run during the self-test procedure (indicated by specifying
133 some random command line argument), we only perform one level of
134 adaptation to speed up the self-test. However, because the original mesh
135 is very coarse, the first mesh adaptation refines \e all elements
136 in the mesh (cf. the animation of the adaptive mesh refinement
137 shown above), so that no hanging nodes are created -- not a good
138 test-case for a validation run! Therefore adjust the error
139 targets so that the first (and only) mesh adaption only refines
140 a few elements and therefore creates a few hanging nodes.
141.
142The main code therefore starts by storing the command line arguments
143and setting the adaptation targets accordingly:
144
145\skipline start_of_main
146\until end max_adapt setup
147
148We then create a \c DocInfo object to specify the labels for the
149output files, and solve the problem, first with Taylor-Hood and then
150with Crouzeix-Raviart elements, writing the results from the
151two discretisations to different directories:
152
153\until end_of_main

193\section constructor The constructor
194
195We start by building the adaptive mesh for the quarter tube domain.
196As for most meshes with curvilinear boundaries,
197the \c RefineablequarterTubeMesh expects the curved boundary to be
198represented by a \c GeomObject. We therefore create an \c EllipticalTube with
199unit half axes, i.e. a unit cylinder and a pass a pointer to the \c
200GeomObject to the mesh constructor. The "ends" of
201the curvilinear boundary (in terms of the maximum and minimum values
202of the Lagrangian coordinates that parametrise the shape of the
203\c GeomObject) are such that it represents a quarter of a cylindrical tube
204of length \f$L=7\f$.
205
206\skipline start_of_constructor
207\until RefineableQuarterTubeMesh
208
209Next, we build an error estimator and specify the target errors
210for the mesh adaptation:
211
212\until min_permitted_error
213
214Now we have to apply boundary conditions on the various mesh
215boundaries. [\b Reminder: If the numbering of the
216mesh boundaries is not apparent from its documentation (as it should
217be!), you can use the function \c Mesh::output_boundaries(...)
218to output them in a tecplot-readable form.
219
220\until some_file.close();
221
222If the mesh has
223N boundaries, the output file will contain N different zones, each
224containing the \f$(x,y,z)\f$ coordinates of the nodes on the
225boundary.]
226
227
228For the \c RefineableQuarterTubeMesh, the boundaries are numbered
229as follows:
230- Boundary 0: "Inflow" cross section; located along the line
231 parametrised by \f$ x_3=\xi_0 = \xi_0^{lo} \f$ on the
232 \c GeomObject that specifies the wall.
233- Boundary 1: Plane \f$ x_1=0 \f$
234- Boundary 2: Plane \f$ x_2=0 \f$
235- Boundary 3: The curved wall
236- Boundary 4: "Outflow" cross section; located along the line
237 parametrised by \f$ x_3=\xi_0 = \xi_0^{hi} \f$ on the
238 \c GeomObject that specifies the wall.
239.
240
241\image html boundaries.gif "Plot of the mesh boundaries. "
242\image latex boundaries.eps "Plot of the mesh boundaries. " width=0.75\textwidth
243
244
245We apply the following boundary conditions:
246- Boundary 0: (\f$ x_3=0\f$) pin all three velocities.
247- Boundary 1: (\f$ x_1=0\f$) pin \f$ u_1\f$.
248- Boundary 2: (\f$ x_2=0\f$) pin \f$ u_2\f$.
249- Boundary 3: (\f$ x_1^2+x_2^2=1\f$) pin all three velocities.
250- Boundary 4: (\f$ x_3=L\f$) pin \f$ u_1 \f$ and \f$ u_2 \f$.
251.
252
253\until end loop over boundaries
254
255Now we assign the \c re_pt() for each element and pin the redundant
256nodal pressures (see
257<A HREF="../../../navier_stokes/adaptive_driven_cavity/html/index.html">
258another tutorial</A> for details).
259
260\until pin_redundant_nodal_pressures
261
262We provide an initial guess for the velocity field by
263initialising all velocity components with their Poiseuille flow values.
264
265\until }
266
267Finally, we set the value of \c Alpha, the exponent that specifies
268the bluntness of the inflow profile and assign
269the equation numbers.
270
271\until end_of_constructor

276\section before_solve Actions before solve
277We use the function \c Problem::actions_before_newton_solve() to re-assign
278the inflow boundary conditions before every solve. In the present
279problem this is an \e essential step because the blunt inflow profile
280(4)
281cannot be represented accurately on the initial coarse mesh (see the
282animation of the axial velocity profiles at the beginning of this
283document). As discussed in the
284<A HREF="../../../unsteady_heat/two_d_unsteady_heat_adapt/html/index.html">
285example that illustrates the use of spatial
286adaptivity for time-dependent problems</A>,
287\c oomph-lib automatically <B>(i)</B> applies the correct boundary conditions
288for newly created nodes that are located on the mesh boundaries
289and <B>(ii)</B> assigns the nodal values at such nodes by interpolation from
290the previously computed solution. This procedure is adequate on
291boundaries where homogeneous boundary conditions are applied,
292e.g. on the curved wall, the symmetry and the outflow boundaries.
293However, on the inflow boundary, the interpolation from the FE
294representation of the blunt velocity profile (imposed on
295the coarse initial mesh) onto the refined mesh, does not yield a
296more accurate representation of the prescribed inflow profile. It is
297therefore necessary to re-assign the nodal values on this boundary
298after every adaptation, i.e. before every solve.
299
300\skipline start_of_actions_before_newton_solve
301\until }

317\section comments Comments and exercises
318-# Suppress the reassignment of the prescribed inflow profile in
319 \c Problem::actions_before_newton_solve() to confirm that this step
320 is essential if the computation is to converge to the exact
321 solution.
322-# Suppress the specification of the parabolic (Poiseuille) velocity
323 profile as the initial guess for the velocity field in the problem
324 constructor to confirm that the assignment of a "good" initial
325 guess for the solution is essential for the convergence of
326 the Newton method. [Hint: You can simply comment out the initialisation
327 of the velocities -- they then retain their default initial values of
328 0.0. When you re-run the code, the Newton iteration will "die"
329 immediately with an error message stating that the maximum residual
330 exceeds the default threshold of 10.0, stored in the protected data member
331 \c Problem::Max_residuals. Try increasing the value of this
332 threshold in the Problem constructor. Is this sufficient to make the
333 Newton method converge?]
