3\mainpage Demo problem: Steady finite-Reynolds-number flow through an iliac bifurcation
4
5The purpose of this tutorial is to demonstrate the simulation of
6cardiovascular fluid mechanics problems with \c oomph-lib. We showed in
7<a href="../../../meshes/mesh_from_vmtk/html/index.html">another tutorial</a>
8how to use <a href="http://www.vmtk.org">VMTK</a>
9together with \c oomph-lib's conversion code \n\n
10<CENTER>
11<a href="../../../../demo_drivers/meshing/mesh_from_vmtk/create_fluid_and_solid_surface_mesh_from_fluid_xda_mesh.cc">demo_drivers/meshing/mesh_from_vmtk/create_fluid_and_solid_surface_mesh_from_fluid_xda_mesh.cc</a>
12</CENTER>
13\n\n
14to generate unstructured fluid and solid meshes for the simulation
15of physiological fluid-structure interaction problems based on data from
16medical images. Here we show how to simulate steady finite-Reynolds-number flow
17through a (rigid) iliac bifurcation. A particular feature of this
18problem is that, unlike the problem considered in
19<a href="../../unstructured_three_d_fluid/html/index.html">
20another tutorial,</a> the in- and outflow boundaries are not
21aligned with any coordinate planes. Parallel in- and outflow
22is therefore enforced by a Lagrange multiplier method, implemented
23using \c oomph-lib's \c FaceElements.
24
25
26The problem studied here also serves as a "warm-up problem" for the
27<a href="../../../interaction/vmtk_fsi/html/index.html">
28corresponding fluid-structure interaction problem </a> in which the vessel
29wall is elastic and deforms in response to the traction
30that the fluid exerts onto it.
31
32We stress that the tutorial focuses on the implementation
33aspects, not the actual physics. Since the driver code discussed
34here is also used in the library's self-tests we
35deliberately use a very coarse mesh and restrict ourselves to
36steady flows. The results shown below are therefore unlikely
37to bear much resemblance to the actual flows that arise
38\e in \e vivo. The section \ref realism at the end of this
39tutorial provides several suggestions on how to make the simulation
40more realistic.

71\section parallel_flow Imposing parallel in- and outflow
72In most of the Navier-Stokes problems we have considered so far,
73the geometry of the fluid domain was such that the in- and outflow
74cross-sections were aligned with the Cartesian coordinate planes.
75In such geometries the imposition of parallel in- and outflow is
76straightforward as it only requires pinning of the transverse
77velocity components. A pressure drop between
78upstream and downstream ends can be applied by attaching
79\c NavierStokesTractionElements to the bulk elements that are
80adjacent to the relevant domain boundaries.
81
82In the current problem, the in- and outflow cross-sections have
83some arbitrary orientation, implying that when the in- or outflow is
84parallel (or, more accurately, orthogonal to the in- or outflow cross
85sections), neither of the three velocity components vanishes.
86The easiest way to enforce parallel outflow in such situations is
87to employ Lagrange multipliers to enforce the two constraints
88\f[
89{\bf u} \cdot {\bf t}_\alpha = 0 \mbox{\ \ \ \ for $\alpha = 1,2$,}
90\f]
91where \f$ {\bf t}_\alpha \f$ (for \f$\alpha = 1,2\f$) are the
92two tangent vectors spanning the in- or outflow cross-sections.
93Physically, the Lagrange multipliers act as tangential tractions
94that enforce the parallel flow. The Lagrange multipliers
95introduce additional degrees of freedom into the problem and their
96implementation as \c ImposeParallelOutflowElements employs a
97technique similar to that used to enforce prescribed boundary
98displacements in solid mechanics problems. (This is discussed in
99<a href="../../../solid/prescribed_displ_lagr_mult/html/index.html">
100another tutorial</a>.) The \c ImposeParallelOutflowElements also
101allow the specification of a pressure that acts on the fluid
102(in the direction opposite to the outer unit normal on the bulk
103fluid element).

239\subsection re What does the Reynolds number mean in this problem?
240\c oomph-lib's implementation of the Navier-Stokes equations is
241based on <a href="../../driven_cavity/html/index.html#equation">their
242non-dimensional form</a> so we typically expect
243the geometry of the problem to have been non-dimensionalised on
244a representative lengthscale, \f$ {\cal L} \f$. When dealing with
245geometries that are obtained from medical images, the vessel coordinates are
246typically provided as dimensional quantities, e.g. in millimetres.
247Discarding the unit of the coordinates (i.e. using
248a non-dimensional coordinate \f$ (x,y,z)=(1,2,3)\f$ to represent
249the point located at \f$
250(x^*,y^*,z^*)=(1\mbox{mm},2\mbox{mm},3\mbox{mm}),\f$ say)
251is therefore equivalent to non-dimensionalising
252all lengths on a reference length of \f$ {\cal L} = 1\mbox{mm}. \f$
253<a href="../../driven_cavity/html/index.html#equation">Recall</a>
254that the Reynolds number is defined as
255\f[
256Re = \frac{\rho {\cal U} {\cal L}}{\mu}
257\f]
258where \f$ \rho \f$ and \f$ \mu \f$ are the fluid density and viscosity,
259respectively. The lengthscale \f$ {\cal L} = 1\mbox{mm} \f$ does obviously
260not provide a measure of the dimension of our blood vessel whose
261diameter (at the inlet) is about \f$ D \approx 16\mbox{mm}. \f$ The
262nominal Reynolds number of \f$ Re=10 \f$ used in the computations
263is therefore equivalent to an actual Reynolds number (formed with
264the vessel diameter)
265of
266\f[
267Re_D = \frac{\rho {\cal U} D }{\mu} = Re \ \frac{D}{\cal L} = 160,
268\f]
269where the velocity scale \f$ {\cal U} \f$ is formed with the
270(dimensional) applied pressure drop \f$ \Delta P^* \f$
271between the in- and outflow cross-sections, as in the
272<a href="../../unstructured_three_d_fluid/html/index.html">problem
273with axis-aligned in- and outflow cross-sections,</a>
274\f[
275{\cal U} = \frac{\Delta P^*\ {\cal L} }{\mu}.
276\f]

280\subsection realism How to make the simulation more realistic
281The simulation presented above is obviously very crude and serves
282primarily as a proof of concept. However, it is straightforward
283to address most of the shortcomings and we encourage you to explore
284the following improvements:
285- Generate a finer fluid mesh using the instructions
286 in \c oomph-lib's
287 <a href="../../../meshes/mesh_from_vmtk/html/index.html">VMTK
288 tutorial </a> We are hoping to make the image files
289 used in this tutorial available soon. Please
290 <a href="../../../contact/html/index.html">contact us</a>
291 if you can't wait. Remember that you will have to update the
292 enumeration of the domain boundaries if you change the mesh.
293 \n\n
294- Attach "flow extensions" to the in- and outflow
295 cross-sections, using the technique described in \c oomph-lib's
296 <a href="../../../meshes/mesh_from_vmtk/html/index.html#add_extensions">
297 VMTK tutorial.</a>
298 \n\n
299- Make the problem time-dependent and apply a period driving pressure
300 drop.
