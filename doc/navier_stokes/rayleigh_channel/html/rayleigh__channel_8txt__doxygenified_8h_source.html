11 \section example The example problem
12 We consider finite-Reynolds-number flow in a 2D channel that is driven
13 by the oscillatory tangential motion of the "upper" wall:
14 
15 <CENTER>
16 <TABLE>
17 <TR>
18 <TD>
19 <CENTER>
20 <B>Unsteady flow in a 2D channel with an oscillating wall.</B>
21 </CENTER>
22 Here is a sketch of the problem:
23 
24 \image html rayleigh_channel.gif "Sketch of the problem. "
25 \image latex rayleigh_channel.eps "Sketch of the problem. " width=0.75\textwidth
26 
27 The flow is governed by the 2D unsteady Navier-Stokes equations,
28 \f[
29 Re\left(St\frac{\partial u_i}{\partial t} +
30 u_j\frac{\partial u_i}{\partial x_j}\right) =
31 - \frac{\partial p}{\partial x_i} +
32 \frac{\partial }{\partial x_j} \left(
33 \frac{\partial u_i}{\partial x_j} +
34 \frac{\partial u_j}{\partial x_i} \right)
35  \ \ \ \ \ \ \ \ \ \ (1)
36 \f]
37 and the continuity equation
38 \f[
39 \frac{\partial u_i}{\partial x_i} = 0
40  \ \ \ \ \ \ \ \ \ \ (2)
41 \f]
42 in the domain
43 \f[
44 D = \bigg\{(x_1,x_2) \ | \ x_1 \in [0,1], \ x_2 \in [0,1] \bigg\}.
45 \f]
46 We apply the Dirichlet (no-slip) boundary condition
47 \f[
48 \left. \mathbf{u}\right|_{\partial D_{lower}}=(0,0),
49 \ \ \ \ \ \ \ \ \ \ (3)
50 \f]
51 on the lower, stationary wall, \f$ \partial D_{lower} =
52 \big\{(x_1,x_2) \ | \ x_2=0 \big\} \f$ , apply the Dirichlet
53 (no-slip) conditions
54 \f[
55 \left. \mathbf{u}\right|_{\partial D{upper}}=(\sin\left(\omega t\right),0),
56 \ \ \ \ \ \ \ \ \ \ (4)
57 \f]
58 on the upper, moving wall , \f$ \partial D_{upper} =
59 \big\{(x_1,x_2) \ | \ x_2=1 \big\} \f$ , and apply
60 periodic boundary condition on the "left" and "right" boundaries:
61 \f[
62 \left.\mathbf{u}\right|_{x_1=0} = \left.\mathbf{u}\right|_{x_1=1}.
63 \ \ \ \ \ \ \ \ \ \ (5)
64 \f]
65 Initial conditions for the velocities are given by
66 \f[
67 {\bf u}(x_1,x_2,t=0) = {\bf u}_{IC}(x_1,x_2),
68 \f]
69 where \f$ {\bf u}_{IC}(x_1,x_2)\f$ is given.
70 </TD>
71 </TR>
72 </TABLE>
73 </CENTER>

77 \subsection solution The exact solution
78 The above problem has an exact, time-periodic parallel flow solution
79 of the form
80 \f[
81 {\bf u}_{exact}(x_1,x_2,t) = U(x_2,t) \ {\bf e}_1 \mbox{ \ \ \ \ \ and
82 \ \ \ \ }
83 p_{exact}(x_1,x_2,t) = 0,
84 \f]
85 where \f$ U(x_2,t) \f$ is governed by
86 \f[
87 Re\, St\frac{\partial U}{\partial t} =
88 \frac{\partial^2 U}{\partial x_2^2},
89 \f]
90 subject to the boundary conditions
91 \f$ U(x_2=0,t)=0\f$ and \f$ U(x_2=1,t)=\sin(\omega t)\f$ .
92 The solution is given by
93 \f[
94 U(x_2,t) = Re\left\{\frac{e^{i\omega t}}{e^{i\lambda}-e^{-i\lambda}}
95 \left(e^{i\lambda y}-e^{-i\lambda y}\right)\right\},
96 \f]
97 where
98 \f[
99 \lambda = i\sqrt{i\omega Re\, St}.
100 \f]

240 We start by building the timestepper, determining its type from the
241 class's second template argument, and pass a pointer to it to
242 the Problem, using the function \c Problem::add_time_stepper_pt(...).
243 
244 \skipline start_of_constructor
245 \until add_time_stepper_pt
246 
247 Next we build the mesh and pass an additional boolean flag to the
248 constructor to indicate that periodic boundary conditions are to be
249 applied in the \f$ x_1\f$ -direction.
250 We will discuss the implementation of this feature in
251 more detail in <A HREF="#periodic"> below</A>.
252 
253 \until time_stepper_pt()
254 
255 We pin both velocity components on the top and bottom boundaries
256 (i.e. at \f$ x_2=0\f$

277 \section IC Initial conditions
278 The application of initial conditions for vector-valued problems
279 is performed by the same procedure that we described for
280 <A HREF="../../../unsteady_heat/two_d_unsteady_heat/html/index.html#IC">
281 scalar problems</A>, except that we now have to assign "history
282 values" for multiple nodal values. For timesteppers from the BDF family,
283 the "history values" represent the solution at previous timesteps.
284 We check that the timestepper is of the appropriate type,
285 loop over previous time levels, determine the
286 velocity at those times and assign them to the "history values"
287 of the velocities. No initial conditions are required
288 for the pressure. Note that we also have to assign "history values"
289 for the nodal positions since \c oomph-lib's Navier-Stokes elements
290 discretise the momentum equations in their ALE form. This aspect
291 was explained in more detail in our discussion of the solution of the
292 <A HREF="../../../unsteady_heat/two_d_unsteady_heat/html/index.html#IC">
293 unsteady heat equation</A>.

301 \section doc Post processing
302 The function \c doc_solution(...) is similar to those used in the
303 <A HREF="../../../unsteady_heat/two_d_unsteady_heat/html/index.html#doc">
304 unsteady heat examples</A>. We plot the computed solution, the
305 time-periodic exact solution and the difference between the two,
306 and record various parameters in the trace file. The plot of the
307 computed solution contains tecplot instructions that generate
308 a blue line in the top-left corner of the plot to indicate how time
309 progresses during the simulation. The trace file contains a record of
310 - the value of the continuous time, \f$ t \f$ ,
311 - the coordinates of a control node, \f$ (x_1^{[c]},x_2^{[c]})\f$ ,
312 - the computed velocity at the control
313 node,\f$( u_1^{[c]},u_2^{[c]})\f$ ,
314 - the time-periodic solution, evaluated at the control node,
315  \f$ (u_1^{[c,exact]},u_2^{[c,exact]})\f$ ,
316 - the difference between the computed velocities and the
317  time-periodic solution at the control node,
318 - the L2 norm of the "error" between the computed and time-periodic
319  solution for the velocity, and
320 - the L2 norm of the time-periodic
321  solution for the velocity.
322 .

365 \subsection periodic Periodic boundaries
366 A key feature of the current problem is the presence of periodic
367 boundary conditions. The application of the periodic boundary
368 condition is performed "inside" the mesh constructor and details of
369 the implementation were therefore "hidden". We will now discuss
370 the steps required to apply periodic boundary conditions
371 and explain why it is easier to apply periodic boundary
372 conditions in the mesh constructor rather than in the "driver code".
373 
374  Periodic boundary conditions arise in problems that are periodic
375 in one of their coordinate directions. It is important to realise that,
376 even though the solution at the corresponding nodes on the two
377 periodic domain boundaries (the left and the right boundary in
378 the above example) are the same, one of their nodal coordinates
379 differs. For instance, in the above example, each of the nodes on the left
380 boundary has the same velocity values and the same
381 \f$ x_2\f$ -coordinate as its corresponding (periodic) node on the
382 right boundary. However, the \f$ x_1 \f$ -coordinate of the nodes on
383 the left boundary is \f$ x_1=0\f$ , whereas that of the (periodic) nodes
384 on the right boundary is \f$ x_1=1\f$ . It is therefore not possible
385 to regard the nodes as identical.
386 
387 In \c oomph-lib we create periodic nodes by allowing two (or more) nodes to
388 access some of the same internal data.
389 One of the nodes should be regarded as the
390 original and the other(s) are set to be its "periodic counterpart(s)"
391 and hence access its internal data. The "periodic counterpart(s)" are
392 created by calling the member function
393 \code
394 BoundaryNode::make_periodic(Node *const& node_pt)
395 \endcode
396 where the pointer to the original node is passed as the argument.
397 Note that the required functionality imposes a slight
398 storage overhead and so in \c oomph-lib we only allow BoundaryNodes to
399 be made periodic.
400 
401 Here is a sketch of a 2D rectangular quad mesh. If this mesh is
402 to be used in a problem with periodic boundary conditions in the
403 horizontal direction (as in the above example), the pointer to
404 node 3 on boundary 3 would have to be used when node 3 on
405 boundary 1 is made periodic, etc. The
406 appropriate commands are
407 \code
408  //Get pointers to the two nodes that are to be "connected" via
409  //a periodic boundary
410  Node* original_node_pt = mesh_pt()->boundary_node_pt(3,3);
411  Node* periodic_node_pt = mesh_pt()->boundary_node_pt(1,3);
412 
413  //Make the periodic_node_pt periodic the data from the original_node_pt
414  periodic_node_pt->make_periodic(original_node_pt);
415 \endcode
416 
417 \image html periodic_mesh.gif "Figure of a mesh that is periodic in the horizontal direction. "
418 \image latex periodic_mesh.eps "Figure of a mesh that is periodic in the horizontal direction. " width=0.75\textwidth
419 
420 Although it is possible to make
421 nodes periodic at any time, it is usually easier to determine which
422 nodes should be "connected" during mesh construction.
423 We therefore strongly recommend to implement periodic boundary conditions
424 inside the mesh constructor. The
425 <A HREF="../../../../src/meshes/rectangular_quadmesh.template.cc">source
426 code</A> for the constructor of the
427 
428 \code
429 RectangularQuadMesh<ELEMENT>
430 \endcode
431 
432 that we used in the above problem, illustrates a possible implementation.
