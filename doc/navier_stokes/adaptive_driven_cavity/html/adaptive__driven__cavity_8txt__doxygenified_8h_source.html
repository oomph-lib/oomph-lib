3\mainpage Example problem: Adaptive solution of the 2D driven cavity problem
4
5In a <A HREF="../../driven_cavity/html/index.html">previous example</A>
6we demonstrated the solution of the 2D driven cavity problem
7using \c oomph-lib's 2D Taylor-Hood and Crouzeix-Raviart Navier-Stokes
8elements on a uniform mesh. The computed solution was clearly
9under-resolved near the corners of the domain where the discontinuity
10in the velocity boundary conditions creates pressure singularities.
11
12In this example we shall re-solve the driven cavity problem with
13the refineable versions of \c oomph-lib's quadrilateral
14Navier-Stokes elements -- the \c RefineableQTaylorHoodElement<2> and the \c
15RefineableQCrouzeixRaviartElement<2>. Enabling spatial adaptivity for
16this problem involves the same straightforward
17steps as for a scalar problem:
18- The domain must be discretised with a refineable mesh, i.e. a mesh
19 that is derived from the \c RefineableMesh class.
20- An \c ErrorEstimator object must be specified.
21.
22Two additional steps tend to be required during the adaptive solution of
23Navier-Stokes problems:
24- Recall that in Navier-Stokes problems in which the velocity is
25 prescribed along
26 the entire domain boundary, the pressure is only determined up an
27 arbitrary constant, making it necessary to "pin" one pressure value.
28 If the "pinned" pressure degree of freedom is associated with an element
29 that is unrefined during the mesh adaptation, the
30 "pinned " degree of freedom may no longer exist in the adapted problem.
31 To ensure that exactly one pressure degree of freedom is pinned
32 when re-solving the adapted problem, we recommend using the
33 function \c Problem::actions_after_adapt() to
34 -# unpin all pressure values, e.g. using the function
35 \code
36 NavierStokesEquations<DIM>::unpin_all_pressure_dofs(...)
37 \endcode
38 -# pin a pressure degree of
39 freedom that is known to exist (e.g. the first pressure degree of
40 freedom in the first element of the mesh -- whichever element this
41 may be), e.g. using the function
42 \code
43 NavierStokesEquations<DIM>::fix_pressure(...)
44 \endcode
45 .
46- The possible presence of hanging nodes in an adapted mesh
47 requires special treatment for elements (e.g. Taylor-Hood elements)
48 in which the pressure is represented by a low-order interpolation
49 between a subset of the element's nodal values. The required tasks
50 are performed by the function
51 \code
52 NavierStokesEquations<DIM>::pin_redundant_nodal_pressures(...)
53 \endcode
54 which should be called
55 -# before assigning the equation numbers for the first time, and
56 -# after every mesh adaptation.
57 .
58 [If the user "forgets" to call this function, a warning is issued
59 if the library is compiled with the PARANOID flag.]
60.
61The driver code discussed below illustrates the use of these functions.
62The section \ref comments provides a more detailed discussion of the
63technical details involved in their implementation.

165\section problem The problem class
166Most of the problem class is identical to that in the
167 <A HREF="../../driven_cavity/html/index.html#problem"> non-adaptive
168 version of the code </A>: We provide a constructor and destructor and
169 use the function \c Problem::actions_before_newton_solve() to (re-)assign
170the boundary conditions.
171
172\dontinclude adaptive_driven_cavity.cc
173\skipline start_of_problem_class
174\until end_of_actions_before_newton_solve
175
176
177As discussed in the introduction, we use the function
178\c Problem::actions_after_adapt() to ensure that, regardless of the
179mesh adaptation pattern, exactly one pressure degree of freedom is
180pinned. We start by unpinning all pressure degrees of freedom:
181
182\skipline After adaptation
183\until unpin_all_pressure_dofs
184
185[Note that this function (implemented as a static member function of
186the \c NavierStokesEquations<DIM> class) unpins the pressure degrees
187of freedom in all elements that are specified by the input argument
188(a vector of pointers to the these elements). This implementation
189allows certain elements in a mesh to be excluded from the procedure;
190this is required in problems where a mesh contains multiple element
191types. In the present problem, the mesh contains only Navier-Stokes
192elements, so we pass a vector of pointers to all elements in the
193mesh (returned by the function \c Mesh::element_pt()] to
194the function.]
195
196
197Following the mesh adaptation any redundant nodal pressures must be
198pinned, so that hanging pressure degrees of freedom are treated
199correctly. We note that calling this function is essential
200for Taylor-Hood elements. The function may be executed without any
201adverse effect for all other Navier-Stokes elements; see
202\ref comments for more details on the implementation.
203
204\until pin_redundant
205
206
207Finally, we pin a single pressure degree of freedom (the first
208pressure value in the first element in the mesh) and set its
209value to zero.
210
211\until }
212
213The remainder of the problem class remains as
214<A HREF="../../driven_cavity/html/index.html#problem"> before</A>.
215
216\until end_of_problem_class

247\subsection hang_vector Hanging nodes in problems with vector-valued unknowns
248We discussed in an earlier example for a scalar
249(<A HREF="../../../poisson/two_d_poisson_adapt/html/index.html">
250Poisson</A>) problem how
251\c oomph-lib's mesh adaptation routines create hanging nodes and how
252the values that are stored at such nodes are automatically constrained
253to ensure the inter-element continuity of the solution. The
254methodology employed for scalar problems is easily generalised to
255problems with vector-valued unknowns, provided that all unknowns are
256represented by the same isoparametric interpolation between the elements' nodal
257values. In such problems, the unknown nodal values at the hanging
258nodes are constrained to be linear combination of the corresponding
259values at their "master nodes". The list of "master nodes" and the
260corresponding "hanging weights" (contained in a \c HangInfo object)
261are the same for all unknowns.
262
263 To allow the use spatial adaptivity for elements
264in which different unknowns are represented by different interpolation
265schemes (e.g. in 2D quadrilateral Taylor-Hood elements where the two velocity
266components are represented by bi-quadratic interpolation between the
267values stored at the element's 3x3 nodes, whereas the pressure is
268represented by bi-linear interpolation between the pressure values
269stored at the element's 2x2 corner nodes) \c oomph-lib allows
270the different nodal values to have their own list of "master nodes"
271and "hanging weights". This is achieved as follows;
272- By default, all nodes are assumed to be non-hanging. This status
273 is indicated by the fact that a \c Node's pointer to its
274 \c HangInfo object, accessible via its member
275 function \c Node::hanging_pt(), is \c NULL.
276- Mesh adaptation may turn a node into a hanging node.
277 A node's "hanging status" is primarily a
278 geometrical/topological property.
279 If a \c Node is found to be hanging, \c oomph-lib's mesh adaptation
280 procedures create a \c HangInfo object that stores the list of
281 the hanging node's "master nodes" and their respective weights.
282 A pointer to the \c HangInfo object is then passed to the \c Node.
283 The list of master nodes and weights, stored in this \c HangInfo
284 object is then used by the function \c Node::position() to
285 determine the \c Node's (constrained) position.
286- Nodes also provide storage for separate pointers to \c HangInfo
287 objects for each of their nodal values. These are accessible via
288 the member function \c Node::hanging_pt(i) which returns the pointer to the
289 \c HangInfo object associated with the node's i-th nodal value.
290 By default, these pointers point to the "geometric" \c HangInfo
291 object, accessible via the argument-free version of this function.
292 This default behaviour is appropriate for isoparametric elements
293 in which all unknowns are represented by interpolation between the
294 elements' nodes, using its geometric shape functions as basis functions.
295- For elements that use different interpolation schemes for
296 different nodal values (e.g. in Taylor-Hood elements), the default
297 assignment for the pointers to the \c HangInfo objects may be
298 over-written. This is task is typically performed by re-implementing
299 (and thus over-writing) the empty virtual function
300 \code
301 RefineableElement::further_setup_hanging_nodes()
302 \endcode
303 for such elements. This function is called automatically during
304 at the end of \c oomph-lib's mesh adaptation procedures.
305.

308\subsection refining_TH Adaptivity for Taylor-Hood elements
309
310\subsubsection issues The issues
311
312In non-adaptive 2D [3D] Taylor-Hood elements, every node stores
313(at least) two [three] nodal values which represent the two [three]
314velocity components. The four corner [eight vertex] nodes store
315an additional third [fourth] value which represents the pressure.
316If the mesh is subjected to non-uniform refinement, some of the
317mid-side nodes in large elements also act as corner nodes
318for adjacent smaller elements, as illustrated in the figure below.
319
320\image html pressure_hang.gif "An adapted mesh, with nodes storing a pressure value denoted by circles, and velocity nodes by squares. "
321\image latex pressure_hang.eps "An adapted mesh, with nodes storing a pressure value denoted by circles, and velocity nodes by squares. " width=0.75\textwidth
322
323The figure illustrates that the "hanging status" of the various
324degrees of freedom can be become fairly involved. For instance
325- Node 7 is geometrically hanging, with master nodes 6, 8 and 12. It
326 is not a pressure node.
327- Node 8 is geometrically non-hanging, but it is a hanging node for
328 the pressure interpolation. Its pressure master nodes are 6 and 12.
329- Node 10 is geometrically hanging, and its geometric master nodes are
330 6, 8 and 12, while its pressure master nodes are 6 and 12.
331.
332To illustrate that \c oomph-lib's automatic mesh adaptation procedures
333are able to deal with these cases, the figure below shows
334a "carpet plot" of the pressure distribution, \f$ p(x_1,x_2) \f$,
335obtained from a (strongly under-resolved) driven-cavity
336computation on the mesh shown above. The figure illustrates
337that the hanging node constraints ensure the inter-element continuity
338of the pressure throughout the domain, and, in particular, along
339the "right" boundary of the largest element.
340
341
342\image html hanging_pressure.gif "The under-resolved pressure distribution for the driven cavity problem, using the above mesh and computed with Taylor-Hood elements. "
343\image latex hanging_pressure.eps "The under-resolved pressure distribution for the driven cavity problem, using the above mesh and computed with Taylor-Hood elements. " width=0.75\textwidth

345\subsubsection implementation Details of the implementation
346
347
348To facilitate the book-keeping for such problems, \e all nodes in the
349refineable
350Taylor-Hood elements store three [four] nodal values, even though,
351depending on the mesh's refinement pattern some of the pressure
352values will not be used. To eliminate the "redundant" pressure degrees
353of freedom from the problem, we provide the function
354
355\code
356NavierStokesEquations<DIM>::pin_redundant_nodal_pressures(...)
357\endcode
358
359which pins the "redundant" pressure degrees of freedom
360in all elements specified by the input argument (a vector of pointers
361to the Navier-Stokes elements). This function must be called
362after the initial mesh has been created, and after each mesh adaptation.
363The function first pins \e all nodal pressure values, using the
364function
365
366\code
367NavierStokesEquations<DIM>::pin_all_nodal_pressure_dofs()
368\endcode
369
370and then unpins the nodal pressure values at the elements' corner
371[vertex] nodes, using the function
372
373\code
374NavierStokesEquations<DIM>::unpin_proper_nodal_pressure_dofs()
375\endcode
376
377These functions are implemented as empty virtual functions
378in the \c NavierStokesEquations<DIM> class which provides a base
379class for all Navier-Stokes elements. The empty functions
380are overwritten for \c QTaylorHoodElement<DIM> and remain empty for
381all other Navier-Stokes elements, therefore they can be called
382for any element type.
383
384\subsection refining_CR Adaptivity for Crouzeix-Raviart elements
385As discussed in the <A HREF="../../driven_cavity/html/index.html#CR">
386previous example</A>, \c oomph-lib's isoparametric 2D [3D]
387Crouzeix-Raviart elements employ a piecewise bi- [tri-]linear, globally
388discontinuous pressure representation. In each element, the pressure
389is represented by bi-[tri-]linear basis functions, multiplied by 3 [4]
390pressure values which are stored in the element's internal \c
391Data. Since the pressure representation is discontinuous, the pressure
392values do not have to be subjected to any constraints to ensure
393inter-element continuity. Each \c Node stores two [three] velocity
394degrees of freedom. Since the velocity representation is
395isoparametric, the default assignment for the nodal values' \c
396HangInfo pointer is appropriate and no further action is required.
