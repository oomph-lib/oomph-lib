3\mainpage Example problem: The deformation of an elastic strip by a periodic traction -- this time with spatial adaptivity
4
5In this tutorial we re-visit a linear elasticity problem that we
6already discussed in detail
7in <a href="../../periodic_load/html/index.html">
8another tutorial</a>: The deformation of an elastic strip loaded by
9a periodic surface traction. We demonstrate how to solve
10the problem with spatial adaptivity and explain how
11to apply periodic boundary conditions in such problems.

56As before, we only discretise the domain over one period of the

64\section results Results
65The figure below shows a vector plot of the displacement field
66near the upper domain boundary. As already observed in the
67<a href="../../periodic_load/html/index.html">computations with a
68spatially uniform discretisation,</a> the displacements decay rapidly with
69distance from the loaded surface. \c oomph-lib's automatic mesh
70adaptation therefore chooses a much finer discretisation near the upper
71domain boundary than in the interior, leading to a significant
72reduction in the number of unknowns in the problem.

110\section constructor The problem constructor -- applying periodic boundary condition in spatially adaptive computations.
111
112A key feature of this problem is the presence of the periodic boundary
113conditions (4) which require that the displacement
114field at the left boundary is the same as that on the right one.
115As discussed in
116<a href="../../../navier_stokes/rayleigh_channel/html/index.html#periodic">
117another tutorial,</a> such constraints can be imposed by
118the function
119\code
120BoundaryNode::make_periodic(...).
121\endcode
122Once this function is called for a \c BoundaryNode, the \c
123BoundaryNode shares its \c Data with the \c Node specified as the
124argument to that function. This "wraps the solution around the domain".
125
126In spatially adaptive computations a complication arises because
127the non-uniform refinement of a mesh creates so-called hanging nodes,
128i.e. nodes in a refined element that have no counterpart in
129an adjacent less-refined element. Within \c oomph-lib, such hanging nodes
130are automatically constrained to maintain the inter-element continuity
131of the solution. The automatic detection of hanging nodes requires the
132identification of the elements' neighbours. This task is performed by
133neighbour finding routines that operate on a tree-(forest-)based
134representation of a mesh's refinement pattern; see the discussion in
135<a href="../../../the_data_structure/html/index.html#QuadTreeInMeshes">
136the tutorial explaining \c oomph-lib's overall data structure.</a>
137The representation of the initial, unrefined mesh as a \c TreeForest,
138required by these routines, is typically generated in the mesh constructor,
139using the function \c TreeBasedRefineableMeshBase::setup_tree_forest().
140In the 4x4 mesh shown below this function would establish that
141within the \c Forest representing this mesh, the \c Tree associated
142with element 11 has three neighbours: Element 16 in the northern (N)
143direction; element 10 in the western (W) direction; element 7 in the
144southern (S) direction. There is no neighbour in the eastern (E)
145direction. If element 11 is refined but its neighbours are not, the
146hanging nodes on the edges with elements 7, 10 and 16 are
147automatically detected and constrained, maintaining the continuity
148of the solution across the element boundaries.
149
150
151\image html wrapped_forest.gif "Sketch illustrating neighbour finding in problems with periodic boundary conditions. "

155If periodic boundary conditions are applied, the "wrapping around" of
156the domain (indicated by the red line) means that element 8 must be

159element 11, using the functions \c TreeRoot::neighbour_pt(...) and

200\section comments Comments and Exercises

202\subsection nondim Comments
203
204When setting up periodic boundary conditions, it is obviously
205important to correctly identify the corresponding nodes and elements
206on the mesh boundaries. The enumeration of these nodes and elements
207is typically performed in the mesh
208constructor. If you are unsure what conventions have been used
209(and are too lazy to read the source code), recall that you can use the
210function
211\code
212Mesh::boundary_node_pt(b,j)
213\endcode
214to obtain a pointer to the \c j -th node on the \c Mesh's \c b -th
215boundary and
216\code
217Mesh::boundary_element_pt(b,j)
218\endcode
219to obtain a pointer to the \c j -th element on the \c Mesh's \c b -th
220boundary.
