69 \subsection tar_file Option 1: Get a tar file
70 
71 
72 You can download a tar file of the latest distribution
73 from the <A HREF="../../download/html/index.html">oomph-lib download
74 page</A>.
75 
76 Once you have downloaded the tar file unpack it:
77 
78 \code
79 tar xvfz oomph-lib-*.tar.gz
80 \endcode
81 
82 The entire distribution will now be contained in the sub-directory
83 <code>oomph-lib-*</code> where the "*" indicates the version/revision
84 number (the same as in the tar file).
85 
86 \subsection subversion Option 2: Get the distribution from the subversion repository
87 
88 You can also get hold of the library from our
89 <a href="http://www.subversion.tigris.org">subversion</a>-based
90 repository. The procedure for checking out from the repository is
91 explained in <a href="../../subversion/html/index.html">another
92 tutorial.</a> When checking out from the repository, you can specify
93 where the distribution is placed. By default subversion will place it
94 into a directory called <code>trunk</code>.

99 \section install Install the distribution
100 
101 \subsection standard_install Option 1: Standard installation
102 
103 Change into the \c oomph-lib directory and run the \c autogen.sh
104 build script:
105 
106 \code
107 cd oomph-lib-*
108 ./autogen.sh
109 \endcode
110 
111 This build script will ask a few questions, e.g. to verify that the default
112 build directory (<code>build </code>, relative to the oomph-lib home
113 directory) is appropriate before starting the build.
114 
115 By default, \c autogen.\c sh will build the library and the
116 demo codes with certain default settings (using the gcc compilers with full
117 optimisation, no debugging, no PARANOIA, and fully installed header files
118 rather than symbolic links). These are appropriate if you wish to install
119 the library once-and-for-all and do not anticipate any changes to their
120 sources. The section \ref custom_install provides more details on the
121 various options). `autogen.sh` will display the configure
122 options and ask for confirmation that they are appropriate. If you are
123 unsure if they are, simply hit return -- the default will be fine.
124 
125 The following flags for `autogen.sh` may be of interest:
126 
127 - `--jobs=n`: Run the build process using `n` cores. This can greatly speed
128  up build times and is strongly recommended if you have a multicore
129  machine.
130 
131 - `--rebuild`: Rebuild the configuration files from scratch. This is
132  useful if you
133  believe you may have somehow broken the build process, but should not be
134  necessary normally.
135 
136 
137 Under the hood `autogen.sh` simply collects settings in a
138 question and answer manner then calls a non-interactive script
139 `non_interactive_autogen.sh` (yes, really!) with the appropriate
140 flags. If you prefer you can simply call `non_interactive_autogen.sh`
141 directly.
142 
143 The self-tests can be initiated outside of `autogen.sh` using
144 `make check -k` (to run on a single core) or `./bin/parallel_self_test.py`
145 (to run on all available cores). These commands will compile and run all
146 the demo codes and verify their output. This is an optional step and can be
147 very time consuming, especially if run on a small number of cores.
148 
149 \subsection manual_install Manual installation
150 
151 If you're an experienced linux/unix user, you may prefer to control the
152 build/install process manually, using the standard \c configure; \c make;
153 \c make \c install; \c make \c check; sequence. Note that this will *only*
154 work when building the tar file version (i.e. not the version from svn
155 which still requires the configure script to be generated). In this case,
156 the library will be compiled with GNU's default compiler options ("-O3 -g")
157 and the libraries and include files will be installed in \c /usr/local/
158 unless you specify a different directory with the \c --prefix argument to
159 the \c configure command. Here's an example of a manual installation
160 
161 - Unpack the distribution (you may have to replace the "*" with the
162  actual revision number)
163  \code
164  tar xvfz oomph-lib-*.tar.gz
165  \endcode
166 - Change into the oomph-lib home directory (again, you may have to
167  replace the "*" with the actual revision information)
168  \code
169  cd oomph-lib-*
170  \endcode
171 - Store the oomph-lib home directory in shell variable -- in a bash
172  shell this is done like this:
173  \code
174  OOMPH_HOME=`pwd`
175  \endcode
176 - Create the build directory -- specify your own here!
177  \code
178  mkdir $OOMPH_HOME/build
179  \endcode
180 - Configure the distribution -- prefix specifies the build directory
181  \code
182  ./configure --prefix $OOMPH_HOME/build
183  \endcode
184 - Make and install the library
185  \code
186  make; make install
187  \endcode
188 - Done! Check if we can make and run the user codes
189  \code
190  cd $OOMPH_HOME/user_drivers/joe_cool
191  make
192  ./joes_poisson_code
193  \endcode
194 - Run the self-tests
195  \code
196  cd $OOMPH_HOME
197  make check -k
198  \endcode
199 .

201 \subsection custom_install Configuration options
202 
203 The build scripts allow you to specify a file of configure options.
204 For instance, you may wish to
205 specify another compiler, change the optimisation level, allow for
206 debugging or range checking, etc. Previously used sets of options are
207 stored in various files in the sub-directory `config/configure_options/`.
208 The default settings are in `default`; the currently used ones are in
209 `current`.
210 
211 What options are there? You can get a complete list by typing
212 
213  ./configure --help
214 
215 in `oomph-lib`'s top level directory. Here are some options that
216 we use frequently:
217 
218 
219 \subsubsection no_doc Suppressing the build of the documentation:
220 Building the online documentation locally is time consuming and
221 requires a significant amount of disk space. Since the documentation
222 is also available from the <A href="http://www.oomph-lib.org">oomph-lib
223 homepage</a> you may wish to suppress this step.
224 To achieve this, specify the configure option
225 
226  --enable-suppress-doc
227 
228 \subsubsection no_doc Suppressing the build of the pdf version of the documentation:
229 By default the tutorials are built as html files (which are best accessed by
230 starting from the local copy of the \c oomph-lib homepage, which is in
231 `doc/html/index.html`) and as pdf files
232 (which are accessible via a link at the bottom of the relevant html-based
233 tutorial). In the past we have sometimes had problems with
234 \c doxygen (and hence the entire build process!) hanging while the
235 pdf files are generated. To avoid this (by not even attempting
236 to create the pdf files) specify the configure option
237 
238  --enable-suppress-pdf-doc
239 
240 (or better: update to a more recent version of \c doxygen -- for
241 instance version 1.8.6 works).
242 
243 \subsubsection sym_links Replace library headers by symbolic links to the sources
244 During the build process \c oomph-lib's various libraries are installed in
245 the subdirectory \c build/lib (or in whatever other directory you may
246 have specified when asked to confirm their location) and the
247 associated header files are copied to \c build/include. This is a
248 sensible default for libraries that are only installed
249 once and then never again tinkered with. Here the situation is
250 slightly different: If you ever decide to add your own "user libraries" to \c
251 oomph-lib (and you are encouraged to do so!), their header files will
252 also be copied to \c build/include.
253 If during code development, any of your header files contain syntax errors,
254 the compiler will complain about the syntax errors in the copied file
255 in \c build/include rather than the one in your source directory.
256 This will encourage you to edit the copied file rather than the
257 original -- clearly a recipe for disaster! To avoid it we provide
258 the configure option
259 
260  --enable-symbolic-links-for-headers
261 
262 In this mode, the copies of the header files in \c build/include
263 are replaced by symbolic links to the actual sources in
264 \c src or \c user_src.
265 
266 \subsubsection paranoia PARANOIA
267 \c oomph-lib provides an extensive range of optional run-time self-tests. The
268 self-tests issue diagnostic error messages if any inconsistencies are
269 detected at run-time and then terminate the code execution (semi-)gracefully
270 by throwing an exception which (if not caught) aborts. This
271 allows backtracking of the call sequence
272 in a debugger during code development. Obviously, the self-tests
273 introduce a slight run-time overhead and are therefore only performed if the
274 C++ code is compiled with a special compiler flag, \c PARANOID.
275 For gcc (and most other compilers we know) this is done by passing the flag
276 \c -DPARANOID to the C++ compiler. This is achieved by adding
277 \code
278 CXXFLAGS="-DPARANOID"
279 \endcode
280 to the configure options. As discussed, during code development,
281 this is most useful if debugging is also enabled, so the combination
282 \code
283 CXXFLAGS="-g -DPARANOID"
284 \endcode
285 is common.
286 
287 
288 \subsubsection bounds Range checking
289 Most of the containers used in \c oomph-lib allow for optional
290 (and very costly!) range checking which is enabled by
291 specifying the C++ compiler flag \c RANGE_CHECKING. You are advised
292 to recompile the code (yes, all of it!) if (and only if) you
293 encounter some mysterious segmentation fault. Again, this is most
294 useful if used together with debugging,
295 \code
296 CXXFLAGS="-g -DRANGE_CHECKING"
297 \endcode
298 (Without the \c -g flag you will only find out that an illegal
299 index has been specified, not where this happened...)

303 \section nav Finding your way through the distribution
304 
305 The \c oomph-lib distribution has several main sub-directories:
306 
307 \subsection src The src directory
308 
309 The \c src sub-directory contains the source code for the
310 various sub-libraries that make up \c oomph-lib.
311 The most important one is the \c generic library which is built from
312 the sources in the sub-directory \c src/generic. This library
313 defines the fundamental \c oomph-lib objects (nodes, elements, meshes,
314 timesteppers, linear and nonlinear solvers, mesh-adaptation routines,
315 etc.)
316 
317 The other libraries (\c poisson, \c navier-stokes,...)
318 define elements for the solution of specific systems of PDEs.
319 
320 Finally, the \c src/meshes subdirectory contains several fully
321 functional \c Meshes (and, where appropriate, the associated
322 \c Domains) that are used in the demo codes.
323 All \c Meshes in this directory are templated by the type of element they
324 contain. Since the element type can only be specified in the driver
325 codes, the meshes cannot be compiled into libraries --
326 the sources are always included as header-like files. Our
327 <A HREF="../../example_code_list/html/index.html">list of
328 example codes</A> contains an
329 <A HREF="../../poisson/two_d_poisson/html/index.html#precompile_mesh">
330 example</A> that explains this in more detail.
331 
332 
333 
334 
335 \subsection ext_src The external_src directory
336 
337 The \c external_src directory contains "frozen" versions of
338 various external libraries (e.g.
339 <A HREF="http://crd.lbl.gov/~xiaoye/SuperLU">SuperLU</A>). Inclusion of these
340 libraries into the distribution facilitates the overall build process:
341 You only have to download and install a single distribution. This
342 is much easier than finding out (typically halfway through the install
343 process), that library A depends on library B which invariably turns
344 out to depend on library C, etc.) Often the sources in the \c external_src
345 subdirectories are sub-sets of the full libraries. For instance,
346 we only include the double precision sources for
347 <A HREF="http://crd.lbl.gov/~xiaoye/SuperLU">SuperLU</A> as
348 neither the complex nor the single-precision versions are required
349 within \c oomph-lib.
350 
351 \subsection ext_dist The external_distributions directory
352 
353 \c oomph-lib provides interfaces to various third-party libraries
354 which have their own build machinery. Some of these libraries are built by
355 default in the course of the oomph-lib installation, using tar
356 files that are distributed with \c oomph-lib; others
357 will only be built if the user places the relevant tar files
358 into the approproriate location within the \c oomph-lib directory
359 structure.
360 
361 \subsubsection cgal External distributions that are built by default
362 
363 By default \c oomph-lib builds CGAL, the Computational Geometry Algorithms
364 Library, <a href="http://www.cgal.org">http://www.cgal.org</a>. This
365 library requires three other libraries which we also install:
366 - The GNU Multiple Precision Arithmetic Library (GMP),
367  <a href="https://gmplib.org">https://gmplib.org</a>.
368 - The GNU MPFR Library <a href="http://www.mpfr.org">https://gmplib.org</a>.
369 - The Boost library, <a href="http://www.boost.org">http://www.boost.org</a>.
370 .
371 Note that the installation of these libraries is not quick. We
372 therefore provide the option to (i) suppress their installation (in which case
373 \c oomph-lib will employ a sub-optimal "locate_zeta" algorithm in its
374 multi-domain algorithms) or to (ii) link against already existing
375 installations of the libraries; see \ref external_dist_cgal_etc for details.
376 
377 \subsubsection trilinos_hypre_external External distributions that are not built by default
378 
379 \c oomph-lib provides interfaces to various optional third-party libraries
380 whose sources we deemed to be too big to be included in the \c oomph-lib
381 distribution. If you wish to use these we expect you to install them
382 yourself. To facilitate this task, we provide the option
383 to let the \c oomph-lib build machinery perform the installation
384 for you. If you place a copy of the tar file into the appropriate
385 sub-directory in \c external_distributions, \c oomph-lib will build
386 and install the library for you; see \ref external_dist for details.
387 (Note that \c oomph-lib is fully functional without these libraries --
388 if the libraries are not available the build process ignores any \c oomph-lib
389 code that depends on them.)
390 
391 
392 \subsection demo_drivers The demo_drivers directory
393 
394 The \c demo_drivers directory contains a large number of demo codes.
395 They are arranged in sub-directories, based on the type of the
396 problem that is being solved. For instance, the \c demo_drivers/poisson
397 subdirectory contains a number of demo problems involving
398 the Poisson equation.
399 
400  All sub-directories in
401 \c demo_drivers contain shell scripts that validate the output
402 from the demo codes by comparing the computed results against the
403 reference results stored in the \c validata sub-directories.
404 The comparison is performed with the \c python
405 script \c bin/fpdiff.\c py which tolerates slight differences due
406 to the unavoidable variations in roundoff error on different
407 platforms and/or at different optimisation levels. The validation
408 scripts can either be executed individually in each sub-directory
409 or for all sub-directories by issuing the command
410 \c make \c check.
411 
412  If the self-test is run at the top-level, a summary of the
413 self-tests is stored in \c self_test/analyse_self_tests/validation.\c log
414

417

419 of \c demo_drivers

444 \section external External (third-party) libraries
445 
446 \c oomph-lib provides interfaces to a number of third-party libraries.
447 Those libraries that are essential for \c oomph-lib are distributed
448 with the library to ensure that the user does not have to install
449 these separately. We also provide interfaces to a number of
450 third-party libraries that are not distributed with \c oomph-lib,
451 typically because they are too big and/or take (too?) long to build.
452 \c oomph-lib will only build the interfaces to these libraries if
453 they are available and their location is specified during the
454 configuration stage (or if the appropriate tar file
455 is dropped into the required directory in which case \c oomph-lib's
456 build process will build and install the library for you). In the
457 latter case, the compiler flags used to build \c oomph-lib will
458 be passed directly to the third-party libraries, so you may wish to compile
459 the libraries separately if you wish to specify different compiler
460 flags, e.g. no debugging information.
461 
462 
463 \subsection external_src External (third-party) libraries that are distributed with oomph-lib
464 
465 \c oomph-lib provides local copies of the following third-party libraries:
466 - <A HREF="http://netlib.org/blas">BLAS</A>
467 - <A HREF="http://netlib.org/lapack">LAPACK</A>
468 - <A HREF="http://crd.lbl.gov/~xiaoye/SuperLU">SuperLU</A>
469 - <A HREF="http://www-users.cs.umn.edu/~karypis/metis/">METIS</A>
470 .
471 By default \c oomph-lib automatically builds and links against these.

473 \subsubsection blas Specifying an existing, local copy of the blas library
474 
475 If a local, possibly optimised version of the
476 <A HREF="http://netlib.org/blas">blas library</A>
477 already exists on your machine you can force \c oomph-lib to link
478 against it and avoid the compilation of \c oomph-lib's own copy.
479 
480 If your local copy of the <A HREF="http://netlib.org/blas">blas library</A>
481 is located at
482 \code
483 /home/mheil/local/lib/blas/blas.a
484 \endcode
485 say, you can link against it by specifying the configure option
486 \code
487 --with-blas=/home/mheil/local/lib/blas/blas.a
488 \endcode
489 
490 \subsubsection lapack Specifying an existing, local copy of the lapack library
491 If a local, possibly optimised version of the
492 <A HREF="http://netlib.org/lapack">lapack library</A>
493 already exists on your machine you can force \c oomph-lib to
494 link against it and avoid the compilation of \c oomph-lib's
495  own copy.
496 
497 For instance, if your local copy of the
498 <A HREF="http://netlib.org/lapack">lapack library</A> is located at
499 \code
500 /home/mheil/local/lib/lapack/lapack.a
501 \endcode
502 you can link against it by specifying the configure option
503 \code
504 --with-lapack=/home/mheil/local/lib/lapack/lapack.a
505 \endcode

508 \subsection external_dist_cgal_etc External (third-party) libraries whose tar files are distributed with oomph-lib
509 
510 By default \c oomph-lib builds CGAL, the Computational Geometry Algorithms
511 Library, <a href="http://www.cgal.org">http://www.cgal.org</a>. This
512 library requires three other libraries which we also install:
513 - The GNU Multiple Precision Arithmetic Library (GMP),
514  <a href="https://gmplib.org">https://gmplib.org</a>.
515 - The GNU MPFR Library <a href="http://www.mpfr.org">https://gmplib.org</a>.
516 - The Boost library, <a href="http://www.boost.org">http://www.boost.org</a>.
517 .
518 These four libraries are built from tar files that we downloaded
519 from the relevant webpages and then included into the \c oomph-lib
520 distribution. We adopted this procedure to ensure that the versions
521 of the libraries are consistent with each other.
522 
523 \subsubsection cgal_default_install Default CGAL installation.
524 
525 By default the four libraries are installed within \c oomph-lib's
526 \c external_distributions directory, and the paths to the relevant
527 \c lib and \c include directories are propagated to \c oomph-lib's
528 \c Makefile s. The installations are deleted by "make clean" or
529 "make distclean", i.e. they are treated like any other \c oomph-lib
530 code.
531 
532 \subsubsection cgal_permanent_install Installing CGAL in a permanent location
533 
534 Given that the installation of the libraries takes a fair amount of
535 time, we also provide the option to install them in a permanent location
536 outside the \c oomph-lib directory structure. This is done by
537 specifying the configure flag:
538 \code
539 --with-cgal-permanent-installation-dir=ABSOLUTE_PATH_TO_PERMAMENT_INSTALL_DIRECTORY
540 \endcode
541 where \c ABSOLUTE_PATH_TO_PERMAMENT_INSTALL_DIRECTORY specifies what
542 it says. So, for instance, specifying
543 \code
544 --with-cgal-permanent-installation-dir=/home/mheil/junk_default_installation
545 \endcode
546 installs the libraries in \c /home/mheil/junk_default_installation.
547 In subsequent rebuilds of \c oomph-lib it is then possible to
548 specify the location of these libraries using configure options.
549 Following an \c oomph-lib installation with
550 \c --with-cgal-permanent-installation-dir these configure options
551 are displayed at the end of the \c oomph-lib build procedure. (They are
552 also contained in the file
553 \c external_distributions/cgal_configure_flags.txt.)
554 For instance, if the libraries have been installed in
555 \c /home/mheil/junk_default_installation the relevant, the configure
556 options are:
557 \code
558 --with-boost=/home/mheil/junk_default_installation/boost_default_installation
559 --with-gmp=/home/mheil/junk_default_installation/gmp_default_installation
560 --with-mpfr=/home/mheil/junk_default_installation/mpfr_default_installation
561 --with-cgal=/home/mheil/junk_default_installation/cgal_default_installation
562 \endcode

577 \subsubsection hypre Hypre
578

795 \section own How to write your own code and link it against oomph-lib's library/libraries
796 
797 If you followed the instructions so far, you will be able to install
798 \c oomph-lib and run the demo codes that are provided in the
799 \c demo_drivers directory. Great! Now on to the next step: How do
800 you write your own codes and link them against \c oomph-lib?
801 There are two options, depending on
802 whether you have (or are willing to install) the gnu autotools
803 <A HREF="http://www.gnu.org/software/autoconf/">autoconf</A>,
804 <A HREF="http://www.gnu.org/software/automake/">automake</A>
805 and <A HREF="http://www.gnu.org/software/libtool/">libtool</A>
806 on your machine.
807 
808 \subsection have_auto_tools Writing/linking your own driver codes under autotools control
809 
810 \subsubsection add_driver Adding your own driver codes
811 Let's start with the straightforward case: You want to use \c
812 oomph-lib to solve one of your own problems. To do this within
813 \c oomph-lib's autotools framework, simply create a new
814 directory in \c user_drivers and write your driver code. To facilitate
815 these steps, the \c user_drivers directory already contains a
816 sample directory \c joe_cool for which all these steps have been
817 performed. If you don't object to the directory name (or if your
818 name is Joe Cool) you can simply work in that directory.
819 If not, we suggest the following sequence of steps:
820 -# Go to the \c user_drivers directory and create a new
821  directory, e.g.
822 \code
823 cd user_drivers
824 mkdir josephine_cool
825 \endcode
826 
827 -# Copy the \c Makefile.am and the driver code \c joes_poisson_code.cc
828  from \c user_drivers/joe_cool to \c user_drivers/josephine_cool.
829 
830 -# Return to \c oomph-lib's top-level directory and re-run `./autogen.sh`
831  to generate the required Makefile etc. in your own directory.
832 
833 -# You may now return to your own directory in \c user_drivers and
834  make your own driver code:
835 \code
836 cd user_drivers/josephine_cool
837 make
838 \endcode
839  This will create the required executable.
840  \n\n
841 -# Unfortunately, the driver code (copied from Joe Cool's directory!) is
842  unlikely to be the one you want but you can now rename it, edit it,
843  or add further driver codes to your directory. In general we
844  suggest that you have a look at the
845  <A HREF="../../example_code_list/html/index.html">list of
846  example codes</A> and try to identify a problem that is similar to the
847  one you want to solve. The associated driver code will be a good starting
848  point for your own. Note that whenever you add new driver codes or
849  rename existing ones you will have to update the local
850  \c Makefile.\c am, though it is not necessary to re-run
851  \c autogen.\c sh. The sample \c Makefile.\c am
852  copied from Joe Cool's directory is well annotated and gives
853  clear instructions how to adapt its contents:
854  \n\n
855 \include "joe_cool_makefile.am"
856  \n\n
857  Note the following points: \n\n
858  - Lines that start with a "#" are comments.
859  \n\n
860  - The first (non-comment) line in the above file specifies the name(s) of
861  the executable(s) that will be created by make. These names must be
862  the same as those in the \c *_SOURCES and \c *_LDADD variables.
863  \n\n
864  - The \c joes_poisson_code_SOURCES variable declares which (local)
865  sources your executable depends on. In the current example there
866  is only a single file, the driver code,
867  \c joes_poisson_code. \c cc, itself.
868  \n\n
869  - The \c joes_poisson_code_LDADD variable declares:
870  - the location of the library directory (automake
871  will convert the macro \c -L\@libdir\@ into the actual
872  directory -- you don't have to change this!).
873  - the libraries (\c oomph-lib or otherwise) that you wish to link
874  against. This is done with the usual \c -l flag that you
875  will be familiar with from your compiler. Have a look at the
876  comments regarding the order of the libraries!
877  \n\n
878  - The \c INCLUDES variable specifies where to find the include
879  header files. This line is again completely generic -- automake
880  will convert the macro \c -I\@includedir\@ into the actual
881  location.
882  \n\n
883  - If you have multiple driver codes, add the name of all
884  executables to the \c noinst_PROGRAMS variable, and specify
885  the \c *_SOURCES and \c *_LDADD variables
886  for each one.
887  \n\n
888  - The \c INCLUDES variable should only be specified once.
889  .
890 
891 
892 \subsubsection add_src Adding new libraries and linking against them from driver codes
893 
894 The above instructions should be sufficient to get you started.
895 You can create multiple sub-directories for different projects
896 and each sub-directory may, of course, contain multiple files,
897 separated into header and source files. \c automake will ensure that
898 only those files that have been
899 changed will be recompiled when you issue the \c make
900 command. However, at some point you may wish to package some of your sources
901 into your own library and maybe even offer it for permanent inclusion into
902 \c oomph-lib. For this purpose the \c oomph-lib distribution provides the
903 sub-directory \c user_src which closely mirrors that of the
904 \c src and \c external_src directories discussed earlier. During the
905 build process, each sub-directory in \c user_src is compiled
906 into its own library and installed in the standard location.
907 
908  The steps required to include your own library into the \c oomph-lib
909 build process are very similar to those required to
910 add additional user drivers. As before, the \c user_src directory
911 already contains a sample directory \c jack_cool, to facilitate
912 the procedure. We therefore suggest the following sequence of steps:
913 -# Go to the \c user_src directory and create a new
914  directory, e.g.
915 \code
916 cd user_src
917 mkdir jacqueline_cool
918 \endcode
919  \n\n
920 -# Copy the \c Makefile.am and the codes \c hello_world.cc and
921  \c hello_world.h
922  from \c user_src/jack_cool to \c user_src/jacqueline_cool.
923  \n\n
924 -# Return to the top-level \c oomph-lib directory and re-run
925  `autogen.sh`.
926 
927 -# You may now return to your own directory in \c user_src and
928  make and install our own library
929 \code
930 cd user_src/jacqueline_cool
931 make
932 make install
933 \endcode
934  This will create the library and install it in \c build/lib
935  \n\n
936  The \c Makefile.\c am for libraries is slightly more complicated
937  (though reasonably well documented) so -- for now -- we'll
938  just list it here and hope that the changes required
939  to include additional sources are obvious. If you really can't
940  figure it out, send us an email and prompt us to complete
941  this bit of the documentation....
942  \n\n
943 \include "linked_make_file.am"
944 
945 Note that the directory \c user_drivers/jack_cool contains
946 an example of a user driver code (\c jacks_own_code \c .cc)
947 that uses a user library.

951 \subsection dont_have_auto_tools Writing/linking user code without autotools
952 Linking directly against \c oomph-lib's (sub-)libraries
953 is slightly complicated by cross-compilation issues arising from
954 the fact that the \c oomph-lib distribution includes a few C and Fortran
955 sources. When linking is done (by the C++ compiler) one usually has to
956 explicitly specify a few compiler-specific Fortran libraries.
957 The beauty of the \c autotools approach described above is that these libraries
958 (and any other flags that need to be passed to the compiler/linker)
959 are determined and specified automatically. Doing this manually
960 is no fun! Have a look at
961 Mike Gerdts's excellent document
962 <A HREF="https://sites.google.com/site/mgerdts/whatgccreallydoes">
963 "How gcc really works"</A> for details.
964 
965  The good news is that \c oomph-lib's installation procedure automatically
966 generates a sample \c Makefile that contains all the relevant
967 information. Once the installation is complete, the sample makefile is
968 located at
969 \code
970 demo_drivers/linking/makefile.sample
971 \endcode
972 Here is the version that was generated one of our machines:
973 
974 \include makefile.sample.biowulf
975 
976 The version that is generated during the build process
977 on your machine provides template for your own customised
978 Makefiles. When modifying the sample to different driver codes,
979 you should not (have to) edit any of the "installation specific"
980 variables. Simply specify the \c oomph-lib (sub-)libraries that you wish to
981 link against in the \c OOMPH_LIB-LIBS variable (in the example shown
982 above, we are linking against the \c generic and \c poisson libraries),
983 and specify the dependencies for your own driver code, following the
984 usual \c Makefile syntax. The executable may then be created by the usual
985 \code
986 make -f makefile.sample my_demo_code
987 \endcode
988 
989 <a name="apple_linking">
990 <b>Note/Disclaimer:</b></a> The sample \c Makefile generated during \c oomph-lib's
991 installation should work for most (if not all) linux machines, though it may
992 require slight tweaks for Darwin (the BSD-derived UNIX core of Apple's
993 OSX operating system). Problems are most
994 likely to arise from the \c SHARED_LIBRARY_FLAGS variable.
995 As mentioned in the comment in the sample \c Makefile, on such
996 machines the fragment <code>-Wl,--rpath -Wl,</code> should be deleted from the
997 \c SHARED_LIBRARY_FLAGS variable.
