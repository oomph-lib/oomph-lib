57 \section install Install the distribution
58 
59 \subsection standard_install Installation with \c autogen.sh
60 
61 Change into the \c oomph-lib directory and run the \c autogen.sh
62 build script:
63 
64 \code
65 cd oomph-lib
66 ./autogen.sh
67 \endcode
68 
69 This build script will ask a few questions, e.g. to verify that the default
70 build directory (<code>build </code>, relative to the oomph-lib home
71 directory) is appropriate before starting the build.
72 
73 By default, \c autogen.\c sh will build the library and the
74 demo codes with certain default settings (using the gcc compilers with full
75 optimisation, no debugging, no PARANOIA, and fully installed header files
76 rather than symbolic links). These are appropriate if you wish to install
77 the library once-and-for-all and do not anticipate any changes to their
78 sources. The section \ref custom_install provides more details on the
79 various options). `autogen.sh` will display the configure
80 options and ask for confirmation that they are appropriate. If you are
81 unsure if they are, simply hit return -- the default will be fine.
82 
83 The following flags for `autogen.sh` may be of interest:
84 
85 - `--jobs=n`: Run the build process using `n` cores. This can greatly speed
86  up build times and is strongly recommended if you have a multicore
87  machine.
88 
89 - `--rebuild`: Rebuild the configuration files from scratch. This is
90  useful if you
91  believe you may have somehow broken the build process, but should not be
92  necessary normally.
93 
94 
95 Under the hood `autogen.sh` simply collects settings in a
96 question and answer manner then calls a non-interactive script
97 `non_interactive_autogen.sh` (yes, really!) with the appropriate
98 flags. If you prefer you can simply call `non_interactive_autogen.sh`
99 directly.
100 
101 The self-tests can be initiated outside of `autogen.sh` using
102 `make check -k` (to run on a single core) or `./bin/parallel_self_test.py`
103 (to run on all available cores). These commands will compile and run all
104 the demo codes and verify their output. This is an optional step and can be
105 very time consuming, especially if run on a small number of cores.
106

108 \subsection custom_install Configuration options
109 
110 The build scripts allow you to specify a file of configure options.
111 For instance, you may wish to
112 specify another compiler, change the optimisation level, allow for
113 debugging or range checking, etc. Previously used sets of options are
114 stored in various files in the sub-directory `config/configure_options/`.
115 The default settings are in `default`; the currently used ones are in
116 `current`.
117 
118 What options are there? You can get a complete list by typing
119 
120  ./configure --help
121 
122 in `oomph-lib`'s top level directory. Here are some options that
123 we use frequently:
124 
125 
126 \subsubsection no_doc Suppressing the build of the documentation:
127 Building the online documentation locally is time consuming and
128 requires a significant amount of disk space. Since the documentation
129 is also available from the <A href="http://www.oomph-lib.org">oomph-lib
130 homepage</a> you may wish to suppress this step.
131 To achieve this, specify the configure option
132 
133  --enable-suppress-doc
134 
135 \subsubsection no_pdf Suppressing the build of the pdf version of the documentation:
136 By default the tutorials are built as html files (which are best accessed by
137 starting from the local copy of the \c oomph-lib homepage, which is in
138 `doc/html/index.html`) and as pdf files
139 (which are accessible via a link at the bottom of the relevant html-based
140 tutorial). In the past we have sometimes had problems with
141 \c doxygen (and hence the entire build process!) hanging while the
142 pdf files are generated. To avoid this (by not even attempting
143 to create the pdf files) specify the configure option
144 
145  --enable-suppress-pdf-doc
146 
147 (or better: update to a more recent version of \c doxygen -- for
148 instance version 1.8.6 works).
149 
150 \subsubsection sym_links Replace library headers by symbolic links to the sources
151 During the build process \c oomph-lib's various libraries are installed in
152 the subdirectory \c build/lib (or in whatever other directory you may
153 have specified when asked to confirm their location) and the
154 associated header files are copied to \c build/include. This is a
155 sensible default for libraries that are only installed
156 once and then never again tinkered with. Here the situation is
157 slightly different: If you ever decide to add your own "user libraries" to \c
158 oomph-lib (and you are encouraged to do so!), their header files will
159 also be copied to \c build/include.
160 If during code development, any of your header files contain syntax errors,
161 the compiler will complain about the syntax errors in the copied file
162 in \c build/include rather than the one in your source directory.
163 This will encourage you to edit the copied file rather than the
164 original -- clearly a recipe for disaster! To avoid it we provide
165 the configure option
166 
167  --enable-symbolic-links-for-headers
168 
169 In this mode, the copies of the header files in \c build/include
170 are replaced by symbolic links to the actual sources in
171 \c src or \c user_src.
172 
173 \subsubsection paranoia PARANOIA
174 \c oomph-lib provides an extensive range of optional run-time self-tests. The
175 self-tests issue diagnostic error messages if any inconsistencies are
176 detected at run-time and then terminate the code execution (semi-)gracefully
177 by throwing an exception which (if not caught) aborts. This
178 allows backtracking of the call sequence
179 in a debugger during code development. Obviously, the self-tests
180 introduce a slight run-time overhead and are therefore only performed if the
181 C++ code is compiled with a special compiler flag, \c PARANOID.
182 For gcc (and most other compilers we know) this is done by passing the flag
183 \c -DPARANOID to the C++ compiler. This is achieved by adding
184 \code
185 CXXFLAGS="-DPARANOID"
186 \endcode
187 to the configure options. As discussed, during code development,
188 this is most useful if debugging is also enabled, so the combination
189 \code
190 CXXFLAGS="-g -DPARANOID"
191 \endcode
192 is common.
193 
194 
195 \subsubsection bounds Range checking
196 Most of the containers used in \c oomph-lib allow for optional
197 (and very costly!) range checking which is enabled by
198 specifying the C++ compiler flag \c RANGE_CHECKING. You are advised
199 to recompile the code (yes, all of it!) if (and only if) you
200 encounter some mysterious segmentation fault. Again, this is most
201 useful if used together with debugging,
202 \code
203 CXXFLAGS="-g -DRANGE_CHECKING"
204 \endcode
205 (Without the \c -g flag you will only find out that an illegal
206 index has been specified, not where this happened...)

210 \section nav Finding your way through the distribution
211 
212 The \c oomph-lib distribution has several main sub-directories:
213 
214 \subsection src The src directory
215 
216 The \c src sub-directory contains the source code for the
217 various sub-libraries that make up \c oomph-lib.
218 The most important one is the \c generic library which is built from
219 the sources in the sub-directory \c src/generic. This library
220 defines the fundamental \c oomph-lib objects (nodes, elements, meshes,
221 timesteppers, linear and nonlinear solvers, mesh-adaptation routines,
222 etc.)
223 
224 The other libraries (\c poisson, \c navier-stokes,...)
225 define elements for the solution of specific systems of PDEs.
226 
227 Finally, the \c src/meshes subdirectory contains several fully
228 functional \c Meshes (and, where appropriate, the associated
229 \c Domains) that are used in the demo codes.
230 All \c Meshes in this directory are templated by the type of element they
231 contain. Since the element type can only be specified in the driver
232 codes, the meshes cannot be compiled into libraries --
233 the sources are always included as header-like files. Our
234 <A HREF="../../example_code_list/html/index.html">list of
235 example codes</A> contains an
236 <A HREF="../../poisson/two_d_poisson/html/index.html#precompile_mesh">
237 example</A> that explains this in more detail.
238 
239 
240 
241 
242 \subsection ext_src The external_src directory
243 
244 The \c external_src directory contains "frozen" versions of
245 various external libraries (e.g.
246 <A HREF="http://crd.lbl.gov/~xiaoye/SuperLU">SuperLU</A>). Inclusion of these
247 libraries into the distribution facilitates the overall build process:
248 You only have to download and install a single distribution. This
249 is much easier than finding out (typically halfway through the install
250 process), that library A depends on library B which invariably turns
251 out to depend on library C, etc.) Often the sources in the \c external_src
252 subdirectories are sub-sets of the full libraries. For instance,
253 we only include the double precision sources for
254 <A HREF="http://crd.lbl.gov/~xiaoye/SuperLU">SuperLU</A> as
255 neither the complex nor the single-precision versions are required
256 within \c oomph-lib.
257 
258 \subsection ext_dist The external_distributions directory
259 
260 \c oomph-lib provides interfaces to various third-party libraries
261 which have their own build machinery. Some of these libraries are built by
262 default in the course of the oomph-lib installation, using tar
263 files that are distributed with \c oomph-lib; others
264 will only be built if the user places the relevant tar files
265 into the approproriate location within the \c oomph-lib directory
266 structure.
267 
268 \subsubsection cgal External distributions that are built by default
269 
270 By default \c oomph-lib builds CGAL, the Computational Geometry Algorithms
271 Library, <a href="http://www.cgal.org">http://www.cgal.org</a>. This
272 library requires three other libraries which we also install:
273 - The GNU Multiple Precision Arithmetic Library (GMP),
274  <a href="https://gmplib.org">https://gmplib.org</a>.
275 - The GNU MPFR Library <a href="http://www.mpfr.org">https://gmplib.org</a>.
276 - The Boost library, <a href="http://www.boost.org">http://www.boost.org</a>.
277 .
278 Note that the installation of these libraries is not quick. We
279 therefore provide the option to (i) suppress their installation (in which case
280 \c oomph-lib will employ a sub-optimal "locate_zeta" algorithm in its
281 multi-domain algorithms) or to (ii) link against already existing
282 installations of the libraries; see \ref external_dist_cgal_etc for details.
283 
284 \subsubsection trilinos_hypre_external External distributions that are not built by default
285 
286 \c oomph-lib provides interfaces to various optional third-party libraries
287 whose sources we deemed to be too big to be included in the \c oomph-lib
288 distribution. If you wish to use these we expect you to install them
289 yourself. To facilitate this task, we provide the option
290 to let the \c oomph-lib build machinery perform the installation
291 for you. If you place a copy of the tar file into the appropriate
292 sub-directory in \c external_distributions, \c oomph-lib will build
293 and install the library for you; see \ref external_dist for details.
294 (Note that \c oomph-lib is fully functional without these libraries --
295 if the libraries are not available the build process ignores any \c oomph-lib
296 code that depends on them.)
297 
298 
299 \subsection demo_drivers The demo_drivers directory
300 
301 The \c demo_drivers directory contains a large number of demo codes.
302 They are arranged in sub-directories, based on the type of the
303 problem that is being solved. For instance, the \c demo_drivers/poisson
304 subdirectory contains a number of demo problems involving
305 the Poisson equation.
306 
307  All sub-directories in
308 \c demo_drivers contain shell scripts that validate the output
309 from the demo codes by comparing the computed results against the
310 reference results stored in the \c validata sub-directories.
311 The comparison is performed with the \c python
312 script \c bin/fpdiff.\c py which tolerates slight differences due
313 to the unavoidable variations in roundoff error on different
314 platforms and/or at different optimisation levels. The validation
315 scripts can either be executed individually in each sub-directory
316 or for all sub-directories by issuing the command
317 \c make \c check.
318 
319  If the self-test is run at the top-level, a summary of the
320 self-tests is stored in \c self_test/analyse_self_tests/validation.\c log
321 
322 
323 \subsection doc The doc directory
324 
325 The structure of the \c doc directory (approximately) mirrors that
326 of \c demo_drivers
327 and contains the source code for the \c doxygen - based detailed
328 explanation of the demo codes. If \c configure locates a
329 sufficiently up-to-date version of \c doxygen on your system,
330 the entire \c oomph-lib documentation will be built locally
331 and can be navigated from the homepage in
332 \c doc/html/index.\c html -- a copy of the \c oomph-lib homepage.

336 \subsection user The user_src and user_drivers directories
337 
338 The \c configure script and the associated \c Makefiles that build and
339 install the \c oomph-lib libraries and demo codes are generated
340 by <A HREF="http://www.gnu.org/software/autoconf/">autoconf</A> and
341 <A HREF="http://www.gnu.org/software/automake/">automake</A>.
342 If these powerful tools are installed on your machine,
343 you can include your own libraries and driver codes into the
344 fully-automated \c oomph-lib build process. Store your code in
345 suitably named sub-directories in \c user_src and \c user_drivers.
346 See \ref own for more details.

351 \section external External (third-party) libraries
352 
353 \c oomph-lib provides interfaces to a number of third-party libraries.
354 Those libraries that are essential for \c oomph-lib are distributed
355 with the library to ensure that the user does not have to install
356 these separately. We also provide interfaces to a number of
357 third-party libraries that are not distributed with \c oomph-lib,
358 typically because they are too big and/or take (too?) long to build.
359 \c oomph-lib will only build the interfaces to these libraries if
360 they are available and their location is specified during the
361 configuration stage (or if the appropriate tar file
362 is dropped into the required directory in which case \c oomph-lib's
363 build process will build and install the library for you). In the
364 latter case, the compiler flags used to build \c oomph-lib will
365 be passed directly to the third-party libraries, so you may wish to compile
366 the libraries separately if you wish to specify different compiler
367 flags, e.g. no debugging information.
368 
369 
370 \subsection external_src External (third-party) libraries that are distributed with oomph-lib
371 
372 \c oomph-lib provides local copies of the following third-party libraries:
373 - <A HREF="http://netlib.org/blas">BLAS</A>
374 - <A HREF="http://netlib.org/lapack">LAPACK</A>
375 - <A HREF="http://crd.lbl.gov/~xiaoye/SuperLU">SuperLU</A>
376 - <A HREF="http://www-users.cs.umn.edu/~karypis/metis/">METIS</A>
377 .
378 By default \c oomph-lib automatically builds and links against these.
379 
380 \subsubsection blas Specifying an existing, local copy of the blas library
381 
382 If a local, possibly optimised version of the
383 <A HREF="http://netlib.org/blas">blas library</A>
384 already exists on your machine you can force \c oomph-lib to link
385 against it and avoid the compilation of \c oomph-lib's own copy.
386 
387 If your local copy of the <A HREF="http://netlib.org/blas">blas library</A>
388 is located at
389 \code
390 /home/mheil/local/lib/blas/blas.a
391 \endcode
392 say, you can link against it by specifying the configure option
393 \code
394 --with-blas=/home/mheil/local/lib/blas/blas.a
395 \endcode
396 
397 \subsubsection lapack Specifying an existing, local copy of the lapack library
398 If a local, possibly optimised version of the
399 <A HREF="http://netlib.org/lapack">lapack library</A>
400 already exists on your machine you can force \c oomph-lib to
401 link against it and avoid the compilation of \c oomph-lib's
402  own copy.
403 
404 For instance, if your local copy of the
405 <A HREF="http://netlib.org/lapack">lapack library</A> is located at
406 \code
407 /home/mheil/local/lib/lapack/lapack.a
408 \endcode
409 you can link against it by specifying the configure option
410 \code
411 --with-lapack=/home/mheil/local/lib/lapack/lapack.a
412 \endcode

415 \subsection external_dist_cgal_etc External (third-party) libraries whose tar files are distributed with oomph-lib
416 
417 By default \c oomph-lib builds CGAL, the Computational Geometry Algorithms
418 Library, <a href="http://www.cgal.org">http://www.cgal.org</a>. This
419 library requires three other libraries which we also install:
420 - The GNU Multiple Precision Arithmetic Library (GMP),
421  <a href="https://gmplib.org">https://gmplib.org</a>.
422 - The GNU MPFR Library <a href="http://www.mpfr.org">https://gmplib.org</a>.
423 - The Boost library, <a href="http://www.boost.org">http://www.boost.org</a>.
424 .
425 These four libraries are built from tar files that we downloaded
426 from the relevant webpages and then included into the \c oomph-lib
427 distribution. We adopted this procedure to ensure that the versions
428 of the libraries are consistent with each other.
429 
430 \subsubsection cgal_default_install Default CGAL installation.
431 
432 By default the four libraries are installed within \c oomph-lib's
433 \c external_distributions directory, and the paths to the relevant
434 \c lib and \c include directories are propagated to \c oomph-lib's
435 \c Makefile s. The installations are deleted by "make clean" or
436 "make distclean", i.e. they are treated like any other \c oomph-lib
437 code.
438 
439 \subsubsection cgal_permanent_install Installing CGAL in a permanent location
440 
441 Given that the installation of the libraries takes a fair amount of
442 time, we also provide the option to install them in a permanent location
443 outside the \c oomph-lib directory structure. This is done by
444 specifying the configure flag:
445 \code
446 --with-cgal-permanent-installation-dir=ABSOLUTE_PATH_TO_PERMAMENT_INSTALL_DIRECTORY
447 \endcode
448 where \c ABSOLUTE_PATH_TO_PERMAMENT_INSTALL_DIRECTORY specifies what
449 it says. So, for instance, specifying
450 \code
451 --with-cgal-permanent-installation-dir=/home/mheil/junk_default_installation
452 \endcode
453 installs the libraries in \c /home/mheil/junk_default_installation.
454 In subsequent rebuilds of \c oomph-lib it is then possible to
455 specify the location of these libraries using configure options.
456 Following an \c oomph-lib installation with
457 \c --with-cgal-permanent-installation-dir these configure options
458 are displayed at the end of the \c oomph-lib build procedure. (They are
459 also contained in the file
460 \c external_distributions/cgal_configure_flags.txt.)
461 For instance, if the libraries have been installed in
462 \c /home/mheil/junk_default_installation the relevant, the configure
463 options are:
464 \code
465 --with-boost=/home/mheil/junk_default_installation/boost_default_installation
466 --with-gmp=/home/mheil/junk_default_installation/gmp_default_installation
467 --with-mpfr=/home/mheil/junk_default_installation/mpfr_default_installation
468 --with-cgal=/home/mheil/junk_default_installation/cgal_default_installation
469 \endcode

482 \subsection external_dist External (third-party) libraries that are not distributed with oomph-lib
483 
484 Note: The third-party libraries discussed here are not installed
485 by default but are built on demand if suitable tar files are placed
486 in the relevant directories in the oomph-lib directory tree. You can
487 download the tar files using the script
488 <CENTER>
489 <a href="../../../bin/get_external_distribution_tar_files.bash">bin/get_external_distribution_tar_files.bash</a>
490 </CENTER>

493 
494 \subsubsection hypre Hypre
495 
496 \c oomph-lib provides wrappers to the powerful solvers and
497 preconditioners from the
498 <a href="https://computation.llnl.gov/casc/linear_solvers/sls_hypre.html">
499 Scalable Linear Solvers Project</a>. The wrappers are only built
500 if \c Hypre is available on your machine. If your local
501 copy of the \c Hypre library installed in
502 \code
503 /home/mheil/local/hypre
504 \endcode
505 i.e. if this directory contains \c Hypre's lib and include
506 directories:
507 \code
508 biowulf:~ 10:44:22$ ll /home/mheil/local/hypre
509 total 8
510 drwxr-xr-x 2 mheil users 4096 Nov 3 2007 include
511 drwxr-xr-x 2 mheil users 4096 Nov 3 2007 lib
512 \endcode
513 you can get \c oomph-lib to link against it (and to compile
514 \c oomph-lib's wrappers to \c Hypre's solvers and preconditioners)
515 by specifying the configure option
516 \code
517 --with-hypre=/home/mheil/local/hypre
518 \endcode
519 
520 
521 \b Note: \c oomph-lib works with version 2.0.0 of the library.
522 If this version of \c Hypre is not available on your machine
523 download the tar file from the our own website:
524 
525 <CENTER>
526 <a href="https://personal.maths.manchester.ac.uk/oomphlib/oomph-lib_external_distfiles/hypre-2.0.0.tar.gz">hypre-2.0.0.tar.gz</a>
527 </CENTER>
528 
529 You can either build the library yourself
530 or get \c oomph-lib to build it for you. To do this simply place a copy of the
531 tar file into the directory
532 \code
533 external_distributions/hypre
534 \endcode
535 and (re-)run \c autogen.sh. The installation procedure will detect
536 the tar file, unpack it, and install the library in
537 \code
538 external_distributions/hypre/hypre_default_installation
539 \endcode
540 Unless you explicitly specified a library location using
541 the \c --with-hypre flag, \c oomph-lib will then link against
542 this newly created version of the library. However, we strongly
543 recommend moving the newly created library
544 to another place (outside the \c oomph-lib distribution) to
545 preserve it for future use. Once this is done you simply specify the
546 (new) location of the library with the \c --with-hypre flag, as
547 discussed above. (Also make sure to delete the tar file
548 from \c external_distributions/hypre, otherwise the library
549 will be re-built.) Note that \c make \c clean will delete the
550 unpacked \c Hypre sources but not the tar file and the library itself.

555 \c oomph-lib provides wrappers to the powerful solvers and

557 <a href="http://trilinos.sandia.gov/">
558 Trilinos Project</a>. The wrappers are only built

712 \section own How to write your own code and link it against oomph-lib's library/libraries
713 
714 If you followed the instructions so far, you will be able to install
715 \c oomph-lib and run the demo codes that are provided in the
716 \c demo_drivers directory. Great! Now on to the next step: How do
717 you write your own codes and link them against \c oomph-lib?
718 There are two options, depending on
719 whether you have (or are willing to install) the gnu autotools
720 <A HREF="http://www.gnu.org/software/autoconf/">autoconf</A>,
721 <A HREF="http://www.gnu.org/software/automake/">automake</A>
722 and <A HREF="http://www.gnu.org/software/libtool/">libtool</A>
723 on your machine.
724 
725 \subsection have_auto_tools Writing/linking your own driver codes under autotools control
726 
727 \subsubsection add_driver Adding your own driver codes
728 Let's start with the straightforward case: You want to use \c
729 oomph-lib to solve one of your own problems. To do this within
730 \c oomph-lib's autotools framework, simply create a new
731 directory in \c user_drivers and write your driver code. To facilitate
732 these steps, the \c user_drivers directory already contains a
733 sample directory \c joe_cool for which all these steps have been
734 performed. If you don't object to the directory name (or if your
735 name is Joe Cool) you can simply work in that directory.
736 If not, we suggest the following sequence of steps:
737 -# Go to the \c user_drivers directory and create a new
738  directory, e.g.
739 \code
740 cd user_drivers
741 mkdir josephine_cool
742 \endcode
743 
744 -# Copy the \c Makefile.am and the driver code \c joes_poisson_code.cc
745  from \c user_drivers/joe_cool to \c user_drivers/josephine_cool.
746 
747 -# Return to \c oomph-lib's top-level directory and re-run `./autogen.sh`
748  to generate the required Makefile etc. in your own directory.
749 
750 -# You may now return to your own directory in \c user_drivers and
751  make your own driver code:
752 \code
753 cd user_drivers/josephine_cool
754 make
755 \endcode
756  This will create the required executable.
757  \n\n
758 -# Unfortunately, the driver code (copied from Joe Cool's directory!) is
759  unlikely to be the one you want but you can now rename it, edit it,
760  or add further driver codes to your directory. In general we
761  suggest that you have a look at the
762  <A HREF="../../example_code_list/html/index.html">list of
763  example codes</A> and try to identify a problem that is similar to the
764  one you want to solve. The associated driver code will be a good starting
765  point for your own. Note that whenever you add new driver codes or
766  rename existing ones you will have to update the local
767  \c Makefile.\c am, though it is not necessary to re-run
768  \c autogen.\c sh. The sample \c Makefile.\c am
769  copied from Joe Cool's directory is well annotated and gives
770  clear instructions how to adapt its contents:
771  \n\n
772 \include "joe_cool_makefile.am"
773  \n\n
774  Note the following points: \n\n
775  - Lines that start with a "#" are comments.
776  \n\n
777  - The first (non-comment) line in the above file specifies the name(s) of
778  the executable(s) that will be created by make. These names must be
779  the same as those in the \c *_SOURCES and \c *_LDADD variables.
780  \n\n
781  - The \c joes_poisson_code_SOURCES variable declares which (local)
782  sources your executable depends on. In the current example there
783  is only a single file, the driver code,
784  \c joes_poisson_code. \c cc, itself.
785  \n\n
786  - The \c joes_poisson_code_LDADD variable declares:
787  - the location of the library directory (automake
788  will convert the macro \c -L\@libdir\@ into the actual
789  directory -- you don't have to change this!).
790  - the libraries (\c oomph-lib or otherwise) that you wish to link
791  against. This is done with the usual \c -l flag that you
792  will be familiar with from your compiler. Have a look at the
793  comments regarding the order of the libraries!
794  \n\n
795  - The \c INCLUDES variable specifies where to find the include
796  header files. This line is again completely generic -- automake
797  will convert the macro \c -I\@includedir\@ into the actual
798  location.
799  \n\n
800  - If you have multiple driver codes, add the name of all
801  executables to the \c noinst_PROGRAMS variable, and specify
802  the \c *_SOURCES and \c *_LDADD variables
803  for each one.
804  \n\n
805  - The \c INCLUDES variable should only be specified once.
806  .
807 
808 It makes sense to create a separate GitHub repository for your user
809 driver directory. This won't interfere with the forked/cloned
810 oomph-lib repository that you're working within. As far as the \c
811 oomph-lib repository is concerned your user driver directory is simply
812 one of possibly many directories that it's not tracking. Similarly, from
813 within your user driver directory, the files outside it are not
814 tracked by your repository, so the two can happily co-exist.
815 
816 \subsubsection add_src Adding new libraries and linking against them from driver codes
817 
818 The above instructions should be sufficient to get you started.
819 You can create multiple sub-directories for different projects
820 and each sub-directory may, of course, contain multiple files,
821 separated into header and source files. \c automake will ensure that
822 only those files that have been
823 changed will be recompiled when you issue the \c make
824 command. However, at some point you may wish to package some of your sources
825 into your own library and maybe even offer it for permanent inclusion into
826 \c oomph-lib. For this purpose the \c oomph-lib distribution provides the
827 sub-directory \c user_src which closely mirrors that of the
828 \c src and \c external_src directories discussed earlier. During the
829 build process, each sub-directory in \c user_src is compiled
830 into its own library and installed in the standard location.
831 
832  The steps required to include your own library into the \c oomph-lib
833 build process are very similar to those required to
834 add additional user drivers. As before, the \c user_src directory
835 already contains a sample directory \c jack_cool, to facilitate
836 the procedure. We therefore suggest the following sequence of steps:
837 -# Go to the \c user_src directory and create a new
838  directory, e.g.
839 \code
840 cd user_src
841 mkdir jacqueline_cool
842 \endcode
843  \n\n
844 -# Copy the \c Makefile.am and the codes \c hello_world.cc and
845  \c hello_world.h
846  from \c user_src/jack_cool to \c user_src/jacqueline_cool.
847  \n\n
848 -# Return to the top-level \c oomph-lib directory and re-run
849  `autogen.sh`.
850 
851 -# You may now return to your own directory in \c user_src and
852  make and install our own library
853 \code
854 cd user_src/jacqueline_cool
855 make
856 make install
857 \endcode
858  This will create the library and install it in \c build/lib
859  \n\n
860  The \c Makefile.\c am for libraries is slightly more complicated
861  (though reasonably well documented) so -- for now -- we'll
862  just list it here and hope that the changes required
863  to include additional sources are obvious. If you really can't
864  figure it out, send us an email and prompt us to complete
865  this bit of the documentation....
866  \n\n
867 \include "linked_make_file.am"
868 
869 Note that the directory \c user_drivers/jack_cool contains
870 an example of a user driver code (\c jacks_own_code \c .cc)
871 that uses a user library.

875 \subsection dont_have_auto_tools Writing/linking user code without autotools: How do I treat \c oomph-lib simply as a library?
876 Linking directly against \c oomph-lib's (sub-)libraries
877 is slightly complicated by cross-compilation issues arising from
878 the fact that the \c oomph-lib distribution includes a few C and Fortran
879 sources. When linking is done (by the C++ compiler) one usually has to
880 explicitly specify a few compiler-specific Fortran libraries.
881 The beauty of the \c autotools approach described above is that these libraries
882 (and any other flags that need to be passed to the compiler/linker)
883 are determined and specified automatically. Doing this manually
884 is no fun! Have a look at
885 Mike Gerdts's excellent document
886 <A HREF="https://sites.google.com/site/mgerdts/whatgccreallydoes">
887 "How gcc really works"</A> for details.
888 
889  The good news is that \c oomph-lib's installation procedure automatically
890 generates a sample \c Makefile that contains all the relevant
891 information. Once the installation is complete, the sample makefile is
892 located at
893 \code
894 demo_drivers/linking/makefile.sample
895 \endcode
896 Here is the version that was generated one of our machines:
897 
898 \include makefile.sample.biowulf
899 
900 The version that is generated during the build process
901 on your machine provides template for your own customised
902 Makefiles. When modifying the sample to different driver codes,
903 you should not (have to) edit any of the "installation specific"
904 variables. Simply specify the \c oomph-lib (sub-)libraries that you wish to
905 link against in the \c OOMPH_LIB-LIBS variable (in the example shown
906 above, we are linking against the \c generic and \c poisson libraries),
907 and specify the dependencies for your own driver code, following the
908 usual \c Makefile syntax. The executable may then be created by the usual
909 \code
910 make -f makefile.sample my_demo_code
911 \endcode
