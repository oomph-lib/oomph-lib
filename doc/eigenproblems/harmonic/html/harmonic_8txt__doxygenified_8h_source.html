3 \mainpage Demo problem: A one-dimensional eigenproblem
4 In this document, we demonstrate how to solve a 1D eigenproblem
5 (eigenvalues of the 1D Laplace operator in a bounded domain) by
6 creating custom elements. The tutorial is similar in spirit to the
7 <a href="../../../quick_guide/html/index.html"> quick guide </a> but far
8 less detailed.

33  An eigenfunction must be non-zero by definition and so the exact
34 solutions are given by the countably infinite set:
35 \f[
36  u_{n} = \sin (\sqrt{\lambda}_{n} x_{1}), \ \ \ \ \ \ \
37  \lambda_{n} = n^{2}\pi^{2}.
38 \f]

40 We provide a detailed discussion of the driver code
41 <A HREF="../../../../demo_drivers/eigenproblems/harmonic/harmonic.cc">
42 harmonic.cc</A> which solves the problem for the first four eigenvalues.
43 
44 In \c oomph-lib all eigenproblems are solved by using interfaces to third-party
45 libraries. The default is LAPACK's (direct) QZ solver which is included with
46 the library. An interface is also provided to the serial version of
47 ARPACK, an iterative solver, but we are not allowed to
48 redistribute the source code, so it
49 must be downloaded separately. That said, much of
50 the functionality of ARPACK has
51 been superseded by the ANASAZI solver that is part of The Trilinos
52 Project. Thus, if you want to solve small problems, use the default QZ
53 solver; for larger problems, we recommend using ANASAZI.

58 \section theory Overview of the theory
59 
60  A discrete (linear) eigenproblem takes the form
61 \f[ J_{ij} V_{j} = \lambda M_{ij} V_{j}, \ \ \ \ \ \ \ \ (3) \f]
62  where \f$ V_{j} \f$ is the vector of discrete unknowns, \f$ \lambda \f$ is
63  the eigenvalue, and \f$ J_{ij} \f$ and \f$ M_{ij} \f$ are two matrices:
64  \f$ J_{ij} \f$ (often denoted \f$ K_{ij} \f$ in the
65  literature) is known as the stiffness matrix and \f$ M_{ij}
66  \f$ is termed the mass matrix. The third-party libraries mentioned
67  above all solve systems of the form (3), and within \c
68  oomph-lib we must simply provide the required mass and stiffness
69  matrices.
70 
71 \subsection linear_stab Linear Stability Theory
72 
73 In many cases, eigenproblems arise from
74 linear stability analyses of steady solutions to our set of nonlinear
75 residuals. For example, if we have a time-dependent system of equations
76 \f[ \frac{\partial u}{\partial t} = \mathcal{F}(u), \f]
77 then the residual formulation is
78 \f[ \mathcal{R}(u) = \mathcal{F}(u) - \frac{\partial u}{\partial t} =
79 0; \ \ \ \ \ \ \ (4) \f]
80 and steady solutions, \f$ u_{s} \f$ are found by setting \f$ \partial
81 u / \partial t =
82 0 \f$. Note that in a nonlinear system there may be many possible
83 steady solutions for the same set of parameter values.
84 
85 A linear stability analysis proceeds by assuming that the steady
86 solution is perturbed by a small amount, so that
87 \f$ u = u_{s} + \epsilon \hat{u} \f$, where \f$ |\epsilon| \ll
88 1\f$. If we substitute this expression into the governing equation
89 (4) and
90 neglect terms that involve \f$ \epsilon^{2} \f$ (on the assumption that they
91 are really small) then we obtain the equation:
92 \f[ \mathcal{F}(u_{s}) + \epsilon \left[\frac{\partial
93 \mathcal{F}(u_{s})}{\partial u} \hat{u} - \frac{\partial
94 \hat{u}}{\partial t} \right] \approx 0. \f]
95 Note that we have used a Taylor expansion of \f$ \mathcal{F} \f$, so
96 we are assuming that such an operation "makes sense". The solution \f$
97 u_{s} \f$ is a steady state, so by definition \f$\mathcal{F}(u_{s}) =
98 0\f$ and then we are left with the \b linear equation
99  \f[ \frac{\partial \mathcal{F} (u_{s})}{\partial u} \hat{u} =
100  \mathcal{J}(u_{s}) \hat{u} = \frac{\partial
101  \hat{u}}{\partial t}. \ \ \ \ \ \ \ \ \ (5) \f]
102  The Jacobian \f$ \mathcal{J} \f$ is exactly the same as that required by the
103  Newton method when solving the steady set of equations, \f$
104  \mathcal{F}(u) = 0.\f$
105 
106 The general solution of equation (5) takes the form \f$
107 \hat{u} = \mbox{e}^{\lambda t} v\f$ because the only function that is
108 proportional to its derivative is the exponential. The necessary
109 functional form of the solution means that equation (5)
110 becomes
111 \f[ \mathcal{J}(u_{s}) v = \lambda v, \f]
112 which is an eigenproblem in which the mass "matrix" is the identity.
113 Thus, in order to assess the stability of a solution to our nonlinear
114 system we must solve an eigenproblem using the Jacobian matrix, which
115 is why we used the notation \f$ J_{ij} \f$ rather than \f$ K_{ij} \f$
116 in equation (3). The solution is said to be linearly stable if the real
117 part of all eigenvalues is negative because then the perturbation
118 decays as \f$ t \to \infty\f$, but it is linearly unstable if even a single
119 eigenvalue has positive real part. Note that in even more general equations
120 the mass "matrix" is not necessarily the identity.

123 \subsection weak Weak formulation of the problem
124 
125  If we wish to solve the problem (1) using finite elements,
126 we must first derive the weak form, which follows from the standard
127 procedure of multiplication by a test function and integration by
128 parts
129 \f[ \int_{D} \frac{\partial u}{\partial x_{1}} \frac{\partial
130 \phi^{(test)}}{\partial x_{1}}\,\mbox{d} x_{1} =
131  \lambda \int_{D} u\, \phi^{(test)}\,\mbox{d} x_{1}. \f]
132 Note that the boundary terms are neglected as a consequence of the
133 Dirichlet boundary conditions.
134 
135 If we expand the \f$ u(x_{1}) = V_{j} \psi_{j}(x_{1}) \f$ in terms of
136 known basis functions \f$ \psi_{j} \f$ and use
137 the same basis functions as our test functions (Galerkin method), then the
138 weak form becomes
139 \f[ \int_{D} \frac{\partial \psi_{j}}{\partial x_{1}} \frac{\partial
140 \psi_{i}}{\partial x_{1}}\,\mbox{d} x_{1} V_{j} = \lambda \int_{D}
141 \psi_{i} \psi_{j}\,\mbox{d} x_{1} V_{j}. \f]
142 On comparison with equation (3), we identify the Jacobian and
143 mass matrices
144 \f[ J_{ij} = \int_{D} \frac{\partial \psi_{j}}{\partial x_{1}} \frac{\partial
145 \psi_{i}}{\partial x_{1}}\,\mbox{d} x_{1}, \ \ \ \ \
146  M_{ij} = \int_{D}
147 \psi_{i} \psi_{j}\,\mbox{d} x_{1}. \ \ \ \ \ (6) \f]

152 \section impl Implementation
153 
154  In \c oomph-lib, eigenproblems are formulated on an
155 element-by-element basis in a similar way to standard
156 problems. Eigenproblems make use of the function
157 \code
158  GeneralisedElement::get_jacobian_and_mass_matrix(
159  Vector<double> &residuals,
160  DenseMatrix<double> &jacobian,
161  DenseMatrix<double> &mass_matrix);
162 \endcode
163  where the Jacobian is the matrix \f$J_{ij}\f$ and the mass matrix is
164  \f$M_{ij}\f$ in equation (3). The residuals need not be
165  returned when solving an eigenproblem, but the interface is chosen so
166  that linear stability of solutions to the nonlinear system can easily be
167  calculated without reformulating the problem.
168 
169  Once again, to avoid reinitialisation in multi-physics problems, the helper
170  function
171 \code
172  GeneralisedElement::fill_in_contribution_to_jacobian_and_mass_matrix(
173  Vector<double> &residuals,
174  DenseMatrix<double> &jacobian,
175  DenseMatrix<double> &mass_matrix);
176 \endcode
177  is used and this will be overloaded in our custom elements below.

223 \section main The driver code
224 
225 In order to solve the 1D eigenproblem using \c oomph-lib, we represent the
226 mathematical problem defined by equations (1)
227 and (2) in a specific \c Problem object, \c
228 HarmonicProblem, which is templated by the element type and the
229 eigensolver. We use \c QHarmonicElement<3>, a quadratic element and our
230 three different eigensolvers, including timing statements to compare
231 the three approaches. Note that if you do not have Trilinos or ARPACK
232 installed then only the LAPACK_QZ solver will do anything.
233 
234 The problem class takes a single argument corresponding to the number
235 of elements used to discretise the domain and contains a member
236 function \c solve that takes an integer used for documentation purposes.
237

251 \section problem The problem class
252 The \c HarmonicProblem is derived from \c oomph-lib's generic
253 \c Problem class and the specific element
254 type and eigensolver are specified as template parameters to make it easy for
255 the "user" to change either of these from the driver code.
256 
257 \dontinclude harmonic.cc
258 \skipline start_of_problem_class
259 \until public Problem
260 
261 The problem class has four member functions:
262 - the constructor \c HarmonicProblem(...)
263 - the destructor \c ~HarmonicProblem()
264 - the function \c solve(...)
265 - the function \c doc_solution(...)
266 .
267 
268  The destructor merely cleans up the memory by deleting the objects that are
269 allocated in the constructor, so we shall not discuss it further
270 here. The \c doc_solution function is also simple and writes the
271 eigenfunction to a file.

276 \section constructor The Problem constructor
277 In the \c Problem constructor, we start by creating the eigensolver
278 specified by the second template parameter.
279 We then discretise the domain using \c oomph-lib's \c 1DMesh object.
280 The arguments of this object's constructor are the number of elements
281 (whose type is specified by the template parameter),
282 and the domain length. Next, we pin the nodal values on the domain
283 boundaries, which corresponds to applying the Dirichlet boundary conditions.
284 Finally we call the generic \c

295 \section solve Solving the problem
296 
297  The \c solve(...) function is where all the action happens
298  and takes a single unsigned integer argument which is used as a
299  label to distinguish the output from different eigensolvers.
300 
301  The function \c Problem::solve_eigenproblem(...) plays an equivalent
302  role to \c Problem::newton_solve(...) in eigenproblems. Here,
303  additional storage must be allocated for the
304  eigenvalues (a vector of complex numbers)
305  and eigenvectors (a vector of double vectors). The vectors will be
306  resized internally depending on the number of eigenvalues
307  returned. The number is not always the same as the number of
308  eigenvalues requested because both parts of a
309  complex conjugate pair of eigenvalues are always returned.
310  In the \c solve(..) function we first
311  allocate the required storage, specify the desired number of
312  eigenvalues and then solve the eigenproblem:
313 
314  \skipline start_of_solve
315  \until solve_eigenproblem(
316 
317  The rest of the function post-processes the output from the
318  eigensolver. In order to ensure repeatability of the output for our
319  self-tests the eigenvalues are sorted on the size of their real
320  part. The eigenfunction associated with the second smallest
321  eigenvalue is normalised to have unit length and then output to a file.
322 
323  In order to output the eigenfunction the values must be transferred
324  to the nodal values so that it can be interpolated. This is performed
325  by the function \c Problem::assign_eigenvector_to_dofs(...). Note
326  that this function overwrites the existing nodal data, so a backup
327  must be taken if it is important. This can be done using the function
328  \c Problem::store_current_dof_values() and the stored values can be
329  recovered via \c Problem::restore_dof_values().
330 
331  Finally, the sorted eigenvalues are reported and also saved to a file.
