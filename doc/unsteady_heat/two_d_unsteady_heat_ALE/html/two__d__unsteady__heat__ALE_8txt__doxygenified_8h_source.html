3\mainpage Example problem: Spatially adaptive solution of the 2D unsteady heat equation with flux boundary conditions in a moving domain: ALE methods.
4
5In this example we consider the solution of the unsteady heat
6equation in a domain with moving boundaries. We demonstrate
7that the presence of moving boundaries only requires trivial changes
8to driver codes for corresponding fixed-mesh computations.

21\sum_{i=1}^2\frac{\partial^2 u}{\partial x_i^2}
22= \frac{\partial u}{\partial t} + f\left(x_1,x_2,t\right),
23 \ \ \ \ \ \ \ \ \ \ (1)
24\f]
25in the domain \f$ D \f$ , bounded by the coordinate axes
26and the time-dependent ellipse
27\f[
28\mathbf{r}_{ellipse}(\xi,t) =
29\left(
30\begin{array}{c}
31\big( a+\hat{a}\sin(2\pi t/\hat{T}) \big) \cos(\xi) \\
32\big( b+\hat{b}\sin(2\pi t/\hat{T}) \big) \sin(\xi)
33\end{array}
34\right),
35 \ \ \ \ \ \ \ \ \ \ (2)
36\f]
37subject to Neumann boundary conditions,
38\f[
39\left. \frac{\partial u}{\partial n}\right|_{\partial D_{Neumann}}=
40- \left. \frac{\partial u}{\partial x_2}\right|_{\partial D_{Neumann}}=
41g_0,
42\ \ \ \ \ \ \ \ \ \ (3)
43\f]
44along the horizontal domain boundary \f$ \partial D_{Neumann} =
45\{ (x_1,x_2) | x_1 \in [0,1], x_2=0 \} \f$ , and to Dirichlet
46boundary conditions,
47\f[
48\left. u\right|_{\partial D_{Dirichlet}}=h_0,
49\ \ \ \ \ \ \ \ \ \ (4)
50\f]
51elsewhere.

54The initial conditions are given by
55\f[
56u(x_1,x_2,t=0)=k_0(x_1,x_2),
57\ \ \ \ \ \ \ \ \ \ (5)
58\f]
59where the functions \f$ f, g_0, \ h_0 \f$ and \f$ k_0 \f$ are given.

65We choose the functions \f$ f, g_0, \ h_0 \f$ and \f$ k_0 \f$
66so that
67\f[
68u_0(x_1,x_2,t) = \tanh\bigg[ 1-\alpha\bigg( \tan\Phi
69\big(x_1-\beta\tanh[ \gamma\cos\left(2\pi
70t\right)]\big)- x_2\bigg)\bigg]
71 \ \ \ \ \ \ \ \ \ \ (6)
72\f]
73is the exact solution.
74
75The solution represents the "usual" tanh profile, whose steepness
76is controlled by the parameter \f$ \alpha \f$ so that for \f$ \alpha
77\gg 1 \f$ the solution approaches a step. The step is oriented
78at an angle \f$ \Phi \f$ against the \f$ x_1- \f$ axis and its position varies
79periodically. The parameter \f$ \beta \f$ controls the amplitude
80of the step's lateral displacement, while \f$ \gamma \f$ determines the
81rate at which its position changes. For \f$ \gamma \gg 1 \f$ ,
82the step remains stationary for most of the period and then
83translates rapidly parallel to the \f$ x_1- \f$ axis, making this a very
84challenging problem.

110\section ale Background: ALE methods and the evaluation of time-derivatives in moving domains
111
112\c oomph-lib's \c UnsteadyHeatEquations, the equation class that forms the
113basis for the isoparametric \c QUnsteadyHeatElements, is based
114on the Arbitrary Lagrangian Eulerian (ALE) formulation of the
115weak form of the governing PDE, (1). Within each element, the
116solution is represented by interpolation between the
117element's \f$ N_{node}^{(E)} \f$ nodal values \f$ U_j^{(E)}(t) \
118(i=1,..., N_{node}^{(E)})\f$ , i.e.
119\f[
120u = \sum_{j=1}^{N_{node}^{(E)}} U_j^{(E)}(t) \ \psi_{j}(s_1,s_2),
121\ \ \ \ \ \ \ \ \ \ (7)
122\f]
123where \f$ s_1 \f$ and \f$ s_2 \f$ are the element's two local coordinates.
124The mapping between the local and global (Eulerian) coordinates is based on
125the interpolation between the nodal coordinates,
126\f[
127x_i(s_1,s_2) = \sum_{j=1}^{N_{node}^{(E)}} X_{ij}^{(E)}(t) \
128\psi_{j}(s_1,s_2),
129\ \ \ \ (i=1,2) \ \ \ \ \ \ (8)
130\f]
131where \f$ X_{ij}^{(E)}(t) \f$ is the i-th global (Eulerian) coordinate of
132node j in the element. In moving-domain problems, where the nodal positions vary as
133function of time, the time-derivative of the nodal value,
134\f$ dU_j(t)/dt \f$ , represents the rate-of-change of \f$ u \f$
135at the moving node, rather than the time-derivative
136of \f$ u \f$ at a fixed Eulerian position,
137\f$ \partial u/\partial t, \f$ the quantity required in the
138PDE, (1).
139
140The rate of change of \f$ u \f$ at a moving node,
141\f$ dU_j(t)/dt \f$ , may also be expressed by the material derivative,
142\f[
143\left. \frac{D u}{Dt}\right|_{\mbox{\small node} \ j} =
144\frac{\partial u}{\partial t}
145+ \sum_{i=1}^2 v_{ij} \ \frac{\partial u}{\partial x_i}
146\f]
147where
148 \f[
149v_{ij} = \frac{dX_{ij}^{(E)}}{dt}
150\f]
151is the i-th velocity component of node j, often referred to
152as the "mesh velocity". The rate of change of \f$ u \f$ ,
153experienced at the (fixed) spatial position that coincides with
154the current position of node j is therefore given by
155\f[
156\frac{\partial u}{\partial t} =
157\frac{d U_j^{(E)}}{dt}
158- \sum_{i=1}^2
159\left(
160\frac{dX_{ij}^{(E)}}{dt}
161\ \sum_{k=1}^{N_{node}^{(E)}} U_k^{(E)} \
162\frac{\partial \psi_{k}}{\partial x_i}
163\right) .
164\f]
165This is the form in which the time-derivative in (1)
166is implemented in \c oomph-lib's unsteady heat elements. The mesh
167velocity is determined automatically, using the \c Node's positional
168\c TimeStepper and the history values of the nodal positions.
169[By default, the positional \c TimeStepper is the same as the \c TimeStepper
170used for the evaluation of the time-derivatives of the nodal values;
171a different one may be assigned with the access function
172\c Node::position_time_stepper_pt().] This is why it is important
173to initialise the "previous nodal positions" in computations
174on fixed meshes, as demonstrated in the
175<A HREF="../../two_d_unsteady_heat/html/index.html">
176exercise in our earlier
177example:</A> The previous nodal positions are
178initialised to zero when a \c Node is first created. Therefore, if the previous
179positions are not overwritten with the actual nodal positions,
180the positional \c TimeStepper would compute non-zero mesh velocities,
181even though the \c Nodes are stationary.
182
183 \c oomph-lib's unsteady Newton solvers automatically
184advance the time-history of the nodal positions when computing
185a new timestep. Performing computations on moving meshes
186is therefore extremely straightforward: The only task to be performed
187by the "user" is to update the nodal positions before a new
188timestep is taken. This is best done in the function
189\c Problem::actions_before_implicit_timestep().
190
191 The update of the nodal positions may be performed "manually",
192by assigning new nodal positions directly, using the function
193\c Node::x(...). However, in most cases, the deformation of the domain
194will be driven by the motion of its boundaries. We discussed
195in an <A HREF="../../../poisson/fish_poisson2/html/index.html">earlier
196example,</A> that in \c oomph-lib curvilinear, moving boundaries are typically
197represented by (time-dependent) \c GeomObjects which define the
198\c MacroElement boundaries of the \c Domain object associated with
199the \c Mesh. In this case, the update of the nodal positions may be
200performed by updating the parameters that control the shape of the
201\c GeomObject and calling the \c Mesh's node-update function
202\c Mesh::node_update(), as illustrated
203in the <A HREF="../../../poisson/fish_poisson2/html/index.html">earlier
204example.</A>
205
206It is also possible (and, in fact, much easier) to include the
207time-dependence of the domain boundaries into the definition of
208the \c GeomObject, by making its shape, described by
209\c GeomObject::position(...), a function of time.
210In that case, the update of the nodal positions in response to the
211boundary motion merely requires a call to \c Mesh::node_update() in
212\c Problem::actions_before_implicit_timestep(). This is the approach we take
213in the present problem.

231\section ellipse Representing the moving curvilinear domain boundary by a time-dependent GeomObject
232
233As discussed above, we will incorporate the time-dependence of the
234moving curvilinear boundary into the specification of the \c GeomObject
235that defines that boundary. For this purpose we represent the
236elliptical boundary by the \c GeomObject \c MyEllipse.
237Its constructor stores the geometric parameters (the
238mean values of the ellipse's half axes, \f$ a \f$ and \f$ b \f$ , the
239amplitude of their temporal variations \f$ \hat{a} \f$ and
240\f$ \hat{b} \f$ , and the period of the
241oscillation, \f$ \hat{T} \f$ , in its private member data. We also store a
242pointer to a \c Time object to give the \c GeomObject access to the
243"current" and "previous" values of the "continuous time".
244The destructor can remain empty.
245
246\dontinclude two_d_unsteady_heat_ALE.cc
247\skipline start_of_MyEllipse
248\until ~MyEllipse
249
250The "steady" version of the \c position(...) function must return
251the position vector to the point on the \c GeomObject, identified
252by its intrinsic coordinate \c xi, and evaluated at the current
253value of the continuous time, which we extract from \c Time::time():
254\skipline Current position
255\until end of position
256
257
258The "time-dependent" version of the \c position(...) function must return
259the position vector to the \c GeomObject, evaluated at the
260\c t - th previous timestep. The value of the continuous time at that timestep
261is available from from \c Time::time(t):
262\skipline Parametrised
263\until end of position
264
265We omit the code that defines the private member data.

331\section set_init Setting the initial condition
332
333The only other change to the code occurs in the assignment of
334the initial conditions. The \c Nodes' positional history values
335are given by the positions at which the \c Nodes would have been at
336previous timesteps. Similarly, the history values themselves
337must be computed by evaluating the exact solution at the position
338at which the \c Nodes would have been at those timesteps.
339
340This is achieved with a few minor changes to the previous version
341of this function. We loop over the previous timesteps, reconstruct
342the value of the continuous time at that timestep, and temporarily
343over-write the value of the continuous time stored in \c
344Time::time(). This ensures that the call to \c
345MyEllipse::position(...) during the node update
346operation returns the position vector to the domain boundary
347at that timestep. Following the update of the nodal positions (which moves them
348into the position they would have occupied at the previous timestep)
349we copy their positions and the value of the exact solution
350into the appropriate history values. Here is the relevant code
351fragment from the \c set_initial_condition() function:
352
353\skipline Loop over current & previous timesteps
354\until end of loop over previous timesteps

361\section comm Comments and Exercises
362While the spatial adaptivity resolves the rapid spatial variations
363in the solution, the time-integration with a fixed timestep
364introduces errors during the phases when the solution undergoes
365rapid temporal variations. The <A HREF="../figures/step_soln.avi">
366animations</A> of the exact and computed solutions show clearly
367that the computed solution lags behind the exact one during these
368phases. In the <A HREF="../../two_d_unsteady_heat_2adapt/html/index.html">
369next example</A> we will therefore demonstrate how to combine
370temporal and spatial adaptivity.
