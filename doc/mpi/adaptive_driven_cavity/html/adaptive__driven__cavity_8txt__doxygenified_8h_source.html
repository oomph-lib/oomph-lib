3 \mainpage Parallel solution of the adaptive driven cavity problem
4 
5 This document concerns the parallel solution of the
6 <a href="../../../navier_stokes/adaptive_driven_cavity/html/index.html">adaptive
7 driven cavity problem</a>. It is the first in a
8 <a href="../../../example_code_list/html/index.html#distributed">
9 series
10 of tutorials</a>
11 that discuss how to modify serial driver codes to distribute the
12 \c Problem object across multiple processors.
13 
14 Most of the driver code is identical to its serial counterpart and
15 we only discuss the changes required to distribute the problem. Please
16 refer to
17 <a href="../../../navier_stokes/adaptive_driven_cavity/html/index.html">
18 another tutorial</a> for a more detailed discussion of the
19 problem and its (serial) implementation.

23 \section main_body The main function
24 
25 As described in the <a href="../../general_mpi/html/index.html">
26 parallel processing document </a> all parallel driver codes must
27 initialise and shutdown oomph-lib's MPI routines
28 by calling \c MPI_Helpers::init(...) and \c MPI_Helpers::finalize().
29 The functions \c MPI_Helpers::init(...)
30 and \c MPI_Helpers::finalize() call their MPI counterparts,
31 \c MPI_Init(...) and \c MPI_Finalize(), which must \b not be called
32 again.
33 
34 In our demo driver codes, we surround all parallel
35 sections of code with \c \#ifdefs to ensure that the code remains
36 functional if compiled without parallel support (in which case
37 the macro \c OOMPH_HAS_MPI is undefined), for example.
38 
39 \dontinclude adaptive_driven_cavity.cc
40 \skipline start_of_main
41 \until #endif
42 
43 In order to distribute the problem over multiple processors a single
44 call to the function \c Problem::distribute() is all that is
45 required. Thus, a minimally-changed serial driver code would be
46 \code
47  // Set output directory
48  DocInfo doc_info;
49  doc_info.set_directory("RESLT");
50 
51  // Set max. number of black-box adaptation
52  unsigned max_adapt=3;
53 
54  // Solve problem with Taylor Hood elements
55  //---------------------------------------
56  {
57  //Build problem
58  RefineableDrivenCavityProblem<RefineableQTaylorHoodElement<2> > problem;
59 
60  //Distribute the problem (only change from serial version)
61  problem.distribute();
62 
63  //Solve the problem with automatic adaptation
64  problem.newton_solve(max_adapt);
65 
66  //Output solution
67  problem.doc_solution(doc_info);
68  }
69 \endcode
70 
71  Finally, we must call \c MPI_finalize() before the end of the main
72  function
73 \dontinclude adaptive_driven_cavity.cc
74 \skipline Finalise MPI
75 \until }
76 
77 The actual driver code is slightly more complicated that the
78 version shown above because it also acts as a self-test. The distribution
79 of individual elements over the processors is determined by
80 \c METIS and we have found that \c METIS occasionally gives slightly
81 different results on different machines. To ensure reproducible
82 results when acting as a self-test, the code (like all our parallel test codes)
83 reads a predetermined element distribution from the disk; see
84 <a href="#distribution"> below </a> for more details.

88 \section problem_class Changes to the problem class
89 
90 A particular feature of the
91 <a href="../../../navier_stokes/adaptive_driven_cavity/html/index.html">
92 driven cavity problem</a> is that the flow is completely enclosed
93 and that a single pressure degree of freedom must be prescribed.
94 In the serial driver code, we arbitrarily pinned the first pressure
95 degree of freedom in the "first" element in the mesh.
96 Once the problem is distributed this element will only be available to
97 particular processors and the pressure degree of freedom must be
98 pinned on each one.
99 Consequently we re-write the \c actions_after_adapt()
100 function as follows:
101 
102 \dontinclude adaptive_driven_cavity.cc
103 \skipline After adaptation: Unpin pressure
104 \until end_of_actions_after_adapt
105 
106 This change ensures that every processor that holds the element
107 containing the node at position (0,0) (i.e. the first element)
108 fixes the pressure for that element. The floating-point comparison
109 does not cause any problems in this case because the \c Node's
110 position is explicitly set to "exactly" (0.0,0.0) in the Mesh
111 constructor and never changes.
112 
113 It is not necessary to change the corresponding statements
114 in the problem constructor because the problem distribution occurs after
115 the problem has been constructed. In fact, the problem constructor
116 is unchanged from the serial version.

180 \subsection from_disk Option II: Using a pre-determined distribution
181 
182 If command line arguments are specified (typically when the code is
183 run in validation mode) we read the distribution from disk,
184 using a file that was written using the procedure shown above.
185 (This is useful because in our experience \c METIS may produce
186 slightly different distribution on different machines. This would
187 cause the self-tests to fail even though the computed results
188 would be correct).

191 in which the problem distribution will be documented and by reading in
