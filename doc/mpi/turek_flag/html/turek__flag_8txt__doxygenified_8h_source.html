8This document provides an overview of
9- how to change the serial driver code for
10 <a href="../../../interaction/turek_flag/html/index.html">Turek &
11 Hron's FSI benchmark problem</a> so that both the fluid and solid meshes
12 can be adapted,
13.
14- how to distribute the problem across multiple processors,
15.
16and
17- how to enable load balancing of the problem once it is distributed.
18.

20<a href="../../../example_code_list/html/index.html#distributed">
21series of tutorials</a>
22that discuss how to modify existing serial driver codes so that the
23\c Problem object can be distributed across multiple processors.

30\section double_adapt Enabling spatial adaptivity for the fluid and solid meshes
31In the original (serial) driver code for
32<a href="../../../interaction/turek_flag/html/index.html">Turek &
33Hron's FSI benchmark problem</a> we only adapted the fluid mesh.
34Before discussing how to modify the code to refine the
35fluid and solid meshes simultaneously, we provide a
36brief reminder of the procedure used to discretise fluid-structure
37interaction problems that involve fluid and solid domains of equal spatial
38dimension (e.g. a 2D fluid domain interacting with a 2D solid domain)
39when using algebraic node update methods to adjust the position of
40the nodes in the fluid mesh.
41We refer to <a href="../../fsi_channel_with_leaflet/html/index.html">
42another tutorial</a> for a discussion of FSI
43problems involving the interaction of fluids with (lower-dimensional)
44shell and beam structures.
45
46\subsection orig General methodology
47
48The figure below shows a sketch of a simple(r)
49fluid-structure interaction problem involving fluid and solid domains
50that meet along a single mesh boundary. We assume that the fluid mesh
51uses an algebraic node
52update function to adjust the position of its nodes
53in response to changes in the domain
54boundary, represented by the \c GeomObject shown in
55magenta. (You may wish to consult
56<a href="../../../interaction/fsi_collapsible_channel_algebraic/html/index.html">
57another tutorial</a> for a reminder of how \c oomph-lib's algebraic
58node update methods work).
59
60
61\image html fsi_with_adaptive_wall_basic.gif "Basic setup for FSI problems involving algebraic node updates for the fluid mesh. "
62\image latex fsi_with_adaptive_wall_basic.eps "Basic setup for FSI problems involving algebraic node updates for the fluid mesh. " width=0.75\textwidth
63
64In an FSI problem, the fluid mesh's free
65boundary is a boundary of the solid mesh, <em> i.e. </em> the
66boundary along which the fluid exerts a traction onto the solid.
67Within \c oomph-lib, the fluid traction is applied to the solid domain
68by attaching \c FSISolidTractionElements to the faces of the
69"bulk" solid elements adjacent to the FSI boundary.
70(In the above sketch the \c FSISolidTractionElements are shown
71in blue.)
72The deformation of the fluid and solid meshes
73is coupled by using the \c MeshAsGeomObject formed from
74the \c FSISolidTractionElements as the \c GeomObject that defines
75the moving boundary of the fluid mesh. (In sketch above, this is
76indicated by the magenta arrow.)
77
78
79\subsection modification1 Modifications to allow adaptivity of the fluid and solid meshes
80
81
82If the solid mesh is not adapted, the adaptation for the fluid
83mesh is straightforward and proceeds fully automatically
84as described
85<a href="../../../interaction/fsi_collapsible_channel_algebraic/html/index.html">elsewhere</a>.
86In particular, the node update data for
87newly-created fluid nodes is created automatically by a call to
88the \c AlgebraicMesh::update_node_update(...) function during the adaptation.
89This function obtains the required
90information about the boundary by using the \c MeshAsGeomObject
91built from the \c FSISolidTractionElements.
92
93If the solid mesh is also adapted, then the existing
94\c FSISolidTractionElements must (at some point) be deleted and new ones
95must be attached to the adapted "bulk" solid mesh. In all other problems, this
96is done by
97deleting the \c FSISolidTractionElements in \c
98Problem::actions_before_adapt() and attaching new ones in
99\c Problem::actions_after_adapt(); see, e.g. the
100<a href="../../../poisson/two_d_poisson_flux_bc/html/index.html">tutorial
101on the solution of a Poisson problem with flux boundary
102conditions.</a> However, in the present problem
103this is not possible because, once the \c FSISolidTractionElements
104have been deleted, the \c MeshAsGeomObject can no longer be used to
105represent the shape and position of the FSI boundary, which would cause
106the adaptation of the fluid mesh to fail.
107
108To avoid this problem, we adopt the following strategy:
109-# When adding the various meshes to the \c Problem's
110 collection of sub-meshes, we add the fluid mesh
111 \b before the solid mesh. (This happens to be what was done
112 already in the original driver code.) Usually, the order in which
113 sub-meshes are added to the \c Problem is irrelevant. Here the order
114 \b does matter because we will exploit the fact that
115 the sub-meshes are adapted individually, in the
116 order in which they were added to the \c Problem.
117 \n\n
118-# The \c FSISolidTractionElements are not deleted in
119 \c Problem::actions_before_adapt() and
120 remain attached to the "bulk" solid elements throughout
121 the "bulk" mesh adaptation procedure.
122 When the fluid mesh is adapted, the appropriate \c MeshAsGeomObject
123 is, therefore, still fully-functional
124 (and refers to the boundary as represented by the solid domain
125 \b before the "bulk" solid mesh is adapted).
126 \n\n
127 Here is a sketch of problem after adaptation of the fluid mesh:
128 \n\n
129\image html fsi_with_adaptive_wall_adapted_fluid.gif "Sketch of the problem following the adaptation of the fluid mesh. The solid mesh has not yet been refined. "
130\image latex fsi_with_adaptive_wall_adapted_fluid.eps "Sketch of the problem following the adaptation of the fluid mesh. The solid mesh has not yet been refined. " width=0.75\textwidth
131 \n\n
132-# The subsequent adaptation of the "bulk" solid mesh is likely
133 to turn some of the \c FSISolidTractionElements into "dangling"
134 elements. (This occurs whenever a \c FSISolidTractionElements
135 is attached to a "bulk" solid elements that disappears during the
136 adaptation, <em> e.g. </em> by being refined.)
137 \n\n
138 Here is a plot of the problem following the adaptation of the
139 solid mesh :
140 \n\n
141\image html fsi_with_adaptive_wall_adapted_solid.gif "Sketch of the problem following the adaptation of the solid mesh -- the `dangling' FSISolidTractionElements are represented by dotted lines. "
142\image latex fsi_with_adaptive_wall_adapted_solid.eps "Sketch of the problem following the adaptation of the solid mesh -- the `dangling' FSISolidTractionElements are represented by dotted lines. " width=0.75\textwidth
143 \n\n
144-# Hence, in \c Problem::actions_after_adapt() we
145 delete the existing \c FSISolidTractionElements and immediately
146 (re-)attach new ones. Now, the \c MeshAsGeomObject that represents the
147 FSI boundary is broken because it still refers to the just deleted
148 \c FSISolidTractionElements.
149 \n\n
150\image html fsi_with_adaptive_wall_new_face_elements.gif "Sketch of the problem following the creation of new FSISolidTractionElements. The fact that the MeshAsGeomObject is broken is indicated by the dashed lines. "
151\image latex fsi_with_adaptive_wall_new_face_elements.eps "Sketch of the problem following the creation of new FSISolidTractionElements. The fact that the MeshAsGeomObject is broken is indicated by the dashed lines. " width=0.75\textwidth
152 \n\n
153-# Thus, we rebuild the \c MeshAsGeomObject from the
154 newly-created \c FSISolidTractionElements, and update the
155 fluid mesh's pointer to this new \c GeomObject that describes the
156 boundary shape.
157 \n\n
158\image html fsi_with_adaptive_wall_done.gif "Sketch of the problem with re-built MeshAsGeomObject. "
159\image latex fsi_with_adaptive_wall_done.eps "Sketch of the problem with re-built MeshAsGeomObject. " width=0.75\textwidth
160 \n\n
161-# Finally, we execute the \c AlgebraicMesh::update_node_update(...)
162 function for all nodes in the fluid mesh to ensure that their
163 node update data refers to the new \c FSISolidTractionElements.
164 \n\n
165-# The remaining tasks (such as the renewed setup of the fluid load on the
166 \c FSISolidTractionElements via a call to
167 \c FSI_functions::setup_fluid_load_info_for_solid_elements(...),
168 etc.) remain the same as in the previous version of the code.
169.

174\section distr Distributing the Problem
175In the present example, there are two "bulk"
176meshes corresponding to the fluid and solid domains
177 and three "surface" meshes of traction
178elements. The traction elements are \c FaceElements
179created from the "bulk" fluid elements and should be deleted before the
180problem is distributed, see the tutorial on applying
181<a href="../../two_d_poisson_flux_bc_adapt/html/index.html"> flux
182boundary conditions in a Poisson problem </a> for more details.
183In the <a href="../../fsi_channel_with_leaflet/html/index.html">
184previous example</a>
185involving the interaction of a 2D fluid domain with a 1D beam
186structure there were only two meshes: a "bulk" fluid mesh
187and a "surface" solid mesh. In that problem
188\b all elements in the
1891D mesh of \c FSIHermiteBeamElements were retained on all processors
190as halo elements by using the function
191\c Mesh::keep_all_elements_as_halos(). The
192same methodology could be used here, but it would be extremely wasteful
193to retain all the solid elements in the "bulk" solid mesh because only
194the elements next to the FSI boundary are required.
195Instead, we use a more fine-grained
196method of retaining elements via the function \c
197GeneralisedElement::must_be_kept_as_halo().

203\section impl Implementation
204Most of the driver code is identical to the original serial version
205discussed in <a href="../../fsi_channel_with_leaflet/html/index.html">
206another tutorial</a>. We therefore only discuss those parts of the
207code that have to be changed to allow (i) the simultaneous adaptation of
208the fluid and solid meshes, and (ii) the problem distribution.
209

212\subsection main_body The main function
213
214As usual in a parallel driver code, the only addition to the \c main()
215function is the inclusion of calls to \c MPI_Helpers::init(), \c
216MPI_Helpers::finalize(), and the \c Problem::distribute()
217functions.

221\subsection problem_class The problem class
222
223The only additions to the serial version of the problem class are the
224functions \c actions_before_distribute() and \c
225actions_after_distribute(), and the helper function \c

247\subsection actions_before_distribute Actions before distribute
248
249 As discussed above, we must ensure that the "bulk" solid
250elements adjacent to the FSI boundary are retained on all processors.
251Hence, the \c actions_before_distribute()
252function starts with a loop over the
253\c FSISolidTractionElements within which we use the
254function \c GeneralisedElement::must_be_kept_as_halo()
255to indicate that the associated bulk elements must be retained.
256
257\dontinclude turek_flag.cc
258\skipline start_of_actions_before_distribute
259\until end of loop over meshes of fsi traction elements
260
261Next, we flush all the meshes from the problem's
262collection of sub-meshes and add only the "bulk" fluid and solid
263meshes (in that order!). The \c FaceElements do not need to be
264distributed, because they will be re-created in \c actions_after_distribute().
265
266\until end of actions before distribute
267

272\subsection actions_after_distribute Actions after distribute
273
274Following the problem distribution, we delete the old
275\c FSISolidTractionElements and then (re-)attach new ones, which will
276be created as halo elements where necessary.
277
278\dontinclude turek_flag.cc
279\skipline start_of_actions_after_distribute
280\until create_fsi_traction_elements()
281
282We complete the build of the \c
283FSISolidTractionElements by passing the FSI parameter and the
284boundary number in the bulk mesh. The relevant code is identical to
285the serial version and we omit its listing here.
286
287Next, we create new \c MeshAsGeomObjects from the
288newly-created \c FSISolidTractionElements and pass them to the
289(algebraic) fluid mesh:
290
291\skipline Turn the three meshes of FSI traction elements
292\until set_tip_flag_pt
293
294The \c MeshAsGeomObjects have changed, so we
295must call the \c update_node_update() function again for each
296node in the fluid mesh:
297
298\until }
299
300Now we add the FSI traction meshes back to the problem and rebuild the
301global mesh.
302
303\until rebuild_global_mesh
304
305Finally, we re-set the fluid load on the solid elements by
306calling \c FSI_functions::setup_fluid_load_info_for_solid_elements(...)
307before re-assigning the auxiliary node update function that imposes the
308no-slip condition for all fluid nodes on the FSI boundaries.
309[<a href="../../fsi_channel_with_leaflet/html/index.html#action_after_adapt">
310Recall</a>
311that the (re-)assignment of the auxiliary node-update function
312must be performed \b after the call to \c
313FSI_functions::setup_fluid_load_info_for_solid_elements(...).]
314
315\until end of (re-)assignment of the auxiliary node update fct
316
317The remainder of the function identifies which processors
318contain the fluid control node whose velocities
319we document in the trace file.

324\subsection actions_after_adapt Actions after adapt
325
326The \c actions_after_adapt() function is very similar to
327\c actions_after_distribute() function, so we omit is listing here.
328 The only significant differences are that (i) the redundant fluid and
329 solid pressures are (re)-pinned; (ii) the identification of the
330fluid control node does not need to
331 be setup; and (iii) the traction meshes were never removed from the
332 problem, so do not need to be added back in.
333

335
336\subsection doc_solution The doc_solution() function
337
338As with the other parallel driver codes, the main modification to the
339post-processing function is the
340addition of the processor number to all output files.
341Furthermore, we only write the trace file on the processors that
342contain the fluid control node. In the interest of brevity
343we omit the listing of the modified function.

366\section load_balance Load balancing
367
368When employing load balancing in this problem,
369we modify the time-stepping loop to perform the procedure
370after each timestep:
371
372\dontinclude turek_flag_load_balance.cc
373\skipline Start of timestepping loop
374\until end of timestepping loop
375
376\subsection build_mesh The build_mesh() function
377
378The function \c Problem::build_mesh() must be supplied by the user if
379they wish to use the load balancing capability. Thus, in this driver
380code, we move all the required code to build the entire global mesh
381into this function, and call it from within the problem constructor:
382
383\dontinclude turek_flag_load_balance.cc
384\skipline start_of_constructor
385\until end_of_constructor
386
387The \c build_mesh() function itself contains all the relevant code
388from within the previous parallel driver code's problem constructor.
389
390\subsection actions_functions Actions before and after load balancing
391
392In this example, all that is required for the \c
393actions_after_load_balance() function is the
394addition of the unpin-repin procedure from the \c actions_after_adapt()
395function to the appropriate part of the \c actions_after_distribute()
396function, since all the other functionality is already identical. The
397\c actions_before_load_balance() function is identical to the \c
398actions_before_distribute() function.
