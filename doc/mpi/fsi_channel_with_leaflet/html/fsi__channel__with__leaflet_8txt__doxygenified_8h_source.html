3 \mainpage Parallel solution of the FSI channel with leaflet problem: Distributing problems with algebraic node updates
4 
5 This document provides an overview of how to distribute
6 fluid-structure interaction problems in which algebraic
7 node update methods are used to deform the fluid mesh in response
8 to changes in the shape of the domain boundary. It is part of a
9 <a href="../../../example_code_list/html/index.html#distributed">
10 series of tutorials</a>
11 that discuss how to modify existing serial driver codes so that the
12 \c Problem object can be distributed across multiple processors.
13 
14 As discussed in the
15 <a href="../../general_mpi/html/index.html#alg_node_update">general
16 MPI tutorial</a>, the parallel implementation of algebraic node update methods
17 for specific meshes is greatly facilitated if the \c GeomObject
18 that describes the motion of the moving domain boundary is available
19 on all processors. In FSI problems, the moving boundary
20 is typically represented by a \c MeshAsGeomObject -- a compound
21 \c GeomObject formed from the lower-dimensional mesh of
22 \c SolidElements that define the moving boundary of the fluid domain.
23 To ensure that the \c MeshAsGeomObject remains available on all
24 processors when the underlying mesh is distributed, we use the
25 function \c Mesh::keep_all_elements_as_halos() to indicate that
26 all elements should be retained on all processors.

50 \section fsi_channel_with_leaflet Revisiting the FSI channel with leaflet problem
51 
52 We demonstrate the methodology for the problem of
53 <a href="../../../interaction/fsi_channel_with_leaflet/html/index.html">
54 flow past an elastic leaflet in a 2D channel</a>.
55 Most of driver code is identical to its serial counterpart
56 and we only discuss the changes required to distribute the problem.
57 Please refer to
58 <a href="../../../interaction/fsi_channel_with_leaflet/html/index.html">
59 another tutorial</a> for a more detailed discussion of the
60 problem and its (serial) implementation.

100 \subsection action_after_adapt Actions after adaptation
101 
102 The \c actions_after_adapt() function requires only a minor change
103 from the serial version --- a simple re-ordering of the sequence
104 in which the various steps are performed. In serial it does not matter
105 in which order the (re-)assignment of the auxiliary node update
106 functions and (re-)setup of the fluid-structure interaction are
107 performed. In the parallel version, however,
108 the assignment of the auxiliary node update functions must take place
109 \b after the (re-)setup of the fluid-structure interaction. This is
110 because the function
111 \c FSI_functions::setup_fluid_load_info_for_solid_elements(...)
112 creates halo copies of the fluid elements (and their nodes!)
113 if the required fluid element
114 is not present on the current processor.
115 Any newly-created halo fluid nodes on the FSI boundary
116 are accessible via the
117 usual boundary lookup schemes and
118 must be told about the auxiliary node update function which applies
119 the no-slip condition.

126 \subsection action_after_distribution Actions after distribution
127 
128  The actions required after distribution are extremely similar to those
129  required after adaptation because distribution deletes certain elements
130 (or replaces them by halo copies). The only difference is that the
131  redundant pressure degrees of freedom do not have to be re-pinned.

138 \subsection doc_solution The doc_solution() function
139 
140 As usual, we add the processor number to the end of the filename for
141 each output file to make sure that the different processors
142 don't over-write each other's output.
143
