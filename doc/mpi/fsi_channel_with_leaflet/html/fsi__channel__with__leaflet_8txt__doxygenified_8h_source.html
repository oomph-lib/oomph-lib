3\mainpage Parallel solution of the FSI channel with leaflet problem: Distributing problems with algebraic node updates
4
5This document provides an overview of how to distribute
6fluid-structure interaction problems in which algebraic
7node update methods are used to deform the fluid mesh in response
8to changes in the shape of the domain boundary. It is part of a
9<a href="../../../example_code_list/html/index.html#distributed">
10series of tutorials</a>
11that discuss how to modify existing serial driver codes so that the
12\c Problem object can be distributed across multiple processors.
13
14As discussed in the
15<a href="../../general_mpi/html/index.html#alg_node_update">general
16MPI tutorial</a>, the parallel implementation of algebraic node update methods
17for specific meshes is greatly facilitated if the \c GeomObject
18that describes the motion of the moving domain boundary is available
19on all processors. In FSI problems, the moving boundary
20is typically represented by a \c MeshAsGeomObject -- a compound
21\c GeomObject formed from the lower-dimensional mesh of
22\c SolidElements that define the moving boundary of the fluid domain.
23To ensure that the \c MeshAsGeomObject remains available on all
24processors when the underlying mesh is distributed, we use the
25function \c Mesh::keep_all_elements_as_halos() to indicate that
26all elements should be retained on all processors.

50\section fsi_channel_with_leaflet Revisiting the FSI channel with leaflet problem
51
52We demonstrate the methodology for the problem of
53<a href="../../../interaction/fsi_channel_with_leaflet/html/index.html">
54flow past an elastic leaflet in a 2D channel</a>.
55Most of driver code is identical to its serial counterpart
56and we only discuss the changes required to distribute the problem.
57Please refer to
58<a href="../../../interaction/fsi_channel_with_leaflet/html/index.html">
59another tutorial</a> for a more detailed discussion of the
60problem and its (serial) implementation.

100\subsection action_after_adapt Actions after adaptation
101
102The \c actions_after_adapt() function requires only a minor change
103from the serial version --- a simple re-ordering of the sequence
104in which the various steps are performed. In serial it does not matter
105in which order the (re-)assignment of the auxiliary node update
106functions and (re-)setup of the fluid-structure interaction are
107performed. In the parallel version, however,
108the assignment of the auxiliary node update functions must take place
109\b after the (re-)setup of the fluid-structure interaction. This is
110because the function
111\c FSI_functions::setup_fluid_load_info_for_solid_elements(...)
112creates halo copies of the fluid elements (and their nodes!)
113if the required fluid element
114is not present on the current processor.
115Any newly-created halo fluid nodes on the FSI boundary
116are accessible via the
117usual boundary lookup schemes and
118must be told about the auxiliary node update function which applies
119the no-slip condition.
120
121\skipline start_of_actions_after_adapt
122\until end_of_actions_after_adapt

126\subsection action_after_distribution Actions after distribution
127
128 The actions required after distribution are extremely similar to those
129 required after adaptation because distribution deletes certain elements
130(or replaces them by halo copies). The only difference is that the
131 redundant pressure degrees of freedom do not have to be re-pinned.
132

134\until end_of_actions_after_distribute

138\subsection doc_solution The doc_solution() function
139
140As usual, we add the processor number to the end of the filename for
141each output file to make sure that the different processors
142don't over-write each other's output.
143
144The trace file documents the time trace of the imposed influx
145and the displacement of the node at the tip of the leaflet. It could be
146written by any processor since all solid elements are retained
147everywhere. We only write to the trace file from processor 0.
