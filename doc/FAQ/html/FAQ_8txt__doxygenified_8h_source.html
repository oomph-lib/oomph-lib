50 \subsection comp_opt How do I specify different levels of optimisation and/or non-standard compiler options?
51 
52 
53 -# <B>Compiler flags/options for the compilation of the library</B> \n\n
54  By default, \c oomph-lib's installation is performed with the
55  <A HREF="http://gcc.gnu.org/">gcc compiler suite</A>, using full
56  optimisation and warning (\c -O3 \c -Wall ).
57  \n\n
58  If you install the library with the \c autogen.sh script you are
59  given the option to overwrite the default compilation flags,
60  either by specifying alternative flags on the command line,
61  or by recycling any of the previously used combinations of flags, stored
62  in the directory <A HREF="../../../config/configure_options/">
63  config/configure_options</A>. The files in this directory
64  also contain more details on available flags and options.
65  \n\n
66  If you prefer the standard linux \c configure/make/make \c install
67  installation procedure, you should already know how to specify
68  flags at the configure stage...
69  \n\n
70 -# <B>Compiler flags/options for the compilation of individual
71  (driver) codes</B> \n\n
72  Within the \c autotools framework,
73  additional flags for the compilation of individual (driver) codes
74  may be specified by setting the appropriate automake variables
75  in the \c Makefile.\c am. For instance, to use the compiler
76  flag \c -DUSE_TAYLOR_HOOD during the compilation of the
77  executable \c test_code, add
78  \code
79  test_code_CXXFLAGS = -DUSE_TAYLOR_HOOD
80  \endcode
81  to the appropriate \c Makefile.\c am. The
82  C++ section of the
83  <A HREF="http://www.gnu.org/software/automake/manual/automake.html">
84  automake manual</A> provides more detail on other \c automake variables.
85  \n\n
86  If you link against \c oomph-lib from outside the \c autotools
87  framework, you can, of course, use whatever technique you
88  prefer to customise the compilation of your driver code.
89 .

95 \subsection cpp11 My compiler doesn't support C++11 by default
96 
97 \c oomph-lib uses the C++11 standard and assumes that your compiler supports
98 this. If you have an older version of the gcc compiler you can add
99 the flag \c -std=c++11 to the \c g++ configure options by adding it to
100 the \c CXXFLAGS in your configure options file. The one that's used
101 to build \c oomph-lib is
102 <center>
103 \c config/configure_options/current.
104 </center>
105 <br>
106 Note that the directory \c config/configure_options also contains many
107 other, generally sensibly named files that specify configure options
108 for other configurations and compilers. You can choose from these when running
109 \c autogen.sh or you can explore them yourself for further options.
110 
111 If you are using a different compiler check its man pages to find out
112 what flag is needed to enable C++11.

118 \subsection other_compilers The compilation fails when I use my xxx [non gcc] compiler
119 
120 \c oomph-lib is developed in a GNU Linux environment, using the
121 <A HREF="http://gcc.gnu.org/">gcc compiler suite</A>. We believe (!)
122 the source code complies with the
123 <A HREF="http://www.open-std.org/jtc1/sc22/wg21/"> C++ standard</A>
124 and it compiles cleanly under the <A HREF="http://gcc.gnu.org/">gcc
125 compilers</A> that we have access to (version 3.2.3 and later).
126 Some warning messages tend to be issued during the compilation of
127 the third-party libraries distributed with \c oomph-lib, but
128 these are not our responsibility!
129 
130 We occasionally compile (or get other people to compile) the library
131 under other compilers and we generally try to rectify any problems that
132 are flagged up in the process. If you encounter any problems
133 (errors or warning messages) while compiling the library with your
134 own (non-gcc) compiler we would <A HREF="../../contact/html/index.html">
135 like to hear from you</A>, especially if you suggest concrete bug
136 fixes.
137 
138  There are problems with version 7 of the intel compiler suite but the
139 library compiles cleanly under version 9. We suggest you upgrade
140 to that if you wish to use the intel compilers.

146 \subsection cygwin The build process fails under cygwin
147 
148 Here are a few things to check:
149 - Make sure the gcc compiler suite is sufficiently up-to-date
150  (we recommend version 3.2.3 or later). \n\n
151 - The build process will fail if you have (foolishly!) chosen
152  a windows username that contains spaces. The build script
153  (executed under linux) makes frequent references to the (absolute!)
154  path to the \c oomph-lib installation directory. Linux does
155  not allow spaces in directory names!

227 \subsection self_tests Some of the self tests fail
228 
229 <B>Tolerance of comparison between floating point numbers</B> \n\n
230 
231 The full \c oomph-lib distribution contains a large number of demo
232 codes in the sub-directory \c demo_drivers. Primarily, these
233 codes serve as demo codes for the \c oomph-lib documentation (contained in the
234 sub-directory \c doc) but they are also used during the library's self-test
235 procedure which checks that \c oomph-lib was installed correctly. The
236 self-tests can either be initiated at the end of the
237 <CODE>autogen.sh</CODE> - based installation or by typing
238 \code
239 make check -k
240 \endcode
241 in \c oomph-lib's top-level directory. The self-test builds
242 and runs all demo codes and compares their results against the reference
243 data stored in the \c validata sub-directories. The comparison
244 is performed with the python script <CODE> bin/fpdiff.py</CODE> which tolerates
245 small (machine- and compiler-dependent) roundoff errors
246 and suppresses the comparison of numbers whose absolute value falls below
247 some threshold (to avoid the comparison of numerical zeroes).
248 
249 If any of the self-test fail, you should first check the output in the file \c
250 validation.log to assess if the differences between the computed
251 data and the reference data are significant. If the
252 discrepancy appears to be due to larger-than-anticipated round-off
253 errors (you'll have to judge this yourself!), modify the
254 \c validate.sh script to specify a larger
255 relative tolerance and/or a larger value for the threshold
256 below which \c fpdiff.py regards numbers as numerical zeroes and
257 excludes them from the comparison. [Type
258 \code
259 bin/fpdiff.py
260 \endcode
261 in \c oomph-lib's top-level directory for instructions on how to use
262 the script].
263 
264 <B>The self-test fails even though the output files produced by
265 the code are correct</B> \n\n
266 
267 This is an odd error that is usually caused by the use of wildcards
268 in the validations scripts,
269 or the comparison of data that is stored in certain STL containers.
270 Typically, the self-test is performed by
271 the \c validate.sh shell script, which runs the executable and
272 concatenates selected output files to a single file whose contents
273 are compared against the reference file in the \c validata directory.
274 
275 While it is tempting to write
276 \code
277 cat RESLT* /soln0.dat > results_file.dat
278 \endcode
279 it is important to realise that the order in which the files are
280 concatenated is machine- and/or operating-system dependent. If
281 the above command is run in a directory with the following structure
282 \code
283 |-- RESLT
284 | |-- soln0.dat
285 | `-- trace.dat
286 |-- RESLT_elastic
287 | |-- soln0.dat
288 | `-- trace.dat
289 \endcode
290 some operating systems will expand the command to
291 \code
292 cat RESLT/soln0.dat RESLT_elastic/soln0.dat > results_file.dat
293 \endcode
294 while others will execute
295 \code
296 cat RESLT_elastic/soln0.dat RESLT/soln0.dat > results_file.dat
297 \endcode
298 In this case the self-test will report a failure, even though the
299 solution files are correct. The <CODE>validate.sh</CODE> scripts
300 should therefore not contain any wildcards.
301 
302 Similar problems can arise if the validation data includes
303 data that is stored in certain STL containers such as sets.
304 The order in which items are stored in such containers may
305 vary from machine to machine and from compiler to compiler.
306 If such data is to be included in a self-test the data should be
307 sorted first, based on a user-controllable sorting criterion.

313 
314 Assume you have followed our advice to explore \c oomph-lib by
315 playing with representative demo driver codes, and have ended up
316 in a directory that contains the following files:
317 \code
318 mheil@biolaptop:~/version339/demo_drivers/poisson/one_d_poisson$ ls -l
319 total 68

325 -rwxrwxr-x 1 mheil mheil 2402 2012-12-08 13:34 validate.sh
326 \endcode
327 The temptation is to compile the driver code \c one_d_poisson.cc
328 with your favourite C++ compiler, e.g. \c g++:
329 \code
330 g++ one_d_poisson.cc
331 \endcode
332 but this does \b not work:
333 \code
334 mheil@biolaptop:~/version339/demo_drivers/poisson/one_d_poisson$ g++ one_d_poisson.cc
335 one_d_poisson.cc:31:21: fatal error: generic.h: No such file or directory
336 compilation terminated.
337 \endcode
338 Why? If you look into the code, you notice that it \#includes
339 \c oomph-lib's \c generic.h header file, and \c g++ obviously doesn't
340 know where to find this. You can, in principle, find
341 out where it (and the associated library files themselves!) live
342 but this is rather against the spirit of the
343 <A HREF="http://www.gnu.org/software/automake/">automake</A>
344 machinery. What you should do instead is to use \c make:
345 \code
346 make one_d_poisson
347 \endcode
348 Note the omitted postfix! The argument to \c make is the name of the
349 target, i.e. the executable that you wish to build -- which source
350 code to compile, where to locate the required headers and which
351 libraries to link against is all encoded in the \c Makefile (which has
352 itself been created by <A HREF="http://www.gnu.org/software/automake/">
353 automake</A> (during the installation of the \c oomph-lib)
354 from the \c Makefile.am file in the directory). As you see
355 from the on-screen output, there is a lot of information that needs
356 to be passed to the compiler/linker:

416 \subsection run2 Why doesn't make make anything? What are the targets called?
417 If you are familiar with makefiles, you are unlikely to have
418 fallen into the trap discussed above. You will almost certainly
419 have spotted the \c
420 Makefile and tried to build the executable(s) with \c make, but
421 oddly this doesn't work as you (probably) expected:
422 \code
423 mheil@biolaptop:~/version339/demo_drivers/poisson/one_d_poisson$ make
424 make: Nothing to be done for `all'.
425 \endcode
426 The reason for this is that
427 -# <A HREF="http://www.gnu.org/software/automake/">automake</A>
428  distinguishes between several types of targets,
429 -# the demo driver codes in the subdirectories of \c demo_drivers
430  double up as self-tests to assert the correct installation of
431  \c oomph-lib when \c make \c check is run from the top-level
432  directory.
433 .
434 The latter requires them to be declared (in the \c Makefile.am)
435 in the \c check_PROGRAMS variable. You can either build (and
436 automatically run) all executables in a \c demo_driver directory
437 by issuing the command
438 \code
439 make check -k
440 \endcode
441 or you can inspect the \c check_PROGRAMS variable in the
442 \c Makefile.am to find out which targets have been declared
443 (and can therefore be built by \c make).
444 For instance, in the \c one_d_poisson directory the first
445 few lines of the \c Makefile.am are
446 \code
447 mheil@biolaptop:~/version339/demo_drivers/poisson/one_d_poisson$ more Makefile.am
448 
449 [...]
450 
451 check_PROGRAMS=one_d_poisson
452 
453 one_d_poisson_SOURCES = one_d_poisson.cc
454 
455 [...]
456 \endcode
457 This declares the target (the executable called \c one_d_poisson)
458 and specifies the source code to be compiled (\c one_d_poisson.cc).
459 The build target needs to specified explicitly:
460 \code
461 make one_d_poisson
462 \endcode
463 We note that the names of the executable and the source code do not
464 have to be as closely related as in this example. While we usually
465 create the name of the executable by dropping the
466 postfix of the source file, this is not guaranteed! Inspect
467 the \c Makefile.am if \c make doesn't give you what you want.

532 \subsection link The oomph-lib distribution includes some third-party libraries. How do I get the code to link against optimised local versions of these libraries that are already installed on my machine?
533 
534 To facilitate the installation, the oomph-lib distribution includes
535 the relevant parts of certain third-party libraries, such as
536 <A HREF="http://crd.lbl.gov/~xiaoye/SuperLU">SuperLU</A> and
537 <A HREF="http://www.netlib.org/blas/">BLAS</A>. These "external libraries"
538 are built and installed along with \c oomph-lib's own sub-libraries.
539 To distinguish them from any already existing local versions of
540 these libraries, their names are pre-fixed with the string \c oomph_.
541

543 This is achieved by defining the variable
544 \c EXTERNAL_LIBS in the \c configure script:
545 
546 \code
547  # Define "external" libraries
548  #----------------------------
549  # These are the third-party libraries distributed with oomph-lib.
550  # Note: If you want to link against another, already existing local
551  # installation of one of these libraries, edit the EXTERNAL_LIBS
552  # assignment below. As an example, if you want to use your own BLAS
553  # library, and you'd usually link against this with
554  #
555  # g++ -o my_code my_code.cc -lblas
556  #
557  # then replace "-loomph_blas" by "-lblas" below:
558  #
559  EXTERNAL_LIBS='-loomph_hsl -loomph_superlu_3.0 -loomph_metis_4.0 -loomph_lapack -loomph_flapack -loomph_blas'
560 \endcode
561 
562 If you wish to link against some other version of the library, edit
563 the \c EXTERNAL_LIBS variable accordingly.

602 \subsection missing_this Missing 'this->'
603 
604 Recent versions of the <A HREF="http://gcc.gnu.org/">gcc compilers</A>
605 enforce the <A HREF="http://www.open-std.org/jtc1/sc22/wg21/">C++
606 standard</A> much more rigorously than earlier versions.
607 Unfortunately, the standard includes some rules that are so counter-intuitive
608 that it is hard get into the habit of using them, especially if code
609 is developed on a compiler that does not enforce the standard
610 as rigorously.
611 
612 The most frequent problem arises in classes that
613 are derived from a templated base class.
614 <A HREF="http://www.open-std.org/jtc1/sc22/wg21/">The C++ standard</A>
615 insists that all references to member functions (or member data) that
616 is defined in the templated base class must be preceded by "this->"
617 when the reference is made in the derived class. Allegedly, this
618 is necessary to avoid ambiguities, though it is not entirely clear what this
619 ambiguity is supposed to be... Here is a driver code that illustrates
620 the problem.
621 
622 \include broken_this_demo.cc
623 
624 If you compile this with sufficiently recent versions of the
625 <A HREF="http://gcc.gnu.org/">gcc compilers</A>, using the
626 flag \c -DUSE_BROKEN_VERSION, the compilation will fail with the
627 following error:
628 \code
629 broken_this_demo.cc: In member function 'void SomeDerivedClass<TEMPLATE_PARAMETER>::output_template_parameter()':
630 broken_this_demo.cc:55: error: there are no arguments to 'say_hello_world' that depend on a template parameter, so a declaration of 'say_hello_world' must be available
631 broken_this_demo.cc:55: error: (if you use '-fpermissive', G++ will accept your code, but allowing the use of an undeclared name is deprecated)
632 \endcode
633 
634 You may not only stumble across this problem in one of your own codes but
635 it is also possible that some code in the library itself still violates
636 this rule. This is because templated classes are only built when
637 needed and it is conceivable that \c oomph-lib's suite of
638 self-tests do not instantiate all templated classes that exist in the
639 library. If you encounter any such problems, check if
640 putting a "this->" in front of the function call fixes the problem.
641 If it does, <A HREF="../../contact/html/index.html">let us know!</A>

672 \subsection seg_fault My driver code compiles but dies with a segmentation fault
673 
674 Suggestions:
675 - Run \c Problem::self_test() before solving the problem. This function
676  performs a large number of sanity checks and reports
677  any inconsistencies in the data structure. \n\n
678 - Recompile the relevant libraries
679  with the PARANOID and RANGE_CHECKING flags set.
680  Segmentation faults are often caused by out-of-bounds access
681  to STL containers. Since > 95% (?) of the containers used in \c oomph-lib
682  are \c Vectors (\c oomph-lib's wrapper to the STL vector class,
683  with optional range checking) they can easily be detected.
684  Make sure to re-compile again with RANGE_CHECKING switched
685  off before you start any production runs -- the run-time overheads
686  incurred by range-checking are significant! \n\n
687 - Recompile the relevant libraries and your driver code with
688  the debugging flag ("-g" for the gnu compiler suite) switched on
689  and all optimisation disabled. Re-run the code in a debugger
690  (\c gdb or its GUI-based equivalent
691  <A HREF="http://www.gnu.org/software/ddd/">ddd</a>) to (try to!) find out
692  where the segmentation fault occurred.
693  [\b Careful: If the segmentation fault is caused by a pointer problem,
694  this naive inspection can be quite misleading -- tell-tale signs are that
695  the traceback displays a non-sensical call stack, e.g. a function being
696  called "out of nowhere"; variables that have just been given
697  values not existing; etc.] \n\n

701 \subsection wrong_results My driver code runs but it produces incorrect/non-sensical results
702 
703 Suggestions:
704 - Have you applied the boundary conditions correctly? Check this by looping
705  over all nodes in your mesh and documenting the pinned-status
706  of their nodal values, using the function \c Node::is_pinned(...). \n\n
707 - Have you passed the required (function) pointers to the elements?
708  Most element constructors assign default values for any physical parameters,
709  e.g. the Reynolds number in the Navier-Stokes
710  elements. Similarly, most source functions etc. default to zero. For
711  instance \c oomph-lib's Poisson elements solve the Laplace equation unless a
712  function pointer to the source function is specified. \n\n
713 - In time-dependent problems, have you assigned suitable initial
714  conditions? Note that, if you use elements that are based on
715  the ALE formulation (and all time-dependent elements should be!)
716  you must initialise the history values for the nodal positions,
717  even if the mesh is stationary!
718  See the discussion of \c oomph-lib's timestepping procedures in the
719  context of <A HREF="../../unsteady_heat/two_d_unsteady_heat/html/index.html">
720  the unsteady heat equation</A> for details. \n\n
721 - Have you implemented the relevant "action" functions, such as
722  \c Problem::actions_before_implicit_timestep()
723  to update any time-dependent boundary conditions?
724 .

728 \subsection newton_divergence The Newton solver diverges
729 
730 Suggestions:
731 - \c oomph-lib's default solver \c Problem::newton_solve(...)
732  will converge quadratically, provided \n\n
733  - a "good" initial guess for the solution has been assigned,
734  .
735  \n
736  or
737  \n
738  - the problem is linear.
739  .
740  \n
741  If the Newton solver fails to converge for a nonlinear problem,
742  try to identify a related linear problem and use continuation
743  to generate a sequence of good initial guesses. For instance,
744  to solve the Navier-Stokes equations at a Reynolds number
745  of 500, say, start by solving the problem for zero Reynolds number
746  (in which case the problem becomes linear so that the Newton
747  method converges in one iteration); increase the Reynolds number
748  by 50, say, and re-solve. Repeat this procedure until
749  the desired value of the Reynolds number is reached.
750  \n\n
751  \b Note: \c oomph-lib also provides automatic continuation methods,
752  based on Keller's arclength continuation, but at the moment,
753  no tutorials exist for these.
754  \n\n
755 - If you have tried the above and the Newton method fails to converge
756  even for a linear version of your problem, the most likely reasons are
757  that
758  \n\n
759  -# You have developed a new element and made a mistake in the
760  implementation of the element's Jacobian matrix.
761  To check if this is the case, comment out the
762  function that computes the element's Jacobian matrix,
763  i.e. the element member functions
764  \c get_jacobian(...) or \c fill_in_contribution_to_jacobian(...).
765  \c oomph-lib will then use the default implementation
766  of these functions in the \c GeneralisedElement base class
767  to compute the Jacobian matrices by finite-differencing.
768  The executable is likely to run more slowly since the
769  finite-difference-based computation
770  is unlikely to be as efficient as the customised
771  implementation for your specific element, but if the
772  Newton method then converges, you know where to look for your
773  bug! You may also want to check for any un-initialised variables.
774  They are the most likely culprits if your code behaves
775  differently at different levels of optimisation as
776  more aggressive optimisation may suppress any default
777  initialisations of data -- in fact, you should never rely on that
778  anyway!
779  \n\n
780  -# Your problem contains "dependent" variables, such as
781  the nodal positions in a free-boundary problem. If the
782  node update in response to changes in the shape of the
783  domain boundary is performed by an algebraic node update
784  procedure (using \c AlgebraicNodes, \c SpineNodes or nodes whose
785  position is updated by a \c MacroElement/Domain - based
786  procedure), the position of the nodes in the "bulk" mesh
787  must be updated whenever the Newton method updates the unknowns.
788  This is most easily done by calling \c Mesh::node_update() in
789  \c Problem::actions_before_newton_convergence_check().
790  .
791 .

891 \subsection element_too_general oomph-lib's implementation of the Navier-Stokes equations (say) is too general (and therefore too expensive) for my application. How do I change this?
892 
893 Many of \c oomph-lib's equations classes (or elements) are implemented
894 in great generality. For instance, our discretisation of the
895 Navier-Stokes equations includes a source term in the continuity equation,
896 and body force terms in the momentum equations; it allows switching between
897 the stress-divergence and simplified forms of the viscous terms;
898 it includes the mesh velocity into the ALE formulation of the
899 time-derivatives; etc. This makes the elements very versatile
900 and robust. However, the generality/robustness comes at a price:
901 Even though we provide default values for most functions
902 (e.g. the body force terms default to zero), their
903 evaluation requires a finite amount of CPU time. If you wish
904 to use the elements in a simple application in which the
905 Navier-Stokes equations are solved in a fixed domain, without
906 any body forces or other source terms, say, you may wish to disable
907 the additional functionality.
908 
909 This is easily done: After all, \c oomph-lib is open-source software and
910 you can therefore change anything you want! In principle, you
911 could edit the source code in the \c src/navier_stokes directory and
912 delete (or at least comment out) all the functionality that you do not require.
913 However, this is probably a risky step as it will break all
914 demo codes (used during \c oomph-lib's self-test procedure) that use
915 some of the features that you are not interested in. We therefore
916 recommend copying the content of the directory \c src/navier_stokes
917 into a new directory, e.g. \c user_src/my_navier_stokes
918 and to edit the copied sources. Follow the instructions on the
919 \c oomph-lib <A HREF="../../the_distribution/html/index.html#add_src">
920 installation page</A> to turn these sources into a separate
921 library against which you can link.

1038 .
1039 \c oomph-lib's <A HREF="../../example_code_list/html/index.html">
1040 tutorials</A> tend to provide a fairly complete annotated
1041 listing of the relevant driver codes; if the function you are
1042 interested in is not mentioned explicitly in the tutorial, it is most
1043 likely to be defined in an include file. You can inspect the driver code
1044 in its entirety by following the link at the end of the
1045 tutorials. If you cannot find the class/function there,
1046 it must be defined in one of the include files listed
1047 at the beginning of the source code.
1048 
1049  The included files themselves can either be located in
1050 the same directory as the demo driver (the directory also tends
1051 to be mentioned at the end of the tutorial) or in one of
1052 \c oomph-lib's sub-libraries. The source code for these
1053 is located in the sub-directories of the \c src directory. Often
1054 the class/function is defined in a source file
1055 with an "obvious" name; if not, use \c grep to find it.
1056 This can, of course, be done recursively. For instance,
1057 the command
1058 
1059 \code
1060 find . \‍( -name '*.h' -o -name '*.cc' \‍) -exec grep -H FiniteElement {} \;
1061 \endcode
1062 
1063 issued in \c oomph-lib's top-level directory
1064 will search through the entire distribution to
1065 locate files that contain the string "FiniteElement".
