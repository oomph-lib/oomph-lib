49 \subsection comp_opt How do I specify different levels of optimisation and/or non-standard compiler options?
50 
51 
52 -# <B>Compiler flags/options for the compilation of the library</B> \n\n
53  By default, \c oomph-lib's installation is performed with the
54  <A HREF="http://gcc.gnu.org/">gcc compiler suite</A>, using full
55  optimisation and warning (\c -O6 \c -Wall ).
56  \n\n
57  If you install the library with the \c autogen.sh script you are
58  given the option to overwrite the default compilation flags,
59  either by specifying alternative flags on the command line,
60  or by recycling any of the previously used combinations of flags, stored
61  in the directory <A HREF="../../../config/configure_options/">
62  config/configure_options</A>. The files in this directory
63  also contain more details on available flags and options.
64  \n\n
65  If you prefer the standard linux \c configure/make/make \c install
66  installation procedure, you should already know how to specify
67  flags at the configure stage...
68  \n\n
69 -# <B>Compiler flags/options for the compilation of individual
70  (driver) codes</B> \n\n
71  Within the \c autotools framework,
72  additional flags for the compilation of individual (driver) codes
73  may be specified by setting the appropriate automake variables
74  in the \c Makefile.\c am. For instance, to use the compiler
75  flag \c -DUSE_TAYLOR_HOOD during the compilation of the
76  executable \c test_code, add
77  \code
78  test_code_CXXFLAGS = -DUSE_TAYLOR_HOOD
79  \endcode
80  to the appropriate \c Makefile.\c am. The
81  C++ section of the
82  <A HREF="http://www.gnu.org/software/automake/manual/automake.html">
83  automake manual</A> provides more detail on other \c automake variables.
84  \n\n
85  If you link against \c oomph-lib from outside the \c autotools
86  framework, you can, of course, use whatever technique you
87  prefer to customise the compilation of your driver code.
88 .

92 \subsection other_compilers The compilation fails when I use my xxx [non gcc] compiler
93 
94 \c oomph-lib is developed in a GNU Linux environment, using the
95 <A HREF="http://gcc.gnu.org/">gcc compiler suite</A>. We believe (!)
96 the source code complies with the
97 <A HREF="http://www.open-std.org/jtc1/sc22/wg21/"> C++ standard</A>
98 and it compiles cleanly under the <A HREF="http://gcc.gnu.org/">gcc
99 compilers</A> that we have access to (version 3.2.3 and later).
100 Some warning messages tend to be issued during the compilation of
101 the third-party libraries distributed with \c oomph-lib, but
102 these are not our responsibility!
103 
104 We occasionally compile (or get other people to compile) the library
105 under other compilers and we generally try to rectify any problems that
106 are flagged up in the process. If you encounter any problems
107 (errors or warning messages) while compiling the library with your
108 own (non-gcc) compiler we would <A HREF="../../contact/html/index.html">
109 like to hear from you</A>, especially if you suggest concrete bug
110 fixes.
111 
112  There are problems with version 7 of the intel compiler suite but the
113 library compiles cleanly under version 9. We suggest you upgrade
114 to that if you wish to use the intel compilers.

120 \subsection cygwin The build process fails under cygwin
121 
122 Here are a few things to check:
123 - Make sure the gcc compiler suite is sufficiently up-to-date
124  (we recommend version 3.2.3 or later). \n\n
125 - The build process will fail if you have (foolishly!) chosen
126  a windows username that contains spaces. The build script
127  (executed under linux) makes frequent references to the (absolute!)
128  path to the \c oomph-lib installation directory. Linux does
129  not allow spaces in directory names!

201 \subsection self_tests Some of the self tests fail
202 
203 <B>Tolerance of comparison between floating point numbers</B> \n\n
204 
205 The full \c oomph-lib distribution contains a large number of demo
206 codes in the sub-directory \c demo_drivers. Primarily, these
207 codes serve as demo codes for the \c oomph-lib documentation (contained in the
208 sub-directory \c doc) but they are also used during the library's self-test
209 procedure which checks that \c oomph-lib was installed correctly. The
210 self-tests can either be initiated at the end of the
211 <CODE>autogen.sh</CODE> - based installation or by typing
212 \code
213 make check -k
214 \endcode
215 in \c oomph-lib's top-level directory. The self-test builds
216 and runs all demo codes and compares their results against the reference
217 data stored in the \c validata sub-directories. The comparison
218 is performed with the python script <CODE> bin/fpdiff.py</CODE> which tolerates
219 small (machine- and compiler-dependent) roundoff errors
220 and suppresses the comparison of numbers whose absolute value falls below
221 some threshold (to avoid the comparison of numerical zeroes).
222 
223 If any of the self-test fail, you should first check the output in the file \c
224 validation.log to assess if the differences between the computed
225 data and the reference data are significant. If the
226 discrepancy appears to be due to larger-than-anticipated round-off
227 errors (you'll have to judge this yourself!), modify the
228 \c validate.sh script to specify a larger
229 relative tolerance and/or a larger value for the threshold
230 below which \c fpdiff.py regards numbers as numerical zeroes and
231 excludes them from the comparison. [Type
232 \code
233 bin/fpdiff.py
234 \endcode
235 in \c oomph-lib's top-level directory for instructions on how to use
236 the script].
237 
238 <B>The self-test fails even though the output files produced by
239 the code are correct</B> \n\n
240 
241 This is an odd error that is usually caused by the use of wildcards
242 in the validations scripts,
243 or the comparison of data that is stored in certain STL containers.
244 Typically, the self-test is performed by
245 the \c validate.sh shell script, which runs the executable and
246 concatenates selected output files to a single file whose contents
247 are compared against the reference file in the \c validata directory.
248 
249 While it is tempting to write
250 \code
251 cat RESLT* /soln0.dat > results_file.dat
252 \endcode
253 it is important to realise that the order in which the files are
254 concatenated is machine- and/or operating-system dependent. If
255 the above command is run in a directory with the following structure
256 \code
257 |-- RESLT
258 | |-- soln0.dat
259 | `-- trace.dat
260 |-- RESLT_elastic
261 | |-- soln0.dat
262 | `-- trace.dat
263 \endcode
264 some operating systems will expand the command to
265 \code
266 cat RESLT/soln0.dat RESLT_elastic/soln0.dat > results_file.dat
267 \endcode
268 while others will execute
269 \code
270 cat RESLT_elastic/soln0.dat RESLT/soln0.dat > results_file.dat
271 \endcode
272 In this case the self-test will report a failure, even though the
273 solution files are correct. The <CODE>validate.sh</CODE> scripts
274 should therefore not contain any wildcards.
275 
276 Similar problems can arise if the validation data includes
277 data that is stored in certain STL containers such as sets.
278 The order in which items are stored in such containers may
279 vary from machine to machine and from compiler to compiler.
280 If such data is to be included in a self-test the data should be
281 sorted first, based on a user-controllable sorting criterion.

289 playing with representative demo driver codes, and have ended up

301 The temptation is to compile the driver code \c one_d_poisson.cc
302 with your favourite C++ compiler, e.g. \c g++:
303 \code
304 g++ one_d_poisson.cc
305 \endcode
306 but this does \b not work:
307 \code
308 mheil@biolaptop:~/version339/demo_drivers/poisson/one_d_poisson$ g++ one_d_poisson.cc
309 one_d_poisson.cc:31:21: fatal error: generic.h: No such file or directory

312 Why? If you look into the code, you notice that it \#includes
313 \c oomph-lib's \c generic.h header file, and \c g++ obviously doesn't
314 know where to find this. You can, in principle, find
315 out where it (and the associated library files themselves!) live
316 but this is rather against the spirit of the
317 <A HREF="http://www.gnu.org/software/automake/">automake</A>
318 machinery. What you should do instead is to use \c make:
319 \code
320 make one_d_poisson
321 \endcode
322 Note the omitted postfix! The argument to \c make is the name of the
323 target, i.e. the executable that you wish to build -- which source
324 code to compile, where to locate the required headers and which
325 libraries to link against is all encoded in the \c Makefile (which has
326 itself been created by <A HREF="http://www.gnu.org/software/automake/">

391 \subsection run2 Why doesn't make make anything? What are the targets called?
392 If you are familiar with makefiles, you are unlikely to have
393 fallen into the trap discussed above. You will almost certainly
394 have spotted the \c
395 Makefile and tried to build the executable(s) with \c make, but
396 oddly this doesn't work as you (probably) expected:
397 \code
398 mheil@biolaptop:~/version339/demo_drivers/poisson/one_d_poisson$ make
399 make: Nothing to be done for `all'.
400 \endcode
401 The reason for this is that
402 -# <A HREF="http://www.gnu.org/software/automake/">automake</A>
403  distinguishes between several types of targets,
404 -# the demo driver codes in the subdirectories of \c demo_drivers
405  double up as self-tests to assert the correct installation of
406  \c oomph-lib when \c make \c check is run from the top-level
407  directory.
408 .
409 The latter requires them to be declared (in the \c Makefile.am)
410 in the \c check_PROGRAMS variable. You can either build (and
411 automatically run) all executables in a \c demo_driver directory
412 by issuing the command
413 \code
414 make check -k
415 \endcode
416 or you can inspect the \c check_PROGRAMS variable in the
417 \c Makefile.am to find out which targets have been declared
418 (and can therefore be built by \c make).
419 For instance, in the \c one_d_poisson directory the first
420 few lines of the \c Makefile.am are
421 \code
422 mheil@biolaptop:~/version339/demo_drivers/poisson/one_d_poisson$ more Makefile.am
423 
424 [...]
425 
426 check_PROGRAMS=one_d_poisson
427 
428 one_d_poisson_SOURCES = one_d_poisson.cc
429 
430 [...]
431 \endcode
432 This declares the target (the executable called \c one_d_poisson)
433 and specifies the source code to be compiled (\c one_d_poisson.cc).
434 The build target needs to specified explicitly:
435 \code
436 make one_d_poisson
437 \endcode
438 We note that the names of the executable and the source code do not
439 have to be as closely related as in this example. While we usually
440 create the name of the executable by dropping the
441 postfix of the source file, this is not guaranteed! Inspect
442 the \c Makefile.am if \c make doesn't give you what you want.

447 \subsection only_compile_dont_run Running the self-tests takes a long time. Is it possible to suppress the execution of the built driver codes and only check if they compile? (Mainly useful for developers).
448 
449 
450 Issuing \c make \c check from the top
451 level \c oomph-lib directory descends into all oomph-lib directories
452 that contain demo driver codes, builds these and then runs them
453 using the script \c validate.sh which typically also compares
454 the output against some reference data
455 that is stored in the \c validata sub-directories. This process
456 (which allows for some tolerances
457 in floating point data) assesses that
458 the installation is working properly, but the entire procedure obviously
459 takes a long time. As a developer, you may want to check (more) quickly
460 if the changes you've made to any source code (in \c src) still compile
461 without running any of the self-tests. It is easy to suppress
462 the execution of the self tests because the \c validate.sh script is, in fact,

508 
509 To facilitate the installation, the oomph-lib distribution includes

577 \subsection missing_this Missing 'this->'
578 
579 Recent versions of the <A HREF="http://gcc.gnu.org/">gcc compilers</A>
580 enforce the <A HREF="http://www.open-std.org/jtc1/sc22/wg21/">C++
581 standard</A> much more rigorously than earlier versions.
582 Unfortunately, the standard includes some rules that are so counter-intuitive
583 that it is hard get into the habit of using them, especially if code
584 is developed on a compiler that does not enforce the standard
585 as rigorously.
586 
587 The most frequent problem arises in classes that
588 are derived from a templated base class.
589 <A HREF="http://www.open-std.org/jtc1/sc22/wg21/">The C++ standard</A>
590 insists that all references to member functions (or member data) that
591 is defined in the templated base class must be preceded by "this->"
592 when the reference is made in the derived class. Allegedly, this
593 is necessary to avoid ambiguities, though it is not entirely clear what this
594 ambiguity is supposed to be... Here is a driver code that illustrates
595 the problem.
596 
597 \include broken_this_demo.cc
598 
599 If you compile this with sufficiently recent versions of the
600 <A HREF="http://gcc.gnu.org/">gcc compilers</A>, using the
601 flag \c -DUSE_BROKEN_VERSION, the compilation will fail with the
602 following error:
603 \code
604 broken_this_demo.cc: In member function 'void SomeDerivedClass<TEMPLATE_PARAMETER>::output_template_parameter()':
605 broken_this_demo.cc:55: error: there are no arguments to 'say_hello_world' that depend on a template parameter, so a declaration of 'say_hello_world' must be available
606 broken_this_demo.cc:55: error: (if you use '-fpermissive', G++ will accept your code, but allowing the use of an undeclared name is deprecated)
607 \endcode
608 
609 You may not only stumble across this problem in one of your own codes but
610 it is also possible that some code in the library itself still violates
611 this rule. This is because templated classes are only built when
612 needed and it is conceivable that \c oomph-lib's suite of
613 self-tests do not instantiate all templated classes that exist in the
614 library. If you encounter any such problems, check if
615 putting a "this->" in front of the function call fixes the problem.
616 If it does, <A HREF="../../contact/html/index.html">let us know!</A>

647 \subsection seg_fault My driver code compiles but dies with a segmentation fault
648 
649 Suggestions:
650 - Run \c Problem::self_test() before solving the problem. This function
651  performs a large number of sanity checks and reports
652  any inconsistencies in the data structure. \n\n
653 - Recompile the relevant libraries
654  with the PARANOID and RANGE_CHECKING flags set.
655  Segmentation faults are often caused by out-of-bounds access
656  to STL containers. Since > 95% (?) of the containers used in \c oomph-lib
657  are \c Vectors (\c oomph-lib's wrapper to the STL vector class,
658  with optional range checking) they can easily be detected.
659  Make sure to re-compile again with RANGE_CHECKING switched
660  off before you start any production runs -- the run-time overheads
661  incurred by range-checking are significant! \n\n
662 - Recompile the relevant libraries and your driver code with
663  the debugging flag ("-g" for the gnu compiler suite) switched on
664  and all optimisation disabled. Re-run the code in a debugger
665  (\c gdb or its GUI-based equivalent
666  <A HREF="http://www.gnu.org/software/ddd/">ddd</a>) to (try to!) find out
667  where the segmentation fault occurred.
668  [\b Careful: If the segmentation fault is caused by a pointer problem,
669  this naive inspection can be quite misleading -- tell-tale signs are that
670  the traceback displays a non-sensical call stack, e.g. a function being
671  called "out of nowhere"; variables that have just been given
672  values not existing; etc.] \n\n

676 \subsection wrong_results My driver code runs but it produces incorrect/non-sensical results
677 
678 Suggestions:
679 - Have you applied the boundary conditions correctly? Check this by looping
680  over all nodes in your mesh and documenting the pinned-status
681  of their nodal values, using the function \c Node::is_pinned(...). \n\n
682 - Have you passed the required (function) pointers to the elements?
683  Most element constructors assign default values for any physical parameters,
684  e.g. the Reynolds number in the Navier-Stokes
685  elements. Similarly, most source functions etc. default to zero. For
686  instance \c oomph-lib's Poisson elements solve the Laplace equation unless a
687  function pointer to the source function is specified. \n\n
688 - In time-dependent problems, have you assigned suitable initial
689  conditions? Note that, if you use elements that are based on
690  the ALE formulation (and all time-dependent elements should be!)
691  you must initialise the history values for the nodal positions,
692  even if the mesh is stationary!
693  See the discussion of \c oomph-lib's timestepping procedures in the
694  context of <A HREF="../../unsteady_heat/two_d_unsteady_heat/html/index.html">
695  the unsteady heat equation</A> for details. \n\n
696 - Have you implemented the relevant "action" functions, such as
697  \c Problem::actions_before_implicit_timestep()
698  to update any time-dependent boundary conditions?
699 .

703 \subsection newton_divergence The Newton solver diverges
704 
705 Suggestions:
706 - \c oomph-lib's default solver \c Problem::newton_solve(...)
707  will converge quadratically, provided \n\n
708  - a "good" initial guess for the solution has been assigned,
709  .
710  \n
711  or
712  \n
713  - the problem is linear.
714  .
715  \n
716  If the Newton solver fails to converge for a nonlinear problem,
717  try to identify a related linear problem and use continuation
718  to generate a sequence of good initial guesses. For instance,
719  to solve the Navier-Stokes equations at a Reynolds number
720  of 500, say, start by solving the problem for zero Reynolds number
721  (in which case the problem becomes linear so that the Newton
722  method converges in one iteration); increase the Reynolds number
723  by 50, say, and re-solve. Repeat this procedure until
724  the desired value of the Reynolds number is reached.
725  \n\n
726  \b Note: \c oomph-lib also provides automatic continuation methods,
727  based on Keller's arclength continuation, but at the moment,
728  no tutorials exist for these.
729  \n\n
730 - If you have tried the above and the Newton method fails to converge
731  even for a linear version of your problem, the most likely reasons are
732  that
733  \n\n
734  -# You have developed a new element and made a mistake in the
735  implementation of the element's Jacobian matrix.
736  To check if this is the case, comment out the
737  function that computes the element's Jacobian matrix,
738  i.e. the element member functions
739  \c get_jacobian(...) or \c fill_in_contribution_to_jacobian(...).
740  \c oomph-lib will then use the default implementation
741  of these functions in the \c GeneralisedElement base class
742  to compute the Jacobian matrices by finite-differencing.
743  The executable is likely to run more slowly since the
744  finite-difference-based computation
745  is unlikely to be as efficient as the customised
746  implementation for your specific element, but if the
747  Newton method then converges, you know where to look for your
748  bug! You may also want to check for any un-initialised variables.
749  They are the most likely culprits if your code behaves
750  differently at different levels of optimisation as
751  more aggressive optimisation may suppress any default
752  initialisations of data -- in fact, you should never rely on that
753  anyway!
754  \n\n
755  -# Your problem contains "dependent" variables, such as
756  the nodal positions in a free-boundary problem. If the
757  node update in response to changes in the shape of the
758  domain boundary is performed by an algebraic node update
759  procedure (using \c AlgebraicNodes, \c SpineNodes or nodes whose
760  position is updated by a \c MacroElement/Domain - based
761  procedure), the position of the nodes in the "bulk" mesh
762  must be updated whenever the Newton method updates the unknowns.
763  This is most easily done by calling \c Mesh::node_update() in
764  \c Problem::actions_before_newton_convergence_check().
765  .
766 .

866 \subsection element_too_general oomph-lib's implementation of the Navier-Stokes equations (say) is too general (and therefore too expensive) for my application. How do I change this?
867 
868 Many of \c oomph-lib's equations classes (or elements) are implemented
869 in great generality. For instance, our discretisation of the
870 Navier-Stokes equations includes a source term in the continuity equation,
871 and body force terms in the momentum equations; it allows switching between
872 the stress-divergence and simplified forms of the viscous terms;
873 it includes the mesh velocity into the ALE formulation of the
874 time-derivatives; etc. This makes the elements very versatile
875 and robust. However, the generality/robustness comes at a price:
876 Even though we provide default values for most functions
877 (e.g. the body force terms default to zero), their
878 evaluation requires a finite amount of CPU time. If you wish
879 to use the elements in a simple application in which the
880 Navier-Stokes equations are solved in a fixed domain, without
881 any body forces or other source terms, say, you may wish to disable
882 the additional functionality.
883 
884 This is easily done: After all, \c oomph-lib is open-source software and
885 you can therefore change anything you want! In principle, you
886 could edit the source code in the \c src/navier_stokes directory and
887 delete (or at least comment out) all the functionality that you do not require.
888 However, this is probably a risky step as it will break all
889 demo codes (used during \c oomph-lib's self-test procedure) that use
890 some of the features that you are not interested in. We therefore
891 recommend copying the content of the directory \c src/navier_stokes
892 into a new directory, e.g. \c user_src/my_navier_stokes
893 and to edit the copied sources. Follow the instructions on the
894 \c oomph-lib <A HREF="../../the_distribution/html/index.html#add_src">
895 installation page</A> to turn these sources into a separate
896 library against which you can link.

1001 \subsection includes Where is this class/function/... defined?
1002 
1003 Assume you have studied one of the
1004 <A HREF="../../example_code_list/html/index.html">example codes</A>
1005 and wish to find out more about the implementation of a particular
1006 class or function that is used there. How do you find
1007 its source code and/or its full documentation?
1008 
1009 Generally, a class/function that is used in a demo code
1010 can only be defined in one of two places:
1011 -# In the demo driver code itself.
1012 -# In an included file and/or an associated library.
1013 .
1014 \c oomph-lib's <A HREF="../../example_code_list/html/index.html">
1015 tutorials</A> tend to provide a fairly complete annotated
1016 listing of the relevant driver codes; if the function you are
1017 interested in is not mentioned explicitly in the tutorial, it is most
1018 likely to be defined in an include file. You can inspect the driver code
1019 in its entirety by following the link at the end of the
1020 tutorials. If you cannot find the class/function there,
1021 it must be defined in one of the include files listed
1022 at the beginning of the source code.
1023 
1024  The included files themselves can either be located in
1025 the same directory as the demo driver (the directory also tends
1026 to be mentioned at the end of the tutorial) or in one of
1027 \c oomph-lib's sub-libraries. The source code for these
1028 is located in the sub-directories of the \c src directory. Often
1029 the class/function is defined in a source file
