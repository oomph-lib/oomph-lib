51  /// A geometric object is an object that provides a parametrised
52  /// description of its shape via the function GeomObject::position(...).
53  ///
54  /// The minimum functionality is: The geometric object has
55  /// a number of Lagrangian (intrinsic) coordinates that parametrise
56  /// the (Eulerian) position vector, whose dimension might
57  /// differ from the number of Lagrangian (intrinsic) coordinates (e.g.
58  /// for shell-like objects).
59  ///
60  /// We might also need the derivatives of the position
61  /// Vector w.r.t. to the Lagrangian (intrinsic) coordinates and interfaces
62  /// for this functionality are provided.
63  /// [Note: For some geometric objects it might be too tedious to work out
64  /// the derivatives and they might not be needed anyway. In other
65  /// cases we might always need the position vector and all
66  /// derivatives at the same time. We provide suitable interfaces
67  /// for these cases in virtual but broken (rather than pure virtual) form
68  /// so the user can (but doesn't have to) provide the required versions
69  /// by overloading.]
70  ///
71  /// The shape of a geometric object is usually determined by a number
72  /// of parameters whose value might have to be determined as part
73  /// of the overall solution (e.g. for geometric objects that represent
74  /// elastic walls). The geometric object
75  /// therefore has a vector of (pointers to) geometric Data,
76  /// which can be free/pinned and have a time history, etc. This makes
77  /// it possible to `upgrade' GeomObjects to GeneralisedElements -- in this
78  /// case the geometric Data plays the role of internal Data in the
79  /// GeneralisedElement. Conversely, FiniteElements, in which a geometry
80  /// (spatial coordinate) has been defined, inherit from GeomObjects,
81  /// which is particularly useful in FSI computations:
82  /// Meshing of moving domains is typically performed by representing the
83  /// domain as an object of type Domain and, by default, Domain boundaries are
84  /// represented by GeomObjects. In FSI computations, the boundary
85  /// of the fluid domain is represented by a number of solid mechanics elements.
86  /// These elements are, in fact, GeomObjects via inheritance
87  /// so that the we can use the standard interfaces of the GeomObject class
88  /// for mesh generation. An example is the class \c FSIHermiteBeamElement which
89  /// is derived from the class \c HermiteBeamElement (a `normal' beam element)
90  /// and the \c GeomObject class.
91  ///
92  /// The shape of a geometric object can have an explicit time-dependence, for
93  /// instance in cases where a domain boundary is performing
94  /// prescribed motions. We provide access to the `global'
95  /// time by giving the object a pointer to a timestepping scheme.
96  /// [Note that, within the overall FE code, time is only ever evaluated at
97  /// discrete instants (which are accessible via the timestepper),
98  /// never in continuous form]. The timestepper is also needed to evaluate
99  /// time-derivatives if the geometric Data carries a time history.

352  /// \short A geometric object may be composed of may sub-objects (e.g.
353  /// a finite-element representation of a boundary). In order to implement
354  /// sparse update functions, it is necessary to know the sub-object
355  /// and local coordinate within
356  /// that sub-object at a given intrinsic coordinate, zeta. Note that only
357  /// one sub-object can "cover" any given intrinsic position. If the position
358  /// is at an "interface" between sub-objects, either one can be returned.
359  /// The default implementation merely returns, the pointer to the "entire"
360  /// GeomObject and the coordinate, zeta
361  /// The optional boolean flag only applies if a Newton method is used to
362  /// find the value of zeta, and if true the value of the coordinate
363  /// s is used as the initial guess for the method. If the flag is false
364  /// (the default) a value of s=0 is used as the initial guess.
365  virtual void locate_zeta(const Vector<double> &zeta,
366  GeomObject* &sub_geom_object_pt,
367  Vector<double> &s,
368  const bool &use_coordinate_as_initial_guess=false)
369  {
370  //By default, the local coordinate is intrinsic coordinate
371  s = zeta;
372  //The sub_object is the entire object
373  sub_geom_object_pt = this;
374  }

376  /// \short A geometric object may be composed of many sub-objects
377  /// each with their own local coordinate. This function returns the
378  /// "global" intrinsic coordinate zeta (within the compound object), at
379  /// a given local coordinate s (i.e. the intrinsic coordinate of the
380  /// sub-GeomObject. In simple (non-compound) GeomObjects, the local
381  /// intrinsic coordinate is the global intrinsic coordinate
382  /// and so the function merely returns s. To make it less likely
383  /// that the default implementation is called in error (because
384  /// it is not overloaded in a derived GeomObject where the default
385  /// is not appropriate, we do at least check that s and zeta
386  /// have the same size if called in PARANOID mode.
387  virtual void interpolated_zeta(const Vector<double> &s, Vector<double> &zeta)
388  const
389  {
390 #ifdef PARANOID
391  if (zeta.size()!=s.size())
392  {
393  std::ostringstream error_message;
394  error_message
395  << "You've called the default implementation of "
396  << "GeomObject::interpolated_zeta() \n"
397  << "but zeta.size()=" << zeta.size()
398  << "and s.size()=" << s.size() << std::endl
399  << "This doesn't make sense! You probably have to \n"
400  << "overload this function in your specific GeomObject\n";
401  throw OomphLibError(error_message.str(),
402  OOMPH_CURRENT_FUNCTION,
403  OOMPH_EXCEPTION_LOCATION);
404  }
405 #endif
406  //By default the global intrinsic coordinate is equal to the local one
407  zeta = s;
408  }
