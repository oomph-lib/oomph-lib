125 /// \short This is a base class for all SolidFiniteElements
126 /// that participate in FSI computations. These elements
127 /// provide interfaces and generic funcionality for
128 /// the two additional roles that SolidFiniteElements play
129 /// in FSI problems:
130 /// -# They parameterise the domain boundary for the fluid domain.
131 /// To allow them to play this role, FSIWallElements are derived
132 /// from the SolidFiniteElement and the GeomObject class,
133 /// indicating that the every specific FSIWallElement must
134 /// implement the pure virtual function GeomObject::position(...)
135 /// which should compute the position vector to a point in the
136 /// SolidFiniteElement, parametrised by its local coordinates.
137 /// -# In FSI problems fluid exerts a traction onto the wall and this traction
138 /// must be added to any other load terms (such as an external pressure
139 /// acting on an elastic pipe) that are already applied to
140 /// the SolidFiniteElements by other means.
141 /// .
142 ///
143 /// The fluid-traction on the SolidFiniteElements
144 /// depends on the fluid variables (velocities and pressures) in
145 /// those fluid elements that are adjacent to the SolidFiniteElements'
146 /// Gauss points. In an FSI problem these velocities and pressures
147 /// are unknowns in the overall problem and the dependency of the
148 /// SolidFiniteElement's residual vector on these
149 /// unknowns must be taken into account when computing the element's
150 /// Jacobian matrix.
151 ///
152 /// For each Gauss point in the FSIWallElement, we therefore store:
153 /// - [a] pointer[s] to the FSIFluidElement[s] that is [are] "adjacent"
154 /// to the Gauss point in the FSIWallElement.
155 /// - the vector[s] of the local coordinates (in the fluid element[s])
156 /// that identify the point in the fluid element that is
157 /// (deemed to be) "opposite" that Gauss point. (Note
158 /// that we do not require the discretisations of the
159 /// fluid and solid domains to match exactly, therefore,
160 /// small "gaps" may occur between fluid and solid elements.)
161 /// .
162 ///
163 /// By default, each FSIWallElement is assumed to be exposed to fluid
164 /// loading only on one of its faces. For elements that are immersed
165 /// into fluid, so that a fluid traction is "exerted from both sides",
166 /// the element can store pointers to multiple adjacent fluid elements
167 /// (and local coordindates in these). This capability must be enabled
168 /// by a call to FSIWallElement::enable_fluid_loading_on_both_sides().
169 ///
170 /// Since the fluid traction can involve derivatives
171 /// of the velocity (think of Newtonian fluids), the traction is
172 /// also affected by changes in the nodal positions of the adjacent fluid
173 /// elements. Since fluid and solid discretisations are not
174 /// required to match, the nodal positions in an adjacent fluid
175 /// element can be affected by the positional variables in
176 /// another FSIWallElement. To capture this influence, we
177 /// provide the function FSIWallElement::node_update_adjacent_fluid_elements()
178 /// which does exactly what it says....
179 ///
180 /// Finally, since oomph-lib's fluid and solid
181 /// elements tend to employ different non-dimensionalisations
182 /// for the stresses, the fluid traction (computed by
183 /// the adjacent fluid element, on the fluid stress-scale) may have to be
184 /// scaled by the ratio \f$ Q \f$ of the stresses used to non-dimensionalise
185 /// the two sets of stresses. For instance, for a fluid stress
186 /// non-dimensionalisation based on the viscous scale \f$ \mu U / L\f$
187 /// (as in oomph-lib's Navier-Stokes elements) and a non-dimensionalisation
188 /// of the solid mechanics stresses, based on a Young's modulus \f$ E \f$,
189 /// as in oomph-lib's KirchhoffLoveBeamElements, the stress ratio is given
190 /// by
191 /// \f[ Q=\frac{\mu U}{LE} \f]
192 /// For other wall/fluid element combinations the definition of \f$ Q \f$
193 /// will differ -- check the documentation and/or implementation to see
194 /// which parameters are used to non-dimensionalise the stresses
195 /// in the respective elements!
196 ///
197 /// The function FSIWallElement::fluid_load_vector(...) computes
198 /// the fluid traction on the wall on the wall stress-scale.
199 /// This function may be called in the get_load (say) function of
200 /// a specific FSIWallElement to add the fluid load to the
201 /// other tractions that may already be applied to the element by other
202 /// means. By default a stress-ratio of \f$ Q = 1 \f$ is used
203 /// but this may be overwritten with the
204 /// access function FSIWallElement::q_pt().
