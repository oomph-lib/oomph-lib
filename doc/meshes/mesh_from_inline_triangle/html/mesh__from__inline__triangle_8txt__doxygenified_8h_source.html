3\mainpage Inline mesh generation and adaptation based on Triangle
4
5In this document we demonstrate how to generate unstructured
62D meshes for \c oomph-lib, using
7<A HREF="http://www.cs.berkeley.edu/~jrs/">Jonathan Shewchuk's</A>
8<A HREF="http://www.cs.cmu.edu/~quake/triangle.html"> Triangle</A>
9library from within an \c oomph-lib driver code.
10This complements the discussion in
11<A HREF="../../../../doc/meshes/mesh_from_triangle/html/index.html">
12another tutorial</a>
13where we illustrated how to build \c oomph-lib meshes using the
14output generated when <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">
15\c Triangle </A> is used to create the mesh externally.
16
17 In addition, we demonstrate
18- how to create meshes with polygonal or curvilinear boundaries
19.
20and
21- how to adapt (re-generate) these meshes based on spatial error estimates.
22.
23Here we restrict ourselves to the solution of problems in fixed
24domains. Other tutorials discuss more advanced applications involving
25moving meshes, e.g.
26- <A HREF="../../../../doc/navier_stokes/jeffery_orbit/html/index.html">
27 The motion of an ellipse in a shear flow</a>
28 \n\n
29- <A HREF=
30 "../../../../doc/navier_stokes/adaptive_bubble_in_channel/html/index.html">
31 The propagation of a bubble in a channel -- mesh generation and adaptation
32 for free-surface flows.</a>
33 \n\n

38\section overview Overview of mesh generation procedures

40\subsection polygonal Meshes with polygonal boundaries
41If the domain has polygonal outer (and possibly internal)
42boundaries, the mesh generation process follows a similar pattern
43to that required in the external use
44of <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">\c Triangle
45</A>, but using \c oomph-lib classes to represent the required data.
46We define the polygonal boundaries using \c
47TriangleMeshPolyLines,
48which represent distinct mesh boundaries and are defined by a series of
49vertex coordinates. Multiple \c TriangleMeshPolyLines can be
50combined to define
51(closed) \c TriangleMeshPolygons. These are then used
52to create an unstructured mesh with a pre-determined target area for
53all elements, via an interface to
54<A HREF="http://www.cs.cmu.edu/~quake/triangle.html">\c Triangle </A>.
55
56One particular feature of \c oomph-lib's interface
57to <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">\c Triangle </A>
58is that each closed \c TriangleMeshPolygon must contain at least two
59distinct \c TriangleMeshPolyLines, each with its own boundary ID.
60This is because every \c Node can only have a single-valued boundary
61coordinate but if the boundary is closed there would be a discontinuity
62in the coordinate value. (For example, a standard representation
63would have \f$ \zeta = [0,2\pi] \f$ , but the node at
64\f$\zeta=0\f$ should also have \f$ \zeta=2\pi \f$ and this is not possible.)
65The sketch below shows a representative domain
66as well as two legal and one illegal representations of the domain
67boundaries. Note that the boundaries can be enumerated in an arbitrary
68fashion.
69

77\subsection curvilinear Meshes with curvilinear boundaries
78It is also possible to discretise domains with curvilinear boundaries
79as shown in the sketch below. Assuming that each curvilinear boundary
80is represented by a \c GeomObject that specifies the position vector
81\f$ {\bf R}(\zeta) \f$ to a point on the curvilinear boundary as a
82function of some surface coordinate \f$ \zeta \f$, we split each
83closed boundary into (at least) two distinct \c TriangleMeshCurviLines --
84the curvilinear equivalents of \c TriangleMeshPolyLines.
85Each \c TriangleMeshCurviLine is constructed from a pointer to the \c
86GeomObject and the start and end values of the boundary coordinate \f$
87\zeta \f$
88along the relevant part of the curvilinear boundary.
89The \c TriangleMeshCurviLines are then combined to a
90\c TriangleMeshClosedCurve -- the general closed curve.
91
92
93\image html curvilinear_mesh.gif "Sketch of a domain bounded by a curvilinear boundary, containing two holes with curvilinear (hole 1) and polygonal (hole 2) boundaries, respectively. "
94\image latex curvilinear_mesh.eps "Sketch of a domain bounded by a curvilinear boundary, containing two holes with curvilinear (hole 1) and polygonal (hole 2) boundaries, respectively. " width=0.75\textwidth
95
96The mesh is then created in a two-stage process: All \c
97TriangleMeshCurviLines are sampled at a certain number of
98points (specified by the user in the constructor) to create the
99vertices for a polyline representation of the boundary.
100This polygonal representation of the boundaries is
101used by \c Triangle to generate the mesh. Finally, nodes on
102curvilinear boundaries are "snapped" onto the actual curvilinear
103boundary.

108\section adapt Overview of mesh adaptation methodology
109The methodology employed to adapt \c oomph-lib's unstructured meshes
110differs from that used for structured meshes. Specifically, rather than
111sub-dividing elements in which the error estimate exceeds a threshold
112and merging elements in which the solution is
113"too accurate", we completely re-generate the mesh and project
114the solution from the old to the new mesh. This is because we
115originally developed the underlying methodology to solve
116free-boundary problems in which the domain undergoes such large
117deformations that re-meshing is required. The ability to
118adjust the element sizes guided by spatial error estimates
119when re-meshing the domain is a simple fringe benefit.
120
121A number of issues are important:
122- To facilitate mesh adaptation in free-boundary problems, we
123 re-generate the polygonal representation of the boundary,
124 using the vertex nodes of the elements on the boundary to re-define
125 the polygon. The number of vertices in the polygons that define the
126 mesh boundaries will therefore generally change during the mesh
127 adaptation process. This is discussed in more detail in
128 <a href=
129 "../../../../doc/navier_stokes/adaptive_bubble_in_channel/html/index.html#adapt">
130 another tutorial.</a>\n\n
131- The mesh (and thus its constituent elements) are completely
132 re-generated when the mesh is adapted, and so it is necessary to "complete
133 the build" of all elements after each mesh adaptation. For instance,
134 pointers to problem parameters (Reynolds numbers, source functions,
135 etc) must be re-set after the adaptation since they cannot
136 (easily) be passed from the old to the new mesh.
137 \n\n
138- When projecting the solution from the old to the new mesh, we
139 project
140 \n\n
141 - all unknowns and their associated history values (if any)
142 \n\n
143 - the history values of the nodal positions. This is important for moving
144 mesh problems where the mesh velocity is required to evaluate
145 the ALE time-derivatives.
146 .
147 \n\n
148- The ability to automatically project the solution from the old to
149 the new mesh requires the elements, of type \c ELEMENT, say, to be wrapped
150 in the templated \c ProjectableElement<ELEMENT> class. This class
151 requires the specification of various element characteristics
152 (such as the number of fields to be projected, the number of history
153 values, etc) in the form of virtual functions. This is much more
154 straightforward than upgrading an existing element to become a refineable
155 element for use in an adaptive computation on a structured
156 mesh because the "mesh adaptation by mesh re-generation" avoids the
157 creation of hanging nodes. See the section \ref wrapper for a
158 slightly more detailed
159 discussion of this aspect.
160 \n\n
161- Note that we do \b not apply any boundary conditions
162 during the projection of these fields. This decision was not taken
163 out of laziness but because the interfaces required to specify
164 which boundary conditions to enforce and which ones to relax during
165 the projection were too unwieldy. It is therefore \b important
166 to re-apply boundary conditions and boundary values after each
167 adaptation.
168 \n\n
169.
170We recommend using the \c Problem::actions_after_adapt() function to
171re-assign boundary conditions and to complete the build of all
172elements after the adaptation.
173
174Apart from these issues, the user interfaces to the mesh adaptation
175functions are exactly the same as for structured meshes. Specifically,
176it is possible to specify maximum and minimum element sizes and
177target values for the error such that the mesh is refined in regions
178where the error estimate is ``too large'' and unrefined where it is
179``too small''.
180
181Typically, the most computationally expensive stage of the mesh
182regeneration procedure is the multi-domain setup procedure which identifies
183corresponding points in the old and new meshes. In "cheap" problems, such as
184the Poisson problem discussed below, the cost of the mesh regeneration
185can exceed the cost of the subsequent solve, but in most "hard"
186problems (such as the ones listed at the beginning of this tutorial)
187the cost of the mesh regeneration is modest (and, in the case
188of large-displacement free-boundary problems, unavoidable anyway).

193\section example An example: The adaptive solution of Poisson's equations on an unstructured mesh
194
195As an example we consider the adaptive solution of Poisson's equation
196\f[
197\frac{\partial^2 u}{\partial x_i^2} = f(x_1,x_2)
198\f]
199in a circular domain that contains an elliptical and a polygonal
200hole. As in many previous examples, we apply Dirichlet boundary
201conditions on all domain boundaries and choose the boundary
202values and the source function \f$ f(x_1,x_2) \f$ such that
203the exact solution of the problem is given by
204\f[
205u(x_1,x_2) = \tanh(1-\alpha(x_1 \tan\Phi - x_2)),
206\f]
207which approaches a step function, oriented at an angle \f$ \Phi \f$
208within the \f$ (x_1,x_2) \f$ plane, as \f$ \alpha \f$ becomes large.
209The figure below shows contour plots of the solution
210for \f$ \alpha = 5 \f$ for various angles \f$ \Phi \f$. It illustrates
211how the mesh adaptation adjusts the mesh such the smallest elements
212are located in the region where the solution undergoes rapid change.
213

369\subsection build_mesh Construct the mesh
370
371To facilitate the construction of the mesh \c TriangleMesh
372object we use the object \c TriangleMeshParameters.
373The only necessary argument for creating this object is the outer
374boundary. The definition of holes, internal boundaries and regions is
375explained in
376<A HREF="../../../../doc/meshes/mesh_from_inline_triangle_internal_boundaries/html/index.html">
377another tutorial</a>.
378The object can also be used to control whether
379additional refinement may be performed on the mesh boundaries. The
380default behaviour is that such refinement will occur so that
381the highest quality mesh is obtained. In some cases,
382e.g. periodic boundary conditions, you may wish to ensure that each
383input boundary segment corresponds to a single element edge in the
384final mesh. This can be achieved for the outer boundary by calling
385\c TriangleMeshParameters::disable_boundary_refinement(). However, Triangle
386will still add additional points to any internal boundaries unless the
387additional function
388\c TriangleMeshParameters::disable_internal_boundary_refinement() is
389also called.
390The functions \c TriangleMeshParameters::enable_boundary_refinement()
391and \c TriangleMeshParameters::enable_internal_boundary_refinement()
392can be used to return to the default behaviour. Note that it is not
393currently possible to suppress refinement on the internal boundaries,
394but refine the outer boundary.

464\subsection wrapper Upgrading elements to become "projectable"
465As discussed above, a key step in the "mesh-adaptation-by-mesh-regeneration"
466procedure is the projection of the solution from the old to the new mesh.
467The ability to perform this projection fully-automatically during the
468mesh adaptation requires the elements to be "wrapped" in the templated
469\c ProjectableElement<ELEMENT> class. This class is derived from the
470\c ProjectableElementBase base class which specifies a number of
471pure virtual functions that must be specified for each specific
472element type. For the Poisson problem considered here these functions
473are already provided in the \c ProjectablePoissonElement<ELEMENT>
474class. Similar wrappers exist for many other equations. If you
475want to "upgrade" your own elements to become projectable, inspect
476the prototypes for the relevant pure virtual functions
477which are defined in
478
479<CENTER>
480<A HREF="../../../../src/generic/projection.h">
481src/generic/projection.h
482</A>
483</CENTER>
484
485while the specific implementation for the projectable Poisson elements
486is provided in
487
488<CENTER>
489<A HREF="../../../../src/poisson/poisson_elements.h">
490src/poisson/poisson_elements.h
491</A>
492</CENTER>

505\subsection how_many_segments How many vertices should I use to sample my curvilinear boundary?
506
507As discussed above, meshes with curvilinear boundaries are created in
508a two-stage process. Initially, Triangle generates a mesh with
509polygonal boundaries using a user-specified number of vertices
510that are evenly distributed along the
511relevant \c GeomObject. The nodes on that boundary are then
512"snapped" onto the actual curvilinear boundary in a post-processing
513step. The decision of how many vertices to choose involves a
514compromise between two conflicting demands:
515- The number of boundary nodes created by Triangle will be at least
516 as big as the number of vertices specified. Triangle may add
517 additional boundary nodes to generate a mesh of sufficient quality
518 but it will not remove any vertices. Using a very large number of vertices
519 can therefore lead to unnecessarily fine meshes.
520 \n\n
521- If the number of vertices is too small, the polygonal representation
522 of the domain boundary may be a poor approximation to the actual
523 curvilinear boundary. Elements near such boundaries may become
524 (too) strongly distorted when nodes are "snapped" onto the
525 curvilinear boundary. This often
526 manifests itself in inverted elements. (An element is
527 considered inverted if the Jacobian of the mapping between local
528 and global coordinates becomes non-positive anywhere.
529 Note that negative Jacobians may
530 occur in the interior of elements (e.g. at their Gauss points) even
531 if a plot of the element, based on its nodal positions, still looks "OK").
532.

586\section curve_sections Appendix A: Generalization for polylines and curvilines
587
588The objects, \c TriangleMeshPolyLine and \c TriangleMeshCurviLine
589inherit the properties of a more general representation called
590\c TriangleMeshCurveSection. This allows one to define more general
591boundaries as a combination of \c TriangleMeshPolyLines and
592\c TriangleMeshCurviLines. Therefore, if we want to define a more general
593closed curve use the \c TriangleMeshClosedCurve object.
594You may notice the use of this object through the example code when defining the outer boundary.
595
