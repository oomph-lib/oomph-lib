3 \mainpage Inline mesh generation and adaptation based on Triangle
4 
5 In this document we demonstrate how to generate unstructured
6 2D meshes for \c oomph-lib, using
7 <A HREF="http://www.cs.berkeley.edu/~jrs/">Jonathan Shewchuk's</A>
8 <A HREF="http://www.cs.cmu.edu/~quake/triangle.html"> Triangle</A>
9 library from within an \c oomph-lib driver code.
10 This complements the discussion in
11 <A HREF="../../../../doc/meshes/mesh_from_triangle/html/index.html">
12 another tutorial</a>
13 where we illustrated how to build \c oomph-lib meshes using the
14 output generated when <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">
15 \c Triangle </A> is used to create the mesh externally.
16 
17  In addition, we demonstrate
18 - how to create meshes with polygonal or curvilinear boundaries
19 .
20 and
21 - how to adapt (re-generate) these meshes based on spatial error estimates.
22 .
23 Here we restrict ourselves to the solution of problems in fixed
24 domains. Other tutorials discuss more advanced applications involving
25 moving meshes, e.g.
26 - <A HREF="../../../../doc/navier_stokes/jeffery_orbit/html/index.html">
27  The motion of an ellipse in a shear flow</a>

29 - <A HREF=

40 \subsection polygonal Meshes with polygonal boundaries
41 If the domain has polygonal outer (and possibly internal)
42 boundaries, the mesh generation process follows a similar pattern
43 to that required in the external use
44 of <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">\c Triangle
45 </A>, but using \c oomph-lib classes to represent the required data.
46 We define the polygonal boundaries using \c
47 TriangleMeshPolyLines,
48 which represent distinct mesh boundaries and are defined by a series of
49 vertex coordinates. Multiple \c TriangleMeshPolyLines can be
50 combined to define
51 (closed) \c TriangleMeshPolygons. These are then used
52 to create an unstructured mesh with a pre-determined target area for
53 all elements, via an interface to
54 <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">\c Triangle </A>.
55 
56 One particular feature of \c oomph-lib's interface
57 to <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">\c Triangle </A>
58 is that each closed \c TriangleMeshPolygon must contain at least two
59 distinct \c TriangleMeshPolyLines, each with its own boundary ID.
60 This is because every \c Node can only have a single-valued boundary
61 coordinate but if the boundary is closed there would be a discontinuity
62 in the coordinate value. (For example, a standard representation
63 would have \f$ \zeta = [0,2\pi] \f$ , but the node at
64 \f$\zeta=0\f$ should also have \f$ \zeta=2\pi \f$ and this is not possible.)
65 The sketch below shows a representative domain
66 as well as two legal and one illegal representations of the domain
67 boundaries. Note that the boundaries can be enumerated in an arbitrary
68 fashion.

77 \subsection curvilinear Meshes with curvilinear boundaries
78 It is also possible to discretise domains with curvilinear boundaries
79 as shown in the sketch below. Assuming that each curvilinear boundary
80 is represented by a \c GeomObject that specifies the position vector
81 \f$ {\bf R}(\zeta) \f$ to a point on the curvilinear boundary as a
82 function of some surface coordinate \f$ \zeta \f$, we split each
83 closed boundary into (at least) two distinct \c TriangleMeshCurviLines --
84 the curvilinear equivalents of \c TriangleMeshPolyLines.
85 Each \c TriangleMeshCurviLine is constructed from a pointer to the \c
86 GeomObject and the start and end values of the boundary coordinate \f$
87 \zeta \f$
88 along the relevant part of the curvilinear boundary.
89 The \c TriangleMeshCurviLines are then combined to a
90 \c TriangleMeshClosedCurve -- the general closed curve.
91 
92 
93 \image html curvilinear_mesh.gif "Sketch of a domain bounded by a curvilinear boundary, containing two holes with curvilinear (hole 1) and polygonal (hole 2) boundaries, respectively. "
94 \image latex curvilinear_mesh.eps "Sketch of a domain bounded by a curvilinear boundary, containing two holes with curvilinear (hole 1) and polygonal (hole 2) boundaries, respectively. " width=0.75\textwidth
95 
96 The mesh is then created in a two-stage process: All \c
97 TriangleMeshCurviLines are sampled at a certain number of
98 points (specified by the user in the constructor) to create the
99 vertices for a polyline representation of the boundary.
100 This polygonal representation of the boundaries is
101 used by \c Triangle to generate the mesh. Finally, nodes on
102 curvilinear boundaries are "snapped" onto the actual curvilinear
103 boundary.

108 \section adapt Overview of mesh adaptation methodology
109 The methodology employed to adapt \c oomph-lib's unstructured meshes
110 differs from that used for structured meshes. Specifically, rather than
111 sub-dividing elements in which the error estimate exceeds a threshold
112 and merging elements in which the solution is
113 "too accurate", we completely re-generate the mesh and project
114 the solution from the old to the new mesh. This is because we
115 originally developed the underlying methodology to solve
116 free-boundary problems in which the domain undergoes such large
117 deformations that re-meshing is required. The ability to
118 adjust the element sizes guided by spatial error estimates
119 when re-meshing the domain is a simple fringe benefit.
120 
121 A number of issues are important:
122 - To facilitate mesh adaptation in free-boundary problems, we
123  re-generate the polygonal representation of the boundary,
124  using the vertex nodes of the elements on the boundary to re-define
125  the polygon. The number of vertices in the polygons that define the
126  mesh boundaries will therefore generally change during the mesh
127  adaptation process. This is discussed in more detail in
128  <a href=
129  "../../../../doc/navier_stokes/adaptive_bubble_in_channel/html/index.html#adapt">
130  another tutorial.</a>\n\n
131 - The mesh (and thus its constituent elements) are completely
132  re-generated when the mesh is adapted, and so it is necessary to "complete
133  the build" of all elements after each mesh adaptation. For instance,
134  pointers to problem parameters (Reynolds numbers, source functions,
135  etc) must be re-set after the adaptation since they cannot
136  (easily) be passed from the old to the new mesh.
137  \n\n
138 - When projecting the solution from the old to the new mesh, we
139  project
140  \n\n
141  - all unknowns and their associated history values (if any)
142  \n\n
143  - the history values of the nodal positions. This is important for moving
144  mesh problems where the mesh velocity is required to evaluate
145  the ALE time-derivatives.
146  .
147  \n\n
148 - The ability to automatically project the solution from the old to
149  the new mesh requires the elements, of type \c ELEMENT, say, to be wrapped
150  in the templated \c ProjectableElement<ELEMENT> class. This class
151  requires the specification of various element characteristics
152  (such as the number of fields to be projected, the number of history
153  values, etc) in the form of virtual functions. This is much more
154  straightforward than upgrading an existing element to become a refineable
155  element for use in an adaptive computation on a structured
156  mesh because the "mesh adaptation by mesh re-generation" avoids the
157  creation of hanging nodes. See the section \ref wrapper for a
158  slightly more detailed
159  discussion of this aspect.
160  \n\n
161 - Note that we do \b not apply any boundary conditions
162  during the projection of these fields. This decision was not taken
163  out of laziness but because the interfaces required to specify
164  which boundary conditions to enforce and which ones to relax during
165  the projection were too unwieldy. It is therefore \b important
166  to re-apply boundary conditions and boundary values after each
167  adaptation.
168  \n\n
169 .
170 We recommend using the \c Problem::actions_after_adapt() function to
171 re-assign boundary conditions and to complete the build of all
172 elements after the adaptation.
173 
174 Apart from these issues, the user interfaces to the mesh adaptation
175 functions are exactly the same as for structured meshes. Specifically,
176 it is possible to specify maximum and minimum element sizes and
177 target values for the error such that the mesh is refined in regions
178 where the error estimate is ``too large'' and unrefined where it is
179 ``too small''.
180 
181 Typically, the most computationally expensive stage of the mesh
182 regeneration procedure is the multi-domain setup procedure which identifies
183 corresponding points in the old and new meshes. In "cheap" problems, such as
184 the Poisson problem discussed below, the cost of the mesh regeneration
185 can exceed the cost of the subsequent solve, but in most "hard"
186 problems (such as the ones listed at the beginning of this tutorial)
187 the cost of the mesh regeneration is modest (and, in the case
188 of large-displacement free-boundary problems, unavoidable anyway).

193 \section example An example: The adaptive solution of Poisson's equations on an unstructured mesh
194 
195 As an example we consider the adaptive solution of Poisson's equation
196 \f[
197 \frac{\partial^2 u}{\partial x_i^2} = f(x_1,x_2)
198 \f]
199 in a circular domain that contains an elliptical and a polygonal
200 hole. As in many previous examples, we apply Dirichlet boundary
201 conditions on all domain boundaries and choose the boundary
202 values and the source function \f$ f(x_1,x_2) \f$ such that
203 the exact solution of the problem is given by
204 \f[
205 u(x_1,x_2) = \tanh(1-\alpha(x_1 \tan\Phi - x_2)),
206 \f]
207 which approaches a step function, oriented at an angle \f$ \Phi \f$
208 within the \f$ (x_1,x_2) \f$ plane, as \f$ \alpha \f$ becomes large.
209 The figure below shows contour plots of the solution
210 for \f$ \alpha = 5 \f$ for various angles \f$ \Phi \f$. It illustrates
211 how the mesh adaptation adjusts the mesh such the smallest elements
212 are located in the region where the solution undergoes rapid change.

464 \subsection wrapper Upgrading elements to become "projectable"
465 As discussed above, a key step in the "mesh-adaptation-by-mesh-regeneration"
466 procedure is the projection of the solution from the old to the new mesh.
467 The ability to perform this projection fully-automatically during the
468 mesh adaptation requires the elements to be "wrapped" in the templated
469 \c ProjectableElement<ELEMENT> class. This class is derived from the
470 \c ProjectableElementBase base class which specifies a number of
471 pure virtual functions that must be specified for each specific
472 element type. For the Poisson problem considered here these functions
473 are already provided in the \c ProjectablePoissonElement<ELEMENT>
474 class. Similar wrappers exist for many other equations. If you
475 want to "upgrade" your own elements to become projectable, inspect
476 the prototypes for the relevant pure virtual functions
477 which are defined in
478 
479 <CENTER>
480 <A HREF="../../../../src/generic/projection.h">
481 src/generic/projection.h
482 </A>
483 </CENTER>
484 
485 while the specific implementation for the projectable Poisson elements
486 is provided in
487 
488 <CENTER>
489 <A HREF="../../../../src/poisson/poisson_elements.h">

505 \subsection how_many_segments How many vertices should I use to sample my curvilinear boundary?
506 
507 As discussed above, meshes with curvilinear boundaries are created in
508 a two-stage process. Initially, Triangle generates a mesh with
509 polygonal boundaries using a user-specified number of vertices
510 that are evenly distributed along the
511 relevant \c GeomObject. The nodes on that boundary are then
512 "snapped" onto the actual curvilinear boundary in a post-processing
513 step. The decision of how many vertices to choose involves a
514 compromise between two conflicting demands:
515 - The number of boundary nodes created by Triangle will be at least
516  as big as the number of vertices specified. Triangle may add
517  additional boundary nodes to generate a mesh of sufficient quality
518  but it will not remove any vertices. Using a very large number of vertices
519  can therefore lead to unnecessarily fine meshes.
520  \n\n
521 - If the number of vertices is too small, the polygonal representation
522  of the domain boundary may be a poor approximation to the actual
523  curvilinear boundary. Elements near such boundaries may become
524  (too) strongly distorted when nodes are "snapped" onto the
525  curvilinear boundary. This often
526  manifests itself in inverted elements. (An element is
527  considered inverted if the Jacobian of the mapping between local
528  and global coordinates becomes non-positive anywhere.
529  Note that negative Jacobians may
530  occur in the interior of elements (e.g. at their Gauss points) even
531  if a plot of the element, based on its nodal positions, still looks "OK").
532 .
