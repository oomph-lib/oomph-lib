3 \mainpage Linking VMTK with oomph-lib
4 
5 The <a href="http://www.vmtk.org">Vascular
6 Modeling Toolkit</a> is a collection of libraries
7 and tools for the 3D reconstruction, geometric analysis, mesh
8 generation and surface data analysis for image-based modelling
9 of blood vessels.
10 
11 This tutorial demonstrates how to open your dataset in VMTK; navigate
12 around a 3D volume; reconstruct the 3D surface of a vascular segment from
13 CT or MR images; process a surface model to generate a mesh; and
14 how to extract the surface mesh. Finally, we demonstrate
15 how to use \c oomph-lib's conversion code
16 \c create_fluid_and_solid_surface_mesh_from_fluid_xda_mesh
17 to generate fluid and solid meshes that allow the simulation of
18 physiological fluid-structure interaction problems in which the
19 (assumed constant thickness) vessel wall deforms in response to the fluid
20 traction.
21 
22 Before starting, make sure that you have VMTK installed on your machine
23 and that you have your DICOM (Digital Imaging and Communications in Medicine)
24 images. Note that this tutorial describes the use of VMTK version 0.8.
25 
26 For further information on VMTK, please refer to VMTK's
27 <a href="http://www.vmtk.org">home page</a>.

54 The generation of high-quality computational meshes from medical
55 images is a non-trivial task. The
56 <a href="http://www.vmtk.org"> Vascular Modeling
57 Toolkit</a> greatly simplifies this task but
58 it is important to remember that (intelligent) user input is required
59 at various stages of the process.
60 
61  This tutorial does \b not provide
62 any guidance on this aspect. We simply demonstrate \b how to generate
63 meshes that can be used in \c oomph-lib - based
64 physiological flow (or fluid-structure interaction) simulations.
65 If you smooth the vessels walls too much, you may lose essential
66 features; if you smooth too little, the shape of the vessel wall
67 may be polluted by imaging artifacts; etc.

83 \code
84  vmtkimagereader -f dicom -d dicom_directory --pipe vmtkimageviewer

96 
97  - Rotate the volume by left-clicking anywhere on the render window
98 (outside the image).

151 \subsection surface The 3D surface reconstruction of a vascular segment
152 
153 VMTK is capable of generating an image based surface model from a \c vti
154 file (obtained using the procedure explained in the
155 section \ref voi above), using the Level Set method and can then
156 extract the surface model using a Marching Cubes algorithm.
157 
158 First, enter:
159 \code
160 vmtklevelsetsegmentation -ifile image_volume_voi.vti -ofile level_sets.vti
161 \endcode
162 
163 \image html voi_display.gif " "
164 \image latex voi_display.eps " " width=0.75\textwidth
165 
166 When the VOI is displayed in a render window, press \c q or \c e to proceed.
167 
168 A message will appear on your terminal:
169 
170 \code
171  Please choose initialization type: (0: colliding fronts; 1: fast
172  marching; 2: threshold; 3: isosurface)
173 \endcode
174 
175 This lets you choose the initialization method used to create the
176 model of the blood vessel. In this tutorial, we will only demonstrate
177 the \c colliding \c fronts method. In this method two seeds are placed on
178 the image. Two fronts are then propagated from the seeds (one front
179 from each) with their speeds proportional to the image intensity.
180 The region where the two fronts cross (or collide), defines the
181 "deformable model" -- the initial representation of the vessel
182 volume. For further information on this and the other
183 initialization methods, please refer to VMTK's
184 <a href="http://www.vmtk.org">tutorial page</a>.
185 
186 Now, enter \c 0 to initialise with \c colliding \c fronts.
187 
188 A message will then appear on the terminal:
189 \code
190  Please input lower threshold ('i' to activate image, 'n' for none):
191 \endcode
192 
193 Wave propagation, used in the \c colliding \c fronts method, can be
194 restricted to a set of intensity levels between two thresholds.
195 Enter the lower threshold, if you want to use
196 one, otherwise enter \c n. Note that if you don't know the appropriate
197 threshold value you can press \c i to activate the image
198 and probe it, then quit
199 with \c q or \c e when probing is done.
200 
201 For our example there is no need for thresholds. Therefore enter \c n.
202 
203 The next message is:
204 \code
205  Please input upper threshold ('i' to activate image, 'n' for none):
206 \endcode
207 
208 We repeat the same steps as for the lower threshold.
209 
210 Next, this message appears:
211 \code
212  Please place two seeds (click on the image while pressing Ctrl).
213 \endcode
214 
215 The render window is now activated. Place the two seeds, one
216 at each of the two ends of a branch. (The geometry of the entire bifurcation
217 is created by "merging" the surfaces of individual branches.)
218 Interact with the image planes to find where you want to place your
219 first seed. When
220 satisfied, left-click on the image while pressing \c Ctrl. A red sphere
221 will then appear.
222 
223 \image html seed1.gif " "
224 \image latex seed1.eps " " width=0.75\textwidth
225 
226 Repeat this procedure for the second seed.
227 Press \c q or \c e when done.
228 
229 \image html seed2.gif " "
230 \image latex seed2.eps " " width=0.75\textwidth
231 
232 Provided everything went OK, a translucent surface will appear in the
233 render window between the two seeds. This is your initial deformable
234 model.
235 
236 \image html deformable_model1.gif " "
237 \image latex deformable_model1.eps " " width=0.75\textwidth
238 
239 Press \c q or \c e to proceed.
240 
241 Next, you will be prompted with:
242 \code
243  Accept initialization? (y/n):
244 \endcode
245 
246 If you are not satisfied with your initialisation, enter \c n to perform
247 it once again, otherwise enter \c y.
248 
249 Now press \c q or \c e again (in fact, you should do this whenever
250 the system appears to be frozen and the message "Displaying" is
251 shown on the terminal.).
252 
253 You are now prompted if you want to initialise another branch:
254 
255 \code
256 Initialize another branch? (y/n):
257 \endcode
258 
259 Press \c n -- we will add the other branches later.
260 
261 The following message will now appear:
262 
263 \code
264 Please input parameters (type return to accept current values, 'e' to
265 end, 'q' to quit):
266  NumberOfIterations(0) [PropagationScaling(1.0) CurvatureScaling(0.0)
267 AdvectionScaling(0.0)]:
268 \endcode
269 
270 These parameters control the deformation of your Level Set:
271 
272 - Number of iterations is the number of deformation steps
273  the model will perform.
274 
275 - Propagation scaling is the weight you assign to model inflation.
276 
277 - Curvature scaling is the weight you assign to model surface
278  regularisation (this will eventually make the model collapse and
279  vanish if it is too large).
280 
281 - Advection scaling regulates the attraction of the surface of the
282  image gradient modulus ridges.
283 
284 Based on our limited experience, we recommended that propagation and curvature
285 should be set to \c 0.0, and advection to \c 1.0. The number of iterations
286 should be set large enough for the level set not to move anymore
287 (if the region is not too big, try with \c 300). We encourage you to
288 experiment with these parameters yourself.
289 
290 Therefore, enter:
291 \code
292  300 0 0 1
293 \endcode
294 
295 The level set will then evolve until the maximum number of iterations is
296 reached. Then the render window will activate, displaying the final
297 model.
298

302 To quit the render window press \c q or \c e as usual.
303 
304 You will then be asked:
305 \code
306  Accept result? (y/n)
307 \endcode
308 
309 If you are not satisfied with your final model, enter \c n to go back to
310 the level set parameter question. Otherwise, enter \c y.
311 
312 Let us assume we are not satisfied, therefore enter \c n. You will
313 then go back to:
314 
315 \code
316 Please input parameters (type return to accept current values, 'e' to
317 end, 'q' to quit):
318  NumberOfIterations(0) [PropagationScaling(1.0) CurvatureScaling(0.0)
319 AdvectionScaling(0.0)]:
320 \endcode
321

323 \code
324  300 0 0.5 1

429 \subsection smooth Smoothing the surface
430 
431 
432 At this point, we have our surface model and we want to generate a
433 computational mesh. In most cases, the surface model has bumpy
434 surfaces, especially if the image quality is not high and if we didn't
435 use the curvature term in the level set evolution. Artificial
436 bumps in the surface can result in spurious flow features and will affect
437 the wall shear stress distribution, so one may want to increase surface
438 smoothness before building the mesh.
439 
440 You can do this in VMTK with :
441 \code
442  vmtksurfacesmoothing -ifile model.vtp -passband 0.1 -iterations 30 \
443 -ofile model_sm.vtp
444 \endcode
445 
446 There are two parameters controlling the amount of smoothing:
447 passband, which is the cut-off spatial frequency of the low pass
448 filter, and iterations, which is the number of smoothing passes.
449 
450 For typical vessels, a passband of 0.1 and a number of iteration of 30
451 should be OK. For example, for the surface model obtained in the
452 section \ref surface above, the resulting surface is:
453 
454 \image html smooth0130.gif " "
455 \image latex smooth0130.eps " " width=0.75\textwidth
456 
457 Remember that we can display the surfaces with :
458 \code
459 vmtksurfacereader -ifile model_sm.vtp --pipe vmtksurfaceviewer
460 \endcode
461 
462 Smoothing with a passband of 0.01 turns this into:
463 
464 \image html smooth00130.gif " "
465 \image latex smooth00130.eps " " width=0.75\textwidth
466 
467 If you want more smoothing, you can increase the passband and/or the
468 number of iterations, but be careful not to remove all surface features
469 by smoothing too much. Also, watch the apex of bifurcations since
470 its curvature may decrease resulting in a shallower apex, affecting
471 the simulated haemodynamics. For example, with a passband of \c 0.001 and
472 a number of iterations of \c 100, the result is:
473 
474 \image html smooth0001100.gif " "
475 \image latex smooth0001100.eps " " width=0.75\textwidth

520 \subsection add_extensions Adding flow extensions
521 
522 VMTK provides the option to generate cylindrical extensions to the
523 inlet and outlet cross-sections. These are typically used to
524 ensure that in the numerical simulation, the boundary conditions
525 are applied at a reasonable distance from the region of interest
526 and do not have a strong effect on the flow field. Exactly what sort
527 of boundary conditions to apply and where to apply them is, of course,
528 a non-trivial problem. Nonetheless, the appropriate
529 VMTK command to add the cylindrical extensions is:
530 
531 \code
532 vmtksurfacereader -ifile model_cl.vtp --pipe vmtkcenterlines \
533 -seedselector openprofiles --pipe vmtkflowextensions -adaptivelength 1 \
534 -extensionratio 8 -normalestimationratio 1 -interactive 0 --pipe \
535 vmtksurfacewriter -ofile model_ex.vtp
536 \endcode
537 
538 where the control parameters are:
539 
540 - \c adaptivelength (here set to 1) is a boolean flag. If set
541  to 1, the length of each flow extension is made proportional to the mean
542  radius of the vessel
543 
544 - \c extensionratio (here set to 8) is the proportionality factor of
545 the length of each flow extension.
546 
547 - \c normalestimationratio: We suggest setting this parameter to 1. For further
548  information, please refer to VMTK's
549  <a href="http://www.vmtk.org">tutorial page</a>.
550 
551 - The flag \c -interactive was set to \c 0: This means that
552  \c vmtkflowextensions will not prompt the user about what inlet or outlet
553  to extend, but it will perform the task on all the available open
554  boundaries. If you use \c -interactive \c 1, VMTK prompts the user
555  to specify which boundaries to extend through a graphical window.
556 
557 When the render window pops up, before pressing \c q, make sure you note
558 the different ids of the inlets and outlets. For our example, we see
559 in the figure below that we have one inlet with an id 0 and two
560 outlets with the ids 1 and 2.
561 
562 \image html extension1.gif " "
563 \image latex extension1.eps " " width=0.75\textwidth
564 
565 
566 A message will appear on your terminal:
567 \code
568 Please input list of inlet profile ids:
569 \endcode
570 
571 Enter the list of ids with a space between them. In our example enter
572 \c 0 . Next you will receive:
573 \code
574 Please input list of outlet profile ids:
575 \endcode
576 
577 Enter the list of ids with a space separating them. In our
578 example enter:
579 \code
580 1 2
581 \endcode
582 
583 You can display the resulting surface using:
584 \code
585 vmtksurfacereader -ifile model_ex.vtp --pipe vmtksurfaceviewer
586 \endcode
587 
588 The result is:
589 
590 \image html extenstion_display.gif " "

611 \section oomph_mesh Creating an oomph-lib mesh based on output files generated by VMTK
612 
613 VMTK exports the mesh for different solvers. The one we chose is the
614 ".xda" format (of <a href="http://libmesh.sourceforge.net/">LibMesh</a>).
615 To export the mesh in this format enter:
616 
617 \code
618  vmtkmeshwriter -ifile mesh.vtu -entityidsarray CellEntityIds -ofile \
619 iliac.xda
620 \endcode
621 
622 \c oomph-lib provides a conversion code
623 \code
624 bin/create_fluid_and_solid_surface_mesh_from_fluid_xda_mesh
625 \endcode
626 
627 which uses the VMTK output mesh in "*.xda" format to generate a file
628 "fluid_*.poly" that can be used to generate the fluid mesh
629 using the open-source mesh generator
630 <a href="http://wias-berlin.de/software/tetgen//">TetGen</a>.
631 
632 When executing the conversion code you will be prompted with:
633 \code
634 Please enter the file name without the file extension '.xda':
635 \endcode
636 
637 In our example, enter \c iliac .
638 
639 To facilitate the simulation of physiological fluid-structure
640 interaction problems in which the vessel walls deform elastically
641 in response to the fluid traction, the conversion
642 code also generates a file "solid_*.poly" that can be used to
643 to generate a mesh for the 3D vessel wall which is assumed to be of
644 constant thickness.
645 
646 The conversion code therefore requests the wall thickness:
647 
648 \code
649 Enter the (uniform) wall thickness
650 \endcode
651 
652 
653 As explained in
654 <a href="../../../interaction/unstructured_three_d_fsi/html/index.html">another
655 tutorial</a>, the consistent generation of surface coordinates in FSI problems
656 requires that all faces with the same boundary ID should be co-planar.
657 For this reason the conversion code provides the option to assign a distinct
658 boundary ID to each surface triangle on the fluid-structure
659 interaction boundary and ensures the IDs are consistent for the fluid and the
660 solid meshes. (The "*.poly" files generated by the conversion code list
661 the relation between the old and boundary IDs at the end.)
662 
663 \code
664 Do you want to create a separate ID for each planar facet
665 on the fluid-structure interaction boundary?
666 Enter y or n:
667 \endcode
668 
669 For single-physics problem, however, it is much easier to assign a
670 single boundary ID to each "physical" surface (<em> i.e. </em>
671 inlet, outlet, vessel boundary). Assuming that we are solving such a
672 single-physics problem we enter \c n.
673 Two files are now generated : \c fluid_iliac.poly
674 which contains the fluid domain and \c solid_iliac.poly containing the
675 solid one.
676

717 \section comm_and_ex Comments and Exercises
718 
719 - In its current form, the conversion code
720  <CENTER>
721  <a href="../../../../demo_drivers/meshing/mesh_from_vmtk/create_fluid_and_solid_surface_mesh_from_fluid_xda_mesh.cc">demo_drivers/interaction/create_fluid_and_solid_surface_mesh_from_fluid_xda_mesh.cc</a>
722  </CENTER>
723  is restricted to bifurcation-like geometries, and makes certain
724  assumptions about the enumeration of the boundaries.
725  Furthermore, it can only deal with constant thickness vessel
726  walls: the outer surface of the blood vessel is assumed to be
727  parallel to the FSI interface, with the wall thickness being
728  measured in the direction normal to that interface. This simplistic
729  construction can easily create overlapping elements if the
730  apex of the bifurcation is highly curved and/or if a large
731  wall thickness is requested. If this happens tetgen will be
732  unable to generate the solid mesh. It should be easy to modify the
733  code to deal with such cases, however.
734  <a href="../../../contact/html/index.html">Get in touch</a> if you
735  need help with this (and feel free to share any improvements
736  with us).
737 - We provide three separate tutorials to demonstrate the use of
738  VMTK-based meshes in \c oomph-lib driver codes:
739  - <a href="../../../solid/vmtk_solid/html/index.html">The inflation
740  of the arterial bifurcation by a constant pressure -- a pure solid
741  mechanics problem.</a>
742  - <a href="../../../navier_stokes/vmtk_fluid/html/index.html">Steady
743  finite-Reynolds number flow through the (rigid) arterial bifurcation --
744  a pure fluid-mechanics mechanics problem.</a>
745  - <a href="../../../interaction/vmtk_fsi/html/index.html">Steady
746  finite-Reynolds-number flow through an elastic
747  arterial bifurcation -- a fluid-structure interaction
748  problem.</a>
749  .

751  \c xda file that was used to create the meshes in the above
752  examples. It is available at:
