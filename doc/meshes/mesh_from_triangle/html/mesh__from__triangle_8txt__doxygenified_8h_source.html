3 \mainpage Mesh generation based on Triangle
4 
5 In this document we demonstrate how to generate unstructured
6 triangular meshes for \c oomph-lib, based on the output from
7 <A HREF="http://www.cs.berkeley.edu/~jrs/">Jonathan Shewchuk's</A>
8 open-source mesh generator
9 <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">\c Triangle.</A>
10 The mesh generation is performed in a two-stage process.
11 First we use <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">
12 \c Triangle </A> to generate the mesh "offline". Then we
13 process the output files generated by
14 <A HREF="http://www.cs.cmu.edu/~quake/triangle.html"> \c Triangle </A>
15 to generate an \c oomph-lib mesh.
16 
17 You should also consult
18 <A HREF="../../../../doc/meshes/mesh_from_inline_triangle/html/index.html">
19 another tutorial</a> where we discuss how to generate (and adapt) unstructured
20 meshes from within \c oomph-lib driver codes.

25 \section triangle_use Quick Guide for the use of Triangle
26 
27 The <A HREF="http://www.cs.cmu.edu/~quake/triangle.html"> \c Triangle
28 home page</A> contains a comprehensive User's Guide for the
29 code and its many options, therefore we only present a brief
30 overview of the code's most basic usage.
31 
32 
33 <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">
34 \c Triangle </A> creates the mesh based on the information
35 about the mesh boundaries provided in an input file,
36 \c filename.poly, say. By default, three output files,
37 \c filename.1.poly, \c filename.1.node, and \c filename.1.ele
38 are created. They contain the information about the polygonal
39 mesh boundaries, the nodal positions and the element connectivity
40 lists, respectively.

45 \subsection input The input file format
46 The input file for <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">
47 \c Triangle </A> (usually a file with the extension \c *. \c poly)
48 has the following format:
49 
50 <TABLE>
51 <TR>
52 <TD>
53 \b Nodes:
54 
55 First line: [number of vertices] [dimension (must be 2)]
56 [number of attributes for nodes] [number of boundary markers for nodes (0 or 1)] \n
57 Following lines: [vertex index] [x] [y] [[attributes]] [[boundary
58 marker]] \n
59 
60 \b Segments:
61 
62 One line: [number of segments] [number of boundary markers for
63 segments (0
64 or 1)] \n
65 Following lines: [segment index] [endpoint] [endpoint] [[boundary
66 marker]] \n
67 
68 \b Holes:
69 
70 One line: [number of holes] \n
71 Following lines: [hole index] [x] [y] \n
72 
73 </TD>
74 </TR>
75 </TABLE>
76 
77 \b Comments:
78 - The data between [ ] must be provided.
79 
80 - The attributes and boundary markers (i.e. data surrounded by [[ ]] )
81  must only be specified if the
82  corresponding number of boundary markers and the number of
83  attributes specified at the beginning of the relevant block is
84  nonzero.
85 
86 - The "segments" define boundary edges, i.e. exterior boundaries and
87  boundaries of holes. All boundary edges should be specified as segments,
88  otherwise \c Triangle may triangulate regions that
89  are not part of the domain.
90 
91 - Boundary markers should be used to identify which nodes are located
92  on which domain boundaries. If domain boundaries are to be
93  identified (this is strongly recommended as \c oomph-lib driver
94  codes tend to require this information to apply boundary
95  conditions), the number of boundary markers should be set to 1,
96  otherwise it must be set to zero.
97 
98 - If the number of boundary markers is set to 1, boundary markers must
99  be specified for every node or segment. To be consistent
100  with \c oomph-lib's (zero-based) boundary numbering schemes,
101  a boundary marker 0 should be used for nodes that are not located
102  on domain boundaries; a boundary marker \c b+1 should be
103  used to indicate that a node is located on the mesh boundary \c b
104  in the final \c oomph-lib mesh.
105 
106 - \c oomph-lib does not currently use the "attributes" so their
107  number should be set to zero. [\b Note: The attributes are likely
108  to be used to define boundary coordinates in future releases
109  of \c oomph-lib.]
110 
111 - If a node is located on multiple boundaries,
112  <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">
113  \c Triangle </A> assigns the boundary marker
114  for the node arbitrarily.
115 
116 - It is important to specify the boundary markers via the segments
117  rather than the nodes if the mesh contains multiple boundaries.
118 
119 - Each hole is identified by the coordinates of a single point in its
120  interior.
121 
122 - Blank lines and comments prefixed by `#' may be placed anywhere.
123 
124 - Input files for \c Triangle may contain optional additional lines
125  which specify, e.g. area constraints. See the
126  <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">
127  \c Triangle home page</A> for further information.
128 .

132 \subsection run_triangle How to run Triangle
133 
134 To create the mesh from a given input file the command is
135 
136 \code
137 ./triangle filename.poly
138 \endcode
139 
140 If the domain contains a hole the argument \c -pc must be added, i.e.
141 
142 \code
143 ./triangle -pc filename.poly
144 \endcode
145 
146 With these commands, \c Triangle will generate as few triangles
147 as possible. Finer meshes may be generated by imposing additional
148 constraints via command line arguments. For instance:
149 - A maximum triangle area can be specified with
150  <CODE>-a</CODE><EM>n</EM> where <EM>n</EM> is the maximum permitted area.
151  (There is no space between \c -a and the number specifying the area!)
152 - A minimum angle for the triangles can be specified with
153  \c -q to avoid angles smaller than 20 degree or
154  <CODE>-q</CODE><EM>n</EM> where <EM>n</EM> is the minimum permitted angle.
155 - ...
156 .
157 Again, we refer to the
158 <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">
159 \c Triangle home page</A> for a comprehensive listing of all
160 available options.
161 
162 When run with an input file \c filename.poly,
163 <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">
164 \c Triangle </A> generates output files called
165 \c filename.1.node, \c filename.1.ele and \c filename.1.node
166 which can be processed to generate an \c oomph-lib mesh.

297 
298 \subsection check Checking the boundary numbers
299 We re-iterate that
300 <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">\c Triangle </A>
301 does not allow nodes to be located on multiple boundaries. It is
302 therefore important to check the boundary numbers allocated by
303 <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">\c Triangle </A>,
304 e.g. by using the function \c Mesh::output_boundaries(...). Boundary
305 nodes should always be placed on the boundary with the most
306 restrictive boundary conditions. If this is not possible,
307 some post-processing of the mesh may be required.
308 
309 
310 \subsection elements Higher-order triangles
311 Currently, \c TriangleMesh can be used to generate
312 three, six and ten-node triangles (i.e. triangles with bi-linear,
313 bi-quadratic and bi-cubic shape functions). The generation of
314 ten-node triangles is currently performed somewhat inefficiently
315 and a warning is issued. Developing a more efficient implementation
316 should be straightforward and you are invited to perform this
317 as an exercise.
