3 \mainpage Demo problem: The two-dimensional Poisson problem with flux boundary conditions revisited -- multiple meshes
4 
5 In this document, we discuss an alternative approach for solving the
6 2D Poisson problem:
7 <CENTER>
8 <TABLE>
9 <TR>
10 <TD>
11 <CENTER>
12 <B>Two-dimensional model Poisson problem with Neumann boundary conditions</B>
13 </CENTER>
14 Solve
15 \f[
16 \sum_{i=1}^2 \frac{\partial^2u}{\partial x_i^2} = f(x_1,x_2),
17  \ \ \ \ \ \ \ \ \ \ (1)
18 \f]
19 in the rectangular domain \f$D = \left\{ (x_1,x_2) \in
20 [0,1] \times [0,2]\right\} \f$. The domain boundary
21 \f$ \partial D = \partial D_{Neumann} \cup \partial D_{Dirichlet} \f$,
22 where \f$ \partial D_{Neumann}
23 = \left\{ (x_1,x_2) | x_1=1, \ x_2\in [0,2] \right\} \f$.
24 On \f$ \partial D_{Dirichlet}\f$ we apply the Dirichlet boundary conditions
25 \f[
26 \left. u\right|_{\partial D_{Dirichlet}}=u_0,
27 \ \ \ \ \ \ \ \ \ \ (2)
28 \f]
29 where the function \f$ u_0 \f$ is given. On \f$ \partial D_{Neumann}\f$
30 we apply the Neumann conditions
31 \f[
32 \left. \frac{\partial u}{\partial n}\right|_{\partial D_{Neumann}} =
33 \left. \frac{\partial u}{\partial x_1}\right|_{\partial D_{Neumann}}
34 =g_0,
35 \ \ \ \ \ \ \ \ \ \ (3)
36 \f]
37 where the function \f$ g_0 \f$ is given.
38 </TD>
39 </TR>
40 </TABLE>
41 </CENTER>

44 In a <A HREF="../../two_d_poisson_flux_bc/html/index.html">
45 previous example,</A> we applied the Neumann boundary conditions
46 by adding \c PoissonFluxElements
47 (elements that apply the Neumann (flux)
48 boundary conditions on surfaces of higher-dimensional "bulk" Poisson
49 elements) to the \c Problem's \c Mesh object.
50 The ability to combine elements of different types in a single
51 \c Mesh object is convenient, and in certain circumstances
52 absolutely essential, but it can cause problems;
53 see the discussion of the \c doc_solution(...) function in the
54 <A HREF="../../two_d_poisson_flux_bc/html/index.html">
55 previous example</A>. Furthermore, it seems strange (if not wrong!) that
56 the \c SimpleRectangularQuadMesh -- an object that is templated by
57 a particular (single!) element type -- also contains elements of a
58 different type.
59 
60 We shall now demonstrate an alternative approach,
61 based on the use of multiple meshes, each containing only one type of
62 element. The ability to use multiple
63 \c Meshes in a single \c Problem is an essential feature of
64 \c oomph-lib and is vital in
65 fluid-structure interaction problems, where the fluid and solid domains
66 are distinct and each domain is discretised by a different
67 element type.
68 
69 
70  We consider the same problem as in the
71 <A HREF="../../two_d_poisson_flux_bc/html/index.html">
72 previous example</A> and choose a source function and
73 boundary conditions for which the function
74 \f[
75 u_0(x_1,x_2) = \tanh(1-\alpha(x_1 \tan\Phi - x_2)),
76 \ \ \ \ \ \ \ \ \ (4)
77 \f]
78 is the exact solution of the problem.

188 \section actions_before "Actions before solve"
189 The only (minor) change to \c Problem::actions_before_newton_solve()
190 is that the nodes on the boundaries of the bulk (!) mesh are
191 now obtained via the \c Bulk_mesh_pt pointer, rather than from the
192 combined \c Mesh, pointed to by \c Problem::mesh_pt().
193 While this may appear to be a trivial change, it is a potentially
194 important one. Recall that the surface mesh is an instantiation of
195 the \c Mesh base class. We created the (empty) mesh in the \c
196 Problem constructor (by calling the default \c Mesh constructor), and
197 used the function \c create_flux_elements(...) to add
198 the (pointers to the) prescribed-flux
199 elements to it. The surface mesh therefore does not have any nodes
200 of its own, and its lookup schemes for the boundary nodes have
201 not been set up. The combined mesh,
202 pointed to by \c Problem::mesh_pt(), therefore only contains the
203 boundary lookup scheme for the bulk mesh. Hence, the combined mesh
204 has four boundaries and their numbers correspond to those
205 in the bulk mesh.
206 
207 If we had set up the boundary lookup scheme in the
208 surface mesh, the constructor of the combined \c Mesh,
209 would have concatenated the boundary lookup schemes of the
210 two sub-meshes so that the four boundaries in sub-mesh 0
211 would have become boundaries 0 to 3 in the combined mesh, while
212 the two boundaries in the surface mesh would have become boundaries
213 4 and 5 in the combined \c Mesh. While the conversion is
214 straightforward, it is obvious that \c Mesh boundaries are
215 best identified via the sub-meshes.

239 \section comments Further comments
240 We mentioned that the \c Mesh constructor that builds a combined
241 \c Mesh from a vector of sub-meshes, concatenates the sub-meshes'
242 element, node and boundary lookup schemes. There are a few
243 additional features that the "user" should be aware of:
244 - The sub-meshes should not contain any duplicate nodes or elements.
245  If they do, the function \c Problem::build_global_mesh()
246  will issue a warning and
247  ignore any duplicates. This is because the \c Problem's global \c Mesh object
248  is used by many functions in which operations must be performed exactly
249  once for each node or element. For instance, in time-dependent
250  problems, the function \c Problem::shift_time_values(),
251  which is called automatically by \c Problem::unsteady_newton_solve(...),
252  advances all "history values" by one time-level to prepare for the
253  next timestep. If this was done repeatedly for nodes that are common
254  to multiple sub-meshes, the results would be incorrect.
255  If your problem requires a combined mesh in which duplicates
256  are allowed, you must construct this mesh yourself.
257 - Recall that the function \c Mesh::add_boundary_node()
258  "tells" the mesh's constituent nodes which boundaries they are
259  located on. What happens if a (sub-)mesh for which this lookup scheme has
260  been set up becomes part of a global \c Mesh? For various (good!) reasons,
261  the \c Mesh constructor \b does \b not \b update this information.
262  The boundary number stored by the nodes therefore always refers to
263  the boundary in the \c Mesh that created them. If this is not
264  appropriate for your problem, you must construct the
265  combined mesh yourself.
