3 \mainpage Demo problem: Adaptive solution of Poisson's equation in a fish-shaped domain.
4 
5 In this document, we discuss the solution of a 2D Poisson problem
6 using \c oomph-lib's powerful mesh adaptation routines:
7 
8 <CENTER>
9 <TABLE>
10 <TR>
11 <TD>
12 <CENTER>
13 <B>Two-dimensional model Poisson problem in a non-trivial domain</B>
14 </CENTER>
15 Solve
16 \f[
17 \sum_{i=1}^2 \frac{\partial^2u}{\partial x_i^2} = -1,
18  \ \ \ \ \ \ \ \ \ \ (1)
19 \f]
20 in the fish-shaped domain \f$D_{fish} \f$, with homogeneous
21 Dirichlet boundary conditions
22 \f[
23 \left. u\right|_{\partial D_{fish}}=0.
24 \ \ \ \ \ \ \ \ \ \ (2)
25 \f]
26 </TD>
27 </TR>
28 </TABLE>
29 </CENTER>

31 \image html fish_poisson_soln.gif "Plot of the solution "
32 \image latex fish_poisson_soln.eps "Plot of the solution " width=0.75\textwidth
33 
34 The sharp corners in the domain create singularities
35 in the solution (its derivatives are unbounded) and so
36 accurate results can only be obtained if we use
37 a sufficiently fine discretisation. Implementing this by
38 uniform mesh refinement would create a huge number of elements in the
39 interior of the domain where the fine discretisation
40 is not required.
41 
42 To avoid this problem, \c oomph-lib provides
43 mesh adaptation routines that
44 automatically adapt the mesh, based on \e a \e posteriori error estimates.
45 Regions in which an error estimator indicates that the
46 solution is not resolved to the required accuracy are refined;
47 automatic unrefinement is performed in regions where the
48 discretisation is unnecessarily fine.
49 
50 We provide a detailed discussion of the driver code
51 <A HREF="../../../../demo_drivers/poisson/fish_poisson/fish_poisson.cc">
52 fish_poisson.cc</A> which illustrates a variety of mesh refinement
53 procedures. [The alternative driver code
54 <A HREF="../../../../demo_drivers/poisson/fish_poisson/fish_poisson_no_adapt.cc">
55 fish_poisson_no_adapt.cc</A> solves the same problem without
56 mesh adaptation. Its structure is very similar to that in
57 the <A HREF="../../../poisson/two_d_poisson/html/index.html">
58 2D Poisson problem considered earlier</A>. It is provided mainly
59 to illustrate how easy to it is incorporate adaptivity into
60 a \c Problem.]
61 
62 In the current example we demonstrate how to \e use existing
63 refineable meshes and elements. Two further examples will demonstrate
64 how easy it is to \e create refineable meshes in
65 <A HREF="../../../poisson/two_d_poisson_adapt/html/index.html">
66 domains with polygonal boundaries</A> and in
67 <A HREF="../../../poisson/fish_poisson2/html/index.html">
68 domains with curvilinear boundaries.</A>

87 \section main The driver code
88 
89 The main code is very short and calls two functions that
90 illustrate two different adaptation strategies:
91 - A black-box approach in which the adaptation cycle
92  -# solve the problem on the initial, coarse mesh
93  -# compute an error estimate
94  -# adapt the mesh
95  -# solve again
96  .
97  is performed automatically until the solution satisfies the
98  required error bounds (or until the maximum permitted number of
99  adaptation steps has been reached).
100 - In the second approach we start by performing a number of uniform
101  mesh refinement steps, and then use incremental adaptations, allowing
102  us to document how the refinement proceeds.
103 .
104 
105 \dontinclude fish_poisson.cc
106 \skipline start_of_main
107 \until end of main

111 \subsection black_box Black-box adaptation
112 
113 We start by creating the Problem object, using the refineable
114 equivalent of the \c QPoissonElement -- the
115 \c RefineableQPoissonElement, which is templated by the dimension and
116 the number
117 of nodes along the element's edges; the \c
118 RefineableQPoissonElement<2,3> is a nine-node (bi-quadratic) quad element.
119 
120 \dontinclude fish_poisson.cc
121 \skipline start_black_box
122 \until RefineableFish
123 
124 
125  After creating the \c DocInfo object, we document the (default)
126 adaptivity targets:
127 
128 \skipline Setup labels
129 \until doc_ad
130 
131 These include
132 - <B>The target for the maximum error:</B>
133  Any elements whose error estimate exceed this value will be split
134  into four "sons".
135 - <B>The target for the minimum error: </B> Any elements whose error
136  estimate lies below this value are deemed to be unnecessarily
137  small and are scheduled for (possible) unrefinement. [Elements can
138  only be unrefined (i.e. merged with their "brothers") if their
139  "brothers" are also scheduled for unrefinement.]
140 - <B>The minimum refinement level:</B> In problems with curvilinear
141  domain boundaries it is often necessary to retain a reasonably accurate
142  representation of the boundary (e.g. for postprocessing
143  purposes), even if the error estimate suggests that the
144  mesh could be unrefined further.
145 - <B>The maximum refinement level:</B> In problems where the solution
146  has singularities, the refinement process would continue indefinitely,
147  therefore an upper bound on the refinement level must be imposed.
148 - Finally, because unrefinement is done purely to speed up the computation,
149  it would not make sense to adapt the mesh if this process would only
150  remove a few elements, while forcing the re-computation of the solution
151  on an only slightly coarsened mesh. Therefore, no mesh adaptation is
152  performed if
153  - the adaptation would only perform unrefinements
154  - <B>and</B> the number of elements scheduled for unrefinement is below
155  a certain threshold.
156  .
157 .
158 These default parameters can be changed by the user; see \ref comments.
159 
160 The fully-adaptive solution of the problem is very simple. We simply pass the
161 maximum number of adaptations to the Newton solver and document the
162 results. Done!
163 
164 \dontinclude fish_poisson.cc
165 \skipline with fully automatic adaptation
166 \until end black box

170 \subsection incremental Incremental adaptation
171 
172 To allow the user more control over the mesh adaptation process,
173 \c oomph-lib provides a number of functions that perform individual
174 adaptation steps without re-computing the solution immediately. This allows
175 the user to
176 - perform uniform mesh refinement and unrefinement,
177 - impose a specific refinement pattern,
178 - monitor/document the progress of the automatic adaptation.
179 .
180 
181 The second driver function illustrates some of these functions. We
182 start by setting up the problem, create the \c DocInfo object and document
183 the adaptivity targets, exactly as before:
184 
185 \dontinclude fish_poisson.cc
186 \skipline start_of_incremental
187 \until doc_ad
188 
189 Next, we solve the problem on the original, very coarse mesh
190 and document the result:
191 
192 \dontinclude fish_poisson.cc
193 \skipline on the initial, very coarse
194 \until doc_info.number()++;
195 
196 
197 We know that the result is unlikely to be very accurate,
198 so we apply three levels of uniform refinement, increasing the number
199 of elements from 4 to 256, and re-compute:
200 
201 \dontinclude fish_poisson.cc
202 \skipline Do three rounds
203 \until doc_info.number()++;
204 
205 The solution looks much smoother but we suspect that the corner
206 regions are still under-resolved. Therefore, we call the
207 \c Problem::adapt() function which computes an error estimate
208 for all elements and automatically performs a single
209 mesh adaptation (refinement/unrefinement) step. If this adaptation changes the
210 mesh, we recompute the solution, using the
211 "normal" Newton solver without automatic adaptation.
212 We document the solution and continue the adaptation
213 cycle until \c Problem::adapt() ceases to change the mesh:
214 
215 
216 
217 \dontinclude fish_poisson.cc
218 \skipline Now do (up to)
219 \until end of incremental
220 
221 The progress of the adaptation is illustrated in the animated gif
222 at the beginning of this document. The first frame displays
223 the solution on the original four-element mesh; the next frame
224 shows the solution on the uniformly refined mesh; the final two
225 frames show the progress of the subsequent, error-estimate-driven
226 mesh adaptation.

232 \section problem The problem class
233 The problem class is virtually identical to that used in the
234 <A HREF="../../../poisson/two_d_poisson/html/index.html">2D Poisson problem
235 without mesh refinement</A>. In the present problem, we leave the function
236 \c Problem::actions_before_newton_solve() empty because the boundary
237 conditions do not change. The function
238 \c RefineableFishPoissonProblem::mesh_pt() overloads the
239 (virtual) function \c Problem::mesh_pt() since it returns a pointer
240 to a generic \c Mesh object, rather than a pointer to the specific
241 mesh used in this problem. This avoids explicit re-casts
242 in the rest of the code where member functions of the
243 specific mesh need to be accessed.

298 \section comments Comments and Exercises
299 
300 The purpose of this example was to provide a high-level overview
301 of \c oomph-lib's mesh adaptation procedures.
302 We demonstrated that the implementation of full adaptivity
303 only required us to
304 - replace the \c FishMesh and the
305  \c QPoissonElement objects by their refineable equivalents,
306  \c RefineableFishMesh and \c RefineableQPoissonElement, respectively
307 - specify the error estimator, and
308 - specify the maximum number of adaptations for the black-box adaptive
309  Newton solver.
310 .
311 
312 (Compare the Problem specification for the current problem
313 to that of its non-refineable equivalent, contained in the
314 alternative driver code
315 <A HREF="../../../../demo_drivers/poisson/fish_poisson/fish_poisson_no_adapt.cc">
316 fish_poisson_no_adapt.cc.</A>)
317 
318 
319 Since most of the "hard work" involved in the mesh adaptation is
320 "hidden" from the user, we highlight some important aspects
321 of the procedure:
322 
323 \subsection adapt_bc Automatic transfer of the solution/boundary conditions during the mesh adaptation
324 The \c Problem::adapt() function automatically determines the correct
325 boundary conditions for newly created nodes on the Mesh boundary;
326 it automatically updates the equation numbering scheme, and
327 interpolates the solution from the original mesh onto the adapted
328 mesh. This is important in nonlinear problems where the provision
329 of a good initial guess for the Newton iteration is vital;
330 and in time-dependent problems where the solution at one timestep
331 provides initial conditions for the next one. See the discussion of
332 <A HREF="../../../unsteady_heat/two_d_unsteady_heat_adapt/html/index.html">
333 the adaptive solution of the unsteady heat equation</A>
334 for more details.
335 Furthermore, the source function pointers are automatically
336 passed to an element's four "son" elements when the element
337 is subdivided. This allows the adaptation to proceed completely
338 automatically, without any intervention by the "user". On return
339 from \c Problem::adapt() the problem can immediately be re-solved.
340 
341 In some special cases, certain actions may need to be performed
342 before or after the mesh adaptation (e.g. if flux boundary
343 conditions are applied by \c FaceElements; this is explained in
344 <A HREF="../../../poisson/two_d_poisson_flux_bc_adapt/html/index.html">
345 another example</A>). To ensure that these steps are performed when
346 the adaptation is controlled by the "black-box" adaptive Newton solver,
347 the \c Problem class provides the two empty virtual functions
348 \code
349 Problem::actions_before_adapt()
350 \endcode
351 and
352 \code
353 Problem::actions_after_adapt()
354 \endcode
355 which are called automatically before and after the adaptation.
356 The "user" can overload these in his/her specific \c Problem class
357 to implement such actions.
358 
359 
360 \subsection how_to Automatic mesh adaptation in domains with curvilinear boundaries
361 The mesh adaptation not only increases the number of elements
362 but also produces a more accurate representation of the
363 curvilinear domain boundary -- new boundary nodes are
364 placed exactly onto the analytically-defined, curvilinear boundary,
365 rather than on the boundaries of the "father" element, which only provides
366 an approximate representation of the exact domain boundary.
367 This is achieved by employing a \c MacroElement-based representation
368 of the \c Domain -- we will discuss this in more detail in
369 <A HREF="../../../poisson/fish_poisson2/html/index.html">
370 another example.</A>
371 
372 \subsection problem_vs_mesh Problem adaptation vs. Mesh adaptation
373 Many adaptation routines in the \c Problem class have
374 equivalents in the \c RefineableMesh class. It is important
375 to appreciate the important differences between them: If adaptation
376 is performed at the \c Problem level, the adapted \c Problem is
377 fully functional, i.e. boundary conditions will have been
378 assigned for newly created nodes on the mesh boundary,
379 the equation numbering scheme will have been updated, etc.
380 The adapted \c Problem can therefore be re-solved immediately.
381 Conversely, if a mesh is refined directly, using the member functions
382 of the \c RefineableMesh class, many of these additional
383 tasks need to be performed "by hand" before the adapted
384 \c Problem can be resolved.

389 \subsection exercises Exercises
390 To familiarise yourself with \c oomph-lib's mesh adaptation procedures
391 we suggest the following exercises:
392 -# When the Poisson problem is solved with the default refinement
393  targets, no elements are unrefined. Increase the minimum
394  permitted error from its default value of \f$ 10^{-5} \f$ to
395  \f$ 10^{-4} \f$ by adding the statement
396  \code
397  problem.mesh_pt()->min_permitted_error()=1.0e-4;
398  \endcode
399  before
400  \code
401  problem.mesh_pt()->doc_adaptivity_targets(cout);
402  \endcode
403  This value forces an unrefinement of several elements in the mesh:
404 \image html fish_poisson_soln2.gif "Plot of the solutions obtained with the modified adaptivity targets. "
405 \image latex fish_poisson_soln2.eps "Plot of the solutions obtained with the modified adaptivity targets. " width=0.75\textwidth
406 -# Convince yourself that \c Problem::adapt() does indeed interpolate the
407  solution from the coarse mesh to the fine mesh -- call
408  \c Problem::doc_solution(...) before and after its execution.
409 -# The \c Problem::refine_uniformly() function has a counterpart
410  \c Problem::unrefine_uniformly(). Why does this function
411  not simply unrefine every single element in the mesh?
412  Explore the action of \c
413  Problem::unrefine_uniformly() by plotting the solution
414  before and after a few executions of this function.
415 \image html fish_poisson_soln3.gif "Uniform unrefinement "
416 \image latex fish_poisson_soln3.eps "Uniform unrefinement " width=0.75\textwidth
417 -# Impose a "user-defined" refinement pattern
418  by calling the function \c Problem::refine_selected_elements(...).
