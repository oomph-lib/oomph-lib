3\mainpage Demo problem: A two-dimensional Poisson problem with flux boundary conditions
4
5In this document, we demonstrate how to solve a 2D Poisson problem
6with Neumann boundary conditions, using existing objects
7from the \c oomph-lib library:
8<CENTER>
9<TABLE>
10<TR>
11<TD>
12<CENTER>
13<B>Two-dimensional model Poisson problem with Neumann boundary conditions</B>
14</CENTER>
15Solve
16\f[
17\sum_{i=1}^2 \frac{\partial^2u}{\partial x_i^2} = f(x_1,x_2),
18 \ \ \ \ \ \ \ \ \ \ (1)
19\f]
20in the rectangular domain \f$ D = \left\{ (x_1,x_2) \in
21[0,1] \times [0,2]\right\} \f$. The domain boundary
22 \f$ \partial D = \partial D_{Neumann} \cup \partial D_{Dirichlet} \f$,
23where \f$ \partial D_{Neumann}
24= \left\{ (x_1,x_2) | x_1=1, \ x_2\in [0,2] \right\} \f$.
25On \f$ \partial D_{Dirichlet} \f$ we apply the Dirichlet boundary conditions
26\f[
27\left. u\right|_{\partial D_{Dirichlet}}=u_0,
28\ \ \ \ \ \ \ \ \ \ (2)
29\f]
30where the function \f$ u_0 \f$ is given. On \f$ \partial D_{Neumann} \f$
31we apply the Neumann conditions
32\f[
33\left. \frac{\partial u}{\partial n}\right|_{\partial D_{Neumann}} =
34\left. \frac{\partial u}{\partial x_1}\right|_{\partial D_{Neumann}}
35=g_0,
36\ \ \ \ \ \ \ \ \ \ (3)
37\f]
38where the function \f$ g_0 \f$ is given.
39</TD>
40</TR>
41</TABLE>
42</CENTER>
43
44
45We provide
46a detailed discussion of the driver code <A HREF="../../../../demo_drivers/poisson/two_d_poisson_flux_bc/two_d_poisson_flux_bc.cc">
47two_d_poisson_flux_bc.cc</A> which solves the problem for
48\f[
49u_0(x_1,x_2) = \tanh(1-\alpha(x_1 \tan\Phi - x_2)),
50\ \ \ \ \ \ \ \ \ (4)
51\f]
52\f[
53f(x_1,x_2) = \sum_{i=1}^2 \frac{\partial^2 u_0}{\partial x_i^2},
54\ \ \ \ \ \ \ \ \ (5)
55\f]
56and
57\f[
58g_0 = \left. \frac{\partial u_0}{\partial n} \right|_{x_1=1}
59 = \left. \frac{\partial u_0}{\partial x_1} \right|_{x_1=1},
60\ \ \ \ \ \ \ \ \ (6)
61\f]
62so that \f$ u_0(x_1,x_2) \f$ is the exact solution of the
63problem. For large values of \f$ \alpha \f$ the solution approaches
64a step function
65\f[
66u_{step}(x_1,x_2) = \left\{
67\begin{array}{rl}
68-1 & \mbox {for $x_2 < x_1 \ \tan\Phi$} \\
691 & \mbox {for $x_2 > x_1 \ \tan\Phi$}
70\end{array}
71 \right.
72\f]
73and presents a serious challenge for any numerical method.
74The figure below compares the numerical and exact solutions for
75\f$ \alpha = 1 \f$ and \f$ \Phi = 45^o \f$.
76
77\image html soln.gif "Plot of the solution "
78\image latex soln.eps "Plot of the solution " width=0.75\textwidth

80Most of the driver code is identical to the code that solves

83Therefore we only provide a detailed discussion of those

93\section global Global parameters and functions
94As in the Dirichlet problem, we define the source function (5)
95and the exact solution (4), together with the
96problem parameters \f$ \tan \Phi \f$ and \f$ \alpha \f$,
97in a namespace \c TanhSolnForPoisson. We add the function
98 \c TanhSolnForPoisson::prescribed_flux_on_fixed_x_boundary(...)
99which computes the
100prescribed flux \f$ g_0 \f$ required in the Neumann boundary
101condition (3). The function evaluates
102\f$ \partial u_0 /\partial n = {\bf N} \cdot \nabla u_0 \f$
103for the normal direction specified by the vector \f$ {\bf N} =
104(1,0)^T. \f$

159\dontinclude two_d_poisson_flux_bc.cc
160\skipline start_of_constructor
161\until SimpleRectangularQuadMesh
162
163Before continuing, we store the number of 2D "bulk" Poisson elements in
164the variable \c FluxPoissonProblem::Npoisson_element:
165
166\skipline Store
167\until Npoisson
168
169Now, we need to apply the prescribed-flux boundary condition
170along the Neumann boundary \f$ \partial D_{Neumann} \f$. The documentation for
171the \c SimpleRectangularQuadMesh shows that this boundary is mesh
172boundary 1. The necessary steps are performed by the function
173\c create_flux_elements(..), described in the section \ref create_flux
174below.
175
176\skipline Create prescribed
177\until create_flux
178
179The rest of the constructor is very similar to its counterpart
180in the <A HREF="../../../poisson/two_d_poisson/html/index.html">pure
181Dirichlet problem</A>. First we apply Dirichlet conditions on the
182remaining boundaries by pinning the nodal values. Next, we finish
183the problem setup by looping over all "bulk" Poisson
184elements and set the pointer to the source function.
185Since we have added the \c PoissonFluxElements to the \c Mesh,
186only the first \c Npoisson_element elements are "bulk" elements
187and the loop is restricted to these. We then perform a second loop
188over the \c PoissonFluxElements which need to be passed
189the pointer to the prescribed-flux function
190\c TanhSolnForPoisson::prescribed_flux_on_fixed_x_boundary(...). Finally,
191we generate the equation numbering scheme.

199\section create_flux Creating the flux elements
200
201\c oomph-lib provides
202an element type \c PoissonFluxElement, which allows the
203application of Neumann (flux) boundary conditions along the "faces"
204of higher-dimensional "bulk" Poisson elements. \c PoissonFluxElements are
205templated by the type of the corresponding higher-dimensional
206"bulk" element, so that a
207\c PoissonFluxElement<QPoissonElement<2,3> >
208is a one-dimensional three-node element that applies
209Neumann boundary conditions along the one-dimensional edge
210of a nine-node quadrilateral Poisson element. Similarly,
211a \c PoissonFluxElement<QPoissonElement<3,2> >
212is a two-dimensional quadrilateral four-node element that applies
213Neumann boundary conditions along the two-dimensional face
214of a eight-node brick-shaped Poisson element; etc.
215
216 The constructor of the \c PoissonFluxElement takes two arguments:
217- a pointer to the corresponding bulk element
218- the index \c face_index of the face that is to be constructed. The
219convention for two-dimensional Q-type elements is that the face_index
220is \f$ -(i+1) \f$ when the coordinate \f$ s_i \f$ is fixed
221at its minimum value over the face and \f$ +(i+1) \f$ when
222\f$ s_i \f$ is fixed at its maximum value over the face
223.
224The layout of the elements in the \c SimpleRectangularQuadMesh
225is sufficiently simple to allow the direct determination of the face
226index: Elements 3, 7, 11 and 15 are located next to mesh
227boundary 1 and along this boundary the element's local coordinate
228\f$ s_0 \f$ has a constant (maximum) value of +1.0. Hence we need to set
229\c face_index=1
230
231In more complicated meshes, the determination of the face index
232can be more difficult (or at least very tedious), especially
233if a \c Mesh has been refined non-uniformly. The generic \c Mesh
234class therefore provides helper functions to determine the
235required face index for all elements adjacent to a specified
236\c Mesh boundary. This allows the creation of the flux elements
237by the following, completely generic procedure: We use
238the function \c Mesh::boundary_element_pt(...) to determine
239the "bulk" elements that are adjacent to the Neumann boundary,
240and obtain \c face_index from the function
241\c Mesh::face_index_at_boundary(...). We pass the parameters to the
242constructor of the \c PoissonFluxElement and add the
243(pointer to) the newly created element to the \c Problem's mesh.
244
245\skipline start_of_create_flux
246\until end of create_flux

264\section doc Post-processing
265The post-processing, implemented in \c doc_solution(...)
266is similar to that in
267<A HREF="../../../poisson/two_d_poisson/html/index.html">pure Dirichlet
268problem</A>. However, since the \c PoissonFluxElements
269are auxiliary elements which are only used to apply
270Neumann boundary conditions on adjacent "bulk" elements,
271their error checking function is not implemented.
272We cannot use the generic \c Mesh member function
273\c Mesh::compute_error() to compute an overall error since this
274function would try to execute the "broken virtual" function
275\c GeneralisedElement::compute_error(...); see the
276section \ref exercises for a more detailed discussion
277of "broken virtual" functions. Error checking would
278therefore have to be implemented "by hand" (excluding the
279\c PoissonFluxElements), or a suitable error measure would have
280to be defined in the \c PoissonFluxElements.
281
282 We do not pursue either approach here because the difficulty
283is a direct consequence of our (questionable) decision to include
284elements of different types in the same \c Mesh object. While this is
285perfectly "legal" and often convenient, the practice introduces
286additional difficulties in refineable problems and we shall demonstrate
287<A HREF="../../../poisson/two_d_poisson_flux_bc2/html/index.html">
288 an alternative approach in another example</A>.

298\section exercises Exercises and Comments
299-# What happens if you do not create the \c PoissonFluxElements
300 but leave the nodes on the Neumann boundary un-pinned?
301 Compare the computational result to those obtained when you set
302 the prescribed flux to zero, \f$ g_0 = 0 \f$. Does this make sense?
303 [Hint: Remember the "natural" boundary conditions for
304 Poisson's equation].
305-# Try to compute the error of the computed solution by
306 re-instating the global error checking procedure
307 \code
308 // Doc error and return of the square of the L2 error
309 //---------------------------------------------------
310 double error,norm;
311 snprintf(filename, sizeof(filename), "%s/error%i.dat",doc_info.directory().c_str(),
312 doc_info.number());
313 some_file.open(filename);
314 mesh_pt()->compute_error(some_file,TanhSolnForPoisson::get_exact_u,
315 error,norm);
316 some_file.close();
317 \endcode
318 in \c doc_solution(...).
319 What happens when you run the code? The code's behaviour
320 illustrates a general convention in \c oomph-lib:
321 <TABLE>
322 <TR>
323 <TD bgcolor="cornsilk">
324 <CENTER><B>\anchor broken_virtual A general convention</B></CENTER>
325 Some \c oomph-lib functions, such as
326 \c GeneralisedElement::compute_error(...),
327 are defined as virtual functions in a base class to allow
328 the implementation of generic procedures such as
329 \c Mesh::compute_error(...), which loops over all
330 of the \c Mesh's constituent elements and executes their
331 specific \c compute_error(...) member functions. In some rare cases
332 (such as the one encountered here), the implementation of a
333 particular virtual
334 function might not be sensible for a specific element. Therefore,
335 rather than forcing the "element-writer" to implement a dummy version
336 of this function in his/her derived class (by declaring it as a pure
337 virtual function
338 in the base class), we provide a <B> "broken virtual" </b>
339 implementation in \c GeneralisedElement. If the function is
340 (re-)implemented in a derived element, the broken version
341 is ignored; if the function is not overloaded, the broken
342 virtual function throws an error, allowing a traceback in a
343 debugger to find out where the broken function was called from.
344 We note that this practice is not universally approved of in the C++
345 community but we believe it to have its place in situations such as
346 the one described here.
347 \n \n
348 Incidentally, the code discussed above contains another
349 (possibly more convincing) example of why
350 "broken virtual" functions can be useful. Recall that the creation of
351 the \c PoissonFluxElements on the Neumann boundary was
352 greatly facilitated by the availability of the helper functions
353 \c Mesh::boundary_element_pt(...) and
354 \c Mesh::face_index_at_boundary(...).
355 These functions are implemented
356 in the generic \c Mesh base class and determine the relevant parameters
357 via lookup schemes that are stored in that class. Obviously,
358 the lookup schemes need to be set up when a specific \c Mesh
359 is built and this task can involve a considerable amount of
360 work (see also \ref footnote). Since the lookup schemes are useful
361 but by no means essential, the three helper functions are again
362 implemented as broken virtual functions. If the functions
363 are called before the required lookup schemes have been set up,
364 code execution stops with a suitable warning message.
365 </TD>
366 </TR>
367 </TABLE>

369 with the way in which the \c Mesh::compute_error(...) function works.

374\subsection footnote Setting up the boundary lookup schemes
375\c oomph-lib provides a range of helper functions that
376set up the boundary lookup schemes for specific \c Mesh classes.
377For instance, the \c QuadMeshBase class forms a base
378class for all \c Meshes that consist of two-dimensional
379quadrilateral elements and has a member function
380\c QuadMeshBase::setup_boundary_element_info()
381which can be called from the constructor of any derived
382\c Mesh class to set up the lookup schemes required
383by \c Mesh::boundary_element_pt(...) and
384\c Mesh::face_index_at_boundary(...).
