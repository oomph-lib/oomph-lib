3 \mainpage Demo problem: How to create simple refineable meshes
4 
5 In a <A HREF="../../../poisson/fish_poisson/html/index.html">
6 previous example</A> we gave an overview of \c oomph-lib's powerful
7 mesh adaptation capabilities and demonstrated the use of
8 the functions
9 - \c Problem::refine_uniformly() which performs automatic,
10  uniform refinement of a given (refineable) mesh.
11 - \c Problem::adapt() which performs automatic mesh
12  adaptation (local refinement or unrefinement), based on
13  error estimates that are computed (automatically) by a chosen
14  error estimator.
15 - \c Problem::newton_solve(...) -- a black-box adaptive Newton
16  solver that automatically adapts the mesh and recomputes
17  the solution until it satisfies the prescribed error bounds.
18 .
19 Provided the problem has been discretised with suitable "refineable mesh"
20 and "refineable element" objects from \c oomph-lib's mesh and
21 finite element libraries, none of these functions require any
22 intervention by the user. Most of \c oomph-lib
23 finite elements are already available in "refineable" and "non-refineable"
24 forms. For instance, the \c RefineableQPoissonElement that
25 we used in the <A HREF="../../../poisson/fish_poisson/html/index.html">
26 previous example</A> is the refineable equivalent of the 2D \c QPoissonElement.
27 <A HREF="../../../to_be_written/html/index.html">
28 Another document</A> describes how to create new
29 refineable elements. Here we shall discuss how to "upgrade" existing
30 meshes to \c RefineableMeshes, i.e. meshes that can be used with
31 \c oomph-lib's mesh adaptation routines.
32 
33 
34 
35 The minimum functionality that must be provided
36 by such meshes is specified by the pure virtual functions
37 in the abstract base class \c RefineableMesh and all refineable
38 Meshes should be derived from this class.
39 Here is a graphical representation of the typical inheritance structure
40 for refineable meshes, illustrated for 2D quad meshes:
41 
42 \image html refineable_mesh_inheritance.gif "Typical inheritance structure for refineable meshes, illustrated for 2D quad meshes. "
43 \image latex refineable_mesh_inheritance.eps "Typical inheritance structure for refineable meshes, illustrated for 2D quad meshes. " width=0.75\textwidth
44 
45 
46 The diagram contains two fully-functional meshes:
47 - The \c SomeMesh is some basic, non-refineable mesh that
48  is derived directly from the generic \c Mesh base class.
49  Typically, it provides a coarse discretisation of a 2D domain
50  with 2D elements from the \c QElement family. Its constructor
51  creates the mesh's nodes and elements and initialises the various
52  boundary lookup schemes. (Consult the
53  <A HREF="../../../quick_guide/html/index.html#mesh">
54  "How to build a mesh"</A> section
55  of the <A HREF="../../../quick_guide/html/index.html">Quick Guide</A> for
56  details of the generic mesh generation process.)
57 - The \c RefineableSomeMesh is the refineable equivalent of the
58  basic \c SomeMesh. It inherits the original mesh layout from the
59  \c SomeMesh class. Refineability is added by inheriting from the \c
60  RefineableQuadMesh class; this class implements the
61  mesh adaptation procedures, specified as pure virtual functions in
62  the \c RefineableMesh class, for 2D quad meshes, employing
63  \c QuadTree - based refinement techniques.
64 .
65 
66 Equivalent inheritance structures can be/are implemented for meshes with
67 different element topologies: For instance, the \c RefineableBrickMesh
68 class is the 3D equivalent of the \c RefineableQuadMesh class:
69 It performs the mesh adaptation for 3D brick meshes
70 by \c OcTree - based refinement techniques.
71 
72  Typically, most of the "hard work" involved in the mesh adaptation
73 process is implemented in the intermediate classes (such as \c
74 RefineableQuadMesh or \c RefineableBrickMesh). Upgrading
75 an existing mesh to a refineable version therefore usually requires
76 very little effort. We demonstrate this by re-visiting the
77  2D Poisson problem that we analysed in an
78 <A HREF="../../../poisson/two_d_poisson/html/index.html"> earlier example:
79 </A>

86 <B>Two-dimensional model Poisson problem</B>
87 </CENTER>
88 Solve
89 \f[
90 \sum_{i=1}^2 \frac{\partial^2u}{\partial x_i^2} = f(x_1,x_2),
91  \ \ \ \ \ \ \ \ \ \ (1)
92 \f]
93 in the rectangular domain \f$D =\left\{ (x_1,x_2) \in
94 [0,1] \times [0,2]\right\}\f$,
95 with Dirichlet boundary conditions
96 \f[
97 \left. u\right|_{\partial D}=u_0
98 \ \ \ \ \ \ \ \ \ \ (2)
99 \f]
100 where
101 \f[
102 u_0(x_1,x_2) = \tanh(1-\alpha(x_1 \tan\Phi - x_2))
103 \ \ \ \ \ \ \ \ \ (3)
104 \f]
105 and
106 \f[
107 f(x_1,x_2) = \sum_{i=1}^2 \frac{\partial^2 u_0}{\partial x_i^2}
108 \ \ \ \ \ \ \ \ \ (4)
109 \f]
110 so that \f$u_0(x_1,x_2) \f$ represents the exact solution of the
111 problem.
112 </TD>
113 </TR>
114 </TABLE>
115 </CENTER>
116 
117 Recall that for large values of \f$ \alpha \f$ the solution approaches
118 a step function
119 \f[
120 u_{step}(x_1,x_2) = \left\{
121 \begin{array}{rl}
122 -1 & \mbox {for $x_2 < x_1 \ \tan\Phi$} \\
123 1 & \mbox {for $x_2 > x_1 \ \tan\Phi$}
124 \end{array}
125  \right.
126 \f]
127 Accurate numerical solution can therefore only be obtained if the
128 mesh is refined -- ideally only in the vicinity of the "step":
129

144 \section refineable Creating the refineable mesh
145 
146 \c QuadTree-based mesh refinement, as implemented in the
147 \c RefineableQuadMesh class, requires the coarse
148 initial mesh to be represented by a \c QuadTreeForest:
149 Each element in the mesh must be associated with a \c QuadTree,
150 and the relative orientation of the various \c QuadTrees relative
151 to each other must be established. This can be done automatically
152 by calling the function \c RefineableQuadMesh::setup_quadtree_forest().
153 The \c SimpleRefineableRectangularQuadMesh class is therefore very
154 compact. The mesh is derived from the \c SimpleRectangularQuadMesh
155 and the \c Refineable1QuadMesh classes, both of which are templated
156 by the element type:
157 
158 \dontinclude two_d_poisson_adapt.cc
159 \skipline start_of_mesh
160 \until RefineableQuadMesh<ELEMENT>
161 
162 The mesh constructor first calls the constructor of the
163 underlying \c SimpleRectangularQuadMesh to create the nodes and elements,
164 and to set up the various boundary lookup schemes. The call to
165 \c RefineableQuadMesh::setup_quadtree_forest() creates the
166 \c QuadTreeForest representation of the mesh. That's all!
167 
168 \skipline public:
169 \until end of constructor
170 
171 The destructor can remain empty, as all memory de-allocation is
172 handled in the mesh base classes.

176 \until ~SimpleRefineableRectangularQuadMesh

273 \section exercises Comments
274 
275 Since most of the "hard work" involved in the mesh adaptation is
276 hidden from "user" we briefly comment on various steps involved
277 in the mesh adaptation process and highlight certain important
278 implications.
279 
280 \subsection splitting The refinement pattern
281 
282 The \c QuadTree - based mesh adaption routines, implemented in
283 the \c RefineableQuadMesh class, split elements into four
284 "son elements" if the error estimate exceeds the acceptable maximum.
285 By default, the position of any newly created nodes
286 is determined from the geometric mapping of the "father" element. For
287 instance, when a four-node quad "father" element is split into four
288 "sons", five new nodes are created and they are located at
289 \f$ (s_0,s_1) = (0,-1), \ (1,0), \ (0,1), \ (-1,0) \f$ and
290 \f$ (0,0)\f$ in the father element's local coordinate system.
291 This procedure is adequate for problems in which
292 the coarse initial mesh provides a perfect representation of the
293 domain (e.g. polygonal domains). If the domain has curvilinear boundaries,
294 successive mesh refinements must generate a more and more
295 accurate representation of these boundaries. This requires
296 slight changes to the mesh adaptation procedures. We will discuss
297 these in <A HREF="../../../poisson/fish_poisson2/html/index.html">
298 another example</A>.
299 
300 The splitting of "father" elements into four equal-sized "sons"
301 maintains the aspect ratio of the elements during the mesh
302 adaptation. The good news is that mesh adaption will
303 not cause a deterioration in the element quality. The bad news is
304 that poorly designed coarse meshes cannot be improved by
305 mesh adaptation. It is therefore worthwhile
306 to invest some time into the initial mesh design. For complicated
307 domains, it may be sensible to perform the initial mesh generation
308 with a dedicated, third-party mesh generator.
309 (We provide <A HREF="../../../meshes/third_party_meshes/html/index.html">
310 another example</A> to illustrate how to build \c oomph-lib meshes
311 based on the output from a third-party mesh generator.)
312

314 \subsection hang Hanging nodes
315 
316 The local splitting of elements can create so-called "hanging
317 nodes" -- nodes on element edges that are not shared by
318 any adjacent elements. The nodal values and coordinates at such
319 nodes must be constrained to ensure the inter-element continuity
320 of the solution. Specifically, the nodal values and coordinates
321 at hanging nodes must be suitable linear combinations of the
322 values at a number of "master nodes". (In the first instance,
323 the master nodes are the nodes on the adjacent element's edge that
324 \e are shared by adjacent elements. If there are multiple levels of
325 refinement, such nodes can themselves be hanging; the ultimate set
326 of master nodes is therefore be determined recursively.)
327 
328  The setup of the hanging node constraints is handled automatically
329 by the mesh adaptation routines and the technical details are
330 therefore of little relevance to the general user.
331 (The <A HREF="../../../the_data_structure/html/index.html">
332 "bottom up" discussion of the data structure</A> provides details
333 if you are interested.) One aspect of the way in which hanging
334 nodes are handled in \c oomph-lib is important, however. Up to now
335 we have accessed nodal values either via the function
336 
337 \code
338 Node::set_value(...)
339 \endcode
340 
341 which sets the values stored at a \c Node, or the pointer-based
342 access function
343 
344 \code
345 Node::value_pt(...)
346 \endcode
347 
348 which returns a pointer to these values.
349 
350 What happens when a node is hanging, i.e. if
351 \c Node::is_hanging() returns \c true?
352 
353 <TABLE>
354 <TR>
355 <TD bgcolor="cornsilk">
356 <CENTER><B>A convention</B></CENTER>
357 The functions
358 
359 \code
360 Node::set_value(...)
361 \endcode
362 
363 and
364 
365 \code
366 Node::value_pt(...)
367 \endcode
368 
369 always refer to the nodal values stored at the \c Node itself.
370 
371 \b Important: If a node is hanging,
372 the value pointed to by \c Node::value_pt(...) is <B>not</B>
373 kept up to date!
374 
375 The correctly constrained nodal value must be
376 computed "on the fly", using the list of master nodes and their
377 respective weights,
378 stored in the node's \c HangingInfo object. This is done
379 automatically by the function
380 
381 \code
382 Node::value(...)
383 \endcode
384 
385 which returns the appropriate value for hanging \e and non-hanging nodes:
386 For non-hanging nodes it returns the value pointed to by
387 \c Node::value_pt(...); for hanging nodes, it computes the
388 correctly constrained values. When developing new
389 elements or writing new post-processing routines, the user should
390 therefore always refer to nodal values with the \c Node::value(...)
391 function to ensure that the code works correctly in the presence
392 of hanging nodes.
393 
394 
395 We provide equivalent functions to access the nodal positions:
396 The function
397 
398 \code
399 Node::x(...)
400 \endcode
401 
402 returns the values of (Eulerian) coordinates stored at the node. These
403 values can be out of date if the node is hanging. The function
404 
405 \code
406 Node::position(...)
407 \endcode
408 
409 should be used to determine a node's Eulerian position -- this
410 function is the equivalent of \c Node::value(...) and determines
411 the nodal coordinates of hanging nodes "on the fly", using
412 the node's list of master nodes and weights.
413 
414 Finally, we note that while the nodal values and coordinates stored
415 at a node might be out of date \e while a node is hanging, the
416 values are automatically assigned up-to-date values when
417 subsequent mesh adaptations change a node's status from
418 hanging to non-hanging.
