3\mainpage Demo problem: A one-dimensional Poisson problem
4In this document, we demonstrate how a solve the 1D Poisson problem
5using existing objects from the \c oomph-lib library:
6
7<CENTER>
8<TABLE>
9<TR>
10<TD>
11<CENTER>
12<B>One-dimensional model Poisson problem</B>
13</CENTER>
14Solve
15\f[
16\frac{\mbox{d}^2u}{\mbox{d}x^2} = f(x),
17 \ \ \ \ \ \ \ \ \ \ (1)
18\f]
19in a one-dimensional domain \f$ x\in[0,1] \f$, with boundary conditions
20\f[
21u(0)=u_0 \ \mbox{\ \ \ and \ \ \ } \ \ u(1)=u_1,
22\ \ \ \ \ \ \ \ \ \ (2)
23\f]
24where \f$ u_0 \f$ and \f$ u_1 \f$ are given.
25</TD>
26</TR>
27</TABLE>
28</CENTER>
29
30We provide a detailed discussion of the driver code
31<A HREF="../../../../demo_drivers/poisson/one_d_poisson/one_d_poisson.cc">
32one_d_poisson.cc</A> which solves the
33problem for the specific source function
34\f[
35f(x) = \pm 30 \sin(\sqrt{30} x), \ \ \ \ (3)
36\f]
37and boundary conditions
38\f[
39u_0= 0; \ \ \ u_1 = \mp 1.
40\f]
41In this case, the problem has the (fish-shaped) exact solution
42\f[
43u(x) = \pm \left[ \left(\sin(\sqrt{30})-1\right) x - \sin(\sqrt{30} x)\right].
44\ \ \ \ \ (4)\f]
45

53\section global Global parameters and functions
54Source functions and problem parameters generally need
55to be accessible to a variety of \c oomph-lib objects and we
56tend to define such functions/quantities in C++ namespaces.
57Here we use the namespace \c FishSolnOneDPoisson to define
58the source function (3) and the exact
59solution (4). Both functions use the integer value
60FishSolnOneDPoisson::Sign, which can be set by the "user" in
61the driver code.

71\section main The driver code
72In order to solve the Poisson problem with \c oomph-lib,
73we represent the mathematical
74problem defined by equations (1) and (2)
75in a specific \c Problem object, \c OneDPoissonProblem.
76\c oomph-lib provides a variety
77of 1D Poisson elements (1D elements with linear, quadratic and cubic
78representations for the unknown function) and we pass the
79specific element type as a template parameter to the \c Problem.
80In the driver code, listed below, we use the \c QPoissonElement<1,4>,
81a four-node (cubic) 1D Poisson element. We pass the number of elements in the
82mesh as the first argument to the \c Problem constructor and the pointer
83to the source function, defined in the namespace \c FishSolnOneDPoisson, as the second.
84
85Once the problem has been created, we execute \c Problem::self_test(),
86which checks that the \c Problem has been properly set up.
87If this test is passed, we proceed to solve the problem:
88Initially, we set the sign of the source function (defined in the
89variable \c FishSolnOneDPoisson::Sign) to -1 and solve,
90using \c oomph-lib's Newton solver. We write the
91solution to output files, using the \c OneDPoissonProblem's
92member function \c doc_solution(), discussed below. We then repeat
93the process, using a positive sign for the source function.
94
95\dontinclude one_d_poisson.cc
96\skipline start_of_main
97\until end of main

103\section problem The problem class
104\c oomph-lib driver codes tend to be
105very compact and "high-level" because all the "hard work" is done
106in the \c Problem specification. For our simple Poisson
107problem, this step is completely straightforward:
108
109The \c OneDPoissonProblem is derived from \c oomph-lib's generic
110\c Problem class and, as discussed above, the specific element
111type is specified as a template parameter to make it easy for
112the "user" to change the element type in the driver code.
113
114\dontinclude one_d_poisson.cc
115\skipline start_of_problem_class
116\until public Problem
117
118The \c OneDPoissonProblemclass has five member functions, only three of which
119are non-trivial:
120- the constructor \c OneDPoissonProblem(...)
121- the function \c actions_before_newton_solve()
122- the function \c doc_solution(...)
123.
124The function \c Problem::actions_after_newton_solve() is a pure
125virtual member function of the \c Problem base class and must
126be provided. However, it is not required in the present problem so
127we leave it empty. Similarly, the problem destructor can remain
128empty as all memory de-allocation is handled in the destructor of the
129\c Problem base class. The Problem only stores one private data
130member, the pointer to the source function.
131
132\skipline public:
133\until end of problem class
134

136 <TABLE>
137 <TR>
138 <TD bgcolor="cornsilk">
139<CENTER><B>\anchor gen_conv A general convention</B></CENTER>
140The type \c PoissonEquations<1>::PoissonSourceFctPt, used to
141define the type of the source function pointer,
142is a public typedef, defined in \c oomph-lib's Poisson equation
143class, as follows:
144
145\code
146
147[...]
148
149public:
150
151 /// \short Function pointer to source function has the form fct(x,f(x)).
152 /// Note that x is a Vector!
153 typedef void (*PoissonSourceFctPt)(const Vector<double>& x, double& f);
154
155[...]
156
157\endcode
158
159This reflects a \b general \b convention in \c oomph-lib: The function types
160of source functions etc. that are required by specific elements,
161are always declared as public types in the element classes.
162The logic behind this is that only the element writer/maintainer knows
163what type of function (i.e. the type of its arguments and its return
164value) a specific element requires. For instance, the source function
165for the Poisson equation requires the spatial coordinate \c x as
166input and computes the (scalar) value of the source function.
167Since the syntax for C++ function pointers is somewhat "non-obvious"
168(that's according to Bjarne Stroustrup, the designer of C++,
169himself!), we use typedefs to give the function pointers more
170intuitive names. Since the typedefs are public, they can be used
171anywhere in the "user's" code.
172
173 While we're at it, here's \b another
174\b convention: If an \c oomph-lib function has input and output
175arguments in its argument list, the input arguments appear first
176(and are usually passed as constant references), while the output
177arguments appear last (and are passed as references).
178</TD>

185\section constructor The Problem constructor
186In the \c Problem constructor, we define the domain length and
187build a \c Mesh, using \c oomph-lib's \c OneDMesh object
188which is templated by the element type. The required number of elements and
189the domain length are passed as arguments to the \c OneDMesh
190constructors. The subsequent lines of code
191pin the nodal values at the two boundary nodes. Next we pass the
192source function pointer to the elements. Finally, we call the generic \c
193Problem::assign_eqn_numbers() routine which does precisely
194what it says...
195
196\skipline start_of_constructor
197\until end of constructor
198
199The cast in the loop over the elements
200is required because \c Mesh::element_pt(...) returns
201a pointer to the element base class \c GeneralisedElement, which does not
202have an access function to the Poisson element's source function pointer.
203
204 <TABLE>
205 <TR>
206 <TD bgcolor="cornsilk">
207<CENTER><B>\anchor const_conv A general convention</B></CENTER>
208 It might seem more natural to pass
209essential parameters (such as the Poisson element's source function
210pointer) to an element when the element is created. If we
211made the source function pointer an argument of the element constructor,
212we would not have to execute this loop in the constructor.
213
214However, \c oomph-lib employs a general \b convention
215that element constructors should not have any arguments.
216This is because elements are usually created
217by the \c Mesh constructor. To
218allow \c Meshes that were originally developed for one particular
219element type (e.g. a quadrilateral Poisson element) to be used with
220other elements of same geometry (e.g. a quadrilateral Navier-Stokes
221element), \c Mesh objects are usually templated by the
222element type. The \c Mesh constructor creates elements
223by calling the element's default (argument-free!)
224constructor. If we were to set any element-specific arguments via arguments
225to the element constructor it would be impossible to re-use the
226\c Mesh with other element types -- Navier-Stokes elements, for instance, do
227not have a source function pointer. The actions performed in
228this loop are therefore
229fairly typical as most non-trivial elements need to be passed
230some additional information to become fully functional.
231
232</TD>
233</TR>
234</TABLE>

239\section actions_before "Actions before solve"
240The pure virtual function \c Problem::actions_before_newton_solve() must
241be implemented for all specific \c Problems and, as the name
242suggests, should perform any actions that need to be performed
243before the system of equations is solved. In the current problem,
244we use \c Problem::actions_before_newton_solve() to update the boundary
245conditions in response to possible changes in the sign of the source function.
246We use the exact solution (specified in the namespace \c FishSolnOneDPoisson)
247to determine the boundary values that are appropriate for the sign
248specified in \c FishSolnOneDPoisson::Sign.

259\section doc Post-processing
260The function \c doc_solution(...) writes the FE solution and the corresponding
261exact solution, defined in \c FishSolnOneDPoisson::get_exact_u(...)
262to disk. The argument \c label is used to add identifiers
263to the output file names. Note that all output functions
264are implemented in the generic \c Mesh class:
265- The function \c Mesh::output(...) executes the \c
266 FiniteElement::output(...) function for each
267 element in a mesh. For 1D Poisson elements, this function
268 writes the values of \f$ x \f$ and \f$ u(x) \f$ at \c npts uniformly spaced
269 points in the element to the specified file.
270- The function \c Mesh::output_fct(...) plots the function specified
271 by the function pointer in its last argument at the specified
272 number of points in each of the constituent elements. This allows
273 point-by-point comparisons between exact and FE solutions. Here
274 we plot the exact solution at a larger number of points to ensure
275 that the exact solution looks smooth even if only a small number of
276 elements are used for the discretisation of the ODE.
277.
278Finally, we call the function \c Mesh::compute_error(...)
279which determines the square of the L2 error, based on the
280difference between the exact solution (specified by a function
281pointer) and the FE solution. We also plot the pointwise error
282in the specified output file.

292\section exercises Exercises
293-# Run the code with different numbers of elements.
294 How does the error between the exact and the analytical solution
295 change?
296-# Compare the error obtained with different element types --
297 replace the four-node Poisson element, \c QPoissonElement<1,4>,
298 by its lower-order counterparts \c QPoissonElement<1,3> and
299 \c QPoissonElement<1,2>.
300-# The fish-shaped exact solution (4) is fairly smooth.
301 Postulate a more rapidly varying "exact" solution, such as
302 \f[
303 u(x) = \tanh\left[\alpha \left(x-\frac{1}{2}\right)\right]
304 \f]
305 which produces a "step" at \f$ x=1/2 \f$ when \f$ \alpha \f$ becomes
306 sufficiently large. Calculate the source function required for this
307 function to be an exact solution and implement both functions in another
308 namespace, \c TanhSolnOneDPoisson, say. Replace the reference
309 to \c FishSolnOneDPoisson by \c TanhSolnOneDPoisson and repeat
310 the above exercises.
311-# Remove the Dirichlet boundary condition at the left end of the
312 domain. What do you observe? [We shall return to this question

314 another example</A>

316 <A HREF="../../../poisson/two_d_poisson_flux_bc/html/index.html">
