56 \section interface_changes Policy on interface changes
57 
58 We obviously try to avoid interface changes as much
59 as possible as they may force users to adjust their code
60 when upgrading to a new version. We will only change interfaces
61 (names of objects and/or member functions, or the number or
62 order of their arguments) if
63 -# The previously chosen name turned out to be too ambiguous.
64  For instance, the change from
65  \code
66  Problem::actions_before_solve()
67  \endcode
68  to
69  \code
70  Problem::actions_before_newton_solve()
71  \endcode
72  became necessary
73  because of the addition of further nonlinear (but non-Newton) solvers
74  (e.g. the segregated FSI solver). Since such solvers may call the Newton
75  solver themselves, a more fine-grained control over what we
76  mean by "solving" was required. \n\n
77 -# The previously chosen name violated our own
78  <a href="../../coding_conventions/html/index.html">
79  coding conventions.</a> For instance,
80  \code
81  FiniteElement::n_nodal_position_type()
82  \endcode
83  had to be
84  changed to
85  \code
86  FiniteElement::nnodal_position_type()
87  \endcode
88  because we do not want underscores after the leading "n" in an access
89  function that returns the number of certain objects. \n\n
90 -# The order of the function arguments violated our own coding
91  conventions. For instance,
92  \code
93  FiniteElement::get_x(Vector<double>& s, const unsigned& t,...)
94  \endcode
95  had to be changed to
96  \code
97  FiniteElement::get_x(const unsigned& t, Vector<double>& s,...)
98  \endcode
99  because the discrete "time" argument always comes first.
100  \n\n
101 .
102 Changes to function or object names are easy to detect because
103 the compiler will not find the old version when linking
104 against a new version of the library, encouraging the user
105 to consult this list to (hopefully) find the appropriate
106 replacement.
107 
108  Changes to the order of function arguments can be very dangerous: if the order
109 of two arguments of the same type is exchanged the compiler cannot
110 detect the change to the interface but the code is likely to compute
111 different results. To facilitate the detection of such changes, the
112 new version of such functions contains an explicit warnings that can
113 be enabled by compiling the library with the macro \c
114 WARN_ABOUT_SUBTLY_CHANGED_OOMPH_INTERFACES. (For a gcc compiler this
115 is done with the compiler flag
116 \c -DWARN_ABOUT_SUBTLY_CHANGED_OOMPH_INTERFACES).
117 
118 Here is an example of a function in which the order of the two
119 unsigned arguments was changed. If compiled with \c
120 WARN_ABOUT_SUBTLY_CHANGED_OOMPH_INTERFACES, the code will issue
121 an appropriate warning every time this function is called.
122 
123 \code
124 
125 // Function of two unsigned arguments whose order has changed
126 // since the previous release
127 void SomeClass::some_function(const unsigned& t, const unsigned& i)
128 {
129 
130  [...]
131 
132  #ifdef WARN_ABOUT_SUBTLY_CHANGED_OOMPH_INTERFACES
133 
134  // Throw an oomph-lib warning to alert the user to the change
135  // in the order of the arguments
136  OomphLibWarning("Warning: Order of interfaces has changed",
137  "SomeClass::some_function(...)",
138  OOMPH_EXCEPTION_LOCATION);
139 
140  #endif
141 
142  [...]
143 
144 }
145 
146 \endcode
147 
148 Obviously, you should only compile the library with
149 \c WARN_ABOUT_SUBTLY_CHANGED_OOMPH_INTERFACES if you encounter
150 problems after upgrading to a new version.

158 \subsubsection new_functionality_two_point_zero Major new functionality/changes
159 
160 - Moved to C++11 standard and now use automatic formatting of code in
161  the \c src directory, using \c clang-format.
162  \n\n
163 - Upgraded all python scripts to python3.
164  \n\n
165 - Included \c CGAL, \c GMG, \c MPFR and \c Boost into the third-party libraries
166  that are shipped and automatically installed with \c
167  oomph-lib. Existing installations can be used instead (via suitable
168  flags in the configure options), but \c oomph-lib
169  is only guaranteed to work with the versions provided. This functionality
170  does not work on macOS so macOS users must add
171  the \c --enable-suppress-cgal-build flag to their configuration options
172  to ensure that this doesn't break their build.
173  \n\n
174  They are used for more efficient implementations of \c
175  locate_zeta(...) functionality.
176  \n\n
177 - Generalised unstructured 3D (tet-)mesh generation procedures so it can use \c
178  tetgen and \c gmsh.
179  \n\n
180 - Provided scripts that (i) download the third-party tar files (for \c MUMPS,
181  \c trilinos and \c hypre) and (ii) move the built installations to
182  permanent locations.
183  \n\n
184 - Provided space-time discretisations for time-periodic unsteady-heat and
185  Navier-Stokes equations.
186  \n\n
187 - Implemented a geometric multigrid solver for scalar PDEs (tested
188  for the Poisson and Helmholtz equations).
189  \n\n
190 - Implemented the \c QuadFromTriangleMesh to construct quad-based meshes
191  from triangle meshes.
192  \n\n
193 - Provided functionality to impose periodicity for three-dimensional
194  brick meshes.
195  \n\n
196 - Provided machinery to create three-dimensional meshes based on the
197  "extrusion" of a two-dimensional mesh; takes time-dependent
198  macro-element behaviour into account.
199  \n\n
200 - Implemented a Krylov-subspace solver for the solution of
201  single-equation-augmented linear systems (\c AugmentedGMRES ).

208 \subsection zero_point_nine_to_one_point_zero Changes between version 0.9 and 1.0.*

210 \subsubsection new_functionality_one_point_zero Major new functionality
211 - Provided the capability to solve the Helmholtz equation (with
212  various ways of implementing the
213  Sommerfeld radiation condition), the time-harmonic equations of linear
214  elasticity, and the interaction between these two systems of
215  equations in time-harmonic acoustic fluid structure
216  interaction problems. All equations are implemented in cartesian and
217  cylindrical polar coordinates (in the latter case, using an azimuthal Fourier
218  decomposition for the solution).
219  \n\n
220 - Provided block preconditioners for the solution of
221  large-displacement fluid-structure interaction problems, based the
222  coupling of full
223  nonlinear 2D/3D elasticity and the Navier-Stokes equations.
224  \n\n
225 - Implemented the linearised Navier-Stokes equations (with the option to study
226  their solutions as perturbations relative to a base state that is itself
227  governed by the full Navier-Stokes equations).
228  \n\n
229 - Provided elements for the solution of the axisymmetric equations of
230  linear elasticity. \n\n
231 - Finished what is hopefully the final rewrite of the (parallel) block
232  preconditioning framework and its extensive tutorial.
233  \n\n
234 - Provided elements for the solution of the Foeppl-von-Karman
235  equations in various formulations (using either an Airy-stress
236  function or the in-plane displacements) and in various coordinate
237  systems.
238  \n\n
239 - Provided elements for Darcy porous media flow and Biot-type
240  poro-elasticity, including the ability to use the latter elements
241  in fluid-structure interaction problems which are governed by the equations
242  of poro-elasticity coupled to the Navier-Stokes equations.
243  \n\n
244 - Implemented the ability adapt unstructured meshes in 2D
245  (in serial and in parallel) and (in a parallel setting) to
246  distribute such meshes over multiple processors, and to perform
247  load balancing in spatially adaptive simulations.
248  \n\n
249 - Provided elements for the solution of elements for the solution
250  of the Navier-Stokes equations with generalised Newtonian
251  constitutive equations (i.e. non-Newtonian behaviour in which
252  the viscosity depends on the invariants of the rate-of-strain
253  tensor).\n\n
254 - Provided machinery to output paraview (vtu and vtp) files directly
255  (implemented for selected elements; broken virtual functions
256  in base classes provide instructions what to do for others). \n\n
257 - Provided machinery to establish what the equations/unknowns in
258  a fully-built problem represent. Calling <code>Problem::describe_dofs()</code>
259  trawls recursively through the oomph-lib object hierarchy to
260  describe what each degree of freedom represents (from the Problem's,
261  Meshes' and Nodes' point of view). Very useful for debugging (e.g.
262  "Why are all the entries in the n-th row/column in the Jacobian
263  zero?" or "The n-th residual is enormous -- which equation does it
264  correspond to?", etc.)
265  \n\n
266 - Provided machinery for hp adaptivity and p-type spectral elements
267  with mortar constraints in 2D and 3D.\n\n
268 - Added interfaces to the Trilinos ANASAZI eigensolver to allow
269  parallel solution of eigenproblems. \n\n
270 - Basic implementation of discontinuous Galerkin methods for general
271  flux transport problems.\n\n
272 - Many new detailed tutorials explaining:\n\n
273  - How to use \t oomph-lib in parallel and how to (optionally) distribute
274  problems over multiple processors and to perform load balancing when
275  performing spatially-adaptive, distributed simulations.
276  \n\n
277  - How to use oomph-lib's block preconditioning framework.
278  \n\n
279  - How to solve large-displacement fluid-structure interaction
280  problems, based the coupling of full nonlinear 2D/3D elasticity
281  and the Navier-Stokes equations, incl. the use of preconditioners
282  and spatial adaptivity. \n\n
283  - How to generate unstructured 2D meshes whose curvlinear boundaries are
284  described my \c GeomObjects; these boundaries are respected under
285  mesh refinement, i.e. the mesh converges to the exact domain geometry,
286  rather than using the geometry defined by the initial, coarse
287  discretisation. \n\n
288  - How to simulate steady and unsteady free-surface flows in various
289  geometries. \n\n
290  - How to simulate the transport of immersed solid particles in
291  viscous flow. \n\n
292  - How to implement and solve surface transport equations, such
293  as insoluble and soluble surfactant problems. \n\n
294  - How to solve the Helmholtz equations in a variety of coordinate
295  systems and how to apply the Sommerfeld radiation condition by
296  ABCs, DTNs, or PMLs. \n\n
297  - How to solve time-harmonic acoustic fluid-structure interaction
298  problems in a variety of coordinate systems. \n\n
299  - ...\n\n
300 .
301 Have a look through the <a href="../../example_code_list/html/index.html">
302 extensive list of example codes</a> to see what else is available.

308 \subsubsection new_functionality_zero_point_nine Major new functionality
309 - Significantly extended \c oomph-lib's parallel processing
310  capabilities. The library's solver and (block) preconditioning framework is
311  now fully parallelised; the parallel assembly of the Jacobian
312  and residual vector has been optimised, and problems can be
313  now distributed by domain decomposition techniques.
314  <a href="../../mpi/general_mpi/html/index.html">A new tutorial</a>
315  and various <a href="../../example_code_list/html/index.html#parallel">
316  new demo driver codes</a> demo driver codes
317  provide an overview of these capabilities.
318  \n\n
319 - Completed the documentation of \c oomph-lib's solid
320  mechanics (large-displacement elasticity) capabilities. We provide
321  an overview of the
322  <a href="../../solid/solid_theory/html/index.html">theory and
323  implementation</a>, as well as
324  <a href="../../example_code_list/html/index.html#solid">numerous
325  tutorials</a> discussing the solution of steady and unsteady,
326  2D and 3D problems with compressible and incompressible
327  materials on structured and unstructured meshes.
328  \n\n
329 - Developed a <a href="../../meshes/mesh_from_vmtk/html/index.html">
330  tutorial</a> that shows how to use the
331  <a href="http://www.vmtk.org">Vascular
332  Modeling Toolkit (VMTK)</a> to generate \c oomph-lib meshes
333  from medical images. The methodology is used in the following driver codes:
334  \n\n
335  - <a href="../../solid/vmtk_solid/html/index.html">The inflation of a
336  blood vessel.</a>
337  \n\n
338  - <a href="../../navier_stokes/vmtk_fluid/html/index.html">
339  Finite Reynolds number flow through a (rigid) iliac bifurcation.</a>
340  \n\n
341  - <a href="../../interaction/vmtk_fsi/html/index.html">
342  Finite Reynolds number flow through an elastic iliac bifurcation.</a>
343  .
344  \n\n
345 - Extended FSI capabilities so that such problems can be solved
346  on unstructured meshes. We provide new tutorials
347  that demonstrate this capability for
348  <a href="../../interaction/unstructured_fsi/html/index.html">2D</a> and
349  <a href="../../interaction/unstructured_three_d_fsi/html/index.html">3D</a>
350  problems.
351  \n\n
352 - Introduced \c ElementWithExternalElement as base class for
353  multi-physics elements in which elements in different meshes/
354  domains interact via source/load functions (e.g. in
355  fluid-structure interaction or thermal convection problems).
356  See the new tutorials discussing the
357  <a href="../../multi_physics/multi_domain_ref_b_convect/html/index.html">
358  serial</a> and
359  <a href="../../mpi/boussinesq_convection/html/index.html">parallel</a>
360  solution of multi-physics problems by multi-domain approaches.
361  \n\n
362 - Parallelised and optimised the \c MeshAsGeomObject class to use
363  bin structures to accelerate the search in \c locate_zeta(...).
364  \n\n
365 - \c FiniteElements are now \c GeomObjects within which their local
366  coordinates act as their intrinsic coordinates.
367  \n\n
368 - Extended the \c Z2ErrorEstimator so that the elemental error estimate can be
369  based on the maximum of various distinct fluxes -- useful for
370  multi-field problems such as Boussinesq convection. See the
371  <a href="../../multi_physics/refine_b_convect/html/index.html#comments">
372  tutorial</a> for details.
373  \n\n
374 - Added a structured, domain-based mesh for the discretisation of
375  tube-like domains. See the <a href="../../meshes/mesh_list/html/index.html">
376  list of structured meshes</a> for details.
377  \n\n
378 - Provided a new Lagrange-multiplier-based element, that allows the
379  imposition of parallel in- or outflow from boundaries that are not
380  aligned with coordinate planes. This is discussed in a
381  <a href="../../navier_stokes/vmtk_fluid/html/index.html">
382  separate tutorial.</a>
383  \n\n
384 - Provided a new mechanism that allows multiple \c FaceElements to be
385  attached to same node, while providing the ability to distinguish
386  between different Lagrange multipliers stored at the same node.
387  This is discussed
388  in a <a href="../../interaction/vmtk_fsi/html/index.html">
389  separate tutorial.</a>
390  \n\n
391 - Significantly extended oomph-lib's bifurcation tracking routines
392  (no tutorials yet, though).
393  \n\n
394 - Provided general framework and driver codes for discontinuous
395  Galerkin methods and explicit timesteppers (no tutorials yet, though).
396  \n\n
