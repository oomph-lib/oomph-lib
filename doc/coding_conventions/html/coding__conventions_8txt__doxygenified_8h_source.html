3 \mainpage Coding conventions and C++-style
4 
5 This document provides an overview of the general coding conventions that
6 are used throughout \c oomph-lib. Knowledge of these conventions
7 will greatly facilitate the use of the library. Contributors
8 to the library are expected to adhere to these standards.
9 
10 <br>
11 Note that following our move to GitHub we have started to use
12 <a href="https://clang.llvm.org/docs/ClangFormat.html">clang-format</a>
13 to automatically format our sources (but only those in the
14 <code>src</code> directory). The formatting rules are contained in the
15 <code>.clang-format</code> file in the <code>oomph-lib</code> home
16 directory. To run this on your own sources you need at least version
17 10.0.0 of <code>clang-format.</code> Note that some of the code shown
18 below would be changed slightly by <code>clang-format</code> but this
19 concerns mainly minor whitespace adjustments and the beauty of
20 automatic formatting means that you don't have to worry about these
21 too much.

25 \section naming Naming conventions
26 
27  \subsection filenames File names
28 
29  All C++ source files end with the standard extensions *.h and *.cc.
30 
31  - \c *.h: Contains the class definitions and any inline functions.
32  - \c *.cc: Contains all non-inline member functions that can be compiled
33  once and for all. This includes
34  - member functions of classes that do not have any template parameters
35  - member functions of templated classes for which it is
36  known a priori which instantiations are required. Examples
37  are classes that are templated by the spatial dimension. In this
38  case we're unlikely to require instantiations for any values
39  other than 0,1,2 and 3.
40  .
41  - \c *.template.cc: Contains any non-inline member function of
42  templated classes. This file must be included (together
43  with the corresponding \c *.h file) when a specific instantiation
44  of a templated class is required. For instance,
45  most specific \c Mesh classes are templated by
46  the element type and the mesh writer can obviously not predict
47  which element types his/her specific mesh is going to be
48  used with.
49

52  \subsection variables General variables
53  - General variables are all lowercase
54  - Variables that contain multiple words contain underscores
55  to separate them, e.g.
56  \code FiniteElement* surface_element_pt; \endcode
57  .
58 
59 
60  \subsection classes Classes
61  - Classes start with capital letter, e.g.
62  \code class Shape \endcode
63 
64  - If the class name contains multiple words, the first letter
65  of any subsequent word also gets capitalised, e.g.
66  \code class AlgebraicElement \endcode
67  \b Note: No underscores in class names.
68 
69 
70 
71 
72  \subsection private_public Private data and access functions to private data
73  Use a capital first letter for private data, and the
74  all-lowercase equivalent for the access functions. Examples:
75  - This is a declaration for a private data member:
76  \code
77  private:
78 
79  /// Pointer to boundary node
80  Node* Boundary_node_pt; \endcode
81  - Here are two public access functions to the private data member:
82  \code
83  public:
84 
85  /// Access to boundary node (const version)
86  Node* boundary_node_pt() const {return Boundary_node_pt;}
87 
88  /// Access to boundary node
89  Node*& boundary_node_pt() {return Boundary_node_pt;}
90 
91  \endcode
92 .
93  \b Note: Do \b not use public data -- ever! Make it private and
94  and provide an access function -- even if it seems "perfectly
95  obvious" at the time of writing the class that the internal
96  storage for the data item is "never going to be changed".
97 
98  \subsection pointers Pointers
99  - Pointers and access functions to pointers are identified
100  explicitly by the postfix \a _pt to the variable names,
101  as shown in the previous examples.
102 
103  \subsection numbering Access functions to containers
104 
105  Many classes have member functions that provide access to data
106  in private containers (e.g. vectors); they are usually accompanied
107  by a member function that returns the number of entries
108  in that container. Naming conventions:
109  - Use singular for the access function to the container, i.e.
110  \code
111  /// Return pointer to e-th element
112  FiniteElement* element_pt(const unsigned& e);
113  \endcode
114  rather than \c elements_pt(...)
115  - Use a prefix `\c n' for the access function for the
116  number of entries in the container, i.e.
117  \code
118  /// Total number of elements
119  unsigned nelement();
120  \endcode
121  \b Notes: (i) No underscore between the "n" and the container's name;
122  you can then use the underscore for the name of the local variable
123  that stores the value, e.g.
124  \c
125  unsigned \c n_element \c = \c my_mesh_pt->nelement();
126 (ii) No trailing \c _pt in the function that returns the number
127 of objects in the container.
128 
129 \subsection template Template parameters
130 - Template parameters are all caps, e.g.
131  \code
132  template<unsigned DIM>
133  class NavierStokesEquations
134  {
135  public:
136  [...]
137 
138  };
139  \endcode
140 - Sometimes it is necessary to expose the template parameter to make it
141  accessible to the user. This should be done by adding a public \c static
142  \c const copy of the parameter to the class's member data, using the
143  prefix \c TEMPLATE_PARAMETER_ , so the template parameter
144  \c DIM in the above example would be exposed like this:
145 \code
146  public:
147  /// Publicly exposed template parameter
148  static const unsigned TEMPLATE_PARAMETER_DIM = DIM;
149 \endcode
150  Note the exposure of template parameters is optional and is only
151  implemented when we needed it. Feel free to modify existing code
152  yourself if required -- \c oomph-lib is open source!
153 .

155 \subsection verbose Use descriptive function/variable names
156 
157 - Make sure you choose descriptive names for functions and variables,
158  even if the names become long.

191 \subsection layout_functions_etc Layout of functions, classes, etc.
192 - Precede all functions by a comment block, enclosed between lines of `==='
193  \code
194  // =============================================================
195  /// \short (add \short to make sure that multi-line descriptions
196  /// appear in doxygen's short documentation. Include lists with items
197  /// - first item
198  /// - second item
199  /// - first item of sublist
200  /// - second item of sublist
201  /// . //end of sublist
202  /// . //end of main list
203  // =============================================================
204  void SomeClass::some_function()
205  {
206  for (unsigned i=0;i<10;i++)
207  {
208  std::cout << "doing something" << std::endl;
209  }
210  }
211  \endcode
212  Note the triple slash "///" in the comment block that preceeds the
213  function definition -- comments contained in such lines are
214  automatically extracted by doxygen and inserted into the
215  code documentation.
216 
217 - All member data is defined (and annotated) on a separate line,
218  otherwise doxygen gets confused. So use
219  \code
220  /// Number of elements
221  unsigned Nelement;
222 
223  /// Number of nodes
224  unsigned Nnode;
225  \endcode
226  rather than
227  \code
228  /// Number of elements and nodes
229  unsigned Nelement, Nnode;
230  \endcode

233 \subsection oomph The oomph-lib namespace
234 - \c oomph-lib is contained in its own namespace, \c oomph,
235  to avoid clashes of class names with those of other
236  (third-party) libraries. If there is no danger of
237  name clashes, the entire \c oomph
238  namespace may be imported at the beginning of each driver
239  code by placing the statement
240  \code
241  using namespace oomph;
242  \endcode
243  at the beginning of the source code (after the included header files).
244  Any additions to the library (this includes the instantiation
245  of templated \c oomph-lib classes inside a driver code!)
246  must be included into the \c oomph namespace
247  by surrounding the code by
248  \code
249  namespace oomph
250  {
251 
252  // Additions to the library go here...
253  [...]
254 
255  }
256  \endcode
257 
258 
259 \subsection std Namespace pollution
260 - To avoid namespace pollution, the namespace \c std \b must \b not be
261  included globally in any header files. The statement
262  \code
263  using namespace std;
264  \endcode
265  may only be used in driver codes, in *.cc files, or inside
266  specific functions in a *.h file.

268 \subsection layout_classes Layout of class definitions and include guards.
269 Here is an example of a complete header file, including include
270 guards and library includes.
271  \code
272  #ifndef OOMPH_SOME_CLASS_HEADER // Assuming that the file is
273  #define OOMPH_SOME_CLASS_HEADER // called some_class.h
274 
275  // Include generic oomph-lib library
276  #include "generic.h"
277 
278  // Add to oomph-lib namespace
279  namespace oomph
280  {
281 
282  // =============================================================
283  /// Waffle about what the class does etc.
284  ///
285  // =============================================================
286  template<class T>
287  class SomeClass : public SomeBaseClass
288  {
289  public:
290 
291  /// Constructor: Pass coefficients n1 and n2
292  SomeClass(const unsigned& n1, const T& n2) : N1(n1), N2(n2)
293  {}
294 
295  /// Access function to coefficient
296  inline unsigned n1() const
297  {
298  return N1;
299  }
300 
301  /// Access function to other coefficient
302  inline T& n2() const
303  {
304  return N2;
305  }
306 
307 
308  protected:
309 
310  /// Coefficient
311  unsigned N1;
312 
313  private:
314 
315  /// Second coefficient
316  T N2;
317 
318  };
319 
320  }
321  #endif
322 
323  \endcode
324 
325  - Order of public/protected/private may be reversed but
326  the declarations should always be explicit (even though
327  everything is private by default).
328  - Note the prefix <CODE> OOMPH_*</CODE> in the include guard. This is to
329  avoid clashes with include guards of other libraries.

340  \subsection paranoia The PARANOID flag and error handling
341  - Implement optional validation routines, self-tests, and other
342  sanity checks via conditional compilation, using the compiler
343  flag PARANOID, so that the relevant statements are only activated if
344  \c -DPARANOID is specified as a compilation flag
345  for the C++ compiler.
346  If errors are detected, a meaningful diagnostic should be issued,
347  by throwing an \c OomphLibError.
348  If the code is compiled without
349  the PARANOID flag, all sanity checks are bypassed -- good for
350  the overall execution speed, bad for error handling... The
351  user can choose.
352  \n\n
353  Here's an example: \n\n
354  \code
355 
356  // Has a global mesh already been built?
357  if(Mesh_pt!=0)
358  {
359  std::string error_message =
360  "Problem::build_global_mesh() called,\n";
361  error_message += " but a global mesh has already been built:\n";
362  error_message += "Problem::Mesh_pt is not zero!\n";
363 
364  throw OomphLibError(error_message,
365  OOMPH_CURRENT_FUNCTION,
366  OOMPH_EXCEPTION_LOCATION);
367  }
368  \endcode
369  - \c oomph-lib also has an object that allows warning messages to
370  be issued in a uniform format. Here's an example of its use:
371  \code
372  // Was it a duplicate?
373  unsigned nel_now=element_set_pt.size();
374  if (nel_now==nel_before)
375  {
376  std::ostringstream warning_stream;
377  warning_stream <<"WARNING: " << std::endl
378  <<"Element " << e << " in submesh " << imesh
379  <<" is a duplicate \n and was ignored when assembling "
380  <<"global mesh." << std::endl;
381  OomphLibWarning(warning_stream.str(),
382  OOMPH_CURRENT_FUNCTION,
383  OOMPH_EXCEPTION_LOCATION);
384  }
385  \endcode

388 \subsection range Range checking
389 
390 - Most access functions that provide indexed access to
391  a private container, do, in fact, access a private STL vector.
392  Explicit range checking for these (frequent!) cases can be avoided
393  by changing to container to Vector class instead. Vectors
394  performs automatic range checking, if the \c generic library
395  is compiled with the \c RANGE_CHECKING flag, i.e. if
396  \c -DRANGE_CHECKING is specified as a compilation flag
397  for the C++ compiler.
398  \b Note: While it is generally a good idea to compile
399  with \c PARANOID while developing code, \c RANGE_CHECKING
400  is \b very \b expensive and is therefore activated via
401  a second independent flag. We only tend to active this flag
402  as a last resort, typically to track down particularly
403  stubborn segmentation faults.
404 
405  \subsection self_test Self test routines
406  - Every sufficiently complex class should come with its
407  own
408  \code unsigned self_test() \endcode
409  routine which returns 1 for failure, 0 for successful test.

419 \subsection constness Const-ness
420 - Use const wherever applicable (arguments, member functions,...)
421 - Always provide const and non-const overloaded subscript operators.
422 - Example:
423 \code
424 
425  // Return i-th coordinate of Point
426  double& operator[](const unsigned& i){return x[i];}
427 
428  // Return i-th coordinate of Point -- const version
429  const double& operator[](const unsigned& i) const {return x[i];}
430 
431 \endcode
432 
433 \subsection unsigned Only use int if a variable can actually take negative values
434 
435 - Just as the name of a variable gives some indication of its
436  likely use, its type does too. For instance this code fragment
437  \n
438  \code
439  // Create a counter
440  int counter=0;
441  \endcode
442  \n
443  immediately raises the question why the programmer anticipates
444  circumstances in which the counter might be negative. Are negative
445  values used to indicate special cases; etc? If the name of the
446  variable was chosen correctly (i.e. if the variable really is
447  used as a counter) then
448  \n
449  \code
450  // Create a counter
451  unsigned counter=0;
452  \endcode
453  \n
454  is much clearer and therefore preferable, even if the two versions
455  of the code would, of course, give the same result.
456 .
457 
458 \subsection pass_by_reference Only use "pass by reference"
459 - Arguments to functions should only be passed "by reference",
460  not "by value". Use "pass by constant reference" if you want
461  to ensure the const-ness of any (input) arguments.
462 - To "encourage" this behaviour, most \c oomph-lib objects
463  have (deliberately) broken copy constructors and assignment
464  operators, making a "pass by value" impossible.
465  The only exceptions are cases in which we could see
466  a good reason why a fully-functional, non-memory-leaking
467  copy/assignment operator might be required.
468 
469 \subsection break_copy Provide fully-functional or deliberately-broken copy constructors and assignment operators
470 
471 - For the reasons mentioned above, "passing by value" is
472  discouraged and we have only implemented copy constructors
473  for very few classes. To make the use of C++'s default
474  copy constructor impossible (as their accidental use may lead to serious
475  memory leaks) all classes should either have a deliberately-broken
476  copy constructor or provide a "proper" implementation (as in the case of
477  \c oomph-lib's \c Vector class). The same applies to assignment
478  operators.
479  \n \n
480  With the advent of C++11, this can be done easily using the \c delete keyword:
481  \n\n
482  \code
483  /// Broken copy constructor
484  Mesh(const Mesh& dummy) = delete;
485 
486  /// Broken assignment operator
487  void operator=(const Mesh&) = delete;
488  \endcode
489 
490 \subsection order_of_args Order of arguments
491 - If values are returned from a function, put them at the end
492  of the argument list.
493 - "Time" arguments always come first, e.g.
494  \code
495  /// \short Return FE interpolated coordinate x[i] at local coordinate s
496  /// at previous timestep t (t=0: present; t>0: previous timestep)
497  virtual double interpolated_x(const unsigned& t,
498  const Vector<double>& s,
499  const unsigned& i) const;
500  \endcode
501 
502 \subsection brackets Access to elements in containers
503  - Avoid access via square brackets (i.e. via operators) and
504  write access functions instead, as they can be overloaded more
505  easily.
506 
507 \subsection boolean Boolean member data
508  - Avoid access to boolean member data via trivial wrapper functions
509  that return references. These constructions lead to somewhat ugly
510  driver codes and can lead to code that appears to set a boolean,
511  when it does not. Instead the status of the boolean should be
512  modified by two set/unset or enable/disable subroutines
513  (i.e. returning void) and tested using a (const) has_ or is_
514  function that returns a bool . For example
515  \code
516  private:
517 
518  /// Boolean to indicate whether documentation should be on or off
519  bool Doc_flag;
520 
521  public:
522 
523  /// Enable documentation
524  void enable_doc() {Doc_flag=true;}
525 
526  /// Disable documentation
527  void disable_doc() {Doc_flag=false;}
528 
529  /// Test whether documentation is on or off
530  bool is_doc_enabled() const {return Doc_flag;}
531  \endcode
532 
533 \subsection dont_use_macros Macros
534 - Don't use macros! There are two exceptions to this rule: We use the
535  macros \c OOMPH_EXCEPTION_LOCATION and \c OOMPH_CURRENT_FUNCTION to make
536  the file name, line number and current function name available to the \c
537  OomphLibException object -- the object that is thrown if a run-time error
538  is detected.
539 
540 
541  \subsection inlining Inlining
542  - Inline all simple set/get functions by placing them into the *.h
543  file.
544  - \b Careful: Inlined functions should not contain
545  calls to member functions of classes that are defined in other files
546  as this can lead to triangular dependencies.
