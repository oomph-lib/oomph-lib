3 \mainpage Coding conventions and C++-style
4 
5 This document provides an overview of the general coding conventions that
6 are used throughout \c oomph-lib. Knowledge of these conventions
7 will greatly facilitate the use of the library. Contributors
8 to the library are expected to adhere to these standards.

14 \section naming Naming conventions
15 
16  \subsection filenames File names
17 
18  All C++ source files end with the standard extensions *.h and *.cc.
19 
20  - \c *.h: Contains the class definitions and any inline functions.
21  - \c *.cc: Contains all non-inline member functions that can be compiled
22  once and for all. This includes
23  - member functions of classes that do not have any template parameters
24  - member functions of templated classes for which it is
25  known a priori which instantiations are required. Examples
26  are classes that are templated by the spatial dimension. In this
27  case we're unlikely to require instantiations for any values
28  other than 0,1,2 and 3.
29  .
30  - \c *.template.cc: Contains any non-inline member function of
31  templated classes. This file must be included (together
32  with the corresponding \c *.h file) when a specific instantiation
33  of a templated class is required. For instance,
34  most specific \c Mesh classes are templated by
35  the element type and the mesh writer can obviously not predict
36  which element types his/her specific mesh is going to be
37  used with.

41  \subsection variables General variables
42  - General variables are all lowercase
43  - Variables that contain multiple words contain underscores
44  to separate them, e.g.
45  \code FiniteElement* surface_element_pt; \endcode
46  .

49  \subsection classes Classes
50  - Classes start with capital letter, e.g.
51  \code class Shape \endcode
52 
53  - If the class name contains multiple words, the first letter
54  of any subsequent word also gets capitalised, e.g.
55  \code class AlgebraicElement \endcode
56  \b Note: No underscores in class names.

61  \subsection private_public Private data and access functions to private data
62  Use a capital first letter for private data, and the
63  all-lowercase equivalent for the access functions. Examples:
64  - This is a declaration for a private data member:
65  \code
66  private:
67 
68  /// Pointer to boundary node
69  Node* Boundary_node_pt; \endcode
70  - Here are two public access functions to the private data member:
71  \code
72  public:
73 
74  /// Access to boundary node (const version)
75  Node* boundary_node_pt() const {return Boundary_node_pt;}
76 
77  /// Access to boundary node
78  Node*& boundary_node_pt() {return Boundary_node_pt;}
79 
80  \endcode
81 .
82  \b Note: Do \b not use public data -- ever! Make it private and
83  and provide an access function -- even if it seems "perfectly
84  obvious" at the time of writing the class that the internal
85  storage for the data item is "never going to be changed".
86 
87  \subsection pointers Pointers
88  - Pointers and access functions to pointers are identified
89  explicitly by the postfix \a _pt to the variable names,
90  as shown in the previous examples.
91 
92  \subsection numbering Access functions to containers
93 
94  Many classes have member functions that provide access to data
95  in private containers (e.g. vectors); they are usually accompanied
96  by a member function that returns the number of entries
97  in that container. Naming conventions:
98  - Use singular for the access function to the container, i.e.
99  \code
100  /// Return pointer to e-th element
101  FiniteElement* element_pt(const unsigned& e);
102  \endcode
103  rather than \c elements_pt(...)
104  - Use a prefix `\c n' for the access function for the
105  number of entries in the container, i.e.
106  \code
107  /// Total number of elements
108  unsigned nelement();
109  \endcode
110  \b Notes: (i) No underscore between the "n" and the container's name.
111 (ii) No trailing \c _pt in the function that returns the number
112 of objects in the container.

164 \subsection layout_functions_etc Layout of functions, classes, etc.
165 - Precede all functions by a comment block, enclosed between lines of `==='
166  \code
167  // =============================================================
168  /// \short (add \short to make sure that multi-line descriptions
169  /// appear in doxygen's short documentation. Include lists with items
170  /// - first item
171  /// - second item
172  /// - first item of sublist
173  /// - second item of sublist
174  /// . //end of sublist
175  /// . //end of main list
176  // =============================================================
177  void SomeClass::some_function()
178  {
179  for (unsigned i=0;i<10;i++)
180  {
181  std::cout << "doing something" << std::endl;
182  }
183  }
184  \endcode
185  Note the triple slash "///" in the comment block that preceeds the
186  function definition -- comments contained in such lines are
187  automatically extracted by doxygen and inserted into the
188  code documentation.

191 \subsection oomph The oomph-lib namespace
192 - \c oomph-lib is contained in its own namespace, \c oomph,
193  to avoid clashes of class names with those of other
194  (third-party) libraries. If there is no danger of
195  name clashes, the entire \c oomph
196  namespace may be imported at the beginning of each driver
197  code by placing the statement
198  \code
199  using namespace oomph;
200  \endcode
201  at the beginning of the source code (after the included header files).
202  Any additions to the library (this includes the instantiation
203  of templated \c oomph-lib classes inside a driver code!)
204  must be included into the \c oomph namespace
205  by surrounding the code by
206  \code
207  namespace oomph
208  {
209 
210  // Additions to the library go here...
211  [...]
212 
213  }
214  \endcode
215 
216 
217 \subsection std Namespace pollution
218 - To avoid namespace pollution, the namespace \c std \b must \b not be
219  included globally in any header files. The statement
220  \code
221  using namespace std;
222  \endcode
223  may only be used in driver codes, in *.cc files, or inside
224  specific functions in a *.h file.

226 \subsection layout_classes Layout of class definitions and include guards.
227 Here is an example of a complete header file, including include
228 guards and library includes.
229  \code
230  #ifndef OOMPH_SOME_CLASS_HEADER // Assuming that the file is
231  #define OOMPH_SOME_CLASS_HEADER // called some_class.h
232 
233  // Include generic oomph-lib library
234  #include "generic.h"
235 
236  // Add to oomph-lib namespace
237  namespace oomph
238  {
239 
240  // =============================================================
241  /// Waffle about what the class does etc.
242  ///
243  // =============================================================
244  template<class T>
245  class SomeClass : public SomeBaseClass
246  {
247  public:
248 
249  /// Constructor: Pass coefficients n1 and n2
250  SomeClass(const unsigned& n1, const T& n2) : N1(n1), N2(n2)
251  {}
252 
253  /// Access function to coefficient
254  inline unsigned n1() const
255  {
256  return N1;
257  }
258 
259  /// Access function to other coefficient
260  inline T& n2() const
261  {
262  return N2;
263  }
264 
265 
266  protected:
267 
268  /// Coefficient
269  unsigned N1;
270 
271  private:
272 
273  /// Second coefficient
274  T N2;
275 
276  };
277 
278  }
279  #endif
280 
281  \endcode
282 
283  - Order of public/protected/private may be reversed but
284  the declarations should always be explicit (even though
285  everything is private by default).
286  - Note the prefix <CODE> OOMPH_*</CODE> in the include guard. This is to
287  avoid clashes with include guards of other libraries.

298  \subsection paranoia The PARANOID flag and error handling
299  - Implement optional validation routines, self-tests, and other
300  sanity checks via conditional compilation, using the compiler
301  flag PARANOID, so that the relevant statements are only activated if
302  \c -DPARANOID is specified as a compilation flag
303  for the C++ compiler.
304  If errors are detected, a meaningful diagnostic should be issued,
305  by throwing an \c OomphLibError.
306  If the code is compiled without
307  the PARANOID flag, all sanity checks are bypassed -- good for
308  the overall execution speed, bad for error handling... The
309  user can choose.
310  \n\n
311  Here's an example: \n\n
312  \code
313 
314  // Has a global mesh already been built?
315  if(Mesh_pt!=0)
316  {
317  std::string error_message =
318  "Problem::build_global_mesh() called,\n";
319  error_message += " but a global mesh has already been built:\n";
320  error_message += "Problem::Mesh_pt is not zero!\n";
321 
322  throw OomphLibError(error_message,
323  OOMPH_CURRENT_FUNCTION,
324  OOMPH_EXCEPTION_LOCATION);
325  }
326  \endcode
327  - \c oomph-lib also has an object that allows warning messages to
328  be issued in a uniform format. Here's an example of its use:
329  \code
330  // Was it a duplicate?
331  unsigned nel_now=element_set_pt.size();
332  if (nel_now==nel_before)
333  {
334  std::ostringstream warning_stream;
335  warning_stream <<"WARNING: " << std::endl
336  <<"Element " << e << " in submesh " << imesh
337  <<" is a duplicate \n and was ignored when assembling "
338  <<"global mesh." << std::endl;
339  OomphLibWarning(warning_stream.str(),
340  OOMPH_CURRENT_FUNCTION,
341  OOMPH_EXCEPTION_LOCATION);
342  }
343  \endcode

346 \subsection range Range checking
347 
348 - Most access functions that provide indexed access to
349  a private container, do, in fact, access a private STL vector.
350  Explicit range checking for these (frequent!) cases can be avoided
351  by changing to container to Vector class instead. Vectors
352  performs automatic range checking, if the \c generic library
353  is compiled with the \c RANGE_CHECKING flag, i.e. if
354  \c -DRANGE_CHECKING is specified as a compilation flag
355  for the C++ compiler.
356  \b Note: While it is generally a good idea to compile
357  with \c PARANOID while developing code, \c RANGE_CHECKING
358  is \b very \b expensive and is therefore activated via
359  a second independent flag. We only tend to active this flag
360  as a last resort, typically to track down particularly
361  stubborn segmentation faults.
362 
363  \subsection self_test Self test routines
364  - Every sufficiently complex class should come with its
365  own
366  \code unsigned self_test() \endcode
367  routine which returns 1 for failure, 0 for successful test.

377 \subsection constness Const-ness
378 - Use const wherever applicable (arguments, member functions,...)
379 - Always provide const and non-const overloaded subscript operators.
380 - Example:
381 \code
382 
383  // Return i-th coordinate of Point
384  double& operator[](const unsigned& i){return x[i];}
385 
386  // Return i-th coordinate of Point -- const version
387  const double& operator[](const unsigned& i) const {return x[i];}
388 
389 \endcode
390 
391 \subsection unsigned Only use int if a variable can actually take negative values
392 
393 - Just as the name of a variable gives some indication of its
394  likely use, its type does too. For instance this code fragment
395  \n
396  \code
397  // Create a counter
398  int counter=0;
399  \endcode
400  \n
401  immediately raises the question why the programmer anticipates
402  circumstances in which the counter might be negative. Are negative
403  values used to indicate special cases; etc? If the name of the
404  variable was chosen correctly (i.e. if the variable really is
405  used as a counter) then
406  \n
407  \code
408  // Create a counter
409  unsigned counter=0;
410  \endcode
411  \n
412  is much clearer and therefore preferable, even if the two versions
413  of the code would, of course, give the same result.
414 .
415 
416 \subsection pass_by_reference Only use "pass by reference"
417 - Arguments to functions should only be passed "by reference",
418  not "by value". Use "pass by constant reference" if you want
419  to ensure the const-ness of any (input) arguments.
420 - To "encourage" this behaviour, most \c oomph-lib objects
421  have (deliberately) broken copy constructors and assignment
422  operators, making a "pass by value" impossible.
423  The only exceptions are cases in which we could see
424  a good reason why a fully-functional, non-memory-leaking
425  copy/assignment operator might be required.
426 
427 \subsection break_copy Provide fully-functional or deliberately-broken copy constructors and assignment operators
428 
429 - For the reasons mentioned above, "passing by value" is
430  discouraged and we have only implemented copy constructors
431  for very few classes. To make the use of C++'s default
432  copy constructor impossible (as their accidental use may lead to serious
433  memory leaks) all classes should either have a deliberately-broken
434  copy constructor or provide a "proper" implementation (as in the case of
435  \c oomph-lib's \c Vector class). The same applies to assignment
436  operators.
437  \n \n
438  The namespace \c BrokenCopy provides two helper functions,
439  \c BrokenCopy::broken_copy(...) and \c
440  BrokenCopy::broken_assign(...) that issue a suitable error message
441  and then throw an \c OomphLibError. The name of the class
442  should be passed to these functions as a string, as in this
443  example from the \c Mesh class: \n\n
444  \code
445  /// Broken copy constructor
446  Mesh(const Mesh& dummy)
447  {
448  BrokenCopy::broken_copy("Mesh");
449  }
450 
451  /// Broken assignment operator
452  void operator=(const Mesh&)
453  {
454  BrokenCopy::broken_assign("Mesh");
455  }
456  \endcode

458 \subsection order_of_args Order of arguments
459 - If values are returned from a function, put them at the end
460  of the argument list.
461 - "Time" arguments always come first, e.g.
462  \code
463  /// \short Return FE interpolated coordinate x[i] at local coordinate s
464  /// at previous timestep t (t=0: present; t>0: previous timestep)
465  virtual double interpolated_x(const unsigned& t,
466  const Vector<double> &s,
467  const unsigned &i) const;
468  \endcode

470 \subsection brackets Access to elements in containers
471  - Avoid access via square brackets (i.e. via operators) and
472  write access functions instead, as they can be overloaded more
473  easily.
474 
475 \subsection boolean Boolean member data
476  - Avoid access to boolean member data via trivial wrapper functions
477  that return references. These constructions lead to somewhat ugly
478  driver codes and can lead to code that appears to set a boolean,
479  when it does not. Instead the status of the boolean should be
480  modified by two set/unset or enable/disable subroutines
481  (i.e. returning void) and tested using a (const) has_ or is_
482  function that returns a bool . For example
483  \code
484  private:
485  /// Boolean to indicate whether documentation should be on or off
486  bool Doc_flag;
487  public:
488  /// Enable documentation
489  void enable_doc() {Doc_flag=true;}
490  /// Disable documentation
491  void disable_doc() {Doc_flag=false;}
492  /// Test whether documentation is on or off
493  bool is_doc_enabled() const {return Doc_flag;}
494  \endcode
495 
496 \subsection dont_use_macros Macros
497 - Don't use macros! There are two exceptions to this rule: We use the
498  macros \c OOMPH_EXCEPTION_LOCATION and \c OOMPH_CURRENT_FUNCTION to make
499  the file name, line number and current function name available to the \c
500  OomphLibException object -- the object that is thrown if a run-time error
501  is detected.
502 
503 
504  \subsection inlining Inlining
505  - Inline all simple set/get functions by placing them into the *.h
506  file.
507  - \b Careful: Inlined functions should not contain
508  calls to member functions of classes that are defined in other files
509  as this can lead to triangular dependencies.
