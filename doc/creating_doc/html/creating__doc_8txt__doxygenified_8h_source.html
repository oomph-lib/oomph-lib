11\section setup Setup

13\subsection dirs_and_cmake Create the directory and update the CMakeLists.txt files
14
15-# Within an appropriate subdirectory of the \c doc directory,
16 create a directory for your documentation, e.g.
17 \code
18 cd doc/axisym_navier_stokes/
19 mkdir spin_up
20 \endcode
21-# Add this new directory to the parent directory's \c CMakeLists.txt file
22-# Create a new \c *.\c txt file with the same name as the
23 newly-created directory, e.g.
24 \code
25 touch spin_up.txt
26 \endcode
27 This is the "source" file from which the documentation will be
28 generated.
29-# From an existing documentation directory, copy across the following
30 files into the newly created directory:
31 - \c CMakeLists.txt
32 - \c Doxyfile
33-# In \c CMakeLIsts.txt update
34 - the name of the directory in the two messages at the start and
35 end of the file
36 - the name of the directory in the \c project command
37 - the name of the directory in the \c DOCFILE variable in the
38 \c oomph_generate-doc_from command.
39 .
40-# In \c Doxyfile update the (relative) path to the demo-driver directory. Two
41 entries must be updated, one following \c "INPUT" and the other
42 following \c "EXAMPLE_PATH". To find these, search for
43 \c "../..".
44-# Return to \c oomph-lib's doc directory and re-configure:
45 \code
46 cd doc
47 cmake -G Ninja -B build
48 cmake --build build
49 \endcode
50
51\subsection figures_setup Figures
52
53If your documentation is to contain figures or animations, the
54 following subdirectories must be created within your documentation
55 directory:
56 - \c figures
57 - This must contain all figures used in the documentation in both
58 \c *.\c gif and \c *.\c eps formats (\c *.\c gif is used for the \c html
59 documentation whilst \c *.\c eps is used for \c LaTeX).
60 - \c non_distfigures
61 - This contains any additional files (etc.) used to create or
62 maintain the documentation. It is good practise to always keep any
63 \c *.\c lay and \c *.\c lpk files that were used to create
64 figures, along with any macro (\c *.\c mcr) files.

68\section write Writing the documentation
69
70The "source" of the documentation is contained in the \c *.\c txt file (a
71mixture of \c html/doxygen markup). The title must follow the tag
72\c \\mainpage and be all on one line, e.g.
73\code
74\mainpage Demo problem: Spin-up of a viscous fluid
75\endcode
76The main body of the text just follows as in a \c LaTeX document, with
77line spacings indicating paragraph breaks.
78
79\subsection html HTML
80
81Any standard html tags can be used (for example, \c <hr> inserts a
82horizontal line). Hyperlinks are inserted in the following way:
83\code
84This is illustrated in an <A HREF="../figures/my_movie.avi">animation</A>.
85\endcode
86Note that the \c ".." is present in the path above because during the
87build process the documentation is created in subdirectories of the
88directory in which the source file is located. See the section \ref
89generate below for more details.
90
91To link to the documentation of another demo driver, \c
92poisson/one_d_poisson for example, hyperlink to the \c index.html file
93within the \c html directory of that demo driver's documentation. Note
94that because the processed (\c html) version of the documentation you are
95working on will live in its own \c html subdirectory, it is necessary
96to go up three directories in order to be in \c oomph-lib's \c doc directory.
97\code
98...see <A HREF="../../../poisson/one_d_poisson/html/index.html">the
99Poisson tutorial</A>.
100\endcode

102\subsection sections Sections
103
104Sections are created as follows:
105\code
106\section section_label This is the title of my section
107\endcode
108
109Likewise, subsections are created in the following way:
110\code
111\subsection subsection_label This is the title of my subsection
112\endcode
113
114Any \c LaTeX section types can be used in this way. To link to a
115section within a document, use the syntax \c \\ref \c section_label,
116as in this example:
117\code
118...can be found in the section \ref theory below.
119\endcode

121\subsection equations Equations
122
123Equations are generated as in \c LaTeX except \c \\f must be added
124before \c \$ or \c [, \c ].
125- E.g. inline maths:
126 \code
127 ...is given by \f$ \sin(x) \f$
128 \endcode
129- E.g. equation environment:
130 \code
131 \f[
132 \sin(x)
133 \f]
134 \endcode
135
136To label equations, the normal \c LaTeX system cannot be used. Instead,
137mark an equation with the tag \c \@E\c [LABEL\c ]\@ and refer to it later
138using \c \@R\c [LABEL\c ]\@, e.g.
139
140\image html labelling_example.gif " "
141\image latex labelling_example.eps " " width=0.71\textwidth
142
143and then later on...
144\code
145...is given by equation (@R[eqn:cont]@).
146\endcode

148\subsection lists Lists
149
150To create bullet point lists, precede each item with a \c -, e.g.
151\code
152- First item
153- Second item
154\endcode
155To create enumerated lists, precede each item with a \c -#, e.g.
156\code
157-# First item
158-# Second item
159\endcode

161\subsection figures Figures
162
163A figure with the filename \c my_figure.\c * is inserted in the
164following way:
165
166\image html insert_figures.gif " "
167\image latex insert_figures.eps " " width=0.71\textwidth
168
169Note the space between the last character in the caption and the
170quotation marks. Like the equation labelling, this line is processed
171by the \c txt2h.sh script (which is run automatically as part of the
172\c make process) and replaced with the necessary commands that tell
173\c doxygen
174to use the \c *.\c gif files for the html documentation and the \c *.\c
175eps files for the \c LaTeX documentation.
176
177\subsection code Code
178
179To insert single words of code into prose, precede the word with a
180\c \\c, e.g.
181\code
182The function \c FiniteElement::output(...) is used to...
183\endcode
184
185To include blocks of code such as the one immediately above this line
186of text, use the \c \\code environment, e.g.
187
188\image html code_environment.gif " "
189\image latex code_environment.eps " " width=0.71\textwidth
190
191To include sections of the demo code which you are documenting,
192e.g. the main function of \c spin_up.\c cc, use the following syntax:
193\code
194\dontinclude spin_up.cc
195\skipline start_of_main
196\until end of main
197\endcode
198This only works if \c start_of_main exists somewhere in \c spin_up.\c cc
199file, but any word(s) can be used as a start/endpoint. However, <B>do
200not use dashes as targets</B> because more recent versions of doxygen
201get very confused by this, so <b>don't</b> do
202\code
203\skipline ----
204\endcode
205say.

208\subsection misc Miscellaneous
209
210- To tell \c doxygen to ignore everything in the source file below a
211 certain point, denote this point with \c \@\@END\@\@.
212- To tell \c doxygen that a certain section of the source file is only
213 to be included in the \c html version of the documentation and
214 omitted in the \c LaTeX version, enclose this section within \c
215 \c \\htmlonly and \c \\endhtmlonly tags. CAREFUL: With recent
216 version of doxygen, this has caused problems with certain commands
217 not being interpreted correctly. Best not to use this...
218.

221\section generate Generating the documentation
222
223Once the source file has been written, simply type
224<code>cmake --build build</code> in the top level
225doc directory to build the \c html and \c LaTeX versions,
226e.g.
227\code
228cd doc
229cmake --build build
230\endcode
231Two subdirectories, \c html and \c latex, are now created containing
232the two versions of the documentation. A \c *.\c pdf file of the
233\c LaTeX version is also placed in the directory.
