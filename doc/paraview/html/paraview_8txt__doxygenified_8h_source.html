3 \mainpage Visualising oomph-lib's output files with Paraview
4 
5 All of \c oomph-lib's existing elements implement the
6 \c GeneralisedElement::output(...) functions, allowing
7 the computed solution to be documented via a simple
8 call to the \c Mesh::output(...) function, e.g.
9 
10 \code
11 // Open output file
12 ofstream output_file("output.dat")
13 
14 // Call the mesh's output function which loops over the
15 // element and calls theirs...
16 Problem::mesh_pt()->output(output_file);
17 \endcode
18 
19 By default, the output is written in a format that is suitable
20 for displaying the data with <A HREF="http://www.tecplot.com">
21 tecplot,</A> a powerful and easy-to-use commercial plotting package
22 -- possibly a somewhat odd choice for a an open-source library.
23 
24 We also provide the capability to output data in
25 a format that is suitable for display with
26 <a href="http://www.paraview.org">paraview</a>,
27 an open-source 3D plotting package.
28 For elements for which the relevant output functions are
29 implemented (they are defined as broken virtual functions in the
30 \c FiniteElement base class) output files for all the elements
31 in a certain mesh (here the one pointed t by \c Bulk_mesh_pt) can
32 be written as
33 
34 \dontinclude rayleigh_instability.cc
35 \skipline Output solution to file in paraview format
36 \skipline some_file.open(
37 \until some_file.close();
38 
39 where the unsigned integer \c npts controls the number of plot points per
40 element (just as in the tecplot-based output functions). If \c npts
41 is set to 2, the solution is output at the elements' vertices. For
42 larger values of \c npts the solution is sampled at greater number of (equally
43 spaced) plot points within the element -- this makes sense for
44 higher-order elements, i.e. elements in which the finite-element solution is not
45 interpolated linearly between the vertex nodes. It is important to note that
46 when displaying such a solution in paraview's "Surface with Edges" mode, the
47 "mesh" that is displayed does not represent the actual finite element mesh but is a
48 finer auxiliary mesh that is created merely to establish the connectivity
49 between the plot points.
50 
51 Paraview makes it possible to animate sequences of plots from
52 time-dependent simulations. To correctly animate results from
53 temporally adaptive simulations (where the timestep varies)
54 paraview can operate on pvd files which provide a list of
55 filenames and the associated time. These can be written automatically
56 from within \c oomph-lib, using the functions in the \c ParaviewHelper
57 namespace:
58 
59 \dontinclude mesh.h
60 \skipline paraview_helper
61 \until }
62 
63 Once the pvd file is opened, call \c
64 ParaviewHelper::write_pvd_header(...) to write the header information
65 required by paraview; then add the name of each output file
66 and the value of the associated value of the continuous time, using \c
67 ParaviewHelper::write_pvd_information(...). When the simulation is
68 complete write the footer
69 information using \c
70 ParaviewHelper::write_pvd_footer(...), then close to the pvd file.
71 
72 Currently, the paraview output functions are only implemented for a
73 relatively small number of elements but it is straightforward
74 to implement them for others.
75 
76 The <a href="../../FAQ/html/index.html">FAQ</a> contain
77 an entry that discusses how to display \c oomph-lib's output
78 with <A HREF="http://www.gnuplot.info">gnuplot</A> and
79 <a href="../../FAQ/html/index.html#tecplot">
80 how to adjust \c oomph-lib's output functions to different
81 formats.</a>

85 Angelo
86 Simone has written a python script that converts \c oomph-lib's
87 output to the vtu format that can be read by
88 <a href="http://www.paraview.org">paraview</a>.
89 This has since been improved
90 and extended significantly with input from Alexandre Raczynski and
91 Jeremy van Chu. The conversion script
92 can currently deal with output from meshes that are composed
93 of 2D triangles and quad and 3D brick and tet elements.
94 
95 The \c oomph-lib distribution contains three scripts:
96 
97 - <a href="../../../bin/oomph-convert.py"><code>bin/oomph-convert.py
98  </code>:</a> The python conversion script itself.
99  \n\n
100 - <a href="../../../bin/oomph-convert"><code>bin/oomph-convert
101  </code>:</a> A shell script wrapper that allows the processing of
102  multiple files.
103  \n\n
104 - <a href="../../../bin/makePvd"><code>bin/makePvd</code>:</a>
105  A shell script the creates the \c * \c .pvd files required by
106  paraview to produce animations.

410 \subsection py_sample_mult_par An example session for data from a parallel computation
411 
412 \c oomph-lib typically outputs results from parallel (distributed)
413 computations on a processor-by-processor basis, resulting in filenames
414 of the form
415 \code
416  soln_proc0_0.dat \
417  soln_proc1_0.dat |
418  : | Data for timestep 0
419  soln_proc[NPROC-1]_0.dat /
420 
421 
422  soln_proc0_1.dat \
423  soln_proc1_1.dat |
424  : | Data for timestep 1
425  soln_proc[NPROC-1]_1.dat, /
426 
427  :
428 
429 \endcode
430 where NPROC is the number of processors. An animation of such data
431 obviously requires the output from different processors (but for the
432 the same timestep) to be combined. Provided, the filenames have
433 the pattern
434 \code
435  [stem]proc[processor_number]_[timestep_number].dat
436 \endcode
437 (note the "proc" and "_", both of which are required), the pvd file
438 can be generated by first processing the files with \c oomph-convert,
439 \code
440  oomph-convert -z [stem]proc*
441 \endcode
442 followed by
443 \code
444  makePvd [NPROC] [stem] myplot.pvd
445 \endcode
446 So, for the files listed above, to produce a pvd file that
447 contains data from a computation with four processors the commands
448 \code
449 biowulf: 12:40:56$ oomph-convert soln_proc*
450 \endcode
451 followed by
452 \code
453 biowulf: 12:40:59$ makePvd 4 soln_ soln.pvd
454 \endcode
455 would create the file soln.pvd from which paraview
456 can create an animation of the solution.
