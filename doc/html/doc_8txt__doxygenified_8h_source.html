55  <h1>Welcome to the oomph-lib homepage</h1>
56  <p><code>oomph-lib</code> is an object-oriented, open-source
57  finite-element library for the simulation of multi-physics
58  problems. It is developed and maintained by <a href=
59  "http://www.maths.man.ac.uk/~mheil">Matthias Heil</a> and <a href=
60  "http://www.maths.man.ac.uk/~ahazel">Andrew Hazel</a> of the
61  <a href="http://www.maths.manchester.ac.uk/">School of
62  Mathematics</a> at <a href="http://www.manchester.ac.uk/">The
63  University of Manchester</a>, along with many <a href=
64  "../people/html/index.html">other contributors</a>.</p>
65  <p>The latest version of the library is 1.0,

86  release.</p>
87  <h2>What is it [and what is it not]?</h2>
88  <code>oomph-lib</code> <b>is</b>:
89  <ul>
90  <li>an object-oriented, open-source finite-element library for the
91  simulation of multi-physics problems.</li>
92  </ul>
93  <code>oomph-lib</code> <b>is not</b>:
94  <ul>
95  <li>a GUI-based, mouse-driven "package".</li>
96  </ul>
97  You must write your own C++ driver codes to specify your problem,
98  using <code>oomph-lib</code>'s high-level objects. Once the problem
99  is formulated, the <code>main</code> function can be as simple as
100  this:
101  <div class="fragment">
102  <div class="line"> main()</div>
103  <div class="line"> {</div>
104  <div class="line"> <span class="comment">// Create the problem object</span></div>
105  <div class="line"> <span class="keywordtype">MyProblem</span> problem;</div>
106  <div class="line"> </div>
107  <div class="line"> <span class="comment">// Solve the problem, using oomph-lib&#39;s default Newton solver</span></div>
108  <div class="line"> problem.newton_solve();</div>
109  <div class="line"> }</div>
110  </div>
111  <code>oomph-lib</code> provides the state of the art infrastructure
112  for the problem formulation and solution. The library contains a
113  large number of elements, meshes, timesteppers, solvers, etc. which
114  can be combined to solve any problem.
115  <p>The library is supplied with extensive on-line documentation
116  which includes:</p>
117  <ul>
118  <li>
119  <h4><a href="../intro/html/index.html"><b>Theory: Finite Elements
120  -- from procedural to object-oriented</b></a></h4>
121  This provides a relatively brief, constructive "top-down"
122  introduction to the maths of the finite element method, and
123  explains its object-oriented implementation in
124  <code>oomph-lib</code>.</li>
125  <li>
126  <h4><a href="../the_data_structure/html/index.html"><b>The Data
127  Structure</b></a></h4>
128  A bottom-up discussion of <code>oomph-lib's</code> overall data
129  structure, including a complete, cross-referenced index of all
130  <code>oomph-lib's</code> classes (generated by Doxygen).</li>
131  <li>
132  <h4><a href="../quick_guide/html/index.html"><b>(Not-So-)Quick
133  Guide</b></a></h4>
134  The <a href="../quick_guide/html/index.html">(Not-So-)Quick
135  Guide</a> provides a "quick" introduction on how to create new
136  instances of <code>oomph-lib's</code> fundamental objects:
137  <code>Problems</code>, <code>Meshes</code>, and
138  <code>Elements</code>.</li>
139  <li>
140  <h4><a href="../example_code_list/html/index.html"><b>Example
141  Codes/Tutorials</b></a></h4>
142  We provide a large number of verified and documented example
143  problems.
144  <ul>
145  <li style="margin-top:10px;">The example problems are arranged in
146  order of increasing complexity and can be used as chapters in a
147  self-study course. Comments and exercises are provided to encourage
148  further exploration of <code>oomph-lib's</code> capabilities.</li>
149  <li style="margin-top:10px;">The individual examples are as
150  self-contained as possible and can be used as quick guides to
151  particular features of <code>oomph-lib</code>. ("How do I solve
152  time-dependent problems?"; "How do I enable spatial adaptivity?";
153  "How do I write my own meshes?"; etc.)</li>
154  <li style="margin-top:10px;">The examples codes should be regarded
155  as templates for your own driver codes: Examine the <a href=
156  "../example_code_list/html/index.html">list of examples</a> and try
157  to find one that resembles the problem that you wish to solve. For
158  most "classical" problems (e.g. the solution of the Navier-Stokes
159  equations with standard boundary conditions), very few
160  modifications are required (e.g. adjusting the mesh to a different
161  domain, changing the boundary and/or initial conditions, etc). If
162  you wish to solve a non-standard problem, or one for which
163  <code>oomph-lib</code> does not (yet) provide elements or meshes,
164  it is easy to write your own, following the detailed instructions
165  provided in the documentation.</li>
166  </ul>
167  </li>
168  </ul>

173  <article class="tab-pane" id="features">
174  <h1>Features</h1>
175  <ul>
176  <li>Large number of fully-developed and carefully-validated element
177  types for the solution of:
178  <ul>
179  <li>The Poisson equation</li>
180  <li>The unsteady heat equation</li>
181  <li>The linear wave equation</li>
182  <li>The advection-diffusion equation</li>
183  <li>The Navier-Stokes equations, incl. free-surfaces, in Cartesian
184  and axisymmetric co-ordinates.</li>
185  <li>The equations of large-displacement solid mechanics, based on
186  the principle of virtual displacements in cartesian and
187  axisymmetric geometries; with arbitrary constitutive
188  equations.</li>
189  <li>Shell and beam elements, based on a geometrically non-linear
190  formulation.</li>
191  <li>...and many others.</li>
192  </ul>
193  [<b>Note:</b> Most elements are implemented in a
194  dimension-independent formulation]</li>
195  <li>It is easy to formulate new elements for problems that are not
196  included in the above list.</li>
197  <li>It is easy to combine any of the above elements to
198  "multi-physics" elements, allowing e.g.
199  <ul>
200  <li>the solution of fluid-structure interaction problems</li>
201  <li>the simulation of unsteady heat conduction in an elastic body
202  that undergoes large displacements</li>
203  <li>free-surface problems</li>
204  <li>etc.</li>
205  </ul>
206  </li>
207  <li>Parallel processing capabilities.</li>
208  <li>Full spatial and temporal adaptivity. All elements listed above
209  support quad-tree/octree-based mesh refinement procedures and
210  adaptive timestepping.</li>
211  <li>A wide range of meshes; new meshes can easily be added to the
212  library. <code>oomph-lib</code> can also use meshes that are
213  generated by third-party (commercial) mesh generators.</li>
214  <li>The ability to solve problems in domains with time-dependent,
215  curvilinear boundaries.</li>
216  <li><code>oomph-lib</code> treats all problems as non-linear
217  problems and uses Newton's method as the default non-linear solver.
218  Within this framework, linear problems are special cases for which
219  the Newton iteration converges in one step. The linear systems
220  arising during the Newton iteration can be solved with a variety of
221  direct and iterative solvers. Continuation methods (Keller's
222  arclength method and displacement-control for solid mechanics
223  problems) are implemented.</li>
224  <li>A large number of <a href=
225  "../example_code_list/html/index.html">fully documented example
226  codes.</a> All examples include an introduction to the
227  mathematical/physical problem, a detailed discussion of the driver
228  code, a discussion of the results (incl. comparisons against
229  analytical solutions where appropriate) and exercises and
230  comments.</li>
231  <li>The library has extensive self-test facilities. It can be
232  compiled with a PARANOID flag switched on or off. If the flag is
233  set during compilation, the code performs a large number of
234  self-tests during its execution. If things go wrong, code execution
235  terminates gracefully and diagnostic output is generated to help
236  pinpoint the problem. Of course, the tests introduce an additional
237  run-time overhead, so can be switched off in production runs.
238  During development, range checking on all array-based objects,
239  (vectors, matrices, tensors, etc) can be enabled separately by
240  using the RANGE_CHECKING flag, but introduces considerable run-time
241  overhead.</li>
242  </ul>

270  <h2>Downloads</h2>
271  <p>We provide <a href="../the_distribution/html/index.html">gzipped
272  tar files</a> of the library. We also provide read-only access to
273  the <a href="../subversion/html/index.html">subversion
274  repository</a> of the oomph-lib project. For more information see
275  the <a href="../the_distribution/html/index.html">installation
276  guide</a>. Note that the <a href=
277  "../the_distribution/html/index.html">gzipped tar files</a> , the
278  latest version of the library available via the public repository
279  and the webpage at <a href=
280  "http://www.oomph-lib.org">http://www.oomph-lib.org</a> are
281  automatically synchronised.</p>
282  <h2>Prerequisites</h2>
283  <p><code>oomph-lib</code> has been/is primarily being developed in
284  Linux so it is well tested there. However several people
285  successfully run it on Windows and Mac OS X. In order to minimise
286  the dependence on external libraries we include "frozen" versions
287  of certain external libraries (<a href=
288  "http://www-users.cs.umn.edu/~karypis/metis/">METIS</a>, <a href=
289  "http://crd.lbl.gov/~xiaoye/SuperLU">SuperLU</a>, ...) with our
290  distribution. The idea is that you should only have to unpack,
291  build and install one distribution to produce the fully compiled
292  library, the html-based documentation and working example
293  codes.</p>
294  For this purpose you <b>must have</b>:
295  <ul>
296  <li>A computer with a linux (or unix) operating system. [<b>Note
297  for windows users:</b> We believe that <code>oomph-lib</code> can
298  be installed in a windows environment, using <a href=
299  "http://www.cygwin.com/">cygwin</a>, though we haven't tried this
300  for a long time.]</li>
301  <li>Compilers for C++, C and Fortran77. The open-source <a href=
302  "http://gcc.gnu.org/">GNU gcc compiler suite</a> version 4.8.4 or
303  later is fine. [ <b>Note:</b> The Fortran77 and C compilers are
304  only required to compile some external libraries --
305  <code>oomph-lib</code> itself is written entirely in C++.]</li>
306  </ul>
307  The following programs are helpful but not essential:
308  <ul>
309  <li>GNU's <a href=
310  "http://www.gnu.org/software/autoconf/">autoconf</a>, <a href=
311  "http://www.gnu.org/software/automake/">automake</a> and <a href=
312  "http://www.gnu.org/software/libtool/">libtool</a> are required if
313  you wish to add additional features to your local copy of
314  <code>oomph-lib</code>; see <a href=
315  "../the_distribution/html/index.html">the <code>oomph-lib</code>
316  download page</a> for more details.</li>
317  <li><a href="http://www.python.org/download/">python</a> is needed
318  to analyse the results of the extensive self-tests that may be
319  performed at the end of the installation to verify that the build
320  process was completed successfully. If python is not available, the
321  self-test will compile and run all test codes but it will not be
322  possible to verify the correctness of the results.</li>
323  <li><a href="http://www.doxygen.org">doxygen</a> is needed to build
324  a local copy of <code>oomph-lib</code>'s extensive online
325  documentation. If doxygen is not installed on your system you can
326  consult the latest version of the online documentation on <a href=
327  "http://www.oomph-lib.org">the <code>oomph-lib</code>
328  homepage.</a></li>
329  </ul>
330  The <code>oomph-lib</code> distribution is built under <a href=
331  "http://www.gnu.org/software/autoconf/">autoconf</a> / <a href=
332  "http://www.gnu.org/software/automake/">automake</a> control,
333  making the installation completely straightforward. We provide two
334  main installation mechanisms:
335  <ul>
336  <li><code>oomph-lib</code> can be built/installed/tested with GNU's
337  standard <code>configure; make; make install; make check</code>
338  procedure.</li>
339  <li>We also provide a customised build script,
340  <code>autogen.sh</code>, that guides the "non-expert" user through
341  the installation. The script facilitates the specification of
342  various build options, such as optimisation levels etc.</li>
343  </ul>
344  Subject to the minimum requirements, listed above, you should be
345  able to install and use our libraries "anywhere". However, as
346  anybody who has ever ported any code to a different platform knows,
347  things are rarely that simple -- even with <a href=
348  "http://www.gnu.org/software/autoconf/">autoconf</a>, <a href=
349  "http://www.gnu.org/software/automake/">automake</a> and <a href=
350  "http://www.gnu.org/software/libtool/">libtool</a>... Having said
351  that, we have successfully installed and tested our installation on
352  the following platforms:
353  <ul>
354  <li>Ubuntu (version 14.04) with gcc compilers (version 4.8.4). The
355  tests included the MPI part of the library, which we compiled with
356  <a href="https://www.open-mpi.org/">OpenMPI</a> (version
357  1.6.5).</li>
358  <li>Redhat (version 4.4.7-3) with gcc compilers (version 4.4.7) The
359  tests included the MPI part of the library, which we compiled with
360  <a href="https://www.open-mpi.org/">OpenMPI</a> (version
361  1.5.4).</li>
362  <li> Ubuntu (version 14.04) with Clang compilers (version 3.5) </li>
363  <li> Arch Linux (05-2016) with gcc compilers (version 6.1.1) </li>
364  <li> Ubuntu (version 16.04) with gcc compilers (version 5.3.1) </li>
365  <li> Debian Jessie (gcc 4.9)</li>
366  </ul>

421  <h1>How do I get started?</h1>
422  <code>oomph-lib</code> contains a lot of code and a lot of
423  documentation. How to get started obviously depends on your
424  background: Are you familiar with the finite element method? How
425  good is your knowledge of C++? Etc. Here are some possible
426  "routemaps" around the library:
427  <ul>
428  <li><b>You are familiar with the finite element method and have a
429  fairly good knowledge of C++</b>
430  <ul>
431  <li>Have a look through the <a href=
432  "../example_code_list/html/index.html">list of example codes</a> to
433  get a feeling for <code>oomph-lib</code>'s capabilities. Pick a
434  problem that interests you and study the associated tutorial. Copy
435  the driver code into your own directory and play with it.</li>
436  <li>Once you have played with a few example codes, you may wish to
437  to learn more about <code>oomph-lib</code>'s <a href=
438  "../the_data_structure/html/index.html">overall data structure,</a>
439  or find out <a href="../optimisation/html/index.html">how to
440  optimise the library</a> for your particular application.</li>
441  </ul>
442  </li>
443  <li><b>You have never used finite element methods but have a fairly
444  good knowledge of C++</b>
445  <ul>
446  <li>Study the <a href="../intro/html/index.html">"Top Down"
447  introduction</a>. This document includes a "low tech" overview of
448  the mathematical/theoretical background and contrasts procedural
449  implementations of the finite element method with the
450  object-oriented approach adopted in <code>oomph-lib</code>.</li>
451  <li>Consult the <a href=
452  "../quick_guide/html/index.html">(Not-So-)Quick-Guide</a> to learn
453  how to construct basic <code>oomph-lib</code> objects for your
454  problem: <code>Problems</code>, <code>Meshes</code>,
455  <code>FiniteElements</code>, etc.</li>
456  <li>Continue with the steps suggested above.</li>
457  </ul>
458  </li>
459  <li><b>You have never used finite element methods and are a
460  newcomer to C++</b>
461  <ul>
462  <li>Buy <a href="http://www.math.wayne.edu/~yang/">Daoqi Yang</a>
463  brilliant book <a href=
464  "http://www.springeronline.com/sgw/cda/frontpage/0,11855,4-40007-22-2105335-0,00.html?changeHeader=true">
465  C++ and Object-Oriented Numeric Computing for Scientists and
466  Engineers.</a> Read it! Pretty much everything in this book is
467  relevant for some parts of <code>oomph-lib</code>. You should at
468  least understand:
469  <ul>
470  <li>The procedural aspects of C++ (basic types, functions and
471  control structures).</li>
472  <li>Namespaces.</li>
473  <li>Classes (private, protected and public members; inheritance and
474  multiple inheritance; virtual and pure virtual functions; base
475  classes and derived classes; static and dynamic casts).</li>
476  <li>Templates and template instantiations.</li>
477  <li>The standard template library (STL).</li>
478  </ul>
479  </li>
480  <li>Continue with the steps suggested above.</li>
481  </ul>
482  </li>
483  </ul>
