55 <h1>Welcome to the oomph-lib homepage</h1>
56 <p><code>oomph-lib</code> is an object-oriented, open-source
57 finite-element library for the simulation of multi-physics
58 problems. It is developed and maintained by <a href=
59 "https://www.matthias-heil.co.uk/new_webpage/index.html">Matthias Heil</a> and <a href=
60 "https://personalpages.manchester.ac.uk/staff/Andrew.Hazel/">Andrew Hazel</a> of the
61 <a href="http://www.maths.manchester.ac.uk/">Department of
62 Mathematics</a> at <a href="http://www.manchester.ac.uk/">The
63 University of Manchester</a>, along with many <a href=
64 "../people/html/index.html">other contributors</a>.</p>
65 <p> See the <a href="../change_log/html/index.html">change
66 log page</a> for an overview of the changes made since the previous
67 release.</p>
68 <h2>What is it [and what is it not]?</h2>
69 <code>oomph-lib</code> <b>is</b>:
70 <ul>
71 <li>an object-oriented, open-source finite-element library for the
72 simulation of multi-physics problems.</li>
73 </ul>
74 <code>oomph-lib</code> <b>is not</b>:
75 <ul>
76 <li>a GUI-based, mouse-driven "package".</li>
77 </ul>
78 You must write your own C++ driver codes to specify your problem,
79 using <code>oomph-lib</code>'s high-level objects. Once the problem
80 is formulated, the <code>main</code> function can be as simple as
81 this:
82 <div class="fragment">
83 <div class="line"> main()</div>
84 <div class="line"> {</div>
85 <div class="line"> <span class="comment">// Create the problem object</span></div>
86 <div class="line"> <span class="keywordtype">MyProblem</span> problem;</div>
87 <div class="line"> </div>
88 <div class="line"> <span class="comment">// Solve the problem, using oomph-lib&#39;s default Newton solver</span></div>
89 <div class="line"> problem.newton_solve();</div>
90 <div class="line"> }</div>
91 </div>
92 <code>oomph-lib</code> provides the state of the art infrastructure
93 for the problem formulation and solution. The library contains a
94 large number of elements, meshes, timesteppers, solvers, etc. which
95 can be combined to solve any problem.
96 <p>The library is supplied with extensive on-line documentation
97 which includes:</p>
98 <ul>
99 <li>
100 <h4><a href="../intro/html/index.html"><b>Theory: Finite Elements
101 -- from procedural to object-oriented</b></a></h4>
102 This provides a relatively brief, constructive "top-down"
103 introduction to the maths of the finite element method, and
104 explains its object-oriented implementation in
105 <code>oomph-lib</code>.</li>
106 <li>
107 <h4><a href="../the_data_structure/html/index.html"><b>The Data
108 Structure</b></a></h4>
109 A bottom-up discussion of <code>oomph-lib's</code> overall data
110 structure, including a complete, cross-referenced index of all
111 <code>oomph-lib's</code> classes (generated by Doxygen).</li>
112 <li>
113 <h4><a href="../quick_guide/html/index.html"><b>(Not-So-)Quick
114 Guide</b></a></h4>
115 The <a href="../quick_guide/html/index.html">(Not-So-)Quick
116 Guide</a> provides a "quick" introduction on how to create new
117 instances of <code>oomph-lib's</code> fundamental objects:
118 <code>Problems</code>, <code>Meshes</code>, and
119 <code>Elements</code>.</li>
120 <li>
121 <h4><a href="../example_code_list/html/index.html"><b>Example
122 Codes/Tutorials</b></a></h4>
123 We provide a large number of verified and documented example
124 problems.
125 <ul>
126 <li style="margin-top:10px;">The example problems are arranged in
127 order of increasing complexity and can be used as chapters in a
128 self-study course. Comments and exercises are provided to encourage
129 further exploration of <code>oomph-lib's</code> capabilities.</li>
130 <li style="margin-top:10px;">The individual examples are as
131 self-contained as possible and can be used as quick guides to
132 particular features of <code>oomph-lib</code>. ("How do I solve
133 time-dependent problems?"; "How do I enable spatial adaptivity?";
134 "How do I write my own meshes?"; etc.)</li>
135 <li style="margin-top:10px;">The examples codes should be regarded
136 as templates for your own driver codes: Examine the <a href=
137 "../example_code_list/html/index.html">list of examples</a> and try
138 to find one that resembles the problem that you wish to solve. For
139 most "classical" problems (e.g. the solution of the Navier-Stokes
140 equations with standard boundary conditions), very few
141 modifications are required (e.g. adjusting the mesh to a different
142 domain, changing the boundary and/or initial conditions, etc). If
143 you wish to solve a non-standard problem, or one for which
144 <code>oomph-lib</code> does not (yet) provide elements or meshes,
145 it is easy to write your own, following the detailed instructions
146 provided in the documentation.</li>
147 </ul>
148 </li>
149 </ul>

154 <article class="tab-pane" id="features">
155 <h1>Features</h1>
156 <ul>
157 <li>Large number of fully-developed and carefully-validated element
158 types for the solution of:
159 <ul>
160 <li>The Poisson equation</li>
161 <li>The unsteady heat equation</li>
162 <li>The linear wave equation</li>
163 <li>The advection-diffusion equation</li>
164 <li>The Navier-Stokes equations, incl. free-surfaces, in Cartesian
165 and axisymmetric co-ordinates.</li>
166 <li>The equations of large-displacement solid mechanics, based on
167 the principle of virtual displacements in cartesian and
168 axisymmetric geometries; with arbitrary constitutive
169 equations.</li>
170 <li>Shell and beam elements, based on a geometrically non-linear
171 formulation.</li>
172 <li>...and many others.</li>
173 </ul>
174 [<b>Note:</b> Most elements are implemented in a
175 dimension-independent formulation]</li>
176 <li>It is easy to formulate new elements for problems that are not
177 included in the above list.</li>
178 <li>It is easy to combine any of the above elements to
179 "multi-physics" elements, allowing e.g.
180 <ul>
181 <li>the solution of fluid-structure interaction problems</li>
182 <li>the simulation of unsteady heat conduction in an elastic body
183 that undergoes large displacements</li>
184 <li>free-surface problems</li>
185 <li>etc.</li>
186 </ul>
187 </li>
188 <li>Parallel processing capabilities.</li>
189 <li>Full spatial and temporal adaptivity. All elements listed above
190 support quad-tree/octree-based mesh refinement procedures and
191 adaptive timestepping.</li>
192 <li>A wide range of meshes; new meshes can easily be added to the
193 library. <code>oomph-lib</code> can also use meshes that are
194 generated by third-party (commercial) mesh generators.</li>
195 <li>The ability to solve problems in domains with time-dependent,
196 curvilinear boundaries.</li>
197 <li><code>oomph-lib</code> treats all problems as non-linear
198 problems and uses Newton's method as the default non-linear solver.
199 Within this framework, linear problems are special cases for which
200 the Newton iteration converges in one step. The linear systems
201 arising during the Newton iteration can be solved with a variety of
202 direct and iterative solvers. Continuation methods (Keller's
203 arclength method and displacement-control for solid mechanics
204 problems) are implemented.</li>
205 <li>A large number of <a href=
206 "../example_code_list/html/index.html">fully documented example
207 codes.</a> All examples include an introduction to the
208 mathematical/physical problem, a detailed discussion of the driver
209 code, a discussion of the results (incl. comparisons against
210 analytical solutions where appropriate) and exercises and
211 comments.</li>
212 <li>The library has extensive self-test facilities. It can be
213 compiled with a PARANOID flag switched on or off. If the flag is
214 set during compilation, the code performs a large number of
215 self-tests during its execution. If things go wrong, code execution
216 terminates gracefully and diagnostic output is generated to help
217 pinpoint the problem. Of course, the tests introduce an additional
218 run-time overhead, so can be switched off in production runs.
219 During development, range checking on all array-based objects,
220 (vectors, matrices, tensors, etc) can be enabled separately by
221 using the RANGE_CHECKING flag, but introduces considerable run-time
222 overhead.</li>
223 </ul>

228 <h1>How do I get/install the code?</h1>
229 <h2>Download/Installation</h2>
230 <p><code>oomph-lib</code> is hosted on and distributed via GitHub at
231 <center><a href="https://github.com/oomph-lib/oomph-lib">
232 https://github.com/oomph-lib/oomph-lib</a></center><br>
233 where we provide detailed build instructions.</p>
234
235 <p><code>oomph-lib</code> has been/is primarily being developed in
236 Linux so it is well tested there. However several people
237 successfully run it on Windows and Mac OS X. Our continuous
238 integration on GitHub currently checks Linux and Mac OS X builds.
239 In order to minimise
240 the dependence on external libraries our CMake-based build machinery
241 provides the option to automatically install all required
242 external libraries (such as <a href=
243 "http://www-users.cs.umn.edu/~karypis/metis/">METIS</a>, <a href=
244 "http://crd.lbl.gov/~xiaoye/SuperLU">SuperLU</a>, ...). This avoids
245 the common and frustrating scenario where to install library A, you
246 first need to install library B, which in turns requires library C etc.
247 With <code>oomph-lib</code> you simply clone our GitHub reposiory and
248 run our build script to produce the fully installed
249 library, the html-based documentation and working example
250 codes.</p>

310 <h1>How do I get started?</h1>
311 <code>oomph-lib</code> contains a lot of code and a lot of
312 documentation. How to get started obviously depends on your
313 background: Are you familiar with the finite element method? How
314 good is your knowledge of C++? Etc. Here are some possible
315 "routemaps" around the library:
316 <ul>
317 <li><b>You are familiar with the finite element method and have a
318 fairly good knowledge of C++</b>
319 <ul>
320 <li>Have a look through the <a href=
321 "../example_code_list/html/index.html">list of example codes</a> to
322 get a feeling for <code>oomph-lib</code>'s capabilities. Pick a
323 problem that interests you and study the associated tutorial. Copy
324 the driver code into your own directory and play with it.</li>
325 <li>Once you have played with a few example codes, you may wish to
326 to learn more about <code>oomph-lib</code>'s <a href=
327 "../the_data_structure/html/index.html">overall data structure,</a>
328 or find out <a href="../optimisation/html/index.html">how to
329 optimise the library</a> for your particular application.</li>
330 </ul>
331 </li>
332 <li><b>You have never used finite element methods but have a fairly
333 good knowledge of C++</b>
334 <ul>
335 <li>Study the <a href="../intro/html/index.html">"Top Down"
336 introduction</a>. This document includes a "low tech" overview of
337 the mathematical/theoretical background and contrasts procedural
338 implementations of the finite element method with the
339 object-oriented approach adopted in <code>oomph-lib</code>.</li>
340 <li>Consult the <a href=
341 "../quick_guide/html/index.html">(Not-So-)Quick-Guide</a> to learn
342 how to construct basic <code>oomph-lib</code> objects for your
343 problem: <code>Problems</code>, <code>Meshes</code>,
344 <code>FiniteElements</code>, etc.</li>
345 <li>Continue with the steps suggested above.</li>
346 </ul>
347 </li>
348 <li><b>You have never used finite element methods and are a
349 newcomer to C++</b>
350 <ul>
351 <li>Buy <a href="https://yangdaoqi.info/">Daoqi Yang</a>
352 brilliant book <a href="https://link.springer.com/book/10.1007/978-1-4613-0189-9">
353 C++ and Object-Oriented Numeric Computing for Scientists and
354 Engineers.</a> Read it! Pretty much everything in this book is
355 relevant for some parts of <code>oomph-lib</code>. You should at
356 least understand:
357 <ul>
358 <li>The procedural aspects of C++ (basic types, functions and
359 control structures).</li>
360 <li>Namespaces.</li>
361 <li>Classes (private, protected and public members; inheritance and
362 multiple inheritance; virtual and pure virtual functions; base
363 classes and derived classes; static and dynamic casts).</li>
364 <li>Templates and template instantiations.</li>
365 <li>The standard template library (STL).</li>
366 </ul>
367 </li>
368 <li>Continue with the steps suggested above.</li>
369 </ul>
370 </li>
371 </ul>
