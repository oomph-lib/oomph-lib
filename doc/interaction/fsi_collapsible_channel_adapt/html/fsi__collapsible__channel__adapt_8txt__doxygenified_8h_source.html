3\mainpage Flow in a 2D collapsible channel revisited -- enabling adaptivity in FSI problems
4
5 In this document we re-visit the collapsible channel problem yet
6again -- this time to demonstrate the use of spatial adaptivity in
7fluid-structure interaction problems. In such problems, two
8additional issues have to be addressed:
9-# \ref history
10-# \ref update_node_update
11.
12We start with a brief reminder of \c oomph-lib's mesh adaptation
13procedures to highlight what additional complications arise
14in fluid-structure interaction problems, and then explain how these
15problems are addressed.

20\section history The assignment of positional history values for newly-created nodes
21
22\subsection what What is the problem?
23
24We first discussed the use of spatial adaptivity for time-dependent
25problems in the context of
26<A HREF="../../../unsteady_heat/two_d_unsteady_heat_adapt/html/index.html#spatial_adapt">
27the unsteady heat equation</A>. We showed that whenever a new node is created,
28\c oomph-lib's mesh adaptation procedures automatically assign the
29nodal values and "history values" (the auxiliary values that are used
30by the \c Node's \c TimeStepper to approximate
31time-derivatives of the nodal values) for the newly-created \c Node
32by interpolating between the corresponding values in the new
33\c Node's father element.
34
35Similarly, an initial assignment for the newly-created \c Node's current
36and previous positions (required for the evaluation of the mesh-velocities
37in the ALE formulation of the time-derivatives) is made by interpolating
38between the current and previous nodal positions of the new \c Node's father
39element. This default assignment is subsequently overwritten if the
40father element is associated with a \c MacroElement, in which case the
41newly-created \c Node's current position is determined by the
42father element's \c MacroElement representation. This ensures
43that the mesh refinement process respects curvilinear domain
44boundaries, as discussed
45<A HREF="../../../poisson/fish_poisson2/html/index.html">elsewhere.</A>
46The same strategy is used to over-write the default assignment for
47a new \c Node's previous positions: They are re-assigned, based on the
48father element's \c MacroElement - representation, evaluated at the
49relevant previous timesteps.
50
51To illustrate the process, consider the sketch below and
52assume that the time-integration of the Navier-Stokes equations is
53performed with a \c BDF<2> timestepper -- a timestepper that requires
54two history values (the solution and the nodal positions at two
55previous timesteps) to evaluate the time-derivative
56\f$ \partial u_i/\partial t \f$ in the momentum equations. The
57position of the nodes in the collapsible
58part of the channel is determined by the \c MacroElement - based
59representation of the \c CollapsibleChannelDomain, using the
60wall shape obtained from the compound \c MeshAsGeomObject that
61we created from the discretisation of the elastic wall,
62shown in green.
63
64Assume that the initial mesh is sufficiently
65fine to resolve the solution at \f$ t=t_0 \f$ and \f$ t=t_0 + \Delta
66t \f$. At \f$ t=t_0 + 2 \Delta t \f$, the automatic mesh adaptation
67splits one of the fluid elements, creating the five new \c Nodes,
68shown in red (for simplicity we only show the fluid elements' vertex
69nodes). Their position is determined by the father element's
70\c MacroElement mapping, using the current position of the
71\c MeshAsGeomObject, obtained from the interpolation between the
72nodal positions of its \c SolidNodes (shown as green circles).
73
74\image html history_values.gif "Sketch illustrating the assignment of positional history values for newly-created nodes in FSI problems. The positional history values of the newly-created Nodes (shown in red) are given by the positions they would have had if they had already existed at previous timesteps. "
75\image latex history_values.eps "Sketch illustrating the assignment of positional history values for newly-created nodes in FSI problems. The positional history values of the newly-created Nodes (shown in red) are given by the positions they would have had if they had already existed at previous timesteps. " width=0.75\textwidth
76
77
78So far, so good! A subtle problem arises when we try to assign
79the positional history values for the newly-created \c Nodes. The evaluation
80of the \c MacroElement mapping at the previous timesteps
81(required to determine the positions the newly-created \c Nodes would have
82had, if they had already existed at previous timesteps) requires
83access to the previous wall shapes. In
84<A HREF="../../../navier_stokes/collapsible_channel/html/index.html">
85the non-FSI problem discussed earlier</A>, the wall shape was given
86analytically and could therefore
87be evaluated at arbitrary times. In the FSI problem considered here,
88the previous wall shape is not available as the previous positions
89of the \c SolidNodes are not required (and are therefore not stored)
90for the solution of the (steady!) wall equations.

95\subsection soln The solution: The Steady<NSTEPS> timestepper
96
97The solution to the problem is simple: Recall that the final entry in
98the argument list of \c oomph-lib's mesh constructors specifies the
99\c TimeStepper to be used for the evaluation of any time-derivatives.
100The \c TimeStepper's member function
101\c TimeStepper::ntstorage() specifies the total number of
102values (the current value plus the number of history values) required
103to evaluate the time-derivatives; it is used by the \c
104FiniteElement::construct_node(...) function to create the required
105amount of storage at the elements' \c Nodes when the \c Nodes are
106created in the mesh constructor. To maximise the
107potential for code-reuse, we always provide a default argument for
108the pointer to the \c TimeStepper -- a pointer
109to a static instantiation of \c oomph-lib's dummy \c TimeStepper, \c
110Steady<0>. This ensures that the "user" can use the mesh for steady problems
111without having to artificially create a dummy \c TimeStepper that
112would be completely irrelevant for his/her problem.
113
114For instance, the full interface to the
115constructor of the \c OneDLagrangianMesh, defined in
116<A HREF="../../../../src/meshes/one_d_lagrangian_mesh.template.h">
117one_d_lagrangian_mesh.template.h</A> is given by
118
119\dontinclude one_d_lagrangian_mesh.template.h
120\skipline Constructor:
121\until &Mesh::Default_TimeStepper);
122
123Thus, when we constructed the wall mesh in the non-adaptive
124version of the driver code
125<A HREF="../../../../demo_drivers/interaction/fsi_collapsible_channel/fsi_collapsible_channel.cc">fsi_collapsible_channel.cc,</A> using the statements
126
127\code
128 //Create the "wall" mesh with FSI Hermite beam elements
129 Wall_mesh_pt = new OneDLagrangianMesh<FSIHermiteBeamElement>
130 (Ncollapsible,Lcollapsible,undeformed_wall_pt);
131\endcode
132
133the use of the dummy \c TimeStepper, \c Steady<0> was implied.
134All time-derivatives computed by this \c TimeStepper
135evaluate to zero, and the \c TimeStepper does not require (or request)
136any storage for history values. This is a sensible default for problems
137that are either inherently steady (such as problems involving the
138Poisson equations, say) or for steady versions of time-dependent PDEs
139(such as the steady Navier-Stokes equations). In the present
140problem, we wish to retain the dummy character of the \c TimeStepper
141so that all time-derivatives in the \c FSIHermiteBeamElements
142are set to zero, while retaining a
143limited history of the \c SolidNode's previous positions so that
144the wall shape at previous timesteps can be evaluated. This is
145possible by creating a \c Steady timestepper with
146a non-zero template argument: The class
147
148\code
149template<unsigned NSTEPS>
150class Steady : public TimeStepper
151\endcode
152
153provides a dummy \c TimeStepper that requires (and maintains) \c
154NSTEPS history values that represent the nodal positions
155(or nodal values) at \c NSTEPS previous timesteps. However, these
156history values are not used to evaluate any time-derivatives --
157all time-derivatives computed by this \c TimeStepper are zero.
158
159If the Navier-Stokes equations are discretised with a \c BDF<2>
160timestepper, the evaluation of the mesh velocity in the
161ALE representation of the time-derivatives \f$ \partial u_i/\partial t
162\f$ requires the nodal position at the two previous timesteps.
163This is achieved by passing a pointer to a specifically-constructed
164\c Steady<2> timestepper to the wall mesh. Here is the relevant
165code fragment from the Problem constructor in the driver code
166<A HREF="../../../../demo_drivers/interaction/fsi_collapsible_channel/fsi_collapsible_channel_adapt.cc">fsi_collapsible_channel_adapt.cc</A>:
167
168\dontinclude fsi_collapsible_channel_adapt.cc
169\skipline Allocate the timestepper
170\until undeformed_wall_pt,wall_time_stepper_pt);

180\section update_node_update Updating the node-update data in refineable AlgebraicMeshes
181
182When discussing
183<A HREF="../../../navier_stokes/algebraic_collapsible_channel/html/index.html">
184the non-FSI version of the collapsible channel problem</A>
185we explained how \c oomph-lib's mesh adaptation procedures
186assign the node-update data for newly-created \c AlgebraicNodes.
187Recall that the node-update data comprises:
188- A pointer to the \c AlgebraicMesh that implements the
189 node-update function. \n\n
190- An ID for the node-update function. This is used in cases in which
191 different regions of the \c AlgebraicMesh are updated by different
192 node-update functions. The ID has a default value of 0; this is
193 used if there is only a single node-update function, as in
194 the \c AlgebraicCollapsibleChannelMesh. \n\n
195- A vector of (pointers to) \c GeomObjects that are involved in
196 the node update. \n\n
197- A vector of reference values, such as the intrinsic coordinates
198 of reference points on the \c GeomObjects. \n\n
199.
200By default, we assume that a newly-created \c AlgebraicNode
201is updated by the same node-update function as the \c AlgebraicNodes
202in its father element. Therefore we pass the pointer to the \c AlgebraicMesh,
203the node-update function ID, and the vector of pointers to \c
204GeomObjects to the newly-created \c AlgebraicNodes, and interpolate
205the reference values between those stored at the \c AlgebraicNodes
206in the father elements.
207
208In most cases this provides a sensible default. For instance, it
209is hard to imagine a situation in which it would be sensible to update the
210position of newly-created \c AlgebraicNodes by a procedure that
211differs from that used for the surrounding
212\c AlgebraicNodes that already existed in the father element.
213Similarly, since the reference values vary from node to node
214(if they were constant we would not store them in the \c AlgebraicNodes'
215node-update data but in the \c AlgebraicMesh that implements the node
216update!) it makes sense to assign the values at newly-created
217\c AlgebraicNodes by interpolation. For instance, in the
218\c AlgebraicCollapsibleChannelMesh one of the reference values
219is the \f$ x_1 \f$ - coordinate of the reference point on the fixed
220lower wall. For the \c AlgebraicNodes that already existed in the
221coarse base mesh, this value is given by the the \c AlgebraicNodes'
222\f$ x_1 \f$ - coordinate in the undeformed mesh. Interpolation
223of this value for the newly-created \c AlgebraicNodes results in
224an axially uniform subdivision of the refined elements.
225
226 The same procedure may be used to assign the reference value that
227represents the intrinsic coordinate of the reference point on the
228upper wall -- at least as long as the upper wall is only ever
229addressed as a compound \c GeomObject, as in our original (slow!) code
230that employed a \c MacroElement - based node update. The
231procedure is illustrated in the sketch below: The upper wall is
232parametrised by a compound \c GeomObject in which the beam's Lagrangian
233coordinate \f$ \xi \f$ doubles as the \c GeomObject's intrinsic
234coordinate \f$ \zeta. \f$
235The green arrows indicate the reference points for nodes I, II, III,
236IV and V in the coarse initial mesh. The reference values
237\f$ \zeta_{I}^{[ref]}, ... , \zeta_{V}^{[ref]} \f$ are stored
238in the node-update data of nodes I,...,V, respectively.
239The red arrow identifies the reference point for the newly-created
240node VI whose reference coordinate \f$ \zeta_{IV}^{[ref]} \f$ is created by
241interpolation between the reference values of nodes II, III, IV and V,
242i.e. the nodes in its father element.
243
244
245\image html update_single.gif "Illustration of the default update procedure for the node-update data: The reference value for newly created node VI is created by interpolation from the reference values for nodes II, III, IV and V in its father element. "
246\image latex update_single.eps "Illustration of the default update procedure for the node-update data: The reference value for newly created node VI is created by interpolation from the reference values for nodes II, III, IV and V in its father element. " width=0.75\textwidth
247
248
249
250Unfortunately, the default assignment is not appropriate for nodes
251that are updated by the (much faster) algebraic node update procedure
252implemented in the \c AlgebraicCollapsibleChannelMesh. Its
253\c AlgebraicNodes store pointers to the sub-\c GeomObjects (i.e. the
254\c FSIHermiteBeamElements in the wall mesh) that contain their
255reference points, and the values of the local coordinate at which
256the reference points are located. This is illustrated in the sketch
257below in which the sub- \c GeomObjects, parametrised by their local
258coordinates \f$ s = \zeta, \f$ are shown in magenta. Note that
259the reference points for nodes II and IV are located in one
260sub- \c GeomObject, those for nodes III and V are located in another.
261
262\image html update_compound.gif "Illustration of the revised update procedure for the node-update data: The intrinsic coordinate within the compound GeomObject can be created by interpolation. The coordinate within the sub-GeomObject has to be determined via the GeomObject::locate_zeta(...) function. "
263\image latex update_compound.eps "Illustration of the revised update procedure for the node-update data: The intrinsic coordinate within the compound GeomObject can be created by interpolation. The coordinate within the sub-GeomObject has to be determined via the GeomObject::locate_zeta(...) function. " width=0.75\textwidth
264
265
266It obviously does not make sense to create the reference value for
267the newly-created node VI by interpolation. The node update data must be
268created from scratch in two steps:
269-# Determine the newly-created Node's intrinsic reference coordinate within
270 the compound \c MeshAsGeomObject (i.e. the continuous beam's
271 Lagrangian coordinate), \f$ \xi_{VI}^{[ref]}\f$ by interpolation.
272 (<A HREF="../../fsi_collapsible_channel_algebraic/html/index.html">Recall
273 </A> that the \c AlgebraicNodes in the non-refineable
274 \c AlgebraicCollapsibleChannelMesh already stored this coordinate,
275 even though it was not used during the node-update itself).
276 \n\n
277-# Use the \c GeomObject::locate_zeta(...) function to determine
278 sub-\c GeomObject and the local coordinate within it, at which
279 the reference point with Lagrangian coordinate \f$
280 \xi_{VI}^{[ref]}\f$ is located.
281.
282To ensure that this procedure is performed automatically after
283every mesh adaptation, it should be implemented in the
284function \c AlgebraicMesh::update_node_update(...). This function is defined
285as a pure virtual function in the \c AlgebraicMesh base class in order to
286force the mesh writer to assess if the default procedure
287for the assignment of the node-update data for newly-created
288\c AlgebraicNodes, described above, is appropriate. If it
289is, the function may, of course, be implemented as an empty function.
290We note that the function \c AlgebraicMesh::update_node_update(...)
291is called \b after the default values for the node-update
292data have been assigned. It is therefore only necessary to correct
293those values for which the default procedure is inappropriate.
294

397\section driver The driver code
398
399The driver code
400<A HREF="../../../../demo_drivers/interaction/fsi_collapsible_channel/fsi_collapsible_channel_adapt.cc">fsi_collapsible_channel_adapt.cc</A>
401for the spatially adaptive problem is a trivial extension of the
402non-adaptive code
403<A HREF="../../../../demo_drivers/interaction/fsi_collapsible_channel/fsi_collapsible_channel.cc">fsi_collapsible_channel.cc,</A>
404therefore we will not provide a detailed listing here.
405Comparing the two source codes (e.g. with \c sdiff) shows that
406spatial adaptivity may be enabled with a few straightforward changes:
407- Change the fluid mesh from the \c
408 MacroElementNodeUpdateCollapsibleChannelMesh to the
409 \c MacroElementNodeUpdateRefineableCollapsibleChannelMesh
410 (or from the \c AlgebraicCollapsibleChannelMesh to the
411 \c RefineableAlgebraicCollapsibleChannelMesh). \n\n
412- Change the element type from \c TaylorHoodElement<2>
413 to \c RefineableQTaylorHoodElement<2> (or
414 \c QCrouzeixRaviartElement<2> to \c
415 RefineableQCrouzeixRaviartElement<2>). \n\n
416- Add the functions \c Problem::actions_before_adapt() and
417 \c Problem::actions_after_adapt(); see below. \n\n
418- Explicitly specify the \c Steady<2> timestepper for the
419 wall mesh, as discussed above. \n\n
420- Create an error estimator, specify the target errors for
421 the adaptation, and call the spatially adaptive Newton solver.
422.

426\subsection before_and_after Actions before and after solve
427
428As in the <A HREF="../../../navier_stokes/collapsible_channel/html/index.html">
429non-FSI problem</A>, we use the function \c
430Problem::actions_before_adapt() to delete the applied traction
431elements before the mesh adaptation:
432

434\skipline start_of_actions_before_adapt
435\until end of actions_before_adapt
436
437
438As usual, we then employ the function \c Problem::actions_after_adapt()
439to re-attach traction elements to the fluid elements that are located
440at the mesh's inflow boundary (mesh boundary 5) when the mesh
441adaptation is complete.
442
443\dontinclude fsi_collapsible_channel_adapt.cc
444\skipline start_of_actions_after_adapt
445\until rebuild_global_mesh();
446
447Next, we pin the redundant pressure degrees of freedom
448(see <A HREF="../../../navier_stokes/adaptive_driven_cavity/html/index.html">
449another tutorial</A> for details) and pass the
450function pointer to the function that defines the prescribed
451traction to the applied traction elements:
452
453\until }
454
455We specify the function
456\c FSI_functions::apply_no_slip_on_moving_wall() as the
457auxiliary node-update function for all fluid nodes that are located
458on the FSI boundary (mesh boundary 3) -- this ensures that
459the fluid velocity is updated (via the no-slip condition)
460whenever the position of a fluid node on this boundary is updated.
461For fluid nodes that already existed before the mesh adaptation
462this statement over-writes the function pointers already stored
463at those nodes. Note the use of compiler flags to distinguish between
464the two node-update strategies.
465
466\until }
467
468Finally, we re-generate the FSI lookup scheme that establishes which
469fluid elements are located next to the Gauss points in the beam
470elements. This is necessary because the previous lookup scheme, set
471up in the problem constructor, becomes invalid if any of the fluid
472elements next the wall are split during the refinement process.
