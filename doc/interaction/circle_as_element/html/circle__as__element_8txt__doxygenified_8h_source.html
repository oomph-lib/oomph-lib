3 \mainpage Demo problem: Upgrading a GeomObject to a GeneralisedElement -- how to parametrise unknown domain boundaries
4 
5 In many previous examples we demonstrated how to use \c GeomObjects
6 to parametrise curvilinear, moving domain boundaries. We
7 used this functionality extensively to perform simulations of
8 problems in moving domains. The techniques illustrated in these examples
9 are adequate for problems in which the motion of the boundary is
10 prescribed; see e.g. the
11 <A HREF="../../../navier_stokes/collapsible_channel/html/index.html">
12 simulation of fluid flow in a 2D channel in which a part of one
13 channel wall performs a prescribed oscillation</A>.
14 
15  Here we shall demonstrate how to use \c GeomObjects to
16 parametrise curvilinear, moving domain boundaries in problem in which
17 the position of the boundary is unknown and has to be determined as
18 part of the overall solution. We will also address the following
19 questions:
20 - What is a \c GeomObject's geometric \c Data?
21 - What is a \c GeneralisedElement's external and internal \c Data?
22 .

28 \section geom_data Part 1: A closer look at the GeomObject class: Geometric Data
29 
30 In an <A HREF="../../../poisson/fish_poisson2/html/index.html">
31 earlier example</A>, we introduced the \c GeomObject as
32 a base class for all objects that provide an analytical parametrisation of a
33 geometric object by specifying its Eulerian coordinates (i.e.
34 the position vector to "material points" in the object,
35 \f$ {\bf r}\f$) as a function of some intrinsic (Lagrangian)
36 coordinates, \f$ {\bf \zeta} \f$, so that \f$ {\bf r} = {\bf r}({\bf
37 \zeta})\f$. Every specific \c GeomObject must implement the pure
38 virtual function
39 
40 \code
41 void GeomObject::position(const Vector<double>& zeta, Vector<double>& r);
42 \endcode
43 
44 where the number of Lagrangian and Eulerian coordinates involved in
45 the parametrisation (i.e. the sizes of the vectors \c zeta and \c r) are
46 stored as unsigned integers in the protected member data,
47 \c GeomObject::NLagrangian and \c GeomObject::Ndim, respectively.
48 
49 
50 Most specific geometric objects store some additional parameters
51 as private data members. For instance, a \c GeomObject that
52 represents a 2D circle, parametrised by the 1D Lagrangian
53 coordinate \f$ \zeta \f$ as
54 \f[ {\bf r}(\zeta)= \left( \begin{array}{c} X_c + R \cos(\zeta) \\
55  Y_c + R \sin(\zeta)
56 \end{array} \right)
57 \f]
58 may be implemented as follows:
59 
60 \dontinclude circle.h
61 \skipline start_of_circle
62 \until end of SimpleCircle class
63 
64 
65 The shape and position of this \c GeomObject is defined by
66 the parameters \f$ X_c, Y_c \f$ and \f$ R \f$ which are stored
67 as double precision numbers in the object's private member data.
68 
69 
70 In many applications (e.g. in free-boundary problems) it is
71 necessary to regard some (or all) of the parameters that
72 specify the object's shape as (an) unknown(s) in the overall problem.
73 Within \c oomph-lib all unknowns are values in \c Data objects.
74 We refer to any \c Data objects whose values affect the
75 shape of a \c GeomObject as the \c GeomObject's geometric \c Data.
76 The \c GeomObject class defines interfaces for two functions that provide
77 access to such \c Data. The function
78 
79 \code
80 virtual unsigned GeomObject::ngeom_data() const;
81 \endcode
82 
83 should return the number of geometric \c Data objects that affect
84 a \c GeomObject's shape and position. The function
85 
86 \code
87 virtual Data* GeomObject::geom_data_pt(const unsigned& j);
88 \endcode
89 
90 should return a pointer to the \c GeomObject's \c j - th geometric
91 \c Data object. Both functions are implemented as
92 "broken" virtual functions in the \c GeomObject base class in order to
93 facilitate the creation of new \c GeomObjects. The functions should be
94 overloaded in \c GeomObjects that actually contain geometric
95 \c Data. If this is not done, the broken versions in the base
96 class issue a suitable explanation before throwing an error.
97 
98

101 allows the the parameters \f$ X_c, Y_c \f$ and \f$ R \f$ to be
102 regarded as unknowns, by storing them as values
103 in a suitable \c Data object:
104

111 The arguments to the first constructor specify the x- and
112 y-coordinates of the ring's centre (\f$ X_c\f$ and \f$ Y_c\f$),
113 and its radius, \f$ R \f$, as double-precision numbers. The constructor creates
114 storage for these values in the \c Geom_data_pt
115 vector which is stored in the object's protected member data.
116 First, the \c Geom_data_pt vector (empty by
117 default) is resized to provide storage for
118 a (pointer to a) single Data item. Next we create a Data object that

120 object in the first (and only) component of the \c Geom_data_pt vector.
121 Finally, we set the values to those specified by the arguments and
122 pin them to reflect the fact that, by default, the values
123 are constants rather than unknowns in the problem.
124 [\b Note: Clearly, there is some ambiguity as to how to distribute
125 the values of the geometric parameters among the geometric \c Data.
126 Here we chose to store the three values in a single \c Data object,
127 but we could also have stored each value in its own \c Data object.]

188 \section example Part 2: Upgrading a GeomObject to a GeneralisedElement
189 
190 We will now consider a (toy!) problem in which the ability to
191 treat a parameter that specifies the shape and/or the position
192 of a geometric object as an
193 unknown (i.e. as a value in a \c Data object) is essential:
194 A circular ring of radius \f$ R \f$ is mounted on a linearly elastic
195 spring of stiffness \f$ k_{stiff} \f$. We wish to find the ring's
196 equilibrium position when it is subjected to a vertical load,
197 \f$ f_{load} \f$.
198 
199 \image html circle_as_element_sketch.gif "Sketch of the example problem. "
200 \image latex circle_as_element_sketch.eps "Sketch of the example problem. " width=0.75\textwidth
201 
202 
203 To solve this problem with \c oomph-lib [Yes, the solution is
204 \f$ Y_c = f_{load}/k_{stiff}\f$ but let's pretend we
205 don't know this...], we employ C++ inheritance
206 to "upgrade" the \c GeneralCircle object, discussed above, to a \c
207 GeneralisedElement in which the vertical position of the ring, \f$ Y_c
208 \f$, is one of the unknowns whose value is determined by the
209 equilibrium equation (in residual form),
210 \f[
211  f_{load} - Y_c \ k_{stiff} = 0. \ \ \ \ \ \ \ \ \ \ \ (1)
212 \f]
213 In the present problem, the load on the ring is a control parameter,
214 but in different applications it may be an unknown whose value
215 has to be determined as part of the solution. (For instance,
216 in fluid-structure interaction problems, the load
217 on an elastic structure depends on the unknowns in the adjacent fluid
218 elements.) Therefore, we also represent \f$ f_{load} \f$ by a \c Data
219 object.
220 
221 To solve the above problem with \c oomph-lib, the residual equation (1)
222  must be implemented in the \c get_residuals(...) function of a suitable
223 \c GeneralisedElement. Recall that the elemental residual
224 vector computed by a \c GeneralisedElement can depend on two types
225 of \c Data:
226 - <B>Internal <EM>Data</EM></B> stores values that are "internal" to the
227  element, i.e. values that the element is "in charge of".
228  Such \c Data is accessible via the pointer-based access function
229  \c GeneralisedElement::internal_data_pt(...)
230 - <B>External <EM>Data</EM></B> stores values that are "external" to the
231  element. The values stored in such \c Data \e affect the element's
232  residual vector but its values are not determined \e by the element.
233  Such \c Data is accessible via the pointer-based access function
234  \c GeneralisedElement::external_data_pt(...)
235 .
236 In the present context, it is most natural to regard the
237 \c GeneralCircle's geometric \c Data as internal to the element and
238 the load \c Data as external.
239 
240 
241 Here is the class definition for the \c ElasticallySupportedRingElement
242 which combines the \c GeneralisedElement and \c GeneralCircle classes by
243 multiple inheritance. Its role as a \c GeomObject allows the object to be used
244 in the parametrisation of domain boundaries; its role as a
245 \c GeneralisedElement allows the value of \f$ Y_c \f$ to be determined
246 as part of the solution.
247

255 parameters, \f$ X_c, Y_c \f$ and \f$ R \f$. We pass them to the
256 constructor of the \c GeneralCircle object and assign a default value for
257 the spring stiffness \f$ k_{stiff}\f$ (stored as a private data
258 member of the class; see below):
259 \dontinclude circle_as_generalised_element.h
260 \skipline short Constructor
261 \until K_stiff(1.0)
262 
263 Next, we add the \c GeneralCircle's geometric \c Data (created automatically
264 by the constructor of the \c GeneralCircle object; see above)
265 into the element's storage for its internal \c Data. This ensures
266 that the geometric \c Data is included in the element's
267 equation numbering procedures. Within the
268 context of the \c GeneralCircle, all geometric parameters were
269 regarded as constants and their values were pinned. Here, the
270 vertical position [stored in the second entry in the geometric \c
271 Data] is unknown, therefore we un-pin it.

326 Finally, we implement the pure virtual functions
327 \c GeneralisedElement::get_jacobian(...) and
328 \c GeneralisedElement::get_residuals(...) which are required to
329 make the element functional. These functions must compute the
330 element's contributions to the \c Problem's global
331 residual vector and Jacobian matrix. As usual, we implement them as wrappers
332 to a single function that computes the residual and (optionally)
333 the Jacobian matrix:
334 
335 \skipline /// Compute element residual vector (wrapper)
336 \until } // end of get_jacobian(...)
337 
338 The "real work" is done in the protected member function
339 \c get_residuals_generic(...), where we distinguish two cases.
340 -# The load is prescribed, i.e. the (single) value in the load \c Data
341  object is pinned.
342  In this case, the element's \c Data contains only one
343  unknown -- the vertical
344  displacement \f$ Y_c \f$, stored as value 1 in the internal \c Data.
345  The element's residual vector has a single entry, given by
346  \f[
347  {\tt r}_0 = f_{load} - Y_c \ k_{stiff}
348  \f]
349  and the element's Jacobian matrix is a 1x1 matrix whose single
350  entry is given by
351  \f[
352  {\tt J}_{00} = \frac{\partial {\tt r}_0}{\partial Y_c} = - k_{stiff}
353  \f]
354 -# If the load is unknown (i.e. an unknown in the overall problem)
355  the element's \c Data contains two unknowns. Recall that an element
356  only makes a
357  contribution to the residuals associated with unknowns that
358  it is "in charge of" -- the external \c Data is assumed to be
359  "determined" by another element.
360  Therefore, the elements residual vectors (i.e. its contribution
361  to the \c Problem's global residual vector) is given by
362  \f[
363  \left( \begin{array}{c}
364  {\tt r}_0 \\
365  {\tt r}_1
366  \end{array} \right)
367  =
368  \left( \begin{array}{c}
369  f_{load} - Y_c \ k_{stiff}\\
370  0
371  \end{array} \right).
372  \f]
373  The element's 2x2 Jacobian matrix contains the derivatives
374  of the residuals with respect to the element's unknowns, \f$ Y_c
375  \f$ and \f$ f_{load}\f$,
376  \f[
377  \left( \begin{array}{cc}
378  {\tt J}_{00} & {\tt J}_{01} \\
379  {\tt J}_{10} & {\tt J}_{11}
380  \end{array} \right)
381  =
382  \left( \begin{array}{cc}
383  \partial {\tt r}_0 / \partial Y_c&
384  \partial {\tt r}_0 / \partial f_{load} \\
385  \partial {\tt r}_1 / \partial Y_c &
386  \partial {\tt r}_1 / \partial f_{load}
387  \end{array} \right)
388  =
389  \left( \begin{array}{cc}
390  - k_{stiff} & 1 \\
391  0 & 0
392  \end{array} \right)
393  \f]
394 .
395 Note that we have (correctly!) assumed that the (fully automatic)
396 local equation numbering procedure, implemented in
397 \c GeneralisedElement::assign_local_eqn_numbers(),
398 regards the internal degree of freedom as local unknown "0" and
399 the external one (if it exists) as unknown "1". However, to be on
400 the safe side, we determine the local equation
401 numbers via the access functions \c internal_local_eqn(i,j)
402 and \c external_local_eqn(i,j), which return the local equation
403 numbers of the j-th value, stored in the element's i-th internal
404 (or external) \c Data object:

407 \until // end of get_residuals_generic(...)
