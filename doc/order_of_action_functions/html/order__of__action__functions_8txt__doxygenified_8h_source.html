3 \mainpage "Action" functions in oomph-lib's black-box Newton solver
4 
5 \c oomph-lib's \c Problem class provides a black-box Newton solver,
6 \c Problem::newton_solve(), which allows the driver code to be as
7 compact as this:
8 \code
9  main()
10  {
11  // Create the problem object
12  MyProblem problem;
13 
14  // Solve the problem, using oomph-lib's default Newton solver
15  problem.newton_solve();
16  }
17 \endcode
18 
19 While the availability of a black-box Newton solver is helpful,
20 it is sometimes necessary to interact with the
21 solution process, e.g. to perform certain additional
22 tasks during the Newton iteration, or to analyse the convergence
23 (or divergence!) of the iteration in more detail. For this
24 purpose \c oomph-lib's \c Problem class has a number of empty
25 virtual member functions that are executed at key stages
26 of the Newton iteration. Two of these functions
27 (\c Problem::actions_before_newton_solve() and \c
28 Problem::actions_after_newton_solve() ) are pure virtual functions that must
29 be implemented in every specific Problem, though they may, of course,
30 be implemented as empty functions. Other functions may be overloaded
31 to provide more detailed control.

394 The function \c Problem::actions_before_newton_solve() is often used to update
395 boundary conditions when performing parameter studies in which these
396 change; \c Problem::actions_after_newton_solve() may be used to
397 automatically perform any post-processing when a solution has
398 been obtained. The function
399 \c Problem::actions_before_newton_convergence_check() makes it
400 possible to update any "dependent" problem parameters, i.e. parameters
401 that depend on one or more of the unknowns in the problem but are not
402 updated automatically when the unknown is changed. This arises most
403 frequently in free-boundary problems in which the position of the
404 nodes in the "bulk" mesh is determined by an algebraic node update
405 procedure. When the Newton method updates the \c Data values that
406 determine the shape of the domain boundary in Step 7, the nodal positions
407 themselves are not updated automatically unless the node update
408 function is executed in \c Problem::actions_before_newton_convergence_check().
409 See the discussion of the
410 <A HREF="../../interaction/macro_element_free_boundary_poisson/html/index.html">free-boundary
411 Poisson problem</A> for an example of its use.

417 \section opt Optimisation for linear problems
418 Recall that, by default, \c oomph-lib regards all problems as
419 nonlinear. Provided a good initial guess for the unknowns
420 is available, the Newton solver will converge quadratically towards
421 the solution. Within this framework, linear problems are simply special
422 cases for which the Newton iteration converges in one step. However,
423 if the problem is known to be linear, a few of the steps in the generic
424 Newton iteration are unnecessary. For instance, it is not necessary
425 to check the residual before or after the Newton step as we know
426 that the exact solution will have been obtained (modulo roundoff
427 errors) after step 7. The computation of the global residual
428 vectors in steps 4 and 11 (which require a finite amount of cpu time)
429 are therefore superfluous and are omitted if the "user" declares
430 a problem as linear by setting the flag
431 \code
432 bool Problem::Problem_is_nonlinear
433 \endcode
434 which is initialised to \c true in the constructor of the \c Problem
435 base class to \c false.
