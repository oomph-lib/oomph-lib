3 \mainpage Demo problem: Solid Mechanics using unstructured meshes
4 
5 The purpose of this tutorial is to demonstrate the solution of
6 solid mechanics problems using unstructured meshes. We focus primarily
7 on two-dimensional meshes,
8 generated using <A HREF="http://www.cs.berkeley.edu/~jrs/">
9 Jonathan Shewchuk's</A> open-source mesh generator
10 <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">\c Triangle,</A>
11 based on the output from the open-source
12 drawing program <A HREF="http://en.wikipedia.org/wiki/Xfig">xfig.</A>
13 An example three-dimensional problem is described
14 <a href="../../unstructured_three_d_solid/html/index.html"> here; </a>
15 see also \ref three_d later in this document.
16 
17 The solid mechanics problem studied here also serves as a
18 "warm-up problem" for the
19 <a href="../../../interaction/unstructured_fsi/html/index.html">corresponding
20 fluid-structure interaction problem </a> in which the solid object
21 is immersed in (and loaded by) a viscous fluid.

39 \section mesh Mesh generation
40 We employ the combination of <A HREF="http://en.wikipedia.org/wiki/Xfig">xfig</A>,
41 \c oomph-lib's conversion code

47 <a href="../../../meshes/mesh_from_xfig/html/index.html">
48 another tutorial.</a>

50 We start by drawing the outline of the solid as a polyline in
51 <A HREF="http://en.wikipedia.org/wiki/Xfig">xfig</A>:

53 \image html xfig_screenshot.gif "xfig drawing of the solid body. "
54 \image latex xfig_screenshot.eps "xfig drawing of the solid body. " width=0.75\textwidth
55 
56 (Note that we draw the solid upside-down because of the way
57 <A HREF="http://en.wikipedia.org/wiki/Xfig">xfig</A> orients its coordinate axes.)
58 
59 We save the figure as a *.fig file and convert it to a *.poly file
60 using \c oomph-lib's conversion code
61 <a href="../../../meshes/mesh_from_xfig/html/index.html">
62 \c fig2poly </a>

66 \code

168 \section main The driver code
169 
170 The driver code is straightforward. We specify an output directory
171 and instantiate a constitutive equation. (Recall that the
172 single-argument constructor to the \c GeneralisedHookean
173 constitutive law implies that all stresses are non-dimensionalised
174 on Young's modulus \f$ E \f$).
175 
176 \dontinclude unstructured_two_d_solid.cc
177 \skipline start_main
178 \until ::Nu);
179 
180 We create the \c Problem object using a displacement formulation of
181 the equations and output the initial
182 configuration
183 
184 \until ()++
185 
186 Finally, we perform a straightforward parameter study, applying
187 a constant gravitational load and slowly increasing the
188 suction (negative pressure) on the upper boundary.
189 
190 \until end_displacement_formulation
191 
192 The parameter study is then repeated for a pressure-displacement
193 formulation with and without an incompressibility constraint, see the
194 <A HREF="../../../../demo_drivers/solid/unstructured_solid/unstructured_two_d_solid.cc">
195 source code </A> for details.

267 \subsection bound Identification/assignment of mesh boundaries
268 This tutorial demonstrates that the use of unstructured meshes
269 for solid mechanics problems is extremely straightforward.
270 The only aspect that requires some care (and not just for solid
271 mechanics applications) is the correct identification/assignment of
272 domain boundaries when the mesh is generated with
273 <A HREF="http://en.wikipedia.org/wiki/Xfig">xfig.</A> The fact that
274 we documented the mesh boundaries and the positions of the pinned
275 nodes in the driver code suggests (correctly!) that we managed to
276 get both assignments (slightly) wrong when we first wrote the
277 driver code. The manual identification of nodes on domain
278 boundaries is tedious and therefore error prone and, as usual,
279 it pays to <B>be as a paranoid as possible</B>! Ignore this
280 advice at your own risk...
281 
282 We also welcome any improvements to our conversion code
283 <A HREF="../../../../demo_drivers/meshing/mesh_from_xfig_triangle/fig2poly.cc">
284 fig2poly.cc
285 </A>
286 that would allow the specification of domain boundaries from
287 within <A HREF="http://en.wikipedia.org/wiki/Xfig">xfig.</A> It's probably
288 not possible but if you have any ideas how to go about this
289 (either by hijacking information that can be generated by
290 <A HREF="http://en.wikipedia.org/wiki/Xfig">xfig</A> or via some other
291 GUI interface),
292 <a href="../../../contact/html/index.html">let us know.</a>
